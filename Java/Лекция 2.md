- **Класс** — это абстрактный тип данных, который определяет пользователь (то есть мы, программисты). Чаще всего класс представляет собой некий объект из реального мира. Если проводить аналогию, то класс — это своего рода _чертеж_ или _прообраз_ того объекта, с которым мы хотим работать в программе.
    
- **Объект** — это уже конкретная _реализация_ этого чертежа. Когда вы создаете объект (это действие называется _созданием экземпляра класса_), под него выделяется память. В этой памяти хранится **состояние** объекта — то есть конкретные значения его полей. С объектом вы можете взаимодействовать, вызывая его **методы**.

В Java же есть несколько ключевых отличий:
1. **Нет разделения на .h и .cpp.** Весь класс, включая поля, конструкторы и методы, пишется в одном файле с расширением `.java`.

2. **Один файл — один класс.** Как правило, в одном `.java`-файле содержится ровно один public-класс. Имя этого public-класса _обязательно_ должно совпадать с именем файла (с учетом регистра!).

3. **Доступ к полям и методам.** В C++ для доступа к членам объекта через указатель использовался оператор `->`. В Java для доступа к полям и методам объекта используется оператор `.`, а `this` является не указателем, а **ссылкой** на текущий объект.

4. **Точка с запятой.** В C++ после объявления класса ставилась точка с запятой `;`. В Java после описания класса фигурные скобки закрываются, и точка с запятой _не ставится_.


В Java существует **четыре** модификатора доступа:

1. `public`
    
2. `private`
    
3. `protected`
    
4. **Уровень доступа по умолчанию** (когда вы не указываете никакой модификатор). Это _не_ ошибка.
	 **По умолчанию (package-private)**: Это уровень доступа, который действует, если вы не написали ни один из модификаторов. В рамках своего **пакета** (package) такие поля, методы и классы ведут себя как `public`. Но если из другого пакета попытаться к ним обратиться — они будут восприниматься как `private` (доступ будет закрыт)

Указывать эти модификаторы можно для:
- **Полей** класса (переменных)
- **Методов**
- **Конструкторов**
- **Самих классов**


Вот проект, который у меня сейчас открыт. Вы видите структуру: папка `src/main/java`. В экосистеме Java, особенно при использовании систем сборки вроде Maven или Gradle, эта папка является корневой для исходного кода.

Если вы создаете внутри нее какую-то папочку, например, `org.example`, то вы по сути создаете **пакет**. Пакет — это просто способ организации классов в пространстве имен, который напрямую соответствует структуре директорий на диске.

Я создам класс `Point` прямо в корне пакета (например, в `src/main/java`). Опишу его просто: объявлю два поля `x` и `y` _без указания модификатора доступа_ — то есть, с уровнем доступа _по умолчанию_.

~~~Java
// Файл: Point.java (в корневой директории пакета)
class Point {
    int x; // Уровень доступа по умолчанию (package-private)
    int y;
}
~~~
Теперь внутри основного класса приложения (`Main`), который находится в том _же самом пакете_, я могу создать объект этой точки и спокойно обратиться к ее полям `x` и `y`.

Давайте я сразу покажу одну распространенную особенность работы с классами. Весь наш код до этого мы писали внутри метода `main`, но часто можно встретить другую конструкцию.

Метод `public static void main` — это статический метод, он вызывается при старте приложения, но объект класса `Main` при этом не создается. Внутри статических методов мы можем создавать объекты других классов (и своего тоже). Когда мы создаем объект с помощью оператора `new`, первым делом вызывается его **конструктор**.

Итак, я создам новую точку и обращусь к ее полю `x`:

~~~Java
// Файл: Main.java (в том же пакете, что и Point)
public class Main {
    public static void main(String[] args) {
        Point myPoint = new Point(); // Создаем объект (вызывается конструктор)
        myPoint.x = 10; // Обращаемся к полю x
        System.out.println(myPoint.x); // И выводим его
    }
}
~~~

В данном случае код компилируется и запускается без ошибок, потому что класс `Point` и класс `Main` находятся в одном пакете, а поле `x` имеет уровень доступа _по умолчанию_, который разрешает доступ внутри своего пакета."

Если мы хотим, чтобы эти поля были доступны отовсюду, мы должны явно указать модификатор `public`.

Часто возникает вопрос: когда нужно скрывать поля, а когда — методы?
Общее правило для полей таково: **поля лучше скрывать**. Вместо прямого доступа к полям `x` и `y` рекомендуется создавать специальные методы — **геттеры** (для получения значения) и **сеттеры** (для установки значения).

#### **Способы передачи данных: вспоминаем C++**
1. **По значению** (`void func(int a)`)
2. **По указателю** (`void func(int* a)`)
3. **По ссылке** (`void func(int& a)`)

- **По значению:** переменные копируются в функцию. Изменения внутри функции не влияют на оригинал.
- **По указателю:** передается адрес переменной в памяти. Копируется сам адрес, а не данные. Через этот адрес мы можем читать и менять значение исходной переменной.
- **По ссылке:** это похоже на указатель, но с автоматическим разыменованием (к адресу обращаемся как к обычной переменной, без символа `*`). Как и с указателем, изменения влияют на оригинал.
### Правила передачи в Java

1. **Прямого доступа к памяти (к адресу) у нас нет.** Указателей, как в C++, в Java нет.
2. **С базовыми (примитивными) типами** (`int`, `double`, `char` и т.д.) все просто: они **всегда передаются по значению**. Внутри метода работает копия.
3. **Со всеми остальными типами (объектами) дело обстоит иначе.** Объекты в Java можно создавать и передавать **только по ссылке**.


В C++ эта строка:
```
Point p; // В C++ это создает объект "по значению" в стеке
```
...создает объект `Point` прямо на стеке.

В Java эта _же самая_ строка:
```
Point p; // В Java это объявляет ССЫЛКУ, а не объект!
```
...означает нечто совершенно иное. Она лишь объявляет **ссылку** типа `Point`, но не создает сам объект. Эта ссылка изначально ни на что не указывает (имеет значение `null`). Если вы попытаетесь у такой ссылки вызвать метод (`p.someMethod()`), вы получите `NullPointerException`, потому что память под объект не была выделена.


**Правильно** с объектами работать нужно так:
1. **Объявить** ссылку на объект.
2. **Создать** сам объект в памяти с помощью оператора `new` (который вызывает конструктор и выделяет память).
3. **Связать** ссылку с созданным объектом.

~~~Java
Point p;         // 1. Шаг: Объявили ссылку (пока что = null)
p = new Point(); // 2. Шаг: Создали объект и присвоили ссылке его адрес
                 // 3. Теперь можно вызывать методы: p.someMethod();
~~~

### Конструкторы
- Конструктор по умолчанию.
- Параметризованные (инициализирующие) конструкторы.

- **Конструктор копирования:** **Не нужен.** Почему? Потому что в Java объекты передаются только по ссылке. Прямого синтаксиса для создания объекта "по значению" (как в C++) нет, следовательно, не нужен и механизм для его автоматического создания. Если вам нужна копия объекта, вы реализуете это отдельно (например, через интерфейс `Cloneable` или метод `copy()`).
- **Конструктор преобразования типа:** **Не используется в том же виде.** Запись `Point p = 10;` в Java **невозможна** по синтаксису языка. Объект всегда нужно создавать явно через `new`. Конструктор с одним параметром создать можно, но он будет обычным параметризованным конструктором, а не механизмом неявного преобразования типа.


### Синтаксис конструкторов в Java и важное правило

- **Инициализация полей:** Вы можете проинициализировать поля значениями по умолчанию прямо при объявлении:
~~~Java
    class Point {
        int x = 0; // Поле инициализировано по умолчанию
        int y = 0;
    }

~~~
- **Конструктор по умолчанию:** Если вы **не написали ни одного конструктора**, компилятор **автоматически сгенерирует** пустой конструктор по умолчанию (`public Point() {}`).
    
- **Важное правило:** Если вы написали **хоть один** конструктор (например, параметризованный), компилятор **перестает** автоматически генерировать конструктор по умолчанию. Поэтому такой код перестанет компилироваться:
~~~Java
    class Point {
        int x, y;
        Point(int x, int y) { this.x = x; this.y = y; } // Наш конструктор
    }
    // ... в другом месте ...
    Point p = new Point(); // ОШИБКА! Конструктора по умолчанию уже нет!
~~~


**Генерация кода:** Современные IDE (как я только что показал, нажав `Alt+Insert`) позволяют автоматически генерировать конструкторы, геттеры, сеттеры и многое другое, что сильно ускоряет разработку


### Отсутствие перегрузки операторов в Java
Перегрузка оператора присваивания (`=`) в Java не нужна.
В Java же огромное счастье в том, что объекты по значению не создаются и не передаются. При присваивании (`Point b = a;`) копируется только **ссылка** на объект, а не сам объект.

```
String a = "hello";
String b = a;
System.out.println(a == b); // Выведет true
```

```
String a = new String("hello");
String b = new String("hello");
System.out.println(a == b); // Выведет FALSE!
```

Почему? В первом случае компилятор оптимизирует код, помещая строку в пул строк, и обе переменные ссылаются на один и тот же объект. Во втором случае мы создали два разных объекта в памяти. Оператор `==` **сравнивает не значения объектов, а ссылки на них**.

Если вы хотите сравнить _содержимое_ объектов, нужно использовать метод `.equals()`.

Явной перегрузки оператора сравнения в Java нет. Для сравнения объектов по значению используется интерфейс `Comparable` (о котором мы поговорим позже).

### Вывод объектов: метод toString()

Все объекты в Java неявно наследуются от класса `Object`. У `Object` есть метод `toString()`, задача которого — преобразовать объект в строку. Поэтому любой объект можно вывести в выходной поток (`System.out.println(point);`).

По умолчанию `toString()` выводит не очень полезную информацию: имя класса и хэш-код объекта (что-то вроде `Point@15db9742`).

Если вы хотите видеть осмысленное содержимое объекта, вам нужно **переопределить (override)** метод `toString()` в своем классе. Сделать это очень легко с помощью IDE (Alt+Insert -> toString()).

~~~Java
@Override
public String toString() {
    return "Point{x=" + x + ", y=" + y + "}";
}
~~~

Теперь при выводе `println(point)` мы увидим понятную строку: `Point{x=10, y=20}`.

Я очень рекомендую реализовывать `toString()` для ваших классов. Это крайне удобный инструмент для отладки и логирования.
### JAR-архивы: упаковка и распространение кода

В Java структура иная. Компилятор Java компилирует каждый класс в отдельный файл с расширением **`.class`** (байт-код). Передавать пользователю кучу таких файлов неудобно.

Поэтому их упаковывают в специальный архив — **JAR-файл**. Это по сути ZIP-архив, содержащий все скомпилированные `.class`-файлы и метаинформацию.

JAR-файлы можно использовать двумя способами:

1. **Как библиотеки:** подключать к вашим проектам для использования их функциональности.
    
2. **Как исполняемые файлы:** если в манифесте JAR-файла указан главный класс (`Main-Class`), его можно запустить командой `java -jar myapp.jar`.
    

Система сборки Maven, которую мы используем, легко создает такие JAR-пакеты.

Кроме удобства, JAR-архивы обеспечивают **безопасность**. Любой JAR-пакет можно подписать электронной цифровой подписью. Если содержимое архива после подписания будет изменено (например, вирусом), виртуальная машина Java это обнаружит и **откажется запускать** такой пакет. Это защищает пользователей.
