## Подходы к обработке ошибок

### Возврат ошибки или хранение

Этот подход предполагает, что функция возвращает специальное значение, указывающее на ошибку, или сохраняет информацию об ошибке в некотором общем хранилище
### Использование исключений

**Исключения** — это механизм, который позволяет программе прервать нормальный ход выполнения и передать управление специальному блоку кода, предназначенному для обработки ошибок 

## Пример обработки деления на ноль

Рассмотрим пример, где из файла нужно прочитать 100 значений и вывести пары текущего и следующего значения, а также результат их деления. Если одно из чисел в файле равно нулю, произойдёт **деление на ноль** .

Для отслеживания этой ситуации можно использовать **`if`** и **`throw`** для генерации исключения. Если исключение не обрабатывается, программа аварийно завершится, и выделенные ресурсы не будут освобождены 

## Использование блоков `try-catch`

Для обработки ошибок **небезопасный код** помещается в секцию **`try-catch`** 

> Если вы хотите любой ключей не ловить, то используются три точки, и в трех точках вы можете выполнить какую-то обработку ошибок, ну а потом в конце, например, закрыть файл. Вот такой код будет более безопасным с точки зрения исполнения и всегда будет заканчиваться тем, что, к примеру, файл был закрыт 

Это гарантирует, что даже при возникновении ошибки, ресурсы, такие как файлы, будут корректно закрыты, предотвращая потерю данных 

## Классовая система для исключений

Исключения могут быть разных типов. Можно построить целую **систему классов** для обозначения различных типов исключений, обеспечивая единый интерфейс для всех исключений 
Например, можно создать базовый класс `Exception` с виртуальным методом `show`, который отображает ошибку. От него можно наследовать различные реализации, такие как `IntegerException` для ошибок с целыми числами или `DoubleException` для ошибок с вещественными числами 
При возникновении ошибки создаётся объект исключения, и в блоке `catch` можно перехватить указатель на базовый класс `Exception` . Это позволяет обрабатывать различные типы исключений единообразно через виртуальный метод `show` . В Java обработка ошибок часто строится именно таким образом 

## Создание пользовательских исключений

Для создания пользовательских исключений в Java, классы исключений наследуются от базового класса `Exception`

### IntegerException (Целочисленное исключение)

Создается класс `IntegerException`, который может сохранять целочисленное значение для отладки. Конструктор принимает сообщение об исключении и целое число. Метод `getMessage` перегружается для возврата сообщения и сохраненного значения
### DoubleException (Вещественное исключение)

Аналогично создается класс `DoubleException`, который сохраняет вещественное значение и перегружает `getMessage`

## Выбрасывание и перехват исключений

### Выбрасывание исключений (`throw`)

В методе `task` демонстрируется, как выбрасывать исключения. Например, если индекс отрицательный, выбрасывается `IntegerException` с сообщением "negative message" и значением индекса. Если индекс равен нулю, выбрасывается `DoubleException`
### Обработка исключений (`try-catch`)
Когда метод выбрасывает исключение, Java требует либо добавить это исключение в **сигнатуру метода** (`throws`), либо обработать его с помощью блока `try-catch`
В данном примере, метод `task` окружен секцией `try-catch`
Можно использовать несколько блоков `catch` для разных типов исключений или один блок `catch` для обработки нескольких типов исключений одновременно
## Важность сигнатуры метода

Если метод потенциально может выбросить исключение (кроме `RuntimeException`), это должно быть указано в его сигнатуре с помощью ключевого слова `throws`. Если исключение не указано в сигнатуре, но метод его выбрасывает, компилятор Java выдаст ошибку, требуя обработать исключение внутри метода

## Паттерн "Цепочка обязанностей"
**Цепочка обязанностей** — это поведенческий паттерн проектирования, который позволяет передавать запросы последовательно по цепочке обработчиков. Каждый обработчик решает, обрабатывать ли запрос самостоятельно или передать его следующему обработчику в цепочке
### Цель паттерна
Основная цель — избежать привязки отправителя запроса к его получателю, давая нескольким объектам шанс обработать запрос

## Пример в бизнес-контексте
Пример описывает предприятие (условный магазин), которое:
- **Закупает материалы**: Например, доски для производства столов 
- **Производит столы**: Из закупленных досок 
- **Продает столы**: Готовую продукцию
## Важность организации функциональности в систему классов
Разделение функциональности на систему классов необходимо для предотвращения создания **очень больших классов**, которые умеют делать всё. Такие монолитные классы трудно отлаживать и расширять

## Демонстрируемые концепции
В рамках этого примера демонстрируются следующие концепции:
- **Интерфейсы**: Определяют контракт, которому должны следовать классы, скрывая детали реализации 
- **Фабрики**: Используются для создания объектов классов, скрывая логику их создания и позволяя не лезть в детали реализации 
- **Обработка ошибок**: Интегрируется в систему для обеспечения надежности и корректного поведения программы при возникновении исключительных ситуаций
