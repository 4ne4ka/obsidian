### Конспект лекции: Процессы и потоки (продолжение)

#### 1. Контекст процесса и переключение контекста (повторение)

**Контекст** — это полный слепок состояния объекта, достаточный для его восстановления «как было» (например, после удаления или выгрузки из процессора).

**Контекст процесса включает:**
- Информация, которой ОС управляет процессом (дескриптор).
- Адресное пространство процесса (виртуальная память).
- Содержимое пользовательской части этого адресного пространства (код, данные, куча, стеки).
- Все ресурсы, которые принадлежат процессу или используются им (файлы, семафоры и т.д.).

**Переключение контекста** — это переход процессора от исполнения одного потока к другому (возможно, из другого процесса).

**Что нужно сделать при переключении:**
1. Сохранить контекст **вытесняемого потока** (чтобы потом вернуться).
2. Если новый поток принадлежит **другому процессу**:
   - Сохранить контекст процесса-владельца старого потока.
   - Загрузить контекст процесса-владельца нового потока.
3. Загрузить контекст **нового потока**, выбранного на исполнение.

**Важный нюанс (из обсуждения):**
При переключении контекста **большая часть информации не разрушается**. ОС просто переустанавливает активные указатели (был активен один процесс → переставили указатели → стал активен другой).
**Единственное, что реально сохраняется и восстанавливается** — это **аппаратный контекст выполнения потока**.
> *Аппаратный контекст* — это состояние регистров ЦП, программный счетчик, указатель стека и т.п. именно для данного потока. Это самая «дорогая» часть переключения (по времени), но она неизбежна.

#### 2. Дескриптор процесса (PCB — Process Control Block)

В Linux это структура `task_struct` (файл `include/linux/sched.h`). Содержит **около 140–150 полей** (раньше говорили про 100). Большинство полей служебные, но есть и те, которые видны пользователю/администратору (например, в диспетчере задач Windows).

**Основные поля дескриптора процесса (понятные человеку):**

| Поле | Что означает | Пример / детали |
|------|--------------|----------------|
| **PID** (Process ID) | Уникальный идентификатор процесса. Целое число, нумерация обычно с 1. Используется как ключ доступа к процессу. | PID могут быть закольцованы: достигнув максимума (например, 32767), нумерация начинается заново, но без повторений для живых процессов. |
| **PPID** (Parent PID) | Идентификатор **родительского процесса**. Кто создал этот процесс? (ОС создаёт первый процесс, остальные — по запросу других процессов). | В Unix это почти всегда так. В Windows — не 100%, но в целом похоже. |
| **Групповые параметры** | Объединение процессов по логическим признакам. | **GID** (Group ID) — группа процессов (лидер + остальные).<br>**SID** (Session ID) — сессия (несколько групп).<br>**Job ID** (Windows) — задание. Пример: при установке программы все процессы помещаются в один job, и для него можно ограничить ресурсы (например, не более 5 минут CPU — после этого все процессы задания убиваются). |
| **Параметры приоритета** | Используются, когда несколько процессов/потоков конкурируют за ресурс. | **Важно (из вопроса лектору)**: за *время CPU* конкурируют **потоки**, а не процессы. Но у процесса обычно есть наследуемые потоками приоритеты. А вот за *память* конкурируют именно процессы. |
| **Состояние процесса** | Влияет на все его потоки. | Создание, исполнение (активен хотя бы один поток), остановлен (всем потокам запрещено CPU), завершение (обработка запроса на завершение), завершён. |
| **Статистические данные** | Накопленная информация. | Суммарное время CPU (всех потоков процесса), пиковый рабочий набор (память), число потоков, загрузка GPU, «рабочий набор» (working set) и его дельта и т.д. В диспетчере задач Windows — до 9–10 параметров памяти. |
| **Описание адресного пространства** | Две части: 1) аппаратно-зависимые структуры для виртуальной памяти; 2) описание внутренней структуры (регионы). | Регионы: код, данные, куча, стеки. В Linux — ещё регион с аргументами командной строки и регион с переменными окружения. |
| **Контекст ввода-вывода** | Не про аппаратный I/O, а про объекты ОС (файлы, сокеты...). | Какие файлы открыты, режим (чтение/запись/дополнение), текущая позиция в файле, код завершения последней операции. |
| **Контекст безопасности** | Две стороны: 1) права и привилегии программы (может ли писать в файл?); 2) доступ к конкретным объектам. | Например, чтобы записать в файл, нужно: а) иметь привилегию на операцию записи, б) иметь право доступа на запись к этому конкретному файлу. |
| **Текущие системные параметры выполнения** | Внутренние флаги, маски сигналов, etc. | — |
| **Код завершения процесса** | Возвращаемое значение при выходе. | 0 — успех, не 0 — ошибка. |

#### 3. Кто управляет процессами и потоками?

- Управление процессами всегда выполняет **ядро ОС**.
- Ядро всегда предоставляет каждому процессу **как минимум один поток** (поток ядра, управляемый ядром).
- Ядро **не обязано поддерживать многопоточность** — но современные ОС поддерживают.
- **Потоки ядра** — те, которыми непосредственно управляет ядро.

**Альтернатива:** можно реализовать многопоточность **на уровне библиотеки пользователя** (user-level threads), вообще без участия ядра.

---

#### 4. Потоки ядра vs. пользовательские потоки — плюсы и минусы

**Пользовательские потоки (user-level threads):**

| Плюсы | Минусы |
|-------|--------|
| **(+)** Операции над потоками (создание, переключение, синхронизация) **не требуют системных вызовов** → работают в **10–100 раз быстрее**, чем потоки ядра. | **(–)** Ядро **ничего не знает** о пользовательских потоках. Оно распределяет время CPU между процессами, игнорируя, сколько внутренних потоков в каждом процессе. |
| **(+)** Можно реализовать **собственный алгоритм планирования** (например, свой диспетчер, который переключает потоки по вашим правилам). | **(–)** Если один пользовательский поток выполняет **блокирующий системный вызов** (например, синхронное чтение с диска или `sleep`), ядро переводит в ожидание **весь поток ядра**, на котором крутятся *все* пользовательские потоки этого процесса. В итоге **блокируются все потоки процесса**, даже если они готовы работать. |

**Итог по пользовательским потокам:**  
Очень быстрые переключения, но фатально ломаются при блокирующем I/O. Поэтому их используют в специфических сценариях (например, в играх или симуляциях, где ввод-вывод предсказуем и не блокирует).

### Итоговые определения (для запоминания)

- **Дескриптор процесса** — структура в ядре (в Linux — `task_struct`), содержащая всё, что ОС нужно знать о процессе: PID, PPID, состояние, приоритеты, контекст безопасности, адресное пространство, открытые файлы и т.д.
- **Переключение контекста** — сохранение состояния текущего потока/процесса и загрузка состояния следующего. Самый затратный элемент — аппаратный контекст CPU.
- **Пользовательские потоки** — потоки, реализованные библиотекой в пользовательском пространстве, без участия ядра. Быстрые, но опасные при блокирующих вызовах.

### 5. Контекст безопасности (вторая сторона) и текущие параметры выполнения

#### Контекст безопасности — две стороны одной монетки

**Сторона 1 (уже обсуждалась):** определяет **права и привилегии самой программы**, выполняющейся в процессе. Например: может ли процесс писать в файл, создавать сокеты, завершать другие процессы.

**Сторона 2 (новая):** определяет **права доступа к самому процессу** как к объекту ОС.  
Над процессом можно выполнять действия: приостановить, изменить приоритет, завершить. Для этого нужно:
- иметь **привилегию** на выполнение операций с процессами вообще;
- иметь **право доступа** к конкретному процессу (аналог прав на файл: кто может читать/изменять/завершать).

> В дескрипторе процесса хранится информация о том, кто и какие операции может выполнять с этим процессом.

#### Текущие системные параметры выполнения

Это значения, которые влияют на выполнение программы в данный момент.

**Главный пример — текущий каталог (current working directory).**

**Зачем он нужен?**  
Для работы с **относительными путями**. Если вы в программе указываете просто имя файла или относительный путь (`data/input.txt`), ОС автоматически приписывает слева текущий каталог, чтобы получить полный путь.

**Важные следствия:**
- Текущий каталог всегда существует (логически непротиворечивое состояние).
- Из-за этого возникают проблемы: если текущий каталог процесса удалить, ОС должна решить, что делать.
- **В Unix:** если текущий каталог процесса удалён, процесс «перекидывается» в корневой каталог (`/`).

### 6. Дескриптор потока (Thread Control Block)

В **Unix / Linux** часто нет отдельной структуры дескриптора потока. Вместо этого в `task_struct` (дескрипторе процесса) часть полей относится к процессу, часть — к потоку. Для многопоточности создаётся **несколько дескрипторов процессов** (с разными TID), у которых совпадают все «процессные» поля. Так получается один процесс с несколькими потоками.

**Как это увидеть в Linux:**  
Команда `ps` показывает два столбца — PID (идентификатор процесса) и TID (идентификатор потока). У многопоточного процесса PID одинаковый у всех потоков, а TID — разные, близкие по значению (например, 1480, 1583, 1584…). Первый поток — первичный (создан вместе с процессом).

#### Основные поля дескриптора потока

| Поле | Что означает | Примечание |
|------|--------------|-------------|
| **TID (Thread ID)** | Уникальный идентификатор потока. Первичный ключ для поиска в базе данных потоков. | |
| **PID процесса-владельца** | Какому процессу принадлежит этот поток. | **Зачем это нужно постоянно?** Потому что все ресурсы (файлы, окна, соединения) принадлежат **процессу**, а не потоку. Когда поток хочет работать с ресурсом, он должен быстро определить, из какого процесса брать таблицу открытых файлов и т.д. Обратная ситуация (найти все потоки процесса) нужна реже — например, при завершении процесса. |
| **Параметры приоритета** | Используются при конкуренции потоков за ресурс. | Главный ресурс — **время CPU**. Также объекты синхронизации (если несколько потоков ждут один мьютекс). Иногда учитывается приоритет потока, иногда процесса. |
| **Статистические данные потока** | Минимум — потреблённое время CPU. Остальное опционально. | |
| **Аппаратный контекст выполнения** | Состояние регистров ЦП, программный счётчик, указатель стека. | Это **самое важное** при переключении потоков. Сохраняется и восстанавливается. Обычно хранится не в самом дескрипторе, а в специальной структуре (например, **TSS — Task State Segment**), а в дескрипторе — указатель на неё. |
| **Код завершения потока** | Пока поток работает — специальное значение (например, `STILL_ACTIVE`). После завершения — код возврата. | |

### 7. Кто управляет процессами и потоками?

- **Управление процессами — всегда ядро ОС.**
- Ядро всегда предоставляет процессу **как минимум один поток** (поток ядра).
- Ядро **может не поддерживать многопоточность** (в очень маленьких/устаревших ОС), но современные системы общего назначения — поддерживают.
- Потоки, которыми непосредственно управляет ядро, называются **потоками ядра (kernel threads)**.

#### Многопоточность на уровне библиотеки пользователя (user-level threads)

Можно реализовать многопоточность **без поддержки ядра** — внутри самой программы через библиотеку.

**Как это выглядит:**
- У процесса есть, скажем, **2 потока ядра** (видимых для ОС).
- Библиотека создаёт **10 пользовательских потоков** и «крутит» их поверх этих 2 потоков ядра.
- Ядро ничего не знает про 10 потоков — оно видит только 2.

**Зачем это придумали?**  
Чтобы писать **единую многопоточную программу**, которая работает и на ОС с поддержкой многопоточности (тогда библиотека использует потоки ядра), и на ОС без поддержки (тогда библиотека эмулирует потоки сама).  
Пример: библиотека **POSIX Threads (pthreads)** в старых Unix — могла работать в обоих режимах.

### 8. Плюсы и минусы пользовательских потоков

#### Плюсы

| Плюс | Пояснение |
|------|-------------|
| **(+) Очень быстро** | Создание/переключение/удаление потока — **без системных вызовов**. В 10–100 раз быстрее, чем у потоков ядра (иногда даже больше 100×). |
| **(+) Свой алгоритм планирования** | Библиотека сама решает, какой пользовательский поток запустить следующим. (Но на практике требуется осторожность — лучше использовать невытесняющее планирование с добровольной передачей CPU). |

#### Минусы

| Минус | Пояснение | Когда это критично? |
|-------|-------------|----------------------|
| **(-) Нечестное распределение CPU** | Ядро не знает о пользовательских потоках и делит CPU между **потоками ядра**. Если в процессе A 10 потоков ядра, а в B — 1 поток ядра + 10 пользовательских, то A получит ≈90% CPU, B — ≈10%. | Только при **100% загрузке CPU**. Если CPU загружен на 50%, всем хватает — нечестность незаметна. |
| **(-) Катастрофа с блокирующим вызовом** | Если любой пользовательский поток делает **блокирующий системный вызов** (например, `read()` с диска), весь **единственный поток ядра**, на котором крутятся все пользовательские потоки, переходит в состояние ожидания. ОС перестаёт давать ему CPU → **все 10 пользовательских потоков замирают** на время блокировки. | Это **всегда катастрофа**, если не принять мер. |

#### Как решают проблему блокирующих вызовов?

Библиотека пользовательских потоков подменяет все блокирующие системные вызовы на **специальные обёртки**:

1. Вместо блокирующего вызова делается **неблокирующий** запрос: «если ресурс недоступен, верни управление сразу, не блокируйся».
2. Библиотека продолжает крутить другие пользовательские потоки.
3. Периодически библиотека **опрашивает** ОС: «а ресурс уже стал доступен?»
4. Когда доступен — возобновляет выполнение потока, который его ждал.

Это **рабочее решение**, но оно сложнее и требует перехвата всех системных вызовов.

#### Где используются пользовательские потоки сегодня?

Не в обычных приложениях, а в специфических областях:

- **Облегчённые потоки (lightweight threads)** — в некоторых языках программирования.
- **Корутины (coroutines)** — во многих современных языках (Go, Kotlin, C++20, Python asyncio).  
  Корутины часто реализованы как пользовательские потоки с **невытесняющим планированием**: корутина сама решает, когда передать управление другой корутине. Это идеально подходит для асинхронного ввода-вывода.

### 9. Операции над процессами и потоками: создание и завершение

Лектор рассматривает две базовые операции: **создание** и **завершение** — в двух стилях: «в стиле Windows» и «в стиле Unix». (Далее по слайдам — формальные последовательности шагов.)

#### Создание процесса (по шагам)

| Шаг | Что происходит |
|-----|----------------|
| 1 | Создать **дескриптор процесса**, поместить в таблицу процессов. |
| 2 | Инициализировать поля общего назначения (PID, PPID, приоритет по умолчанию…). |
| 3 | Создать **виртуальное адресное пространство (ВАП)** процесса и сформировать его структуру (аппаратно-зависимые таблицы страниц). |
| 4 | Заполнить ВАП необходимыми данными: разместить код, данные, аргументы командной строки, переменные окружения. |
| 5 | Выделить процессу **ресурсы по умолчанию** (например, стандартные дескрипторы ввода/вывода). |
| 6 | Оповестить подсистемы ядра, что появился новый процесс (чтобы они могли его учитывать). |
| 7 | Создать **первичный поток** процесса (свой стек, свой аппаратный контекст). |

#### Создание потока (внутри уже существующего процесса)

| Шаг | Что происходит |
|-----|----------------|
| 1 | Создать **дескриптор потока** (заполнить TID, приоритет…). |
| 2 | Создать **стек потока** (в адресном пространстве процесса). |
| 3 | Инициализировать **аппаратный контекст** (записать в структуру начальные значения регистров, указатель стека, точку входа — адрес функции потока). |

> **Важно:** код, куча, глобальные данные — **общие** для всех потоков процесса. Новый стек создаётся **индивидуально**.

#### Завершение потока

| Шаг | Что происходит |
|-----|----------------|
| 1 | Сохранить **статистику потока** и **код возврата** в дескрипторе. |
| 2 | Перевести **все ресурсы, принадлежащие потоку**, в непротиворечивое состояние (например, закрыть ещё не закрытые объекты). |
| 3 | Освободить все ресурсы, которыми владел поток. |
| 4 | Оповестить подсистемы ядра о завершении потока. |
| 5 | Установить **состояние потока** = «завершён». |
| 6 | **Если этот поток был последним активным потоком в процессе** → завершить процесс (перейти к завершению процесса). |

**Важный нюанс:**  
После всех этих действий **дескриптор потока не уничтожается сразу**. Он остаётся в системе, содержа код возврата и статистику. Другой поток (или родитель) может его «перехватить» и узнать, как завершился поток.  
**Момент окончательного уничтожения дескриптора зависит от реализации ОС** (обычно когда кто-то вызовет `pthread_join()` или аналогичный системный вызов).

#### Завершение процесса

| Шаг | Что происходит |
|-----|----------------|
| 1 | **Завершить выполнение всех потоков процесса** (принудительно или дождаться их завершения). |
| 2 | Сохранить **статистику процесса** и **код возврата** в дескрипторе процесса. |
| 3 | Перевести все ресурсы процесса в непротиворечивое состояние (закрыть все файлы, освободить синхронизаторы…). |
| 4 | Освободить все ресурсы, принадлежавшие процессу (память, файлы, сокеты, мьютексы…). |
| 5 | **Освободить виртуальное адресное пространство (ВАП)** и уничтожить его структуры. |
| 6 | Оповестить подсистемы ядра о завершении процесса. |
| 7 | Установить **состояние процесса** = «завершён». |

**И снова:**  
Дескриптор процесса **не удаляется сразу**. Он остаётся с кодом возврата и статистикой, чтобы родительский процесс мог вызвать `wait()` и узнать, как завершился ребёнок. Окончательное удаление — когда родитель «прочитал» статус (или когда система решит, что это больше никому не нужно).

### Итог (что важно запомнить после этой части)

1. **Контекст безопасности** — двусторонний: что процесс может делать (привилегии) и что можно делать с процессом (права доступа к объекту).
2. **Текущий каталог** — нужен для относительных путей. При удалении текущего каталога процесс в Unix перекидывается в корень.
3. **Дескриптор потока** (TCB) — содержит TID, PID владельца, приоритет, статистику, аппаратный контекст (указатель на TSS), код завершения.
4. **Пользовательские потоки** — быстрые (нет системных вызовов), но опасные (блокировка одного блокирует все). Проблема решается неблокирующими обёртками.
5. **Создание процесса** — 7 шагов, от дескриптора до первичного потока.
6. **Создание потока** — 3 шага: дескриптор, стек, аппаратный контекст.
7. **Завершение потока/процесса** — ресурсы освобождаются, но дескриптор остаётся (код возврата, статистика) до тех пор, пока кто-то его не «заберёт» (join/wait).

### 10. Создание процесса в стиле Windows (пошагово)

> *Исходная ситуация:* какой-то поток какого-то процесса выполняет системный вызов `CreateProcess` (или аналогичный). Это запрос на создание нового процесса.

#### Шаг 1. Создание дескриптора процесса и инициализация полей

- Создаётся структура `_EPROCESS` (в ядре Windows) — аналог `task_struct` в Linux.
- Поля заполняются:
  - **Значениями по умолчанию** (если не указано иное).
  - **Значениями, указанными в системном вызове** (флаги, параметры).  
    В `CreateProcess` есть два места с флагами — там может быть несколько страниц описаний! (очень много опций).
  - **Унаследованными от родителя**:  
    - приоритет (типично наследуется)  
    - пользователь-владелец  
    - группа (если есть понятие группы)  
    - Job ID в Windows (если родитель был в job'е, то и новый процесс в него попадает).
  - **Вычисленными заново**:  
    - уникальный новый PID (идентификатор процесса).

#### Шаг 2. Создание виртуального адресного пространства (ВАП)

- Имеется в виду **аппаратно-зависимая часть**: таблицы страниц, дескрипторы сегментов (если нужно) — всё, что обеспечивает работу виртуальной памяти.
- В этот момент ВАП существует, но оно **пустое** (нет ни регионов, ни данных).

#### Шаг 3. Формирование структуры ВАП и её заполнение

Здесь возникает два важных вопроса.

**Вопрос 1. Откуда берутся начальные адреса регионов (кода, данных)?**

**Ответ:** их определяет **линковщик** на этапе сборки программы.

- Линковщик планирует, где в виртуальном адресном пространстве будут лежать:  
  - функции (регион кода)  
  - глобальные статические переменные и константы (регион данных).
- В заголовке **исполнимого файла (PE-файл в Windows)** хранится информация об этих регионах: их виртуальные адреса, размеры, права доступа.
- Когда ОС загружает программу, она читает этот заголовок и создаёт регионы **именно там, где запланировал линковщик**.

> Это справедливо для современных ОС с **линейным виртуальным адресным пространством** (страничная память). В других моделях адресации может быть иначе, но сейчас мы рассматриваем типичный случай.

**Вопрос 2. Почему на слайде (картинка с процессом) нет стека?**

**Ответ:** стек — это **собственность потока**, а не процесса.  
- На данном этапе процесса как такового уже есть, но **потоков ещё нет**.  
- Первичный поток будет создан позже, и **тогда для него создастся стек**.  
- У каждого потока — свой стек.

**Дополнительный вопрос (из лекции): может ли быть несколько регионов кода?**

Да. Откуда они берутся?

- **Динамически подключаемые библиотеки (DLL в Windows, .so в Linux)**.  
  Когда программа загружает DLL, она подключает:
  - ещё один регион кода (код библиотеки)
  - ещё один регион данных (глобальные переменные библиотеки)
  - Эти регионы **не сливаются** с основными регионами программы — они располагаются отдельно, в других местах ВАП.

- Также может быть **несколько статических регионов кода**, если линковщик так настроен (редко, но возможно).

Таким образом, итоговая структура ВАП — это **набор регионов**:
- основной код (.text)
- основные данные (.data, .bss)
- регионы DLL
- куча (может создаваться стартовым кодом или явно)
- стек(и) — по одному на поток

#### Шаг 4. Выделение ресурсов по умолчанию

Даже если программа ничего не запрашивала, ей **уже выделены некоторые ресурсы**.

**Главный пример — стандартные потоки ввода/вывода/ошибок** (`stdin`, `stdout`, `stderr`).

- Для консольной программы на C/C++ вы сразу можете писать `printf("Hello World\n")`, хотя вы не открывали файл `stdout` руками.
- ОС (или стартовый код библиотеки времени выполнения) предоставил их автоматически.

> Это не абсолютное правило для всех систем, но для практических задач — работает.

#### Шаг 5. Создание первичного потока (и переход к созданию потока)

После того как процесс создан и ВАП сформировано, ОС создаёт **первичный поток**.  
Далее лектор переходит к описанию создания потока.

### 11. Создание потока в стиле Windows

Поток может быть создан:
- как **первичный поток процесса** (при создании процесса)
- или позже, по запросу пользователя (`CreateThread` в Windows, `pthread_create` в Linux).

**Шаги создания потока (общие для обоих случаев):**

| Шаг | Действие | Пояснение |
|-----|----------|-----------|
| 1 | **Создать дескриптор потока** | Заполнить TID, приоритет (в Windows — приоритет по умолчанию, не наследуется от родительского потока, а назначается системой). |
| 2 | **Создать структуры данных, необходимые для работы потока на данной архитектуре** | Сюда входит: регион стека и структура для хранения аппаратного контекста. |
| 3 | **Инициализировать аппаратный контекст** | Записать начальные значения регистров, указатель стека (на начало созданного стека), программный счётчик (адрес точки входа — функции потока). |

#### Особое внимание: стек и аппаратная архитектура

Лектор делает важное отступление: **не все архитектуры одинаково поддерживают стек**.

**Сценарий A: архитектура со стеком (например, x86, x86_64, ARM в обычном режиме)**
- В ISA есть специальные инструкции: `push`, `pop`, `call`, `ret`.
- Регион стека может иметь специальные аппаратные атрибуты (например, защита от переполнения).
- ОС выделяет регион под стек, и процессор сам управляет стековым указателем (`RSP`, `SP`).

**Сценарий B: архитектура без аппаратного стека** (встречается в некоторых RISC-архитектурах, во встроенных системах, в некоторых DSP)
- Нет инструкций `push`/`pop`, нет `call`/`ret`.
- Вместо этого: есть инструкция **условного перехода с сохранением адреса возврата** (например, `bl` (branch with link) в ARM, но там всё же есть понятие стека; в совсем экзотических — да, нет).
- Программист (или компилятор) **сам вручную выделяет область памяти** под «стек» и сохраняет туда адреса возврата.
- Для возврата из функции: загрузить сохранённый адрес и выполнить косвенный переход.

**Что делает ОС в таком случае?**
- Даже если архитектура не требует стека, ОС **всё равно создаёт регион под стек** (потому что логика работы языков высокого уровня и библиотек предполагает стек).
- Но аппаратной поддержки нет — значит, переключение контекста и сохранение регистров будет реализовано **программно**, через явное сохранение/восстановление.

**Вывод для конспекта:**  
При создании потока ОС всегда выделяет:
1. **Регион стека** (даже если архитектура его не поддерживает — это делается для единообразия).
2. **Структуру для аппаратного контекста** (место, куда будет сохраняться состояние регистров ЦП при переключении потоков).

### 14. Создание потока: завершающие шаги (инициализация аппаратного контекста)

Продолжаем разбор шагов создания потока. После того как создан дескриптор потока, выделен стек и структура для аппаратного контекста, нужно **инициализировать сам аппаратный контекст**.

#### Что такое аппаратный контекст потока?

Это структура, которая содержит:
- **SP (Stack Pointer)** — указатель стека.
- **IP (Instruction Pointer)** — программный счётчик (адрес следующей инструкции).
- Значения всех общих регистров (обычно заполняются значениями по умолчанию, но на них лучше не полагаться — они могут меняться в зависимости от архитектуры или стартового кода языка программирования).

#### На что устанавливается SP?

- Мы только что создали **регион стека** для этого потока. Мы знаем его начальный адрес и размер.
- SP устанавливается на **вершину стека**.  
  Обычно стек растёт вниз (в сторону младших адресов), поэтому SP указывает на **конец выделенного региона** (самый старший адрес). Если стек пустой, то SP указывает на «дно» стека.

#### Чему равен IP? Откуда берётся адрес первой инструкции?

Это зависит от того, **какой поток создаётся**:

**Случай 1: Создаётся первичный поток процесса**
- IP берётся из **заголовка исполнимого файла**.
- В PE-файле (Windows) или ELF-файле (Linux) записана точка входа — адрес, с которого начинается выполнение.
- **Важно:** для программы на C/C++ это не функция `main`! Это **стартовый код** (обычно `_start` или `mainCRTStartup`), который:
  - инициализирует среду выполнения (например, вызывает конструкторы глобальных объектов)
  - получает аргументы командной строки
  - **затем вызывает** `main`.
- Таким образом, IP = адрес стартового кода.

**Случай 2: Создаётся дополнительный поток (не первый)**
- Поток создаётся по запросу программы (например, `CreateThread` или `pthread_create`).
- В системном вызове явно передаётся **адрес функции**, которую должен выполнять новый поток.
- IP устанавливается на **адрес первой инструкции этой функции**.
- (Технически иногда добавляется небольшая обёртка, но по сути — да, это адрес пользовательской функции).

#### Финальный шаг после инициализации контекста

- Установить **состояние потока** в «готов к выполнению» (ready).
- Теперь планировщик может выделить потоку процессор, и он начнёт работать.

### 16. Завершение потока

Завершение потока — это почти «обратная анимация» по сравнению с созданием. Выполняются следующие шаги (слайд ОС6 (11).png):

| Шаг | Действие | Пояснение |
|-----|----------|-----------|
| 1 | **Сохранить статистические данные потока и код возврата** в дескрипторе | Код возврата и статистика (например, время CPU) могут понадобиться другим потокам/процессам после завершения. |
| 2 | **Перевести все ресурсы, принадлежащие потоку, в непротиворечивое и стабильное состояние** | **Важно:** это с точки зрения ОС, а не вашей программы. Пример: если вы делали `fprintf` и не сделали `fflush`, данные могли не дойти до диска — ОС может просто выбросить буфер. Не думайте, что «стабильное состояние» сохранит ваши незавершённые операции. |
| 3 | **Освободить все ресурсы, принадлежавшие потоку или использовавшиеся потоком** | Обычно ресурсы принадлежат процессу. Но есть ресурсы, созданные на имя потока, или временно захваченные потоком (например, блокировки). |
| 4 | **Оповестить подсистемы, участвующие в управлении потоками**, о завершении | Выполняются внутренние действия ядра (дисклеймер). |
| 5 | **Установить состояние потока = «завершён»** | |
| 6 | **Если данный поток — последний активный поток в процессе → завершить процесс** | Инициируется процедура завершения процесса (см. ниже). |

#### Что остаётся после завершения потока?

**Дескриптор потока не удаляется сразу!**  
Он остаётся в системе, потому что содержит **код возврата** и **статистику**, которые кто-то может захотеть прочитать.

#### Когда дескриптор потока удаляется на самом деле?

**В Windows:** дескриптор потока — это объект ядра (Kernel Object). У него есть счётчик ссылок. Чтобы объект был удалён, **все процессы должны закрыть свои хендлы (handles) на этот поток**. Пока хотя бы один процесс держит хендл открытым, объект существует.

**В общем случае:** момент уничтожения дескриптора зависит от реализации. Обычно нужно, чтобы **все «пользователи» дескриптора явно от него отказались** (закрыли хендл, вызвали `pthread_join()` и т.п.).

> **Аналогия:** вы уволились с работы, но ваше личное дело ещё лежит в отделе кадров — вдруг понадобится справка. Его удалят только когда все, кому оно могло понадобиться, скажут, что оно больше не нужно.

### 17. Завершение процесса

Завершение процесса — аналогичный процесс, но в более крупном масштабе (слайд ОС6 (12).png).

| Шаг | Действие | Пояснение |
|-----|----------|-----------|
| 1 | **Завершить выполнение всех потоков процесса** | Если завершение инициировано процессом (последний поток завершился) — то их просто добивают. Если внешнее завершение (`TerminateProcess`) — принудительно останавливают. |
| 2 | **Сохранить статистику процесса и код возврата** в дескрипторе | Код возврата нужен родительскому процессу (через `WaitForSingleObject` в Windows или `wait` в Unix). |
| 3 | **Перевести все ресурсы процесса в непротиворечивое и стабильное состояние** | Опять же, с точки зрения ОС. Ваши незаписанные данные в буферах могут быть потеряны. |
| 4 | **Освободить все ресурсы, принадлежавшие процессу** | Память, открытые файлы, сокеты, мьютексы, объекты синхронизации и т.д. |
| 5 | **Освободить виртуальное адресное пространство (ВАП) и уничтожить его** | Разрушаются: структуры регионов (код, данные, кучи, стеки потоков) и аппаратно-зависимые таблицы страниц. |
| 6 | **Оповестить подсистемы, участвующие в управлении процессами**, о завершении | Много внутренних действий ядра. |
| 7 | **Установить состояние процесса = «завершён»** | |

#### Что остаётся после завершения процесса?

**Дескриптор процесса остаётся** — в нём хранятся код возврата и статистика.  
Он будет удалён только тогда, когда все «пользователи» откажутся от него.  
В Windows это означает, что все процессы, открывшие хендл к этому процессу (например, родительский процесс или отладчик), должны его закрыть.

> **Типичная проблема:** если родительский процесс не вызывает `WaitForSingleObject` (или не закрывает хендл), дескриптор завершённого процесса остаётся в системе — это называется «зомби-процесс» в Unix или просто утёкший хендл в Windows.

### 18. Функции Windows для работы с процессами и потоками (кратко)

Лектор упоминает их, чтобы вы знали, где искать, но детали предлагает посмотреть в документации или книге (например, «Inside Windows»).

#### Функции для процессов

| Функция | Назначение |
|---------|-------------|
| `CreateProcess` | Создание нового процесса (10 параметров — очень много). |
| `ExitProcess` | Завершение **текущего** процесса. |
| `TerminateProcess` | Завершение **другого** процесса (требуются права доступа). |
| … и многие другие (приоритет, ожидание завершения и т.д.) |

#### Функции для потоков

| Функция | Назначение |
|---------|-------------|
| `CreateThread` | Создание нового потока внутри текущего процесса. **4 параметра**, два самых важных: адрес функции потока и аргумент для неё. |
| `ExitThread` | Завершение **текущего** потока. |
| `TerminateThread` | Завершение **другого** потока (может быть из другого процесса, но нужны соответствующие права доступа). |

> **Важно:** `TerminateThread` — опасная функция. Она не даёт потоку освободить ресурсы (блокировки, память). Используйте только в крайних случаях.

### 19. Итог всей лекции (процессы и потоки) — что нужно вынести

1. **Дескриптор процесса (`_EPROCESS` / `task_struct`)** — огромная структура (~150 полей). Основные видимые поля: PID, PPID, состояние, приоритеты, контекст безопасности, статистика, описание ВАП, контекст I/O.
2. **Дескриптор потока (TCB)** — TID, PID владельца, приоритет, аппаратный контекст (SP, IP, регистры), статистика, код завершения.
3. **Пользовательские потоки** — быстрые (без системных вызовов), но:
   - ядро их не знает → несправедливое распределение CPU при 100% загрузке
   - блокирующий системный вызов одного потока **останавливает все** (лечится неблокирующими обёртками)
4. **Создание процесса (Windows)** — 7 шагов: дескриптор → ВАП → структура ВАП → данные → ресурсы по умолчанию → оповещение → первичный поток.
5. **Создание потока** — 6 шагов: дескриптор → инициализация полей → стек + структура контекста → инициализация SP/IP → оповещение → состояние ready.
   - **IP для первичного потока** — из заголовка исполнимого файла (точка входа, обычно стартовый код).
   - **IP для дополнительного потока** — адрес функции, переданной в `CreateThread`.
6. **Завершение потока** — статистика и код возврата сохраняются в дескрипторе, ресурсы освобождаются, но **дескриптор остаётся**, пока все пользователи не закроют на него хендлы.
7. **Завершение процесса** — сначала завершаются все потоки, потом освобождается ВАП и ресурсы, дескриптор процесса остаётся с кодом возврата (аналог «зомби» в Unix).
8. **Главный принцип:** дескрипторы (и процессов, и потоков) **не удаляются сразу после завершения** — они нужны для получения кода возврата и статистики. Удаление происходит только после явного отказа всех держателей ссылок/хендлов.
