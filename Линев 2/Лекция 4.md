# Архитектура операционных систем (Windows и Unix)
## Архитектура Windows (современная)

### Три уровня (общая структура)

1. **Аппаратный уровень (Hardware Level)** — физическое оборудование.
2. **Уровень ядра (Kernel Level)** — компоненты, работающие в привилегированном режиме.  
   Здесь выполняется **исполнительная часть Windows NT (Windows Executive)**.
3. **Пользовательский уровень (User Level)** — прикладные программы и **защищенные подсистемы**.

### Идея микроядра и реальность

**Миф:**  
При разработке Windows NT использовалась концепция микроядра. ОС разделена на подсистемы, большинство серверов работают в пользовательском режиме.

**Реальность:**
- Терминология (микроядро, приоритеты реального времени) пришла от команды разработчиков, ранее занимавшихся ОС реального времени.
- **Windows не является ОС реального времени**.
- Архитектура Windows — **монолитная**.  
  Если код выполняется на уровне ядра, он имеет доступ ко всему и может «все сломать».
- Часть, работающая в режиме ядра, называется **исполнительная часть (Executive)**.

### Современная структура пользовательского уровня (исправленная схема)

#### Основная подсистема — Win64
- **Нативная подсистема** современных Windows (64-битная).
- Большинство современных приложений работают именно в среде Win64.

#### Подсистема Win32 (для старых 32-битных программ)
- Позволяет выполнять 32-битные приложения.
- Работает **поверх Win64**.
- Эта технология называется **WOW (Win32 on Win64)**.

#### Подсистема для Linux-программ (WSL)
Раньше существовала подсистема **POSIX**, которая с ограничениями позволяла выполнять Unix-приложения.  
Сейчас ее нет. Вместо этого используется **WSL (Windows Subsystem for Linux)**.

- **WSL 1:**  
  Вызовы Linux транслировались в вызовы Win64 и выполнялись через ядро Windows.

- **WSL 2 (современная версия):**  
  Microsoft совместно с **Canonical** (разработчики Ubuntu) создали **специальное ядро Linux**, которое работает внутри Windows.
  - Ядро выполняется как виртуальная машина на уровне ядра Windows.
  - Используется **паравиртуализация**: часть вызовов транслируется напрямую в ядро Windows.
  - Для приложения Linux создается полная среда выполнения.

- **Графические приложения Linux под Windows:**
  - До недавнего времени WSL поддерживал только консольные приложения.
  - Теперь возможен запуск графических приложений Linux.
  - Для этого используется **X-сервер** (например, встроенный в некоторые SSH-клиенты).
  - Приложение выполняется под Windows, а его интерфейс отображается через X-сервер, что выглядит почти как нативное окно.


## Исполнительная часть (Kernel Level)

Это **монолитное ядро**, где выполняются все основные сервисы ОС.  
Компоненты, которые здесь находятся (по слайдам и лекции):

- **Менеджер объектов** — управляет именованными объектами ядра (файлы, события, семафоры и т.д.).
- **Монитор безопасности** — проверяет права доступа.
- **Менеджер процессов** — создает и завершает процессы и нити, приостанавливает и возобновляет их, хранит информацию о них.
- **Средства LPC (Local Procedure Call)** — механизм межпроцессного взаимодействия внутри системы.
- **Менеджер виртуальной памяти** — управляет виртуальным адресным пространством.
- **Менеджер ввода-вывода (I/O Manager)** — предоставляет независимые от устройств средства ввода-вывода.
  - Включает:
    - **Редиректор** и **сетевой сервер** — для удаленного ввода-вывода.
    - **Драйверы файловых систем**.
    - **Кэш-менеджер**.
    - **Драйверы устройств**.

- **Ядро (Kernel)** — обеспечивает:
  - Планирование процессов (диспетчеризацию).
  - Обработку прерываний и исключений.
  - Синхронизацию для многопроцессорных систем.
  - Восстановление после сбоев.

- **Уровень аппаратной абстракции (HAL)** — скрывает различия в аппаратуре (разные чипсеты, контроллеры прерываний и т.д.) от остального ядра.

## Архитектура Unix

### Общая структура

Unix традиционно имеет **два уровня**:
- **Пространство ядра** (Kernel Space)
- **Пространство пользователя** (User Space)

### Компоненты ядра Unix

- **Планировщик процессов** — управляет выполнением процессов, переключение контекста.
- **Управление памятью** — виртуальная память, страничная организация.
- **Файловая подсистема** — работа с файлами, права доступа, inode.
- **Подсистема ввода-вывода** — работа с драйверами устройств.
- **Межпроцессное взаимодействие (IPC)** — сигналы, каналы (pipes), сокеты, очереди сообщений.

### Уровень пользователя

- **Системные вызовы (syscalls)** — интерфейс между приложениями и ядром.
- **Библиотеки** (например, libc) — обертки над системными вызовами.
- **Прикладные программы** — выполняются в пользовательском режиме.

> В Unix **нет защищенных подсистем**, как в Windows. Все, что не ядро, — это пользовательские процессы, даже многие сервисы.

## Важные термины и пояснения

- **Защищенные подсистемы** — компоненты ОС, работающие в пользовательском режиме, но предоставляющие системные сервисы.
- **Исполнительная часть (Windows Executive)** — все, что работает в режиме ядра Windows.
- **WSL (Windows Subsystem for Linux)** — технология выполнения Linux-приложений в Windows.
- **WOW (Win32 on Win64)** — выполнение 32-битных Windows-приложений в 64-битной среде.
- **HAL (Hardware Abstraction Layer)** — уровень абстракции от аппаратуры.


# Исполнительная часть Windows (Executive)

## Уровень аппаратной абстракции (HAL — Hardware Abstraction Layer)

**Назначение:**  
HAL — это программный компонент, который предоставляет вышележащим компонентам ядра **единые абстракции аппаратного обеспечения**, скрывая детали конкретной платформы.

### Зачем это нужно?
- Аппаратные средства (процессоры, контроллеры прерываний, чипсеты) различаются.
- В разные периоды Windows поддерживала разные платформы:
  - Раньше: x86, PowerPC, Alpha, MIPS (до 4 платформ одновременно).
  - Сейчас: **x86-64** и **ARM** (например, ARM64).
- Чтобы операционная система была **переносимой (кросс-платформенной)**, большинство компонентов ядра не должны «знать», на каком железе они работают. Вместо этого они используют абстракции, предоставляемые HAL.

### Пример: модели адресации
- Размер структур данных (например, таблиц страниц) зависит от архитектуры.
- HAL предоставляет набор макросов или функций, которые работают единообразно для всех платформ.
- Вышележащие компоненты (менеджер памяти и др.) используют только эти абстракции и не зависят от деталей реализации на уровне железа.

> **Вывод:** HAL — это прослойка, обеспечивающая переносимость ядра между разными аппаратными архитектурами.


## Ядро (Kernel)

**Важное замечание:**  
Вопреки терминологии «микроядро», в Windows ядро не является минимальным. Оно выполняет ряд критических функций, но не всё — многие сервисы вынесены в другие компоненты исполнительной части.

### Основные функции ядра:

#### Планирование (диспетчеризация)
- Здесь слово «процессов» лишнее. Планирование — это распределение времени центрального процессора между **потоками (нитями)**, а не процессами.
- Ядро определяет, какой поток получит процессорное время следующим.

#### Обработка прерываний и исключительных ситуаций (первичная обработка)
- **Прерывание** — событие, возникающее вне логики выполнения программы.
  - Может быть связано с выполнением самой программы (исключение, например, деление на ноль).
  - Может быть внешним событием (от оборудования: клавиатура, мышь, сетевая карта, таймер).
- Ядро выполняет **первичную обработку**: принимает прерывание, сохраняет контекст, определяет, какой компонент должен обработать событие.
- **Содержательная обработка** обычно выполняется в других компонентах:
  - Внешние прерывания (от устройств) обрабатываются в **драйверах устройств**.
  - Исключения могут обрабатываться в пользовательском режиме или в других частях ядра.

#### Синхронизация процессоров для многопроцессорных и многоядерных систем
- Ядро поддерживает работу на системах с несколькими процессорами или ядрами.
- Это включает не только планирование, но и механизмы синхронизации (спин-блокировки, очереди, атомарные операции) для безопасного доступа к общим данным.

#### Восстановление системы после сбоев
- **Что происходит при критическом сбое в Windows?**
  - Синий экран смерти (BSOD — Blue Screen of Death). В современных версиях экран может быть черным, но суть та же.
  - Система выводит диагностическую информацию (код остановки, параметры, состояние регистров, адрес ошибки) и затем завершает работу или перезагружается.
- **Исторический контекст:**  
  Появление BSOD было революционным для своего времени. До этого при сбое система просто зависала без какой-либо информации. Администраторы фотографировали экран на пленочные фотоаппараты, печатали снимки и приносили разработчикам для анализа.

### Горячая замена оборудования (Hot Swap) — почему это не про Windows

Может ли Windows на лету заменить процессор, память или диск без выключения?

**Ответ:**
- **Замена жестких дисков «на лету»** — обычная практика.
  - В дисковых массивах (RAID) вышедший из строя диск можно вытащить и заменить новым без остановки системы.
- **PCI Express** поддерживает горячее подключение (опционально, но в серверах реализовано).
- **С памятью и процессорами** сложнее. Такая возможность требует:
  1. Поддержки со стороны аппаратуры (специальные контроллеры, архитектура).
  2. Поддержки со стороны ОС.
  3. Поддержки со стороны прикладного ПО.
- **Где это необходимо?**
  - На объектах, требующих **непрерывной работы 24/7**, где остановка недопустима.
  - Примеры: системы управления ядерным реактором, оборонные комплексы.
  - В таких системах оборудование выполнено в виде «горячезаменяемых» блоков (в шкафу загорается лампочка на отказавшем блоке, его вытаскивают и вставляют новый).
- **Серверы в дата-центрах** — плохой пример. Для них обычно планируют обслуживание: задачи мигрируют на другие серверы, систему выключают, чистят, меняют компоненты, затем включают обратно.

> **Вывод:** Windows не поддерживает горячую замену процессоров и памяти. Это не является недостатком, так как для подавляющего большинства сценариев такая возможность не требуется, а для критических систем используются специализированные аппаратно-программные комплексы.

## Другие компоненты исполнительной части


- **Менеджер объектов** — управляет именованными объектами ядра (файлы, события, семафоры, разделяемая память и т.д.).
- **Монитор безопасности** — проверяет права доступа к объектам.
- **Менеджер процессов** — создание и завершение процессов и потоков, управление их состоянием.
- **Средства LPC (Local Procedure Call)** — механизм межпроцессного взаимодействия внутри одной системы.
- **Менеджер виртуальной памяти** — управляет виртуальными адресными пространствами.
- **Менеджер ввода-вывода (I/O Manager)** — предоставляет независимый от устройств ввод-вывод, работает с драйверами, редиректорами, файловыми системами, кэш-менеджером.

## Важные тезисы для экзамена

1. **HAL (Hardware Abstraction Layer)**:
   - Обеспечивает переносимость ОС.
   - Скрывает различия между аппаратными платформами.
   - Предоставляет единые абстракции для работы с железом.

2. **Ядро (Kernel)**:
   - Выполняет планирование потоков.
   - Первичная обработка прерываний и исключений.
   - Обеспечивает синхронизацию в многопроцессорных системах.
   - Формирует аварийный вывод (BSOD) при фатальных ошибках.

3. **Обработка прерываний**:
   - Ядро — первичная обработка.
   - Драйверы устройств — основная обработка внешних прерываний.

4. **Горячая замена**:
   - Windows не поддерживает замену CPU и памяти на лету.
   - Диски и некоторые PCIe-устройства менять можно (при поддержке аппаратуры).
   - Непрерывная работа критических систем требует специализированных решений.

# Менеджер объектов Windows

## Общая концепция: объекты и управление ресурсами

Операционная система управляет ресурсами разных типов:
- Аппаратные ресурсы (процессорное время, память, устройства)
- Информационные ресурсы (файлы, разделяемая память)
- Программные ресурсы (процессы, потоки, синхронизационные объекты)

Для управления этими разнородными ресурсами Windows использует **единую объектную модель**.

### Что такое объект в Windows?
- **Объект** — это структура данных в ядре, которая описывает конкретный ресурс.
- Все объекты имеют общую часть (базовый объект) и специализированную часть (зависит от типа).
- Это напоминает **наследование в C++**: есть «базовый класс» объекта, от которого наследуются объекты конкретных типов (файлы, процессы, события, семафоры и т.д.).

### Менеджер объектов (Object Manager)
- Это компонент исполнительной части Windows.
- Выполняет функции **базы данных объектов ядра**:
  - Создание объектов
  - Удаление объектов
  - Хранение объектов
  - Обеспечение доступа к объектам

## Типы объектов и их содержимое

У каждого объекта есть **тип**. От типа зависят:
- **Данные**, хранящиеся в объекте (содержательная часть)
- **Операции**, допустимые над объектом (например, файл можно читать и писать, процесс — завершить или изменить приоритет)

### Важное уточнение про объект «файл»
Объект ядра типа «файл» — это **не сам файл на диске**, а **контекст доступа** к нему.
- Когда процесс открывает файл, создается объект файла в ядре.
- Этот объект хранит информацию о текущей позиции чтения/записи, правах доступа, указатель на файловую систему и т.д.
- При закрытии файла объект уничтожается. Сам файл на диске остается.

### Объект «процесс»
- Когда запускается программа, создается объект процесса.
- В нем хранится информация о процессе: идентификатор, приоритет, состояние, счетчики использования ресурсов и т.д.
- К объекту процесса можно применять операции: приостановка, завершение, изменение приоритета.
- При завершении процесса объект уничтожается.


## Безопасность объектов

Каждый объект ядра имеет **дескриптор безопасности** (security descriptor), который включает:
- **Access Control List (ACL)** — список управления доступом
- **Access Control Entries (ACE)** — отдельные записи в списке

Каждая ACE содержит:
1. **Субъект безопасности** — пользователь или группа
2. **Права доступа** (например, чтение, запись, удаление)
3. **Флаг разрешения или запрещения**

### Особые случаи ACL:

| Состояние ACL | Значение |
|---------------|---------|
| ACL пуст (нет записей) | **Доступ запрещен всем**. Исключение: владелец объекта всегда может изменить права доступа (изменить ACL). |
| ACL отсутствует (NULL) | **Доступ разрешен всем** на все операции. |

> Это полностью соответствует модели безопасности, которую вы видите в свойствах файлов в проводнике Windows. Файловая система — лишь одно из внешних представлений внутренней объектной модели Windows.

## Механизм доступа: хэндлы (Handles)

Программы в пользовательском режиме **не работают с объектами ядра напрямую**. Вместо этого они используют **хэндлы (handles)**.

### Что такое хэндл?
Хэндл — это **номер записи в таблице хэндлов процесса**.  
Это **не указатель на объект**, а индекс, который ядро использует для поиска ссылки на объект ядра.

### Структура: два уровня
- **Пользовательский режим:** процесс имеет таблицу хэндлов.
- **Режим ядра:** объекты ядра хранятся в памяти ядра, недоступной напрямую из пользовательского режима.

```
ПРОЦЕСС 1 (User)          ПРОЦЕСС 2 (User)
┌─────────────────┐       ┌─────────────────┐
│ Вирт. адресное  │       │ Вирт. адресное  │
│ пространство    │       │ пространство    │
│                 │       │                 │
│ Таблица хэндлов │       │ Таблица хэндлов │
│ ┌───────┐       │       │ ┌───────┐       │
│ │ h1    │───────┼────┐  │ │ h2    │───────┼────┐
│ └───────┘       │    │  │ └───────┘       │    │
└─────────────────┘    │  └─────────────────┘    │
                       │                         │
                       ▼                         ▼
                ┌─────────────────────────────────┐
                │        ЯДРО (Kernel)            │
                │ ┌─────────────────────────────┐ │
                │ │ Объект ядра (типа XXX)      │ │
                │ │ Имя: "My_Lovely_Object"     │ │
                │ │ Счетчик ссылок: 2           │ │
                │ │ Дескриптор безопасности      │ │
                │ └─────────────────────────────┘ │
                └─────────────────────────────────┘
```

## Жизненный цикл работы с объектом (пример)

### Создание объекта
```c
HANDLE h1;
h1 = CreateXXX(
    ...,
    "My_Lovely_Object",  // имя объекта (строка Unicode)
    ...
);
```
- Ядро создает объект указанного типа (XXX).
- Если указано имя, объект становится **именованным**.
- В таблицу хэндлов **текущего процесса** добавляется запись со ссылкой на объект.
- Функция возвращает номер этой записи — **хэндл**.

### Использование объекта
Через хэндл процесс может выполнять операции над объектом:
- Для файла: чтение, запись
- Для процесса: изменение приоритета, завершение
- Для синхронизационных объектов: ожидание

### Подключение другого процесса к тому же объекту
Если объект **именованный**, другой процесс может подключиться к нему:
```c
HANDLE h2;
h2 = OpenXXX(
    ...,
    "My_Lovely_Object",  // то же имя
    ...
);
```
- Система проверяет права доступа.
- Во втором процессе создается **новая запись** в его таблице хэндлов.
- Хэндл **будет другим** (гарантированно).  
  Windows гарантирует, что хэндлы для одного объекта в разных процессах — разные числа.

### Закрытие доступа
```c
CloseHandle(h1);   // в первом процессе
CloseHandle(h2);   // во втором процессе
```
- Удаляется запись из таблицы хэндлов соответствующего процесса.
- Уменьшается **счетчик ссылок** объекта в ядре.

### Уничтожение объекта
У каждого объекта ядра есть **счетчик использования (reference count)**.  
Когда счетчик становится равен 0, объект автоматически уничтожается.

**Исключение:** объекты процессов и потоков  
У них есть дополнительная «внутренняя» единица счетчика, пока процесс или поток активен.  
Даже если все хэндлы закрыты, объект процесса не удаляется, пока процесс не завершится.

## Связь с умными указателями (reference counting)

Лектор проводит аналогию с **умными указателями** в C++ (например, `std::shared_ptr`):
- Один объект, несколько ссылок на него.
- При удалении последней ссылки объект автоматически удаляется.
- Это универсальный паттерн управления ресурсами, используемый не только в ОС, но и в прикладном программировании.

## Именованные объекты

Одно из ключевых свойств объектов Windows:  
**Объектам можно назначать имена в виде строк Unicode.**

### Зачем это нужно?
- Позволяет разным процессам находить один и тот же объект без передачи хэндлов через другие механизмы IPC.
- Достаточно знать имя объекта и вызвать `OpenXXX`.

### Особенности
- Имена видны в пределах всего системы (глобальное пространство имен объектов).
- Существуют иерархические каталоги объектов (например, `\Device`, `\BaseNamedObjects`).
- Менеджер объектов управляет этим пространством имен.

## Менеджер объектов (завершение)

### Проблема уникальности имен объектов

Именованные объекты удобны для взаимодействия между процессами: достаточно придумать имя и использовать его в `CreateXXX` и `OpenXXX`. Однако возникает проблема:

> **Что если два программиста независимо придумают одно и то же имя для объекта?**

#### Реальная ситуация
- Программисты часто используют **Stack Overflow** и другие ресурсы, копируя готовые примеры кода.
- В примерах часто используются одинаковые имена объектов (например, `"MySharedMemory"`, `"Global\\MyEvent"`).
- Если две разные программы (или две копии одной программы) используют одинаковое имя, они начнут работать с **одним и тем же объектом ядра**, что может привести к непредсказуемым последствиям.

#### Исторический пример
Лектор приводит пример с программой **Nero Burning ROM**. Она конфликтовала с другой программой, потому что обе использовали одинаковые имена объектов. При исследовании выяснилось, что оба разработчика скопировали код с одного и того же веб-сайта, не изменив имена объектов.

### Как гарантировать уникальность имени?

Просто придумать уникальное слово недостаточно — кто-то другой может придумать то же самое.

**Решение:** использовать **уникальные идентификаторы (GUID/UUID)**.

- **GUID (Globally Unique Identifier)** — 128-битное число, которое с высокой вероятностью является уникальным во всем мире.
- В Windows есть API для генерации GUID:
  - `CoCreateGuid` (COM)
  - `UuidCreate` (RPC)
- В командной строке: `uuidgen`
- GUID формируется на основе:
  - Аппаратного обеспечения
  - Инсталляции Windows
  - Текущего времени
  - Других источников энтропии

**Рекомендация:** использовать GUID в качестве имени объекта, чтобы избежать конфликтов.

### Утечка хэндлов (Handle Leak)

Если процесс не вызывает `CloseHandle` для объекта, ссылка на объект остается в таблице хэндлов, а счетчик использования объекта не уменьшается.

#### Последствия
- Если программа работает **недолго**, после завершения процесса все хэндлы закрываются автоматически — проблема не проявляется.
- Если программа работает **длительное время** (например, сервер), утечки накапливаются.

## Монитор безопасности (Security Monitor)

**Функции:**
- Устанавливает **правила защиты** на локальном компьютере.
- Защищает **ресурсы операционной системы** и **объекты** (как исполняемые, так и неисполняемые).

## Менеджер процессов (Process Manager)

**Функции:**
- Управляет **процессами и потоками** (в лекции лектор использует термин «потоки» вместо «нити»).
- Создает и завершает процессы и потоки.
- Приостанавливает и возобновляет их выполнение.
- Хранит информацию о них (состояние, приоритет, идентификаторы, статистику использования ресурсов).

## LPC (Local Procedure Call)

### Контекст: вызов функций в разных процессах

- Внутри одного процесса вызов функции — это просто переход по адресу (`f1();`).
- Если функция `f1` находится в **другом процессе**, вызвать ее напрямую невозможно, так как у каждого процесса свое виртуальное адресное пространство.

### RPC (Remote Procedure Call) — общий механизм

**RPC** позволяет:
- Процессу «экспортировать» функцию для вызова извне.
- Другому процессу «подключиться» к этой функции и вызвать ее.
- Вызов может происходить **как на том же компьютере, так и через сеть** (на другом компьютере).

**Пример использования RPC:** сетевые файловые системы, распределенные вычисления.

### LPC — оптимизированная версия для Windows

**LPC (Local Procedure Call)** — это **оптимизированный вариант RPC**, предназначенный для работы **в пределах одного компьютера**.

#### Где используется LPC?
- **Обработка сообщений графического интерфейса Windows**.
- Когда пользователь двигает мышь или нажимает кнопки, сообщения (например, `WM_MOUSEMOVE`, `WM_LBUTTONDOWN`) отправляются потоку, который обрабатывает интерфейс окна.
- Этот механизм фактически реализован через LPC: сообщение передается между процессами как вызов процедуры.

**Определение LPC:**  
LPC — это механизм, обеспечивающий вызов функции, выполняющейся в другом процессе, оптимизированный для высокочастотной передачи сообщений в Windows.

## Менеджер виртуальной памяти (Virtual Memory Manager)

**Основные функции:**
1. Управление **физической оперативной памятью**.
2. Управление **виртуальными адресными пространствами процессов**.
3. Поддержка **виртуальной памяти** (подкачка страниц на диск).

> Это один из самых сложных компонентов ядра (сотни тысяч строк кода).

Лектор рекомендует книгу **Мела Гормана «Understanding the Linux Virtual Memory Manager»** для глубокого понимания работы менеджера памяти (хотя книга про Linux, концепции во многом схожи).

**Важно:** понимание работы менеджера памяти необходимо для понимания работы других компонентов, в частности, **файловой подсистемы**.

## Менеджер ввода-вывода (I/O Manager)

### Общая функция
Предоставляет **средства ввода-вывода, независимые от устройств**.

### Основная задача
Создать **окружение (среду)**, в котором выполняются все компоненты, связанные с вводом-выводом.

Это окружение определяет:
- Как должны быть организованы компоненты ввода-вывода.
- Каким правилам они должны подчиняться.
- Какие ресурсы им доступны.

**Ключевой элемент:** стандартная структура данных, используемая при вводе-выводе всеми компонентами (например, **IRP — I/O Request Packet**).

## Компоненты подсистемы управления файлами (сверху вниз)

### Редиректор (Redirector)

**Функция:** определяет, какой нижележащий компонент будет обрабатывать запрос к файлу.

#### Алгоритм работы
1. Программа пытается открыть файл (например, `C:\folder\file.txt` или `\\server\share\file.txt`).
2. Редиректор определяет **тип устройства**:
   - **Локальный диск** (C:, D:)
   - **Съемный диск** (флешка)
   - **Сетевой диск** (подключенный сетевой каталог)

3. В зависимости от типа:
   - Если **сетевой** → запрос передается **сетевому редиректору** (клиентская часть, рабочая станция), который общается с сетевым сервером на удаленной машине.
   - Если **локальный или съемный** → определяется **тип файловой системы** (NTFS, FAT, exFAT и т.д.) и запрос передается соответствующему **драйверу файловой системы**.

### Драйвер файловой системы (File System Driver)

**Функция:** организация хранения **иерархической файловой системы** на **линейном массиве блоков**.

#### Пояснение
- На нижнем уровне (драйвер устройства) физический диск представляется как **массив блоков** (обычно по 512 байт или 4 КБ).
- Драйвер файловой системы берет этот линейный массив блоков и организует на нем:
  - Каталоги (папки)
  - Файлы
  - Права доступа
  - Метаданные

**Кратко:** преобразование линейного адресного пространства блоков в иерархическую структуру файлов и каталогов.

### Кэш-менеджер (Cache Manager)

**Функция:** временное хранение данных из **медленной памяти** (устройства долговременного хранения — диск, SSD) в **более быстрой памяти** (оперативная память).

#### Особенности
- В данном случае «медленная память» — это диск, «быстрая память» — ОЗУ.
- Кэш-менеджер ускоряет доступ к часто используемым данным.
- Это стандартное определение кэширования: более быстрая память обычно дороже и имеет меньший объем.

**Функция:** непосредственно управляют **аппаратными устройствами**.

- Пример: драйвер контроллера жесткого диска (SATA, NVMe) — обеспечивает работу с диском на уровне команд контроллера.
- Драйверы устройств работают на уровне ядра и имеют полный доступ к аппаратуре.

- **WSL (Windows Subsystem for Linux)** — для выполнения Linux-приложений.

# Архитектура Unix

## Общая структура (три уровня)



1. **Уровень пользователя (User Level)** — выполнение прикладных программ и библиотек.
2. **Уровень ядра (Kernel Level)** — основные подсистемы ОС.
3. **Уровень аппаратного обеспечения (Hardware Level)** — физическое оборудование.

**Ключевое отличие:** в Unix поддерживается **только одна операционная среда** (в отличие от Windows, где их несколько).
## Уровень пользователя (User Level)

### Прикладные программы
- Программы пользователя выполняются в пользовательском режиме.

### Библиотеки и интерфейс системных вызовов
- Прикладные программисты обычно используют **функции стандартных библиотек** (например, libc в C/C++), которые скрывают детали системных вызовов.
- **Интерфейс системных вызовов (System Call Interface)** — это граница между пользовательским режимом и ядром.
- При необходимости можно вызывать системные вызовы напрямую (через специальные инструкции), но обычно это не требуется.
## Уровень аппаратной абстракции

Как и в Windows, в Unix есть уровень, предоставляющий **приемлемые абстракции аппаратного обеспечения** вышележащим компонентам. Это обеспечивает переносимость ОС на разные аппаратные платформы.


## Уровень ядра (Kernel Level)

Ядро Unix разделено на **две основные подсистемы**:

### Подсистема управления процессами (Process Management)

#### Планировщик (Scheduler)
- Распределяет время центрального процессора между процессами/потоками.

#### Управление памятью (Memory Management)
- Управление физической оперативной памятью.
- Поддержка **виртуальных адресных пространств процессов**.
- Поддержка адресного пространства ядра.

#### Межпроцессное взаимодействие (IPC — Interprocess Communication)
- Обеспечивает механизмы взаимодействия между процессами:
  - Сигналы (signals)
  - Каналы (pipes)
  - Очереди сообщений (message queues)
  - Семафоры (semaphores)
  - Разделяемая память (shared memory)
  - Сокеты (sockets) — для сетевого и локального взаимодействия

### Подсистема управления файлами (File Management)

#### Общая структура
Аналогична подсистеме ввода-вывода Windows, но с важным дополнением: **разделение устройств на символьные и блочные**.

#### Символьные устройства (Character Devices)
- Поддерживают **потоковую передачу данных** (байт за байтом).
- Не поддерживают произвольный доступ (прямую адресацию).
- **Примеры:** клавиатура, мышь, последовательные порты, терминалы.

#### Блочные устройства (Block Devices)
- Могут быть представлены в виде **массива блоков одинакового размера** (например, 512 байт или 4 КБ).
- Поддерживают **произвольный доступ** (можно прочитать блок №N).
- **Примеры:** жесткие диски, SSD, CD/DVD, флешки.

> **Важно:** сетевая карта, звуковая карта, видеокарта **не классифицируются** как символьные или блочные устройства. Для них существуют отдельные подсистемы (сетевая подсистема, звуковая подсистема, графическая подсистема).

#### Драйверы устройств
- Управляют физическими устройствами.
- Для блочных устройств драйвер предоставляет интерфейс массива блоков.

#### Кэш (Cache)
- Временное хранение данных диска в оперативной памяти.
- В современных Unix (и Linux) кэш **тесно интегрирован с подсистемой управления памятью**, а не существует как отдельный компонент.

#### Virtual File System (VFS) — виртуальная файловая система
- **Функция:** обеспечивает **единую иерархическую файловую систему** из нескольких различных файловых систем.
- В Windows эта функция выполняется редиректором (но с другой реализацией).

**Как это работает в Unix:**
- Каждая файловая система (ext4, XFS, FAT, NTFS и т.д.) — это отдельное дерево каталогов.
- VFS позволяет «смонтировать» (mount) одно дерево в определенную точку (точку монтирования) главного дерева.
- В результате формируется **единое иерархическое дерево**, доступное всем процессам.

#### Доступ процессов к файлам
- При открытии файла ядро создает структуры данных (дескрипторы файлов, таблицы открытых файлов), аналогичные хэндлам в Windows.
- Это обеспечивает контролируемый доступ к файлам и устройствам.

## Исторический контекст: почему символьные и блочные устройства?

Разработчики Unix вдохновлялись простотой интерфейса файловой системы:
- Открыть файл, прочитать, записать, закрыть.
- Они решили, что **устройства ввода-вывода** было бы удобно использовать **через тот же самый интерфейс**.

Так появилась концепция:
- **Символьные устройства** — для потоковых данных (клавиатура, мышь).
- **Блочные устройства** — для устройств с произвольным доступом (диски).

Сетевые и мультимедийные устройства оказались сложнее, поэтому они были вынесены в отдельные подсистемы.

## Современные изменения в архитектуре Unix


- **Кэш интегрирован в подсистему управления памятью**, а не существует как отдельный компонент.
- Взаимодействие между подсистемой управления файлами и подсистемой управления памятью **очень тесное**.
- Для понимания работы драйверов файловых систем необходимо понимание работы управления памятью (функции типа `readpage`, `writepage`, `commit`).
- Современные диаграммы содержат **в два раза больше компонентов**, чем показано на старых слайдах.


## Сравнение архитектур Windows и Unix (ключевые отличия)

| Аспект                    | Windows                                                                   | Unix                                                                                   |
| ------------------------- | ------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| **Операционные среды**    | Несколько (Win64, Win32, WSL)                                             | Одна (нативная Unix-среда)                                                             |
| **Объектная модель**      | Единая объектная модель (менеджер объектов)                               | Нет единой объектной модели; файлы, процессы, IPC — отдельные интерфейсы               |
| **Управление файлами**    | Редиректор, драйверы ФС, кэш-менеджер                                     | VFS (Virtual File System), символьные/блочные устройства, кэш (интегрирован с памятью) |
| **IPC**                   | LPC (оптимизирован для сообщений), RPC, конвейеры, сокеты, почтовые ящики | Сигналы, каналы, очереди сообщений, семафоры, разделяемая память, сокеты               |
| **Аппаратная абстракция** | HAL (Hardware Abstraction Layer)                                          | Аппаратно-зависимый код в ядре, аналогичный HAL                                        |

## Ключевые термины для запоминания (дополнение)

| Термин (Unix) | Значение |
|---------------|---------|
| **VFS (Virtual File System)** | Подсистема, объединяющая несколько файловых систем в единое иерархическое дерево. |
| **Символьные устройства (character devices)** | Устройства с потоковым вводом-выводом (клавиатура, мышь, последовательные порты). |
| **Блочные устройства (block devices)** | Устройства, представляемые как массив блоков с произвольным доступом (диски, SSD). |
| **IPC (Interprocess Communication)** | Механизмы взаимодействия между процессами (сигналы, каналы, очереди сообщений, семафоры, разделяемая память, сокеты). |
| **Планировщик (scheduler)** | Компонент ядра, распределяющий процессорное время. |
| **Управление памятью** | Управление физической памятью и виртуальными адресными пространствами. |
