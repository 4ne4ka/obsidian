## 1. Введение: процессы в UNIX vs Windows

- В **Windows** создание процесса и запуск в нём новой программы — **одно действие** (`CreateProcess`).
- В **UNIX** это **два разных системных вызова**:
  1. `fork()` — создание копии текущего процесса.
  2. `exec*()` — замена текущей программы в процессе на другую.

> **Важно:** В классическом UNIX не было многопоточности (появилась позже, в Linux поток — это «облегчённый процесс», разделяющий адресное пространство и ресурсы с другими). На лекции рассматриваем **однопоточные процессы**.

---

## 2. Системные вызовы UNIX для работы с процессами

| Вызов | Что делает |
|-------|-------------|
| `fork()` | Создаёт **копию** текущего процесса. Родитель и потомок выполняют одну и ту же программу с одной и той же точки входа. |
| `exec*()` | Заменяет программу в текущем процессе на другую (загружает её из файла). Старая программа полностью стирается из ВАП. |
| `exit()` | Завершает текущий процесс с кодом возврата. |
| `wait()` | Родитель ждёт завершения любого (или конкретного) потомка, получает его код возврата. |

> **Важное обязательство:** каждый родитель **обязан** вызвать `wait()` для своих потомков. Если родитель завершается раньше, все его потомки «усыновляются» процессом `init` (PID=1), который вызывает `wait()` за них.

## 3. Типичный шаблон кода: запуск другой программы в потомке

```c
int ChildPID, ChildRetCode, RetCode = 0;
ChildPID = fork();

if (ChildPID == 0) {
    // Дочерний процесс
    exec*(programme, ...);   // заменяет программу на programme
    // если exec* вернул управление — значит, ошибка
} else {
    // Родительский процесс
    wait(&ChildRetCode);     // ждём завершения потомка
}
// Дальнейший код родителя (после wait)
exit(RetCode);
```

### Пошаговое объяснение работы этого кода (как в лекции)

1. **Вызов `fork()`**  
   - Создаётся **точная копия** процесса.  
   - Теперь есть **два** экземпляра одной и той же программы (родитель и потомок).  
   - В родительском процессе `fork()` возвращает **PID потомка** (положительное число).  
   - В дочернем процессе `fork()` возвращает **0**.

2. **Ветвление по `if`**  
   - `ChildPID == 0` → это потомок: вызывает `exec*()`.  
   - Иначе → это родитель: вызывает `wait()` и блокируется до завершения любого потомка.

3. **В потомке:**  
   - `exec*()` **полностью заменяет** текущую программу на `programme`.  
   - Старый код (включая проверку `if`) исчезает.  
   - Начинает выполняться `programme` с её точки входа.  
   - Когда `programme` завершается, потомок завершается, а родитель получает сигнал.

4. **В родителе:**  
   - `wait()` возвращает управление, когда потомок завершился.  
   - В `ChildRetCode` попадает код завершения потомка.  
   - Родитель продолжает своё выполнение и в конце вызывает `exit()`.

> **Важное замечание лектора:** Если вызвать `wait()`, а потомков нет, процесс **зависнет навсегда**. Всегда проверяйте.

## 4. Детали работы `fork()` (под капотом)

Исходное состояние (однопоточный процесс):
- Виртуальное адресное пространство (ВАП): регионы кода, данных, стека, куча.
- Аппаратный контекст потока: IP (следующая инструкция), SP (указатель стека), регистры.
- Таблица дескрипторов процессов, PID, PPID, статистика.
- Файловая система + пространство подкачки (swap).

**Что делает `fork()` по шагам (без оптимизаций):**

1. Резервирует место в **пространстве подкачки** для регионов данных, стека и кучи **нового процесса** (код — только для чтения, его можно не копировать, а отобразить на тот же исполняемый файл).
2. Создаёт новую запись в таблице дескрипторов процессов:
   - Новый уникальный PID.
   - PPID = PID родителя.
   - Обнуляет статистику, сбрасывает сигналы и блокировки.
   - Остальные атрибуты — либо по умолчанию, либо копируются с родителя.
3. Создаёт **таблицы страниц** (аппаратно-зависимые структуры) для нового ВАП.
4. Копирует **атрибуты родителя** потомку:
   - Открытые файлы (разделяют **общий контекст ввода-вывода**: текущая позиция в файле общая!).
   - Сетевые соединения (если оба читают из одного сокета — кто получит данные, непредсказуемо).
5. Создаёт **структуру ВАП** потомка:  
   - Регион кода — отображается на **тот же** исполняемый файл.  
   - Регионы стека, данных, кучи — на **новые** области в подкачке.
6. Копирует **содержимое** всех регионов из родителя в потомка (все данные, стек, кучу — одинаковы).
7. Создаёт аппаратный контекст для единственного потока потомка:
   - Копирует все регистры из родителя.
   - **Отличие:** регистр возврата (`fork()` в родителе возвращает PID, в потомке — 0).
8. Состояние потомка — «готов к выполнению».

> **Оптимизация (COW — Copy‑on‑Write) в современных UNIX:**  
> На самом деле данные не копируются сразу. Родитель и потомок **разделяют** одну и ту же физическую память, пока кто-то из них не попытается **записать**. Тогда страница копируется. Это сильно экономит время и память.

## 5. Что происходит при `exit()` (завершении процесса)

Слайд «UNIX exit()» описывает последовательность:

1. Отключаются все сигналы.
2. Закрываются все открытые файлы.
3. Сохраняется статистика использования ресурсов и код возврата в глобальной структуре (для `wait()`).
4. Состояние процесса меняется на **«зомби»** — он уже не выполняется, но запись в таблице процессов сохраняется (нужна для `wait()`).
5. Все **непосредственные потомки** текущего процесса передаются процессу `init` (PID=1), который становится их новым родителем.
6. Уничтожается **ВАП процесса** (освобождается память).
7. Родителю посылается сигнал `SIGCHLD` — уведомление о завершении потомка.
8. Если родитель уже ждал в `wait()` — он пробуждается.
9. Происходит переключение контекста.

> **Зомби (zombie)** — процесс, который завершился, но его запись в таблице процессов ещё существует, потому что родитель не вызвал `wait()`. Если зомби накапливаются, это проблема.

## 6. Итог по процессам и потокам (со слайда «Заключение»)

- **Процесс** — пассивный объект, владелец ресурсов (память, файлы, сокеты).
- **Поток** — активный объект, выполняет код внутри процесса.
- С точки зрения ОС, и процессы, и потоки — это ресурсы, требующие специального управления.

## 7. Важные практические моменты (из слов лектора)

- `fork()` + `exec()` — классический способ запустить другую программу в UNIX.
- Если вы хотите создать **много параллельных обработчиков** (например, веб-сервер), вы делаете `fork()` 50 раз, и все потомки слушают один сокет — кто первый примет соединение, тот и обрабатывает.
- `wait()` обязателен. Если родитель не вызывает `wait()`, потомки после завершения становятся **зомби**.
- Если родитель завершился раньше потомков, их усыновляет `init` — он вызовет `wait()` за родителя.
- В многопоточном приложении поведение `fork()` в UNIX **не определено стандартом POSIX** (реализация может быть разной). На лекции рассматривается только однопоточный случай.

## 8. Системный вызов `exec*()` — замена программы в процессе

`exec*()` **не создаёт новый процесс**. Он **заменяет** текущую программу в том же процессе на другую.  
ВАП (виртуальное адресное пространство) переиспользуется, но его содержимое полностью пересоздаётся.

### 8.1. Предварительные действия перед `exec*()`

#### Шаг 1. Трансляция имени файла
- Если передано относительное имя — система ищет файл по переменной окружения `PATH` (или аналогичной).
- Результат — **полный путь** к исполняемому файлу.
- По этому пути ищется объект в файловой системе.

#### Шаг 2. Чтение заголовка файла (распознавание формата)
Три варианта, что может быть в файле:

| Вариант | Что значит | Действие ОС |
|---------|------------|--------------|
| **1. Двоичный исполняемый файл** (обычно ELF, старый a.out уже не поддерживается) | Содержит заголовок с точкой входа, расположением регионов кода/данных | Разбираем заголовок, берём адрес точки входа, размеры регионов |
| **2. Shebang (`#! /путь/к/интерпретатору`)** | Файл — скрипт для указанного интерпретатора | Запускается интерпретатор, а содержимое скрипта подаётся ему на `stdin` |
| **3. Неизвестный формат** (нет ELF, нет shebang) | Не распознан | Запускается **командный интерпретатор по умолчанию** (`/bin/sh` или `/bin/bash`), ему на `stdin` подаётся содержимое файла |

#### Шаг 3. Проверка битов setuid / setgid (можно опустить при ответе, но для понимания)
- Некоторые программы (`ping`, `passwd`) должны работать с правами **владельца файла**, а не запустившего пользователя.
- Если бит установлен — процесс временно получает эффективный UID/GID владельца файла.

#### Шаг 4. Сохранение аргументов вызова и переменных окружения
- В старом ВАП есть регион с переменными окружения и аргументами командной строки.
- При замене программы этот регион **будет уничтожен**.
- Поэтому ОС **сохраняет** эти данные в **адресном пространстве ядра** (временно).
- Потом, после создания нового ВАП, они будут скопированы обратно, чтобы новая программа получила `argc, argv, envp` в `main()`.

> **Важно:** Сам код `exec*()` находится в старом регионе кода, его параметры — в стеке/данных/куче. Их нужно сохранить ДО разрушения старого ВАП.

---

### 8.2. Основные действия `exec*()` (после предварительных)

#### 1. Освобождение ресурсов старого процесса
- Разрушается **внутренняя структура** ВАП (таблицы страниц, регионы).
- Освобождаются **блоки в пространстве подкачки** (swap), которые принадлежали старой программе (стек, куча, данные).
- Регион кода старой программы тоже уничтожается.

#### 2. Резервирование места в пространстве подкачки для **новой** программы
- **Регион кода** — не в swap, а отображается прямо на исполняемый файл (только для чтения).
- **Регион данных** — размер берётся из заголовка ELF.
- **Стек и куча** — размер по умолчанию (например, 4 КБ) или из ELF (если указано).
- Для стека и кучи выделяется место **в пространстве подкачки**.

#### 3. Создание новых регионов ВАП
- Регион кода и регион данных — по адресам, указанным в ELF.
- Регион стека и кучи — **адреса могут быть случайными** (см. ниже).
- Все регионы отображаются на соответствующие места на диске (исполняемый файл или swap).

#### 4. Инициализация аппаратного контекста
- Указатель инструкции (IP) = **точка входа** из ELF.
- Указатель стека (SP) = вершина нового (пока пустого) стека.
- Остальные регистры — в начальное состояние.
- В регионы стека и кучи пока **нет данных** (они заполнятся при работе).

#### 5. Восстановление аргументов и окружения
- Сохранённые ранее аргументы и переменные окружения копируются в новое ВАП (обычно в начало стека).
- Формируется стандартный `main(argc, argv, envp)`.

#### 6. Процесс переводится в состояние «готов к выполнению»
- После того как планировщик даст ему CPU, начнётся выполнение **новой программы** с её точки входа.

---

### 8.3. Важное отступление: почему стек создаётся в **случайном** месте?

> Это прямая защита от атак **переполнения буфера** (buffer overflow).

**Как работает атака (упрощённо):**
1. В стеке расположен локальный массив (например, `char buf[64]`).
2. Злоумышленник передаёт больше 64 байт.
3. Переполнение затирает **адрес возврата** из стека.
4. Если заранее известно, где лежит стек, можно подставить адрес на свой вредоносный код.
5. При возврате из функции управление переходит на код злоумышленника.

**Защита:**  
Если адрес стека **случайный** (рандомизированный), злоумышленник не может предсказать, куда прыгать.  
Современные ОС используют **ASLR** (Address Space Layout Randomization) — стек, куча, библиотеки загружаются в случайные адреса при каждом запуске.

## 9. Завершение процесса: `exit()`

Слайд «UNIX exit()» — это **последовательность действий** при штатном завершении (добровольный вызов `exit()`).

### 9.1. Пошагово, что делает `exit()`

| № | Действие | Пояснение |
|---|----------|------------|
| 1 | **Отключение всех сигналов** | Процесс больше не реагирует на сигналы (SIGTERM, SIGINT и др.) |
| 2 | **Закрытие всех открытых файлов** | Но! Это на уровне ОС. Буферы **высокоуровневого ввода-вывода** (stdio) могут не сброситься, если не был вызван `fclose()`. При штатном `exit()` стандартная библиотека регистрирует обработчик, который сбрасывает буферы. При `_exit()` — нет. |
| 3 | **Сохранение кода возврата и статистики** | В структуру `proc` в таблице процессов записывается `status` (код завершения) и данные о времени CPU, использованной памяти и т.п. |
| 4 | **Изменение состояния на «зомби»** | Процесс больше не выполняется, но запись в таблице процессов сохраняется. Это нужно, чтобы родитель мог вызвать `wait()` и получить код возврата. |
| 5 | **Назначение процесса `init` родителем для всех потомков** | У текущего процесса могут быть дочерние процессы. Теперь их «усыновляет» процесс с PID=1 (`init`, `systemd` и т.д.). Он гарантированно вызовет для них `wait()`, когда они завершатся. |
| 6 | **Уничтожение ВАП процесса** | Освобождаются все регионы памяти (код, данные, стек, куча), таблицы страниц, пространство подкачки. |
| 7 | **Посылка сигнала SIGCHLD родителю** | ОС уведомляет родительский процесс: «твой потомок завершился». |
| 8 | **Если родитель вызывал `wait()` — пробуждение** | Родитель, который был заблокирован в `wait()`, переходит в состояние «готов» и получает код завершения потомка. После этого запись процесса-зомби может быть удалена. |
| 9 | **Переключение контекста** | Планировщик выбирает другой процесс для выполнения. О завершённом процессе можно забыть. |

### 9.2. Что такое «зомби» (zombie)

- Процесс, который **завершился**, но его запись в таблице процессов **всё ещё существует**.
- Ресурсы (память, файлы) уже освобождены, остался только PID, код возврата, статистика.
- Нужен для того, чтобы родитель мог вызвать `wait()`.
- Если родитель **никогда не вызывает `wait()`**, зомби накапливаются (проблема).
- `init` не допускает зомби — он сразу вызывает `wait()` для усыновлённых процессов.

### 9.3. `exit()` vs `_exit()`

| Вызов | Обработчики `atexit()` | Сброс буферов stdio |
|-------|------------------------|----------------------|
| `exit()` | Да, вызываются | Да |
| `_exit()` | Нет | Нет (потеря данных) |

В лекции подчёркивается: при штатном завершении через `return` из `main()` или вызов `exit()` данные в буферах будут сброшены. При жёстком `_exit()` — нет.

## 10. Полная схема: `fork()` + `exec()` + `wait()` + `exit()` (повтор с пояснениями)

```
Родительский процесс:
    ChildPID = fork()
    if (ChildPID != 0) {
        wait(&ChildRetCode)   // блокируется
        // ... код после wait
        exit(0)
    }

Дочерний процесс (ChildPID == 0):
    exec*("programme", ...)   // заменяет себя на programme
    // programme выполняется, затем завершается (явно или неявно)
```

**Что происходит по времени:**

1. **`fork()`** — создаётся потомок-копия.
2. **Потомок** вызывает `exec()` — его старая программа исчезает, загружается `programme`.
3. **Родитель** блокируется в `wait()`.
4. `programme` работает, затем завершается (вызывает `exit()` или `return`).
5. ОС посылает `SIGCHLD` родителю.
6. Родитель выходит из `wait()` с кодом завершения потомка.
7. Родитель продолжает выполнение, затем сам вызывает `exit()`.

## 12. Итоговое резюме по процессам UNIX

- `fork()` — создаёт копию (COW-оптимизация).
- `exec()` — заменяет программу в процессе.
- `wait()` — родитель ждёт потомка (обязательно, иначе зомби).
- `exit()` — штатное завершение (с обработчиками, сбросом буферов).
- Процесс-зомби — только запись в таблице, без ресурсов.
- `init` (PID=1) — усыновляет сирот и вызывает `wait()`.

## 2. Идеализированная физическая память (простейший случай)

**Свойства идеальной физической памяти:**
- **Адресуемая** – у каждой ячейки есть уникальный адрес.
- **Линейная** – адреса — целые неотрицательные числа (0, 1, 2, …).
- **Непрерывная** – все адреса от 0 до N-1 существуют (нет «дырок»).
- **Однородная** – все ячейки памяти имеют **одинаковые характеристики**:
  - Права доступа (логическая характеристика) – одинаковые.
  - **Время доступа** (физическая характеристика) – одинаковое для любого адреса.

> Это **идеализация**. На практике почти всегда что-то нарушается.

---

## 3. Реальная физическая память: нарушения однородности и непрерывности

### 3.1. Гарвардская архитектура (отдельно память команд и память данных)

| Архитектура | Память команд | Память данных | Особенности |
|-------------|---------------|---------------|--------------|
| **Принстонская (фон Неймана)** | Общая | Общая | Универсальность, но конфликты на стадии выборки и работы с памятью |
| **Гарвардская** | Раздельная | Раздельная | Можно одновременно выбирать команду и читать/писать данные → **выше производительность** |

**Как это реализовано в современных процессорах (x86, ARM):**
- На уровне **оперативной памяти** – общая (фон Нейман).
- На уровне **кэша L1** – гарвардская архитектура:
  - L1 I-cache (кэш инструкций)
  - L1 D-cache (кэш данных)

➡ **Нарушается однородность**:  
  - К одной области памяти можно только исполнять (читать команды),  
  - к другой – читать и писать данные.

---

### 3.2. ПЗУ (ROM) vs ОЗУ (RAM)

- Есть области памяти, в которые **нельзя писать** (например, образ BIOS при старте).
- В современных системах BIOS может копироваться в RAM, но физически на старых/встраиваемых системах такие области сохраняются.

➡ **Нарушается однородность** (разные права доступа к разным адресам).

---

### 3.3. Отображение устройств ввода-вывода на адресное пространство (MMIO)

**Механизм:**  
Диапазон *физических адресов* не ведёт к оперативной памяти, а ассоциирован с устройством (сетевая карта, видеокарта, контроллер диска).

- При **записи** по такому адресу – устройство воспринимает это как команду или данные.
- При **чтении** – получает состояние устройства или данные от него.

**Пример:**  
В диспетчере устройств Windows для сетевой карты указан «диапазон памяти». Любое обращение по этим адресам – это обмен с картой, а не с RAM.

➡ **Нарушается однородность** (разное поведение при чтении/записи).

> На архитектуре Intel также был отдельный механизм (порты ввода-вывода, отдельные команды `in`/`out`), но MMIO сейчас превалирует.

---

### 3.4. Видеопамять (Video RAM)

- Буфер в видеокарте, содержащий цвета всех пикселей (RGB).
- Отображается в общее физическое адресное пространство (начиная с некоторого адреса, зависящего от режима).
- **Прямой доступ к видеопамяти** позволяет менять изображение на экране без помощи драйвера.

**Исторически:**  
В DOS можно было писать в видеопамять напрямую. В современных ОС (Windows, Linux, macOS) это делает только драйвер видеокарты.

➡ Опять нарушение однородности (это не RAM, а память устройства).

---

### 3.5. Ограничения внешних устройств при прямом доступе к памяти (DMA)

**DMA (Direct Memory Access):**  
Устройство само (без участия процессора) читает/пишет оперативную память через специальный контроллер.

**Проблема:**  
У старых шин (ISA, ранние PCI) адресация была **32-битной** – устройство может работать только с первыми 4 ГБ физической памяти.

**Последствие для ОС:**  
Ядро должно **резервировать диапазоны в первых 4 ГБ** для DMA-буферов таких устройств. В Linux есть специальный пул «низкой памяти» (ZONE_DMA).

➡ Нарушение **однородности** и **непрерывности** с точки зрения ОС:  
  - Не любой адрес доступен для любого устройства.
  - Есть «дырки» в физическом адресном пространстве (занятые устройствами или зарезервированные).

---

## 4. Итог по физической памяти (для ОС)

| Идеальное свойство | Реальность |
|--------------------|------------|
| Линейная | В целом да (но есть DMA-ограничения) |
| Непрерывная | Нет – есть «дырки» под MMIO, ROM, видеопамять |
| Однородная | **Нет** – разное время доступа, разные права, разное поведение |

**Зачем это знать:**  
Операционная система при управлении памятью **обязана учитывать** эти неоднородности:
- Не выделять DMA-буфер за пределами 4 ГБ для старых устройств.
- Не отдавать приложению область MMIO (это вызовет сбой или вмешательство в работу устройства).
- Защищать критические области (например, код ядра).

## 1. Классификация микропроцессорных систем (по доступу к памяти)

Лектор выделяет **три типа архитектур**, на которые будет ссылаться в дальнейшем.

### 1.1. Система с общей памятью (Symmetric Multiprocessing, SMP)

**Как устроено:**
- Несколько CPU (или ядер) подключаются к **единой оперативной памяти** через коммутатор/контроллер.
- Все процессоры видят одну и ту же физическую память.

**Плюсы:**
- Простота программирования многопоточных программ.
- Легко передавать данные между потоками/процессами (общая память).
- **Миграция потоков** проста: поток можно приостановить на одном CPU и продолжить на другом — данные уже в общей памяти.

**Минусы:**
- **Производительность** упирается в пропускную способность памяти.
- Кэширование усложняется: нужны **протоколы когерентности кэшей** (чтобы изменения в кэше одного CPU стали видны другим).
- С ростом числа CPU растёт конкуренция за шину памяти.

> **Историческая справка (из лекции):**  
> 25 лет назад один CPU загружал память на 50–60%, два CPU — на 80%, три — 90%, четыре — почти 100%.  
> С появлением **Intel Core 2** один CPU стал забирать ~80% пропускной способности, а два — почти 100%. С тех пор общая память стала **критическим узким местом**.

---

### 1.2. Система с индивидуальной памятью (Distributed Memory, кластер)

**Как устроено:**
- Каждый CPU имеет **свою собственную оперативную память**.
- Адресные пространства независимы (у каждого CPU адреса от 0 до N).
- CPU **не могут** напрямую обращаться к памяти другого CPU.
- Для обмена данными — **пересылка сообщений** (через сеть или шину).
- Топология соединений часто **неполный граф** (например, линейная цепочка).

**Плюсы:**
- Высокая производительность: каждый CPU работает со своей «близкой» памятью без конкуренции.

**Минусы:**
- **Нельзя** запустить потоки одного процесса на разных CPU (нет доступа к коду и данным).
- **Миграция процесса** (с одного CPU на другой) требует копирования всего ВАП (очень дорого).
- Обмен данными между процессами — только через сообщения (медленно).

> **Важно:** Современные «компьютеры» с такой архитектурой — это, по сути, **кластеры** или **сети компьютеров**. Отдельный сервер с несколькими CPU так не устроен (кроме очень старых или экзотических систем).

---

### 1.3. Система с неоднородной памятью (NUMA — Non-Uniform Memory Access)

**Как устроено (комбинация первых двух):**
- Каждый CPU имеет свою «близкую» память (как в системе с индивидуальной памятью).
- Но через **дополнительный коммутатор** каждый CPU может обращаться к памяти любого другого CPU (как в системе с общей памятью).
- **Общее адресное пространство** на всех CPU (например, 0–12 ГБ).
- Разница: **время доступа** к своей памяти и к чужой — разное.

**Характеристики (типичные):**
- **Близкая память** (локальная): высокая пропускная способность (например, 100 ГБ/с), низкая латентность (~100 тактов).
- **Далёкая память** (удалённая): пропускная способность в 2 раза ниже (~50 ГБ/с), латентность может быть в 5 раз выше (~500 тактов).

**Плюсы NUMA:**
- Единое адресное пространство → можно запускать потоки одного процесса на разных CPU.
- Миграция потоков возможна (но если поток мигрирует на другой CPU, его данные могут остаться в «далёкой» памяти → проседание производительности).
- Разделяемая память между процессами.

**Минусы NUMA:**
- Производительность может **сильно проседать** при обращении к «далёкой» памяти.
- Нужны специальные механизмы ОС для **управления размещением памяти и потоков** (привязка потоков к CPU, размещение стека в локальной памяти и т.д.).

**Как ОС борется с проблемами NUMA (примеры из лекции):**

1. **Планировщик потоков учитывает «родной» CPU:**
   - У каждого потока есть «идеальный» CPU (где его данные локальны).
   - Сначала пытаемся запустить на идеальном CPU.
   - Если занят — на последнем CPU, где он выполнялся.
   - Если нет — на любом свободном (но это хуже).

2. **Размещение стека:**  
   Стек потока размещается в памяти того CPU, где поток работает (или «идеального» CPU).

3. **Размещение кучи и данных:**
   - Если один поток выделяет память, а потом другой поток к ней обращается — данные могут оказаться в «далёкой» памяти.
   - **Хитрость:** в ОС с copy-on-write или отложенным выделением (lazy allocation) память физически выделяется **при первом обращении**, а не при `malloc`.  
     ⇒ Если каждый поток сам сделает первое обращение к своему блоку данных, этот блок будет выделен в **его локальной памяти**.
   - Программист под NUMA должен **проектировать структуры данных** так, чтобы каждый поток работал в основном со своим диапазоном адресов.

> **Вывод лектора:** NUMA — это современный компромисс. Большинство серверных процессоров (Intel Xeon, AMD EPYC) используют NUMA. ОС умеет с этим работать, но для максимальной производительности программист тоже должен учитывать NUMA.

## 2. Модели адресации (переход к новой теме)

**Определение модели адресации:**
> Модель адресации определяет:
> - **какой вид** имеет виртуальное адресное пространство процесса,
> - **как виртуальные адреса отображаются** на физические адреса оперативной памяти.

Формула:  
`физический адрес = f(виртуальный адрес, контекст)`

Лектор рассматривает две базовые модели: **линейную** и **сегментную**.

---

### 2.1. Линейная модель адресации

**Варианты преобразования (простые):**

1. **Физический адрес = виртуальный адрес**  
   → Программа работает напрямую с физической памятью.  
   → Нет защиты, нет изоляции процессов.

2. **Физический адрес = виртуальный адрес + константа** (смещение)  
   → Например, ОС заняла первые 1 МБ, а программы начинаются с 1 МБ.  
   → Программа «думает», что начинается с адреса 0.  
   → Защиты всё ещё нет, но удобнее для программиста.

**Где используется:**  
Простые встраиваемые системы, загрузчики, ранний этап загрузки ОС.

> **Важно:** Линейная модель **не обеспечивает**:
> - защиту памяти между процессами,
> - изоляцию сбоев,
> - виртуальную память с подкачкой.

---

### 2.2. Сегментная модель адресации (введение)

**Определение сегмента (со слайда):**
> **Сегмент** — часть общего адресного пространства, содержащая внутри себя линейную последовательность адресов от 0 до некоторого максимума.

**Ключевые свойства сегментов (в физической памяти):**
- Сегменты могут **перекрываться** (в отличие от регионов ВАП, которые не перекрываются).
- Сегменты могут иметь **разные характеристики** (даже для одной и той же области физической памяти):
  - один сегмент — для кода (только выполнение),
  - другой сегмент, перекрывающийся с ним, — для данных (чтение/запись).

**Пример из лекции (реальное использование):**
- При загрузке программы ОС создаёт:
  - **сегмент кода** (только чтение, выполнение),
  - **сегмент данных** (чтение/запись),
  - **сегмент стека**,
  - возможно, сегмент кучи.
- Все эти сегменты могут находиться в разных местах физической памяти, но виртуальное адресное пространство процесса выглядит как непрерывное (благодаря сегментации).

**Зачем нужно перекрытие сегментов?**
- Например, чтобы иметь **разные права доступа** к одной и той же физической области (например, ядро ОС видит память и как код, и как данные, а пользовательская программа — только как данные).

**Недостаток сегментной модели (классической):**
- Фрагментация памяти.
- Сложность реализации «расширяющихся» сегментов (стек, куча).
- Современные ОС в основном используют **страничную модель** (о ней будет позже), но элементы сегментации остаются (например, сегментные регистры в x86).

## 1. Сегментная модель адресации (продолжение и углубление)

### 1.1. Реальный пример использования двух сегментов для одной области памяти

**Ситуация:** при загрузке программы ОС создаёт **сегмент кода** (только выполнение). Но ей нужно считать код из исполняемого файла в память. Для этого ОС создаёт **временный сегмент данных**, перекрывающий ту же физическую область, записывает туда код из файла, а затем удаляет этот временный сегмент.

**Итог:** одна и та же область физической памяти была описана двумя разными сегментами:
- сегмент данных (для записи ОС) — временный,
- сегмент кода (для выполнения программой) — постоянный.

> Это иллюстрирует, что сегменты могут **перекрываться** и иметь **разные типы/права доступа**.

---

### 1.2. Дескриптор сегмента (аппаратная структура)

Для описания сегмента ОС заполняет **дескриптор сегмента** — аппаратно-зависимую структуру в памяти. Типовые поля:

| Поле | Смысл |
|------|-------|
| **Адрес начала сегмента** | Физический адрес в памяти |
| **Размер сегмента** | Предел (limit), контролируется выход за границы |
| **Тип информации** | Код, данные, стек, другие (в x86 — 2 бита: 0,1,2,3) |
| **Уровень привилегий** | (подробнее позже) |
| **Права доступа** | Чтение (R), запись (W), выполнение (X) — отдельные биты |
| **Другие атрибуты** | Зависят от архитектуры |

> **Важно:** Все проверки (границы, тип, права, привилегии) выполняются **аппаратно** при каждой трансляции адреса.

---

### 1.3. Таблицы дескрипторов сегментов

Дескрипторы хранятся в **таблицах** — массивах в оперативной памяти. Адреса этих таблиц — в специальных регистрах CPU.

| Таблица | Регистр | Содержание |
|---------|---------|-------------|
| **GDT** (Global Descriptor Table) | `gdtr` | Сегменты, общие для всех процессов (ядро ОС, разделяемые ресурсы) |
| **LDT** (Local Descriptor Table) | `ldtr` | Своя для каждого процесса (код, данные, стек, куча процесса) |

**Переключение контекста:**
- При переключении **процессов** → загружается новый `ldtr` (указывает на LDT нового процесса).
- У потоков **одного процесса** — общая LDT (видят одно и то же виртуальное адресное пространство).

**Виртуальное адресное пространство процесса =**
```
все сегменты из GDT (глобальные) + все сегменты из LDT (локальные)
```
Но на практике процесс обычно имеет доступ только к своей LDT и к ограниченному набору записей GDT (например, к сегменту ядра для системных вызовов).

---

### 1.4. Формат виртуального адреса в сегментной модели

**Виртуальный адрес состоит из двух частей:**

| Селектор сегмента | Смещение (offset) |
|------------------|------------------|

- **Селектор** — указывает, какую таблицу (GDT или LDT) и какой номер записи (индекс) использовать.
- **Смещение** — адрес **внутри сегмента** (от 0 до размера сегмента минус 1).

**Алгоритм трансляции виртуального адреса → физического:**

1. По селектору определяем таблицу (GDT или LDT) и индекс.
2. Читаем дескриптор сегмента из таблицы (находится в памяти).
3. Из дескриптора берём **физический адрес начала сегмента**.
4. Прибавляем смещение → получаем физический адрес.
5. Выполняем аппаратные проверки:
   - не вышли ли за `limit` (размер сегмента)?
   - разрешена ли операция (чтение/запись/выполнение) для этого сегмента?
   - достаточен ли текущий уровень привилегий?

---

### 1.5. Пример: сколько обращений к памяти при выполнении `mov reg, [addr]`?

**Линейная модель адресации (без сегментации):**
1. Чтение самой инструкции из памяти.
2. Чтение данных по адресу `addr`.
→ **2 обращения к памяти**.

**Сегментная модель адресации (с холодным кэшем дескрипторов):**

| Шаг | Что делаем | Обращений |
|-----|-----------|-----------|
| 1 | Чтение дескриптора сегмента **кода** (из GDT/LDT) | 1 |
| 2 | Чтение инструкции по физическому адресу (начало сегмента кода + смещение IP) | 2 |
| 3 | Чтение дескриптора сегмента **данных** (из GDT/LDT) | 3 |
| 4 | Чтение самих данных (начало сегмента данных + смещение из `addr`) | 4 |

**Итого: 4 обращения к памяти** (в два раза больше, чем в линейной модели).

> **Проблема:** память и так медленная, а тут ещё дополнительная нагрузка.

**Решение:** внутри CPU есть **кэш дескрипторов сегментов** (обычно уровня L1). При первом использовании сегмента его дескриптор читается из памяти и кэшируется. При повторных обращениях к тому же сегменту — обращений к памяти за дескриптором **не происходит**.

**Однако:** при переключении контекста (вытеснении потока) этот кэш может быть перезаписан, и при возврате потока дескрипторы придётся загружать заново → дополнительные накладные расходы.

---

### 1.6. Преимущества сегментной модели (с точки зрения ОС)

| Возможность | Как реализовано |
|-------------|----------------|
| **Разделение виртуальных адресных пространств процессов** | У каждого процесса своя LDT. Доступ к чужим сегментам невозможен. |
| **Защита ОС от пользовательских программ** | Сегменты ядра имеют высокий уровень привилегий, пользовательский код не может их использовать. |
| **Перемещение сегментов в физической памяти** | Меняем адрес начала в дескрипторе → программа ничего не замечает (прозрачно). |
| **Контроль границ** | Аппаратная проверка `offset < limit`. |
| **Разные права доступа** | Биты R/W/X на сегмент. |

**Недостатки:**
- Внешняя фрагментация памяти.
- Сложность с динамически растущими сегментами (стек, куча).
- Современные ОС (Linux, Windows) используют **страничную модель**, но элементы сегментации остаются (например, в x86: FS/GS для TLS).

> **Итог по сегментной модели:**  
> Это важный исторический и концептуальный шаг. Она даёт изоляцию и защиту, но страдает производительностью из-за множества обращений к памяти и фрагментации.

## 2. Страничная модель адресации (начало)

### 2.1. Базовые понятия

**Для каждого процесса** создаётся своё **виртуальное адресное пространство (ВАП)**:
- Линейное, непрерывное.
- Адреса от **0** до некоторого максимума (например, для 32 бит — 0 … 4 ГБ).

**ВАП разбивается на блоки одинакового размера — страницы (page).**  
**Физическая память (RAM)** разбивается на блоки **того же размера** — кадры (frame) или физические страницы.

Типичный размер страницы: **4 КБ** (реже 2 КБ, 8 КБ, 16 КБ).  
Размер страницы — степень двойки.

**Ключевое:** размер ВАП обычно **гораздо больше** размера физической памяти.

---

### 2.2. Таблица страниц (page table)

**Таблица страниц** — структура данных (в памяти), которая для **каждой виртуальной страницы** процесса содержит **дескриптор страницы**.

Дескриптор страницы включает (как минимум):
- **Физический адрес** кадра (если страница в RAM).
- Биты признаков: присутствует/отсутствует, доступ на чтение/запись/выполнение, «грязная», «использовалась» и т.д.

**Трансляция:**  
Виртуальный адрес разбивается на две части:
- **Номер виртуальной страницы** (индекс в таблице страниц).
- **Смещение внутри страницы** (младшие биты).

Процессор:
1. Берёт номер виртуальной страницы.
2. Ищет запись в таблице страниц процесса.
3. Читает физический адрес кадра.
4. Прибавляет смещение → физический адрес.

---

### 2.3. Куда может быть отображена виртуальная страница?

| Состояние | Что значит | Где находится содержимое |
|-----------|-----------|--------------------------|
| **Отображена на RAM** | Страница в оперативной памяти | В каком-то кадре RAM |
| **Отображена на диск (swap)** | Страница вытеснена из RAM | В области подкачки (swap-файл/раздел) |
| **Не отображена никуда** | Страница не используется (белая область ВАП) | Нигде (обращение → fault) |

> **Важно:** В начальный момент после `exec()` **ни одна страница ВАП не находится в RAM**. Весь образ процесса — на диске (исполняемый файл для кода, swap для данных/стека/кучи). Страницы подгружаются **по требованию** (demand paging) при первом обращении.

---

### 2.4. Связь с регионами ВАП (вспоминаем начало лекции)

У процесса есть регионы:
- код (отображается на исполняемый файл),
- данные (частично на файл, частично на swap),
- стек (на swap),
- куча (на swap),
- «белые зоны» (не отображены никуда).

**Страничная организация:**  
- Для региона кода: страницы изначально не в RAM, но есть образ в исполняемом файле. При первом чтении — подгрузка с диска.
- Для стека/кучи: образ в swap.
- Для «белых зон»: таблица страниц помечает страницы как «не присутствуют», при обращении — исключение (segmentation fault).

**Соотношение размеров:**  
«Серые» (используемые) регионы обычно занимают **ничтожную долю** от всего ВАП (сотые доли процента). Остальное — «белое» пространство, которое никуда не отображено.

## 3. Итог по теме «Модели адресации» (на данный момент)

| Модель | Принцип | Защита | Производительность | Современное использование |
|--------|---------|--------|--------------------|----------------------------|
| **Линейная** | Физический адрес = виртуальный (+ сдвиг) | Нет | Высокая (прямой доступ) | Только в спецслучаях (загрузчики, bare-metal) |
| **Сегментная** | (селектор, смещение) → дескриптор → адрес | Есть (границы, права, привилегии) | Ниже (доп. обращения к памяти, но есть кэш) | Частично в x86 (FS/GS, GDT/LDT) |
| **Страничная** | (номер страницы, смещение) → таблица страниц → адрес | Есть (биты защиты в PTE) | Хорошая (аппаратный TLB) | **Основная** в современных ОС |


## 1. Страничная модель адресации (продолжение)

### 1.1. Где хранится таблица страниц?

**Таблица страниц (page table)** — это структура данных, которую **операционная система** формирует для каждого процесса.

- Таблица страниц хранится **в оперативной памяти**.
- У каждого процесса — **своя таблица страниц** (изоляция адресных пространств).
- В процессоре x86 есть **специальный регистр** — **CR3** (управляющий регистр 3).
  - В CR3 хранится **физический адрес** начала таблицы страниц текущего процесса.
  - При переключении процессов ОС загружает в CR3 адрес таблицы страниц нового процесса.

> **Важно:** CR3 указывает на **физический адрес**, потому что таблица страниц нужна процессору **до** включения виртуальной памяти (иначе «курица и яйцо»).

---

### 1.2. Формат виртуального адреса и трансляция

**Размер страницы** — степень двойки (обычно 4 КБ = 2¹²).  
Значит:
- **Младшие 12 бит** виртуального адреса — **смещение внутри страницы** (offset).
- **Старшие биты** — **номер виртуальной страницы** (индекс в таблице страниц).

**Пример для 32-битной системы (4 ГБ ВАП, страница 4 КБ):**
- Всего страниц: 2³² / 2¹² = 2²⁰ = 1 048 576.
- Номер страницы — 20 бит, смещение — 12 бит.

**Алгоритм трансляции:**
1. Взять номер виртуальной страницы из адреса.
2. Вычислить адрес записи в таблице страниц:
   `адрес_записи = (содержимое CR3) + (номер_страницы × размер_дескриптора)`
3. Прочитать дескриптор страницы из памяти (первое обращение к памяти).
4. Из дескриптора взять физический адрес начала кадра (страницы в RAM).
5. Прибавить смещение → **физический адрес**.
6. Прочитать/записать данные по этому адресу (второе обращение к памяти).

---

### 1.3. Сколько обращений к памяти для одной инструкции?

Рассмотрим простую инструкцию: `mov reg, [addr]` (прочитать данные по адресу `addr`).

**Пошагово (при холодном TLB, без кэша):**

| Шаг | Что делаем | Обращений к памяти |
|-----|-----------|---------------------|
| 1 | По адресу инструкции (IP) → номер страницы кода | – |
| 2 | Чтение дескриптора страницы **кода** из таблицы страниц | 1 |
| 3 | Чтение самой инструкции по полученному физическому адресу | 2 |
| 4 | Из адреса `addr` → номер страницы **данных** | – |
| 5 | Чтение дескриптора страницы **данных** из таблицы страниц | 3 |
| 6 | Чтение данных по физическому адресу (начало кадра + смещение) | 4 |

**Итого: 4 обращения к памяти** (как и в сегментной модели без кэша).

> **Но** это только начало. В реальности:
> - TLB (Translation Lookaside Buffer) — аппаратный кэш трансляций — запоминает отображения для недавно использованных страниц. При повторном обращении к той же странице TLB даёт ответ без похода в память.
> - Многоуровневые таблицы страниц (2–4 уровня) — каждое обращение к дескриптору может потребовать нескольких чтений из памяти (по одному на каждый уровень). Но TLB и это кэширует.

**Вывод:** страничная модель тоже добавляет накладные расходы, но они приемлемы благодаря TLB.

---

### 1.4. Дескриптор страницы (Page Table Entry, PTE)

Аппаратно-зависимая структура. Типовые поля:

| Поле | Смысл |
|------|-------|
| **Адрес кадра** | Физический адрес страницы в RAM (младшие биты обнулены, т.к. страница выровнена) |
| **Признак присутствия (Present)** | 1 = страница в RAM, 0 = не в RAM (на диске или не отображена) |
| **Признак изменения (Dirty)** | 1 = в страницу была запись (устанавливается CPU, сбрасывается ОС) |
| **Признак обращения (Accessed)** | 1 = страница читалась или писалась (устанавливается CPU, сбрасывается ОС) |
| **Права доступа** | Чтение (R), запись (W), выполнение (X) — отдельные биты |
| **Другие атрибуты** | Зависят от архитектуры (например, бит глобальности, кэшируемость) |

**Важные замечания лектора:**

- **Признак присутствия** — единственный, который процессор использует **автоматически**. Если `Present = 0` → page fault.
- **Dirty и Accessed** используются ОС для алгоритмов замещения страниц (например, приближённый алгоритм «вторая возможность» или NRU).
- **Тип информации (код/данные/стек)** и **уровень привилегий** на уровне страницы — возможны, но на практике редко используются (обычно на уровне сегментов или в отдельных битах).
- **Хранение номера блока на диске** в PTE — теоретически можно, но практически никто так не делает (ОС хранит эту информацию отдельно, в своих структурах).

---

### 1.5. Отображение виртуальных страниц: три варианта (с точки зрения ОС)

| Состояние | Present (CPU) | Для ОС | Действие при обращении |
|-----------|---------------|--------|------------------------|
| **В RAM** | 1 | – | Успешный доступ |
| **На диске (swap/файл)** | 0 | Страница принадлежит региону, есть образ на диске | Page fault → подкачка с диска → перезапуск инструкции |
| **Не отображена (дырка в ВАП)** | 0 | Адрес вне объявленных регионов | Page fault → ОС убивает процесс (segmentation fault) |

> **Важно:** Процессор **не различает** два варианта `Present=0`. Это задача ОС — по контексту (адресу и структуре ВАП) решить, подкачивать страницу или убивать процесс.

## 2. Страничный сбой (Page Fault)

**Определение:**  
Ситуация, когда при трансляции адреса `Present = 0`. Процессор **не может** продолжить выполнение инструкции.

**Механизм:**
1. Процессор генерирует **аппаратное прерывание** (исключение).
2. Управление передаётся обработчику page fault в ОС.
3. ОС анализирует причину.

**Два варианта обработки (как в лекции):**

### Вариант А. Корректная страница, но не в RAM (подкачка)

- Адрес принадлежит одному из объявленных регионов (код, данные, стек, куча).
- Страница есть на диске (в исполняемом файле или swap).
- ОС:
  1. Находит свободный кадр в RAM (если нет — вытесняет какую-то страницу на диск).
  2. Считывает содержимое страницы с диска в этот кадр.
  3. Обновляет PTE: ставит `Present = 1`, записывает физический адрес кадра.
  4. **Перезапускает** ту же инструкцию, которая вызвала page fault.
- Процесс продолжает работу, не зная, что страницы не было в памяти.

### Вариант Б. Неверный адрес (нарушение защиты)

- Адрес **не принадлежит** ни одному из объявленных регионов (дырка в ВАП).
- ОС определяет, что это ошибка программы.
- Типичные действия:
  - В UNIX: посылается сигнал `SIGSEGV` (Segmentation Fault) → процесс убивается.
  - В Windows: возникает исключение `EXCEPTION_ACCESS_VIOLATION`. Для потока GUI может появиться окно «Программа выполнила недопустимую операцию», но иногда поток можно продолжить (если он не повредил критические данные).

> **Опасность продолжения после ошибки (из личного опыта лектора):**  
> Если поток интерфейса случайно получил доступ по неверному адресу, но ОС разрешила продолжить, этот поток может **испортить внутренние структуры данных**. Ошибки начнут возникать чаще, возможно повреждение файлов и БД. Поэтому в серьёзных приложениях лучше сразу завершать процесс.

## 3. Итог по страничной модели (на данный момент)

| Характеристика | Описание |
|----------------|----------|
| **ВАП** | Линейное, непрерывное, от 0 до максимума (например, 4 ГБ) |
| **Разбиение** | Страницы фиксированного размера (степень двойки, обычно 4 КБ) |
| **Таблица страниц** | Хранится в RAM, у каждого процесса своя. Адрес — в регистре CR3 (x86) |
| **Трансляция** | (номер страницы, смещение) → PTE → адрес кадра + смещение |
| **Аппаратная поддержка** | TLB (кэш трансляций) для ускорения |
| **Защита** | Биты R/W/X в PTE, изоляция через разные таблицы страниц |
| **Подкачка (swap)** | Страницы могут быть вытеснены на диск, при обращении — page fault и подкачка |
| **Ошибки** | Page fault: либо подкачка, либо убить процесс (SIGSEGV) |

**Преимущества страничной модели:**
- Нет внешней фрагментации (все кадры одинакового размера).
- Удобно реализовать виртуальную память, больший объём, чем физическая RAM.
- Защита и изоляция процессов.
- Эффективная поддержка разделяемой памяти (несколько процессов могут отображать одни и те же физические страницы с разными правами).

**Недостатки:**
- Таблицы страниц могут быть большими (для 32 бит — 4 МБ на процесс, для 64 бит — проблема многоуровневых таблиц).
- Накладные расходы на трансляцию (но TLB смягчает).
- Page fault — дорогая операция (обращение к диску).
