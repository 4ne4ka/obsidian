
# Раздел "Введение"


## 13. Недетализированные примеры архитектур – UNIX
*(Лекция 4, раздел "Архитектура Unix")*

### Схема архитектуры (упрощенная)

1. **Уровень пользователя (User Level)**
    
    - Прикладные программы
        
    - Стандартные библиотеки (реализуют интерфейс системных вызовов)
        
2. **Граница: Интерфейс системных вызовов (System Call Interface)**
    
3. **Уровень ядра (Kernel Level)**
    
    - **Подсистема управления процессами (Process Management Subsystem)**:
        
        - Планировщик (Scheduler)
            
        - Управление памятью (Memory Management)
            
        - Межпроцессное взаимодействие (IPC)
            
    - **Подсистема управления файлами (File Management Subsystem)**:
        
        - Виртуальная файловая система (VFS)
            
        - Кэш (Buffer Cache)
            
        - Драйверы устройств (Device Drivers)
            
4. **Уровень аппаратной абстракции (HAL)**
    
5. **Аппаратное обеспечение (Hardware)**
    

### Описание компонентов (из лекций)

**Общее:**

- Поддерживается только **одна операционная среда** (в отличие от Windows, где их несколько).
    
- Прикладные программы обычно вызывают функции ОС не напрямую, а через **стандартную библиотеку** языка C (которая выполняет системные вызовы).
    

**Подсистема управления процессами:**

- **Планировщик (Scheduler):** Отвечает за распределение времени центрального процессора между потоками/процессами.
    
- **Управление памятью (Memory Management):** Управление физической памятью, поддержка адресного пространства ядра и виртуальных адресных пространств процессов.
    
- **Межпроцессное взаимодействие (IPC):** Обеспечивает механизмы обмена данными и синхронизации между процессами (сигналы, каналы, сокеты и т.д.).
    

**Подсистема управления файлами:**

- В отличие от Windows, в UNIX используется **единая иерархическая файловая система**. Несколько физических файловых систем (разделов) «монтируются» в ветки единого дерева каталогов.
    
- **Виртуальная файловая система (VFS):** Абстрактный слой, который обеспечивает единый интерфейс для работы с разными типами файловых систем (EXT4, XFS и др.) и разными устройствами.
    
- **Классификация устройств:**
    
    - **Символьные устройства (Character devices):** Поддерживают **потоковую** передачу данных (байт за байтом). Примеры: клавиатура, мышь, последовательный порт.
        
    - **Блочные устройства (Block devices):** Представлены в виде **массива блоков** одинакового размера. Поддерживают произвольный доступ. Примеры: жесткие диски, SSD, флешки (через драйвер). Работа с блочными устройствами часто кэшируется.
        
    - _Сетевые устройства (сетевые карты) не классифицируются ни как символьные, ни как блочные; для них существует отдельная сетевая подсистема._
        
- **Драйверы устройств (Device Drivers):** Программы, непосредственно управляющие аппаратурой.
    
- **Кэш (Buffer Cache):** Временное хранение данных с медленных устройств (дисков) в более быстрой оперативной памяти для ускорения доступа.

## 14. Недетализированные примеры архитектур – Windows NT
_Важное замечание из лекций: Microsoft использует терминологию из ОС реального времени («микроядро», «приоритеты реального времени»), но **Windows не является ОСРВ**. Архитектура Windows — это **монолитное ядро** с некоторыми элементами микроядерного дизайна (защищенные подсистемы на пользовательском уровне)._

### Схема архитектуры (упрощенная)

1. **Уровень пользователя (User Mode)**
    
    - Приложения Win32 / Win64 / WSL
        
    - **Защищенные подсистемы (Protected Subsystems)**:
        
        - Win32 (поддержка 32-битных приложений поверх Win64 — механизм WOW)
            
        - WSL (Windows Subsystem for Linux — выполнение Linux-приложений)
            
2. **Граница: Интерфейс системных вызовов**
    
3. **Уровень ядра (Kernel Mode) — «Исполнительная часть Windows» (Windows Executive)**
    
    - **Ядро (Kernel)** — планирование, обработка прерываний, синхронизация процессоров.
        
    - **Менеджер объектов (Object Manager)** — управляет всеми ресурсами как объектами.
        
    - **Монитор безопасности (Security Monitor)** — управление правами доступа.
        
    - **Менеджер процессов (Process Manager)** — создание/завершение процессов и потоков.
        
    - **Менеджер виртуальной памяти (Virtual Memory Manager)** — управление физической памятью и ВАП.
        
    - **Менеджер ввода-вывода (I/O Manager)** — обеспечивает окружение для драйверов устройств и файловых систем.
        
    - **LPC (Local Procedure Call)** — оптимизированный механизм удаленного вызова процедур для сообщений в GUI.
        
4. **Уровень аппаратной абстракции (HAL — Hardware Abstraction Layer)**
    
5. **Аппаратное обеспечение (Hardware)**
    

### Описание компонентов (из лекций)

**Уровень пользователя:**

- Основная среда — **Win64** (нативная). Подсистема **Win32** работает поверх Win64 (механизм WOW64).
    
- **WSL** (Windows Subsystem for Linux) позволяет выполнять Linux-приложения. Современные версии используют специализированное ядро Linux, работающее как виртуальная машина на уровне ядра Windows.
    

**Уровень ядра (Исполнительная часть):**

- **Ядро (Kernel):** Первичная обработка прерываний и исключений, **планирование** (распределение времени ЦП), синхронизация процессоров для многопроцессорных систем, аварийное завершение (Blue Screen of Death — BSoD).
    
- **Менеджер объектов (Object Manager):** Все ресурсы (процессы, потоки, файлы, события, семафоры) представлены как **объекты**. У каждого объекта есть тип, атрибуты безопасности, счетчик ссылок. Доступ к объектам из приложения осуществляется через **хендлы** (handle) — номера записей в таблице хендлов процесса. _Аналогия с умными указателями в C++_.
    
- **Монитор безопасности (Security Monitor):** Реализует избирательное управление доступом (DACL) с использованием списков управления доступом (ACL). Также управляет **привилегиями** (например, право изменения системного времени).
    
- **Менеджер процессов (Process Manager):** Создание, завершение, приостановка и возобновление процессов и потоков.
    
- **Менеджер виртуальной памяти (VMM):** Управление физической памятью, создание и поддержка виртуальных адресных пространств процессов (включая разделяемую память и copy-on-write).
    
- **Менеджер ввода-вывода (I/O Manager):** Предоставляет унифицированную среду для драйверов файловых систем, драйверов устройств и сетевой подсистемы. Управляет буферизацией и кэшированием (Cache Manager). Использует **редиректор** для определения — локальный или удаленный (сетевой) ресурс запрашивается.
    
- **LPC (Local Procedure Call):** Оптимизированная версия RPC для внутренних нужд Windows, используемая, в частности, для передачи сообщений между потоками графического интерфейса.
    

**HAL (Hardware Abstraction Layer):**

- Изолирует платформо-зависимый код (различия между архитектурами x86, x86-64, ARM, различные чипсеты).
    
- Предоставляет вышележащим компонентам унифицированные абстракции аппаратного обеспечения (например, единый интерфейс для работы с таймерами, контроллером прерываний, отображением памяти на устройства ввода-вывода).
    
- Благодаря HAL, большая часть ядра Windows не зависит от конкретной аппаратной платформы.

# Раздел "Процессы и потоки"

## 15. Поток (определение)

**Определение:** Поток (нить, thread) — это абстракция, представляющая **последовательное выполнение команд программы, развертывающееся во времени**. Это **активный объект**
**У каждого потока есть своё:** программный счетчик (IP), стек, значения регистров общего назначения (GPR).

**Потоки делят (внутри одного процесса):** глобальные/статические переменные, кучу (heap), код программы, ресурсы ОС (файлы, сокеты).

## 16. Процесс (определение)

**Определение:** Процесс — это абстракция, представляющая программу во время её выполнения. Это **пассивный объект** — «контейнер» или «владелец ресурсов».

**Процесс владеет:** адресным пространством (код, данные, куча, стеки), открытыми файлами, сетевыми соединениями, устройствами ввода-вывода.

Процесс жив, пока в нем есть хотя бы один активный поток.

## 17. Многопоточные процессы

**Структура:** Одно виртуальное адресное пространство процесса, внутри которого находятся **множество стеков** (по одному на поток). Куча, данные и код — общие для всех потоков.

**Ресурсы, разделяемые всеми потоками процесса:**

- **Регион кода** (исполняемый код всех функций) — все потоки могут выполнять любую функцию из этого региона.
    
- **Регион данных** (глобальные и статические переменные, константы) — изменения, сделанные одним потоком, сразу видны всем остальным.
    
- **Регион кучи** (динамически выделяемая память) — память, выделенная одним потоком, может быть использована любым другим потоком.
    
- **Системные ресурсы процесса** (открытые файлы, сетевые соединения, окна) — например, если один поток сдвинул позицию в файле, это изменение видно всем потокам.
    

**Ресурсы, индивидуальные для каждого потока:**

- **Собственный стек** — используется для вызовов функций, хранения локальных переменных, адресов возврата. _Два потока не могут использовать один стек (это привело бы к хаосу)._
    
- **Собственный аппаратный контекст** — состояние центрального процессора (значения регистров, указатель стека, счетчик команд).
    

**Типовые шаблоны использования многопоточности (из ОС6):**

1. **Параллельная обработка однотипных запросов** (сетевые серверы) — главный поток принимает запросы и создает дочерние потоки для их обслуживания.
    
2. **Разделение исполнительных активностей** — отдельный поток для пользовательского интерфейса и отдельные потоки для долгих вычислений (чтобы интерфейс не «зависал»).
    
3. **Эффективное использование многоядерных процессоров** — один поток на одно ядро.
4.  Улучшение структуры программы.

**Пример:** Поток 1 выполняет `main`, поток 2 — функцию `f1`, поток 3 — функцию `f2`. Они логически выполняются одновременно (concurrent), а физически — либо параллельно на разных ядрах (parallel), либо по очереди на одном ядре.


## 18. Контекст процесса. Контекст потока. Переключение контекста потока.
*(Лекция 5, части 3 и 4; Лекция 6, раздел 1)*

**Контекст** — это множество информации, полностью описывающее состояние объекта (достаточное для его восстановления в случае удаления).

**Контекст процесса** включает:

- Дескриптор процесса и дополнительную служебную информацию.
    
- Виртуальное адресное пространство процесса (аппаратно-зависимые структуры данных).
    
- Внутреннюю структуру ВАП (регионы кода, данных, стека, кучи).
    
- Множество ресурсов, используемых процессом, и их состояния (контекст файлового ввода-вывода, сетевых соединений и т.д.).
    
- Контекст безопасности (права и привилегии).
    

**Контекст потока** включает:

- Дескриптор потока.
    
- **Аппаратный контекст выполнения потока** — состояние центрального процессора с точки зрения данного потока (значения регистров, программный счетчик, указатель стека).
    
- _Важно:_ В современных ОС прикладные программы не имеют прямого доступа к устройствам, поэтому аппаратный контекст ограничивается только состоянием процессора.

**Переключение контекста потока** (происходит при передаче ЦП от одного потока другому):

1. **Сохранить** аппаратный контекст вытесняемого потока (состояние регистров) в его дескриптор (в оперативную память).
    
2. Если потоки принадлежат **разным процессам** — переустановить указатели на структуры данных нового процесса (таблица страниц, таблица дескрипторов и т.д.).
    
3. **Восстановить** аппаратный контекст нового потока из его дескриптора в регистры ЦП.
    

_Важно:_ Контекст процесса — это просто набор указателей на структуры данных в памяти. При переключении между процессами сами структуры данных не копируются, а только меняются активные указатели.

## 19. Диаграмма состояний потока в многозадачной среде
*(Лекция 5, части 5 и 8)*

**Состояния:**
1.  **Создание** – выделение ресурсов, создание дескриптора.
2.  **Готов (Ready)** – есть все ресурсы, кроме CPU (ждет в очереди).
3.  **Выполнение (Running)** – активно выполняется на CPU.
4.  **Ожидание (Waiting/Blocked)** – заблокирован до наступления события (ввод-вывод, ресурс).
5.  **Завершение** – освобождение ресурсов, дескриптор остается с кодом возврата.

**Переходы:**

- `Создание -> Готов` — поток создан, но ЦП пока не выделен.
- `Готов -> Выполнение` — планировщик выделил потоку ЦП.
- `Выполнение -> Готов` — ЦП отобран у потока (истек квант времени, либо поток добровольно уступил процессор).
- `Выполнение -> Ожидание` — поток запросил ресурс/событие, которое недоступно немедленно (блокирующий системный вызов).
- `Ожидание -> Готов` — наступило ожидаемое событие (завершился ввод-вывод, освободился ресурс).
- `Выполнение -> Завершение` — поток завершился сам (вызов exit_thread).
- `Готов -> Завершение` — другой поток запросил завершение этого потока (terminate_thread).
- `Ожидание -> Завершение` — _редкий случай_; обычно сначала переход в `Готов`, затем завершение.

**Дополнительные состояния в Windows** (не обязательны для базового ответа, но упоминаются в лекциях):

- **Standby** — поток предварительно выбран для запуска на процессоре.
    
- **Deferred** — связаны с миграцией между процессорами в многопроцессорных системах.
## 20. Диаграмма состояний потока в однозадачной среде
*(Лекция 5, часть 7А)*

**Состояния:** Создание → Выполнение ⇄ Ожидание → Завершение.

**Особенность:** Нет состояния «Готов». Если поток уходит в ожидание, процессор простаивает (или занят ядром), так как нет другого потока, которому можно передать управление.

**Переходы:**
- Создание → Выполнение (сразу после создания).
- Выполнение → Ожидание (блокирующий системный вызов).
- Ожидание → Выполнение (ОС завершила обслуживание).
- Выполнение → Завершение (добровольно).

## 21. Дескриптор процесса
*(Лекция 6, раздел 2)*

**Определение:** Структура данных в ядре, содержащая всю информацию, необходимую ОС для управления процессом. В Linux — `task_struct` (~140-150 полей).

**Основные поля:**
- **PID** – уникальный идентификатор процесса.
- **PPID** – идентификатор родительского процесса.
- **Групповые параметры** – GID (группа), SID (сессия), Job ID.
- **Параметры приоритета** – влияют на планирование потоков.
- **Состояние процесса** – создание, исполнение, остановлен, завершение.
- **Статистические данные** – время CPU, рабочий набор (working set), число потоков.
- **Описание адресного пространства** – таблицы страниц + структура регионов.
- **Контекст ввода-вывода** – открытые файлы, текущая позиция.
- **Контекст безопасности** – права и привилегии + права доступа к самому процессу.
- **Код завершения процесса**.

## 22. Дескриптор потока
*(Лекция 6, раздел 6)*

**Определение:** Структура данных в ядре для управления потоком. В Unix/Linux часто нет отдельной структуры — в `task_struct` часть полей относится к процессу, часть к потоку. Для многопоточности создаётся несколько `task_struct` с одинаковыми «процессными» полями.

**Основные поля:**
1. **Идентификатор потока (TID)** — уникальный целочисленный номер.
2. **Идентификатор процесса-владельца** — ссылка на процесс, которому принадлежит поток.
3. **Параметры приоритета** — используются при конкуренции за время ЦП и объекты синхронизации.
4. **Статистические данные** — потребленное время ЦП.
5. **Аппаратный контекст выполнения** — указатель на структуру, хранящую состояние регистров ЦП (сегмент состояния задачи — TSS).
6. **Код завершения** — пока поток работает, хранится специальное значение (STILL_ACTIVE); после завершения — код возврата.


## 23. Создание процесса, завершение процесса (Windows-style)
*(Лекция 6, разделы 10, 17)*

**Создание процесса (7 шагов):**
1.  Создать дескриптор процесса (`_EPROCESS`), поместить в таблицу процессов.
2.  Инициализировать поля (PID, приоритет, унаследованные от родителя).
3.  Создать виртуальное адресное пространство (ВАП) – аппаратно-зависимые структуры (таблицы страниц).
4.  Заполнить ВАП данными: прочитать заголовок PE-файла (адреса регионов кода, данных), разместить код и данные. Стека пока нет.
5.  Выделить ресурсы по умолчанию (stdin, stdout, stderr).
6.  Оповестить подсистемы ядра о новом процессе.
7.  Создать первичный поток.

**Завершение процесса (7 шагов):**
1.  Завершить выполнение всех потоков процесса (принудительно или дождаться).
2.  Сохранить статистику и код возврата в дескрипторе.
3.  Перевести ресурсы процесса в непротиворечивое состояние (с точки зрения ОС).
4.  Освободить все ресурсы (память, файлы, сокеты, мьютексы).
5.  Освободить ВАП и уничтожить его структуры.
6.  Оповестить подсистемы ядра.
7.  Установить состояние процесса = «завершён». Дескриптор остаётся (зомби), пока все пользователи не закроют хендлы.

## 24. Создание потока, завершение потока (Windows-style)
*(Лекция 6, разделы 11, 14, 16)*

**Создание потока (6 шагов):**
1.  Создать дескриптор потока (TID, приоритет по умолчанию).
2.  Инициализировать поля дескриптора.
3.  Создать и выделить структуры данных для работы на данной архитектуре: **регион стека** + структура для аппаратного контекста.
4.  Инициализировать аппаратный контекст: SP = вершина стека, IP = точка входа (для первичного потока – из заголовка PE-файла, для дополнительного – адрес функции из `CreateThread`).
5.  Оповестить подсистемы ядра.
6.  Перевести поток в состояние «Готов».

**Завершение потока (6 шагов):**
1.  Сохранить статистику и код возврата в дескрипторе.
2.  Перевести ресурсы потока в непротиворечивое состояние.
3.  Освободить ресурсы, принадлежавшие потоку.
4.  Оповестить подсистемы ядра.
5.  Установить состояние потока = «завершён».
6.  Если поток был последним активным – завершить процесс. Дескриптор потока остаётся (пока все хендлы не закрыты).

## 25. Системный вызов fork(). Системный вызов exec(). Системный вызов exit(). Системный вызов wait()
*(Лекция 7, разделы 2-9)*

### `fork()` — создание копии процесса

_«Создание копии процесса»_. После вызова `fork()` создается дочерний процесс, который является точной копией родительского на момент вызова.

**Отличия:**
- В родительском процессе `fork()` возвращает **PID дочернего процесса**.
- В дочернем процессе `fork()` возвращает **0**.

**Что происходит при `fork()` (ОС7):**

1. Резервируется место в пространстве подкачки для регионов данных, стека и кучи нового процесса (регион кода отображается на тот же исполняемый файл).
    
2. Создается новый дескриптор процесса, назначается новый уникальный PID.
    
3. Создается новое виртуальное адресное пространство.
    
4. Создается аппаратный контекст для единственного потока нового процесса (копируется с родительского, отличается только возвращаемое значение `fork()`).
    
5. Все ресурсы родителя (открытые файлы, сетевые соединения) копируются или разделяются с потомком.
    
6. Дочерний процесс переводится в состояние `Готов`.

**Оптимизация:** Современные Unix используют механизм **Copy-on-Write** (копирование при записи) — страницы памяти реально копируются только тогда, когда один из процессов пытается в них что-то записать.

### `exec()` — замена исполняемого кода

_«Использовать ресурсы текущего процесса для выполнения указанной программы»_.

1. Найти исполняемый файл на файловой системе и прочитать его заголовок.
    
2. Разрушить текущее виртуальное адресное пространство (освободить регионы кода, данных, стека, кучи и пространство подкачки).
    
3. Создать **новое** виртуальное адресное пространство для новой программы (на основе заголовка исполняемого файла):
    - Регион кода — отображается на исполняемый файл.
    - Регионы данных, стека, кучи — резервируются в пространстве подкачки.
        
4. Сохранить аргументы командной строки и переменные окружения в новом адресном пространстве.
    
5. Инициализировать аппаратный контекст (установить IP на точку входа новой программы, SP на вершину нового стека).
    
6. Перевести процесс в состояние `Готов`.
    

_Важно:_ После успешного `exec()` **старая программа больше не существует**. Управление никогда не возвращается в код, вызвавший `exec()`.

### `exit()` — завершение процесса (штатное)

1. Отключить все сигналы.
    
2. Закрыть все открытые файлы (сбросить буферы данных, если это штатное завершение).
    
3. Сохранить код завершения и статистику в дескрипторе процесса.
    
4. Перевести процесс в состояние **«Зомби»** (Zombie) — у процесса отобраны все ресурсы, кроме дескриптора.
    
5. Для всех непосредственных потомков процесса родителем назначается процесс `init` (PID=1).
    
6. Послать родительскому процессу сигнал `SIGCHLD` (уведомление о завершении потомка).
    
7. Уничтожить виртуальное адресное пространство.
    
8. Дескриптор процесса будет удален только после того, как родитель вызовет `wait()`.
    

### `wait()` — ожидание завершения дочернего процесса

_«Обязанность родительского процесса по отношению к потомкам»_.

- Родительский процесс вызывает `wait()` (или `waitpid()`) и **блокируется**, пока какой-либо (или конкретный) дочерний процесс не завершится.
    
- Когда дочерний процесс завершается, `wait()` возвращает **код завершения** потомка.
    
- После вызова `wait()` дескриптор завершившегося процесса (зомби) окончательно удаляется из системы.
    
- Если родительский процесс завершается раньше своих потомков, все его потомки «усыновляются» процессом `init` (PID=1), который регулярно вызывает `wait()` для них.


**В UNIX создание процесса и запуск программы — два разных вызова.**

**`fork()`:** Создаёт копию текущего процесса (точную копию). Родитель получает PID потомка, потомок получает 0. Используется Copy-on-Write (COW) для оптимизации. Шаги: резервирование места в swap, создание дескриптора, таблиц страниц, копирование атрибутов, создание ВАП потомка, копирование содержимого (или COW).

**`exec*()`:** Заменяет текущую программу в процессе на другую. Не создаёт новый процесс. Шаги: трансляция имени файла (по PATH), чтение заголовка (ELF / shebang / скрипт), проверка setuid/setgid, сохранение аргументов/окружения, освобождение старого ВАП, резервирование места под новый ВАП, создание регионов (код – из файла, стек/куча – в swap), инициализация контекста (IP = точка входа, SP = случайный адрес для защиты), восстановление аргументов.

**`exit()`:** Завершает текущий процесс. Шаги: отключение сигналов, закрытие файлов, сохранение кода возврата и статистики, изменение состояния на «зомби», передача потомков процессу `init` (PID=1), уничтожение ВАП, посылка сигнала `SIGCHLD` родителю, пробуждение `wait()`, переключение контекста.

**`wait()`:** Родитель ждёт завершения любого (или конкретного) потомка, получает его код возврата. Родитель **обязан** вызвать `wait()`, иначе потомки становятся зомби.


# Раздел "Недетализированные модели аппаратного уровня"


## 26. Линейная модель адресации
*(Лекция 7, раздел 2.1)*

**Определение:** Виртуальные адреса отображаются на физические простым образом:
- Физический адрес = виртуальный адрес (прямой доступ).
- Или физический адрес = виртуальный адрес + константа (смещение).

**Где используется:** Простые встраиваемые системы, загрузчики, ранний этап загрузки ОС.

**Недостатки:** Нет защиты памяти между процессами, нет изоляции сбоев, нет виртуальной памяти с подкачкой.

## 27. Сегментная модель адресации
*(Лекция 7, разделы 2.2, 1.1-1.6)*

**Определение:** Виртуальный адрес = (селектор сегмента, смещение). Селектор указывает на дескриптор сегмента в GDT (глобальной) или LDT (локальной). Дескриптор содержит: физический адрес начала, размер (limit), тип (код/данные/стек), уровень привилегий, права доступа (R/W/X).

- **Селектор сегмента:** Номер записи в таблице дескрипторов (GDT или LDT). Также содержит бит, указывающий, глобальная это таблица или локальная.
    
- **Смещение:** Адрес внутри сегмента (начинается с 0).
    

**Дескриптор сегмента (ОС8):** Аппаратная структура данных (8 байт), описывающая сегмент. Содержит:

- **Базовый адрес** (начало сегмента в физической памяти).
    
- **Лимит** (размер сегмента).
    
- **Уровень привилегий (DPL)** — 0,1,2,3 (0 — самый высокий).
    
- **Тип:** Сегмент кода, данных, стека, или системный (шлюз, TSS).
    
- **Права доступа:** Чтение, запись, выполнение.
    

**Где хранятся дескрипторы (ОС8):**

- **GDT (Global Descriptor Table):** Одна на всю систему. Содержит сегменты ОС, сегменты ядра, LDT.
    
- **LDT (Local Descriptor Table):** Одна на каждый процесс. Содержит сегменты кода, данных, стека пользовательского процесса.
    
- Регистры процессора: `GDTR` (указывает на GDT), `LDTR` (указывает на текущую LDT).
    

**Преимущества:**

- Защита (нельзя выйти за лимит сегмента → защита от переполнения буфера на уровне сегмента).
    
- Разные уровни привилегий для разных сегментов.
    
- Прозрачное перемещение сегмента в физической памяти (достаточно изменить базовый адрес в дескрипторе).
    

**Недостатки:**

- Внешняя фрагментация (сегменты разного размера трудно перемещать).
    
- Сложность управления большим количеством сегментов.


**Трансляция:** По селектору → дескриптор → адрес начала + смещение = физический адрес. Аппаратные проверки: выход за limit, права, привилегии.

**Преимущества:** Защита (границы, права), изоляция процессов (своя LDT), возможность перемещения сегментов в памяти (меняем адрес в дескрипторе).

**Недостатки:** Внешняя фрагментация, сложность с растущими сегментами (стек, куча), дополнительные обращения к памяти (4 вместо 2), хотя есть кэш дескрипторов.

## 28. Страничная модель адресации
*(Лекция 7, разделы 2.1-2.4, 3)*

**Определение:** ВАП разбито на блоки одинакового размера – **страницы** (обычно 4 КБ). Физическая память разбита на **кадры** (frame) того же размера. Таблица страниц (page table) – структура в памяти (адрес в CR3 на x86) – для каждой виртуальной страницы содержит дескриптор (PTE).

**Дескриптор страницы (PTE) содержит:** физический адрес кадра (если в RAM), признак присутствия (Present), Dirty, Accessed, права доступа (R/W/X).

**Трансляция:** Виртуальный адрес = (номер страницы, смещение). Номер страницы → индекс в таблице страниц → PTE → адрес кадра + смещение = физический адрес.

**Page Fault (страничный сбой):** если Present=0. ОС анализирует: если адрес принадлежит региону и есть образ на диске – подкачка; если адрес вне регионов – убить процесс (SIGSEGV).
**Механизм работы (ОС8):**

1. Процессор получает виртуальный адрес.
    
2. Разбивает его на уровни (например, 10-10-12).
    
3. Загружает `CR3` (адрес корневой таблицы).
    
4. Проходит по уровням, читая таблицы страниц из памяти (или из TLB-кэша).
    
5. На финальном уровне достает физический адрес кадра, добавляет смещение.
    
6. Если на каком-то уровне Valid=0 → **Page Fault** (прерывание → ОС загружает страницу с диска или убивает процесс).
## 29. Сегментно-страничная модель адресации
*(Лекция 8, раздел "Сегментно-страничная модель адресации")*

**Определение:** Комбинация сегментной и страничной моделей, где **сначала** работает страничное преобразование (формируя _линейное_ виртуальное адресное пространство), а **внутри** этого пространства уже определяются сегменты.

- **Виртуальный адрес** (то, с чем работает программа) = пара **(селектор сегмента + смещение)**.
    
- **Линейный адрес** = результат преобразования виртуального адреса через сегментацию (32-битное число от 0 до 4ГБ в 32-битных системах). Затем этот линейный адрес транслируется через страничную модель в физический.
    

**Как это выглядит в 32-битной системе (реальный пример):**

- В GDT создаются сегменты:
    
    - Сегмент 1: всё 4 ГБ, уровень привилегий 0 (код ОС).
        
    - Сегмент 2: всё 4 ГБ, уровень 0 (данные ОС).
        
    - Сегмент 3: младшие 3 ГБ, уровень 3 (код прикладной программы).
        
    - Сегмент 4: младшие 3 ГБ, уровень 3 (данные прикладной программы).
        
- **Что это даёт:** Старший 1 ГБ доступен только коду с высоким уровнем привилегий (ОС). Прикладная программа (Ring 3) не может ни выполнять код из этого региона, ни читать/писать данные ОС.
    

**Как программист этого не замечает:**

- В архитектуре x86 для каждой инструкции, работающей с памятью, есть **сегментный регистр по умолчанию** (CS для кода, DS для данных, SS для стека).
    
- ОС при запуске процесса загружает в эти регистры **селекторы нужных сегментов**.
    
- Программа использует только **смещения** (offset), а процессор автоматически подставляет селектор из соответствующего регистра.
    

**Современное состояние (64-битные системы):** Роль сегментов сильно урезана. Основную роль играет страничная модель с 4-5 уровнями. Сегментация осталась только для отдельных регистров (FS, GS), например, для указания на блок окружения потока (TEB).

## 30. Численные характеристики моделей

**Дано:**

- Размер страницы = 4 КБ = 2^12.
- Размер дескриптора страницы (PTE) = 4 байта.
- ВАП = 4 ГБ = 2^32.
- Физическая память = до 4 ГБ.

**Расчеты:**

- Число виртуальных страниц = 2^32 / 2^12 = 2^20 = 1 048 576.

- Размер плоской таблицы страниц = 1 048 576 * 4 байта = 4 194 304 байта = **4 МБ**.

- Размер блока, описываемого одной таблицей страниц (при двух уровнях) = 1024 записи * 4 КБ = **4 МБ**.

- Количество таблиц страниц второго уровня (максимально) = 1024 (т.к. 4 ГБ / 4 МБ = 1024).

- Размер Таблицы таблиц страниц (Page Directory) = 1024 записи * 4 байта = **4 КБ** (ровно одна страница).


**Вывод:** В 32-битной модели размер страницы, размер таблицы страниц и размер каталога таблиц страниц подобраны так, чтобы все идеально помещалось в страницы. В 64-битной модели (48-битный ВАП) плоская таблица заняла бы 512 ГБ, поэтому используются 4 уровня.

**Для сегментной модели:** Размер дескриптора сегмента = 8 байт (в 32-битном режиме). GDT может содержать до 8192 записей (64 КБ максимум). LDT — аналогично.

Для уменьшения используемой памяти используется многоуровневая таблица страниц. Страницы ВАП объединяются в блоки, для каждого блока формируется собственная таблица страниц. Затем формируется таблица таблиц страниц, в которой хранятся ФА таблиц страниц.

Адрес таблицы таблиц страниц хранится в CR3. ВА остался линейным, но теперь представляет собой три компонента [№ таблицы страниц : № страницы : смещение]. Каждая таблица делается по размеру такой, чтобы она помещалась в страницу (4 КБ).

ВАП = 232 Б, размер страницы ВАП -  212 Б, страниц -  220, размер блока = 1024 стр. * 4 Б = 4 КБ (умещается в страницу), кол-во блоков = 220/210=1024. Записей в таблице таблиц 1024, суммарный размер = 4 КБ * 1024 = 4 мб.

ВАП = 248Б, размер страницы ВАП -  212 Б, страниц -  236, размер блока = 512 записей * 8Б = 4Кб, кол-во блоков = 236 / 29 = 227. Записей в таблице таблице таблиц 227. Формируется ещё один уровень - 227 / 29 = 218 записей, суммарный размер 4 МБ... Всего 4 уровня таблиц. На каждом уровне таблица по размеру равна одной странице и содержит 512 записей.

Размер блока = размер страницы/размер дескриптора = 4 кб / 4 б = 1024 страницы х32

Размер блока = размер страницы / размер дескриптора (размер адреса) = 4 кб / 8 б = 512 страницы х64
## 31. Уровни привилегий ЦП
_(Лекция 9, раздел "Уровни привилегий (архитектура x86)")_

**Определение:** Аппаратно поддерживаемые уровни доступа (кольца защиты, rings). В архитектуре x86 поддерживается **4 уровня** (Ring 0 — самый привилегированный, Ring 3 — самый низкий).

**Использование на практике:**

- **Идеальная модель:** Ring 0 — ядро ОС, Ring 1 — драйверы, Ring 2 — защищенные подсистемы, Ring 3 — прикладные программы.
    
- **Реальность:** Используются в основном **два уровня**.
    
    - **Ring 0** — ядро, драйверы, системное ПО.
        
    - **Ring 3** — прикладные программы.
        
- **Исторический пример (Windows NT):**
    
    - NT 3.5/3.51: драйверы на Ring 1 (устойчиво, но медленно).
        
    - NT 4.0: драйверы перенесли на Ring 0 ради производительности (ценой синих экранов).
        

**Чем отличаются уровни (что можно/нельзя):**

1. **Привилегированные команды:** Выполняются **только** на Ring 0. Примеры: загрузка регистра глобальной таблицы дескрипторов (`LGDT`), остановка процессора (`HLT`).
    
    - _Про `HLT`: выполняется много раз в секунду при простое CPU для энергосбережения. Разница в температуре процессора с `HLT` и без неё достигала 20 градусов._
        
2. **Чувствительные команды:** Поведение зависит от уровня привилегий.
    
    - `IN`/`OUT` (ввод-вывод): на Ring 0 без ограничений; на Ring 1-3 используется маска ввода-вывода (разрешены только определённые порты).
        
    - `CLI`/`STI` (запрет/разрешение прерываний): на Ring 0 работают; на Ring 1-3 выполняются "успешно", но ничего не меняют (программа думает, что запретила прерывания, но на самом деле нет).
## 32. Шлюз вызова
_(Лекция 9, раздел "Главный механизм: шлюз вызова")_

**Определение:** Механизм для безопасного перехода из менее привилегированного кольца (Ring 3) в более привилегированное (Ring 0). Это дескриптор сегмента **специального вида** в GDT или LDT, который описывает точку входа в привилегированную функцию.

**Что содержит дескриптор шлюза вызова (4 ключевых поля):**

1. **Селектор сегмента** — в каком сегменте находится вызываемая функция (например, сегмент кода ядра).
    
2. **Смещение в сегменте** — точный адрес **первой инструкции** вызываемой функции (запрещает вызов с середины).
    
3. **Уровень привилегий, достаточный для использования шлюза (DPL)** — типично DPL=3, чтобы пользоваться могли все.
    
4. **Количество байт, копируемых из стека вызывающего уровня в стек вызываемого уровня**.
    

**Алгоритм вызова через шлюз:**

1. Программа (Ring 3) кладёт аргументы в свой стек Ring 3.
    
2. Выполняет инструкцию `CALL` с селектором шлюза.
    
3. Процессор переключается на стек Ring 0 (у вызываемой функции).
    
4. Процессор автоматически копирует указанное количество байт из стека Ring 3 в стек Ring 0.
    
5. Выполняется функция Ring 0.
    
6. При `RETF` процессор очищает скопированные аргументы и переключает стек обратно на Ring 3.

## 33. Шлюз задачи

_(Лекция 9, раздел "Шлюз задачи (Task Gate) и сегмент состояния задачи (TSS)")_

**TSS (Task State Segment):** Аппаратно-зависимая структура данных, описывающая **состояние центрального процессора** для конкретной задачи (потока). Размер ~1-1.5 КБ на современных CPU (из-за векторных регистров SSE/AVX). Хранит: CR3, LDT, указатели на стеки для разных уровней привилегий, сегментные регистры, EIP, EFLAGS, регистры общего назначения.

**Шлюз задачи (Task Gate):** Дескриптор сегмента **специального вида** в таблице дескрипторов (обычно в GDT), который **указывает на TSS** задачи.

**Переключение задачи через `JMP`:**

1. В коде задачи 1 встречается `JMP селектор_шлюза_задачи_2`.
    
2. Процессор автоматически сохраняет своё состояние (все регистры, EIP и т.д.) в **TSS задачи 1**.
    
3. Загружает состояние из **TSS задачи 2** (CR3, EIP, ESP, регистры).
    
4. Обновляет регистр задачи `TR`.
    
5. Продолжает выполнение **задачи 2** с того места, где она была остановлена.
    

**Связь с потоками ОС:** Каждому потоку ОС выделяет свой TSS. У потоков одного процесса общие CR3 и LDT, но разные стеки Ring 0 и Ring 3.
## 34. Классификация прерываний
_(Лекция 9, раздел "Прерывания (Interrupts)")_

                        ПРЕРЫВАНИЯ
                           │
            ┌──────────────┴──────────────┐
            │                             │
      ВНУТРЕННИЕ                      ВНЕШНИЕ
    (внутри процессора)            (от внешних устройств)
            │                       (клавиатура, мышь, таймер...)
            │
    ┌───────┴───────┐
    │               │

(ошибки при     (инструкция INT)

    │
    ├── Fault
    ├── Trap

**Внутренние аппаратные:**

- **Fault (ошибка/сбой):** После обработки выполнение продолжается **с той же самой инструкции**, которая вызвала прерывание (пример: **Page Fault** — страница не в памяти, ОС подкачивает её, и инструкция повторяется).
    
- **Trap (ловушка):** После обработки выполнение продолжается **со следующей инструкции** (пример: отладка, точка останова).
    
- **Abort (аварийное прекращение):** Продолжение выполнения **не предполагается** (пример: недопустимая инструкция, хотя ОС может "простить" некоторые ошибки).
    

**Программные прерывания:** Инструкция `INT N`. Симулирует возникновение прерывания с номером N. Исторически использовалось для системных вызовов (`INT 0x80` в Linux), сейчас заменено на `SYSENTER`/`SYSCALL`.

## 35. Обработка прерываний контроллером прерываний

_(Лекция 9, раздел "Стратегии работы контроллера прерываний")_

**Стратегии :**

1. **Неблокирующая стратегия (вложенные прерывания):** Контроллер немедленно доставляет прерывание процессору, даже если обрабатывается другое. Обработчики прерываний должны быть очень короткими, а длинная работа откладывается в DPC (Deferred Procedure Call).
    
2. **Блокирующая обработка:** После доставки прерывания контроллер прекращает доставку любых следующих. После завершения обработки процессор сообщает контроллеру сигнал EOI (End of Interrupt). Контроллер доставляет одно из накопившихся прерываний.
    - Контроллер блокирует доставку новых прерываний, пока текущее не обработано.
	- Запоминает поступившие прерывания (ставит в очередь).
	- После `EOI` (End Of Interrupt) от процессора контроллер выбирает следующее прерывание для доставки

    - **FIFO (First In — First Out):** Первым пришло — первым доставлено.
        
    - **LIFO (Last In — First Out):** Доставляется **последнее** (самое свежее) прерывание. Эффективно для частых прерываний (таймер), чтобы не пропустить следующее.
        
    - **Приоритетная доставка:** Доставляется прерывание с **наивысшим приоритетом**. Высокоприоритетное прерывание может прервать обработку низкоприоритетного, но не наоборот.
## 36. Обработка прерывания центральным процессором

_(Лекция 9, раздел "Обработка прерываний на центральном процессоре")_

**Поиск обработчика:** Через **IDT (Interrupt Descriptor Table)** — таблица из 256 записей (шлюзов вызова или задач). Адрес таблицы в регистре `IDTR`.

**Аппаратные действия (процессор автоматически):**

1. Сохраняет минимальный контекст (обычно `SS:ESP`, `EFLAGS`, `CS:EIP`) в стек (текущего уровня привилегий или целевого, если идет смена уровня).
    
2. Определяет номер прерывания (из контроллера или из кода инструкции).
    
3. Ищет обработчик в **IDT (Interrupt Descriptor Table)**. IDT — таблица из 256 записей, каждая из которых является:
    
    - **Шлюзом прерывания (Interrupt Gate):** Для внешних прерываний. Автоматически запрещает прерывания (сбрасывает `IF`).
        
    - **Шлюзом ловушки (Trap Gate):** Для исключений и программных прерываний. Не запрещает прерывания.
        
    - **Шлюзом задачи (Task Gate):** Для аппаратного переключения задач (редко).
        
4. Переходит по адресу из шлюза (селектор сегмента + смещение), выполняет смену привилегий (если нужно) и переключает стек (через TSS).
    

**Программные действия (обработчик ОС):**

5. Сохраняет остальной контекст (регистры общего назначения, которые будет портить).
    
6. Изменяет состояние контроллера прерываний (подтверждает прием, маскирует/размаскирует).
    
7. Устанавливает контекст для обработчика (переключает `CR3` на ядро, если это системный вызов и т.д.).
    
8. Вызывает функцию обработки (C-функцию в драйвере или ядре).
    
9. **Планировщик:** Точка, где может произойти переключение потоков. ОС может решить, что текущий поток больше не должен выполняться, и выбрать другой.
    
10. Изменяет состояние контроллера прерываний (посылает `EOI`).
    
11. Восстанавливает контекст (регистры).
    
12. Выполняет инструкцию **`iret`** (или `iretq` в 64-битном режиме), которая:
    
    - Восстанавливает `SS:ESP`, `EFLAGS`, `CS:EIP`.
        
    - Возвращает управление прерванной программе (или другому потоку, если планировщик решил переключиться).

**Алгоритм обработки (10 пунктов):**

|№|Действие|Кто выполняет|
|---|---|---|
|1|Аппаратное сохранение состояния (EIP, CS, EFLAGS, возможно SS/ESP)|**Аппаратно**|
|2|Определение адреса обработчика из IDT и переход на него|**Аппаратно**|
|3|Программное сохранение состояния (дополнительные регистры, контекст ОС)|Программно (ОС)|
|4|Изменение состояния контроллера прерываний (сигнал получен, обработка началась)|Программно|
|5|Установка контекста для обработчика (например, для драйвера устройства)|Программно|
|6|Запуск обработчика (содержательная работа)|Программно|
|7|**Выбор задачи (потока) на исполнение — здесь работает планировщик!**|Программно|
|8|Изменение состояния контроллера прерываний (EOI — End of Interrupt)|Программно|
|9|Программное восстановление состояния выбранной задачи|Программно|
|10|Аппаратное восстановление состояния и переход к выполнению выбранной задачи|**Аппаратно**|
Ключевой пункт 7:** Обработка прерывания (особенно от таймера) становится **естественной точкой для переключения контекста**. Планировщик может решить передать CPU другому потоку, а не тому, который был прерван. Это основа вытесняющей многозадачности.

# Раздел "Планирование"


## 37. Критерии оценки алгоритмов планирования
_(Лекция 10, раздел "Критерии оценки алгоритмов планирования")_

**Общие критерии:**

- **Справедливость (Fairness):** Равное распределение времени (спорно, как измерять? на каком уровне?). Всем потокам выделяется равное количество времени ЦП.
    
- **Отсутствие голодания (No Starvation):** Ни один поток не ждёт бесконечно.
    
- **Баланс (Balance):** Загрузка всех устройств (CPU, диск, сеть, принтер) параллельно.
    
- **Накладные расходы (Overhead):** Время на работу планировщика + переключения контекста.
    
- **Масштабируемость (Scalability):** Сложность алгоритма (O(1), O(log n), O(n)).
    

**Для систем пакетной обработки:**

- **Пропускная способность (Throughput):** N / T (число задач в единицу времени).
    
- **Оборотное время (Turnaround Time):** Среднее время от запуска до завершения.
    
- **Эффективность (Efficiency):** Загрузка CPU (суммарное время CPU / общее время).
    

**Для интерактивных систем:**

- **Время отклика (Response Time):** От события (нажатие клавиши) до реакции (появление буквы). Зависит от планировщика только на этапе "время ожидания в очереди готовых".
    
- **Время ожидания (Waiting Time):** Среднее время между **переводом потока в состояние "готов"** и **моментом получения CPU**.
    
- **Отзывчивость (Responsiveness):** Субъективное ощущение пользователя.
    

**Для систем реального времени (RTOS):**

- **Предсказуемость (Predictability):** Максимальное время предоставления ЦП потоку реального времени должно быть гарантированно не больше заданного интервала.
    
- **Соблюдение сроков (Meeting Deadlines):** Окончание обслуживания задачи к установленному сроку.
## 38. Невытесняющее и вытесняющее планирование
_(Лекция 10, раздел "Классификация алгоритмов планирования")_

**Невытесняющее (non-preemptive):** Планировщик выбирает поток и позволяет ему работать **до добровольной блокировки** (I/O) или **до завершения**.

- **Плюс:** Нет лишних переключений.
    
- **Минус:** Бесконечный цикл в потоке → зависание всей системы.
    

**Вытесняющее (preemptive):** Планировщик позволяет потоку работать **не более фиксированного максимального времени (кванта)**. По истечении кванта поток **принудительно приостанавливается**.

- **Плюс:** Надёжность (зависший поток не блокирует систему), отзывчивость.
    
- **Минус:** Накладные расходы на частые переключения.
    
- **Требование:** Аппаратный таймер, генерирующий прерывания (например, 1000 раз в секунду).
- **Выбор кванта:** Очень маленький квант → высокие накладные расходы (частые переключения). Очень большой квант → ухудшение времени отклика (интерактивность). Типичные значения: Windows client — 12 мс, Windows server — 36 мс, Linux — 100-150 мс.
## 39. Алгоритмы FCFS, SJF, SRT, RR
_(Лекция 10, разделы 12, 13, 14, 3)_

**FCFS (First Come First Served):**

- **Тип:** Невытесняющий.
    
- **Суть:** Очередь FIFO (первым пришёл — первым обслужен). Новые потоки и проснувшиеся после I/O становятся в **конец** очереди.
    
- **Голодание:** Невозможно (при разумных предположениях).
    
- **Проблема:** "Эффект конвоя" — короткий I/O-bound поток может долго ждать за длинным CPU-bound потоком.
    

**SJF (Shortest Job First):**

- **Тип:** Невытесняющий.
    
- **Описание:** Выбирается поток с наименьшим **общим ожидаемым временем выполнения** (оценка должна быть известна априори).
    
- **Идея:** Завершение коротких задач быстрее освобождает ресурсы (память, ввод-вывод) для остальных, снижая давление на систему.
    
- **Проблемы:** Голодание длинных задач (постоянно приходят короткие). Невозможность точного предсказания времени выполнения в общем случае. На практике используется в специализированных системах (например, планирование SQL-запросов в СУБД).
    
- **Борьба с голоданием:** Если поток потребил больше заявленного времени, он становится в конец очереди (FCFS для "долгих").

**SRT (Shortest Remaining Time):**

- **Тип:** Вытесняющий (вариант SJF).
    
- **Описание:** Выбирается поток с наименьшим **оставшимся временем выполнения** (общее время минус уже потребленное). Текущий поток может быть вытеснен, если появился новый поток с меньшим оставшимся временем.
    
- **Проблема:** Та же, что у SJF (голодание, необходимость оценки).

**RR (Round Robin):**

- **Тип:** Вытесняющий аналог FCFS. циклический.
    
- **Суть:** Потоки выполняются по очереди, каждый **не более кванта** (например, q=4). Если поток не завершился за квант, он вытесняется и идёт в **конец** очереди.
    
- **Размер кванта:** Windows (клиентские): 12 мс, Windows (серверные): 36 мс, Linux (старые данные): 150 мс.
## 40. Приоритетное планирование
_(Лекция 10, разделы IV, V, VI, VII)_

**Идея:** Потоки имеют приоритеты. CPU всегда выделяется готовому потоку с **наивысшим уровнем приоритета**.

**Реализация (классический вариант):**

- Несколько уровней приоритета.
    
- **Внутри каждого уровня** — Round-Robin (часто с одинаковым квантом).
    
- **Между уровнями** — приоритетное планирование.
    
- Если появляется готовый поток с более высоким приоритетом, чем у текущего выполняющегося → **вытеснение**.
  
- Между уровнями приоритетов — строгая иерархия: если есть хотя бы один готовый поток с приоритетом 3, потоки с приоритетом 1 и 2 не получат ЦП.

**Побочные эффекты:**

- **Голодание (starvation):** Если в системе **всегда есть** готовый высокоприоритетный поток, низкоприоритетные **никогда не получат CPU**.
    

**Модификации для борьбы с голоданием:**

1. **Понижение приоритета за потребление CPU:** С каждым квантом эффективный приоритет уменьшается.
    
2. **Повышение приоритета за долгое ожидание (Boosting):** Поток, который долго не получал CPU, временно повышается до высокого приоритета на один квант, затем его приоритет понижается обратно. Используется в Windows.
    
3. **Адаптивное / динамическое планирование:** Приоритеты постоянно пересчитываются на основе статистики потребления CPU, типа I/O и других параметров.
## 41. Планирование с использованием многоуровневых очередей и обратной связи
_(Лекция 10, разделы IX, X)_

**MLFQ (Multi-Level Feedback Queue):** Поток может **мигрировать** между очередями (уровнями приоритета) в зависимости от его поведения (сколько CPU потребляет, какие I/O выполняет).

**Что нужно задать для реализации MLFQ:**

1. Количество очередей.
    
2. Алгоритм между очередями (обычно приоритетный).
    
3. Алгоритм внутри каждой очереди (может быть свой: RR с разным квантом, FCFS).
    
4. Правила размещения нового потока (обычно в очередь с наивысшим приоритетом).
    
5. **Правила перевода потока из очереди в очередь (самое важное).**
    

**Пример правил перевода:**

- **Вниз (понижение):** Поток полностью потребил квант на уровне 4 → переместить на уровень 3; на уровне 3 потребил квант → на уровень 2; на уровне 2 потребил квант → на уровень 1 (FCFS).
    
- **Вверх (повышение):** Поток дождался **пользовательского ввода-вывода** (клавиатура, мышь) → **на самый верхний уровень**. Дисковый I/O → на уровень 3. Другие события → на уровень 2.


## 42. Алгоритм планирования Windows
_(Лекция 10, раздел XI)_

**Общая характеристика:** Приоритетное вытесняющее планирование.

**Диапазоны приоритетов:**

- **16–31:** Приоритеты реального времени (фиксированные, не изменяются).
    
- **1–15:** Изменяемые приоритеты (здесь работает эффективный приоритет).
    
- **0:** Специальный приоритет (System Idle Process).
    

**От базового приоритета к эффективному:**

1. Базовый приоритет процесса (наследуется от родителя).
    
2. Относительный приоритет потока (Idle, Lowest, Below normal, Normal, Above normal, Highest, Time-critical) → бонус (-2, -1, 0, +1, +2, специальный).
    
3. Вычисленное значение = базовый приоритет + бонус.
    
4. **Эффективный приоритет** = вычисленное значение + **бонусы**.
    

**Бонусы к приоритету в Windows:**

- **Бонус за интерактивность:** Клиентские версии → +2 для интерактивных потоков, серверные → +1.
    
- **Бонус за владение фокусом ввода:** Поток, обрабатывающий сообщения от элемента с фокусом (обслуживающий активное окно) → +2.
    
- **Бонус за завершение I/O:** В зависимости от типа устройства (+8 для микрофона, +4, +2, +1 для других). Бонус временный и убывающий (с каждым квантом -1).
    

**Борьба с голоданием:** Периодическая проверка потоков, которые долго не получали CPU → временное повышение приоритета (boosting).
- При вытеснении более приоритетным потоком, вытесненный поток становится в **начало** очереди своего уровня (а не в конец), чтобы продолжить выполнение сразу после освобождения ЦП.
## 43. Оригинальный алгоритм планирования UNIX
_(Лекция 10, раздел XV)_

**Общая архитектура:** Приоритетное планирование. **Невыгружаемость ядра** (non-preemptible kernel) — поток в режиме ядра не может быть вытеснен.

**Три диапазона приоритетов:**

1. **Самый высокий:** Приоритеты реального времени (фиксированные).
    
2. **Средний (66–95):** Системные вызовы (фиксированные, зависят от типа вызова).
    
3. **Самый низкий (0–65):** Пользовательские приоритеты (динамические).
    

**Формула пользовательского приоритета:** `приоритет = base + nice + CPU_usage`

- **nice** — параметр "любезности". Чем меньше nice, тем выше приоритет. Диапазон от -20 (высокий) до +19 (низкий).
    
- **CPU_usage** — статистика потребления CPU. **Чем больше потреблял, тем ниже приоритет**. Устаревание: раз в секунду `CPU_usage = CPU_usage / 2`.
    

**Проблема:** Голодание возможно, так как разница в `nice` (39) может быть больше, чем компенсирует `CPU_usage` (максимум ~100-199).

**Устаревание (aging) CPU_usage:**

- Чтобы `CPU_usage` не рос бесконечно, раз в секунду происходит его уменьшение.
    
- **Простая формула (ранний Unix):** `CPU_usage = CPU_usage / 2`. Это позволяло потоку "отдохнуть" и быстро сбросить счетчик, но все же голодание было возможно.
    
- **Улучшенная формула (позже):** `CPU_usage = CPU_usage * (2 * load) / (2 * load + 1)`, где `load` — число активных потоков. При высокой загрузке устаревание происходит медленнее. Доказано, что эта формула исключает голодание.
## 44. Алгоритм планирования Linux
_(Лекция 10, раздел XVI)_

**Тип:** Приоритетное вытесняющее планирование. 

**Три класса потоков (по убыванию приоритета):**

1. **Потоки реального времени (FIFO):** FCFS внутри уровня.
    
2. **Потоки реального времени (RR):** Round-Robin с квантом.
    
3. **Потоки разделения времени (Time Sharing):** Динамические приоритеты.
    

**Атрибуты TS-потока:**

- **nice:** от -20 до +19.
    
- **priority (статический):** вычисляется из nice (диапазон ~101-140, где 140 — наивысший).
    
- **quantum (квант):** количество тиков (`jiffy`, обычно ~2 мс), которое поток может выполняться.
    
- **goodness** — **динамический приоритет** (используется планировщиком).
    

**Формула goodness:**
```c
- Для потоков реального времени: `goodness = 1000 + priority` (всегда больше, чем у обычных).
    
- Для потоков разделения времени:
    
    - Если `quantum > 0`: `goodness = quantum + priority`
        
    - Если `quantum = 0`: `goodness = 0` (поток не получит ЦП до следующего перерасчета).
```
**Алгоритм работы:**

1. Выбирается поток с максимальным `goodness`.
    
2. С каждым тиком квант уменьшается на 1.
    
3. Переключение/вытеснение, когда: квант стал 0, ИЛИ блокирующий вызов, ИЛИ появился поток с более высоким `goodness`.
    
4. Если у всех TS-потоков `quantum == 0`:
    
    - Перерасчёт квантов для **всех** TS-потоков: `quantum = quantum / 2 + priority`.
        
    - `quantum / 2` — "долг" (недоиспользованный квант из прошлого цикла). Поток, который долго ждал, может накопить большой `quantum` и временно получить более высокий `goodness`, чем высокоприоритетный поток.
        


**Перерасчет квантов (Aging):**

- Когда у **всех** готовых к выполнению потоков `quantum = 0`, планировщик выполняет перерасчет для **всех потоков** (включая ожидающие):  
    `новый_quantum = старый_quantum / 2 + priority`
    
- Это позволяет потокам, которые долго ждали (находились в состоянии ожидания), **накопить большой quantum**, что со временем повышает их `goodness` и предотвращает голодание. Даже низкоприоритетный поток, долго не получавший ЦП, может накопить quantum ~ 2*priority и временно стать высокоприоритетным.

**Голодание:** **Невозможно**. Даже самый низкоприоритетный поток обязательно получит CPU в своём цикле (когда все высокоприоритетные исчерпают кванты).

## 45. Необходимость синхронизации

**Проблема:** В многопоточных приложениях результат выполнения может зависеть от того, в каком порядке и в какие моменты времени переключаются потоки (диаграмма выполнения потоков).

**Классический пример (из лекции):**  
Два потока работают с одной записью в базе данных. Поток A увеличивает поле на 3, поток B — на 5. Последовательность операций (чтение → изменение в локальной памяти → запись) — это критическая секция.

**Возможные результаты в зависимости от диаграммы выполнения:**

- Если сначала полностью выполнился B, потом A → увеличение на 8.
    
- Если A прочитал, потом B прочитал, B записал, потом A записал → увеличение на 3 (изменения B потеряны).
    
- Если B прочитал, потом A прочитал, A записал, потом B записал → увеличение на 5 (изменения A потеряны).
    

**Почему это проблема:**

- Результат вычислений **неоднозначен**, зависит от случайных факторов (моменты прерываний, попадание в кэш, загрузка ядер).
    
- Разные запуски одной и той же программы могут давать разные результаты.
    
- Проблема возникает не только с базами данных, но и с обычными переменными (например, `i++` при многократном повторении).
    

**Сложность отладки:** Проблемы синхронизации трудно воспроизводимы, так как они зависят от конкретной диаграммы выполнения. Отладочный вывод (printf) может изменить диаграмму и "спрятать" ошибку.

_(Лекция ОС13, начало)_

**Проблема:** В многопоточных приложениях результат выполнения может зависеть от того, в каком порядке и в какие моменты времени переключаются потоки (диаграмма выполнения). Это называется **состоянием гонки (race condition)**.

## 46. Постановка задачи взаимного исключения

**Цель:** Согласовать работу M параллельных потоков при использовании некоторого критического ресурса.

**Условия (7 требований):**

|№|Условие|Смысл|
|---|---|---|
|**1**|**Взаимное исключение (Mutual Exclusion):** Одновременно внутри критической секции (КС) должен находиться **не более одного** потока.|Главное условие — никакие два потока не могут одновременно выполнять КС.|
|**2**|**Отсутствие приоритета:** Критические секции не должны иметь приоритета в отношении друг к другу. Код, добавляемый к различным КС, должен быть одинаковым или симметричным.|Ни одна операция (чтение, запись, пополнение, списание) не должна иметь приоритета над другой.|
|**3**|**Локализация синхронизирующего кода:** Остановка какого-либо потока вне его критической секции не должна влиять на дальнейшую работу других потоков с критическим ресурсом.|Синхронизирующий код должен быть локализован непосредственно вокруг КС.|
|**4**|**Конечность времени принятия решения:** Решение о входе потоков в критические секции не откладывается на неопределённый срок.|Алгоритм не должен приводить к ситуации, когда потоки не могут решить, кому входить (например, вечная "гонка").|
|**5**|**Выполнимость кода выхода:** Освобождение критического ресурса и выход из КС должны быть произведены потоком, использующим ресурс, за **конечное время**.|Код выхода из КС не должен зависать.|
|**6**|**Произвольное состояние вне КС:** Любой поток может переходить в любое состояние, отличное от активного, вне предела своей КС.|Мы не делаем предположений о том, что происходит с потоками вне КС.|
|**7**|**Произвольная относительная скорость:** Относительная скорость развития потоков неизвестна и произвольна.|Решение должно работать при любой возможной диаграмме выполнения потоков.|

**Важное замечание лектора:** Все эти условия предполагают, что **поток не может упасть или бесконечно заблокироваться внутри критической секции**. Если это возможно, то рассмотренные механизмы синхронизации неприменимы.


## 47. Проверка корректности работы алгоритмов синхронизации

**Метод:** Построить **диаграмму выполнения потоков** (или перечислить последовательность операций), которая демонстрирует нарушение одного из условий задачи взаимного исключения.

- Чтобы показать, что решение **неправильное**, достаточно **одной** диаграммы, где два потока одновременно входят в КС (нарушение условия 1) или возникает взаимоблокировка (нарушение условия 4).
    
- Чтобы показать, что решение **правильное**, нужно убедиться, что при **любой** возможной диаграмме все условия выполняются.

**Пример проверки (из лекции):**  
Рассмотрим наивный алгоритм с общим флагом `flag` (изначально `false`).

```c
// Поток 1                  // Поток 2
while (flag) {};            while (flag) {};
flag = true;                flag = true;
// Критическая секция       // Критическая секция
flag = false;               flag = false;
```

**Диаграмма, ломающая алгоритм:**

1. Поток 1 проверяет `flag` (он `false`) и выходит из цикла.
    
2. Планировщик передает управление Потоку 2.
    
3. Поток 2 проверяет `flag` (все еще `false`) и выходит из цикла.
    
4. Оба потока устанавливают `flag = true` и входят в критическую секцию одновременно.  
    **Нарушено условие №1** (взаимное исключение).
## 48. Алгоритм Петерсона
**Назначение:** Решение задачи взаимного исключения для **двух потоков**.

**Данные:**

- `flag[0]`, `flag[1]` — флаги готовности потоков войти в КС (инициализированы `false`).
    
- `turn` — переменная очереди (чей сейчас черёд). Инициализируется, например, 0.
    

**Код для потока i (где i = 0 или 1, а j = 1 − i):**

```c
while (true) {
    flag[i] = true;           // Я хочу войти в КС
    turn = j;                 // Но я уступаю очередь другому потоку
    while (flag[j] && turn == j) { // ПРОВЕРКА ФЛАГА ДОЛЖНА БЫТЬ ПЕРВОЙ
        // ждать (busy wait)
    }
    // КРИТИЧЕСКАЯ СЕКЦИЯ
    flag[i] = false;          // Я вышел из КС
    // НЕКРИТИЧЕСКАЯ СЕКЦИЯ
}
```

**Логика работы:**

1. Поток i заявляет о желании войти в КС (`flag[i] = true`).
    
2. Поток i уступает очередь другому потоку (`turn = j`).
    
3. Поток i ждёт, пока:
    
    - другой поток **хочет войти** в КС (`flag[j] == true`)
        
    - **И** сейчас **его очередь** (`turn == j`).
        
4. Как только одно из условий нарушается (другой поток не хочет входить **или** сейчас наша очередь), поток i входит в КС.
    
5. После выхода из КС поток сбрасывает свой флаг.
    

**Свойства:**

- Обеспечивает взаимное исключение.
    
- Отсутствует голодание (ограниченное ожидание).
    
- Не требует специальной аппаратной поддержки (только атомарность чтения/записи переменных, что есть в большинстве архитектур).

**Важные замечания из лекции:**

- Алгоритм теоретически корректен.
    
- На практике может быть сломан оптимизациями компилятора или процессора (перестановка операций). Чтобы избежать этого, нужно использовать барьеры памяти (memory barriers).
    
- Если поменять местами `flag[i] = true` и `turn = j`, алгоритм **перестанет работать**.
    
- Алгоритм работает на многопроцессорных системах, если операции записи в память видны другим ядрам.
## 49. Алгоритм булочной

**Назначение:** Обобщение идеи Петерсона на **N потоков**.

**Структуры данных и логика:**
- Есть два массива, индексируемые по номеру потока `i`:
    1.  `number[i]` — хранит позицию (номерок) потока в очереди. Изначально у всех `0`, что означает «не конкурирует за вход».
    2.  `choosing[i]` — булево значение. `true`, когда поток *выбирает* себе номерок. Изначально у всех `false`.

- **Процесс получения номерка:**
    1.  Поток устанавливает `choosing[i] = true`, заявляя о начале выбора.
    2.  Вычисляет свой `number[i]` как `max(number[0], ..., number[N-1]) + 1`. Здесь возникает возможность коллизий: несколько потоков могут увидеть одинаковый максимум и получить один и тот же номер.
    3.  После вычисления поток устанавливает `choosing[i] = false`.

- **Разрешение коллизий (лексикографическое сравнение):**
    - Позиция потока в очереди — это пара `(number[i], i)`.
    - Сравнение: сначала по номерку. Если номерки равны — по уникальному идентификатору потока `i`. Поток с меньшим `i` имеет приоритет.
    - **Проблема нечестности:** такой подход даёт постоянное преимущество потокам с меньшими ID. Это «нечестно», но алгоритм обеспечивает разрешение любой коллизии и гарантирует отсутствие deadlock'а.

```C
choosing[i] = true;
number[i] = max(number[0], number[1], ..., number[N-1]) + 1;
choosing[i] = false;

for (j = 0; j < N; j++) {
    while (choosing[j]) {}; // Ждем, пока j выбирает номер
    
    while (number[j] != 0 && 
          (number[j] < number[i] || 
          (number[j] == number[i] && j < i))) {
        // Ждем, пока поток j не завершит КС
    }
}

// --- Критическая секция ---

number[i] = 0; // Выхожу из КС
```

**Сравнение позиций:** лексикографическое (сначала по `number`, при равенстве — по индексу потока). Это **нечестно** (потоки с меньшим ID имеют преимущество), но коллизии разрешаются.

**Пример из практики (лектор ОС12):** В эксперименте с 4 потоками первый получал ресурс в 90% случаев, второй — в 80%, третий — 25%, четвёртый — 5%. При смене ядра Linux соотношение стало 55-45-... — механизм изменился, но остался нечестным.

## 50. Использование аппаратной поддержки (test&set)

**Идея:** Операционная система/процессор предоставляет атомарную инструкцию, которая выполняет две операции как единое целое (неделимо).

**test_and_set (TAS):** Атомарно читает значение переменной, устанавливает её в 1, возвращает старое значение.

```c
int test_and_set(int *target) {  
        int tmp = *target;  
        *target = 1;  
        return tmp;
```
По указателю записывается 1, а возвращается то, что было записано ранее. Это 1 инструкция ЦП.

**Реализация спин-блокировки (spinlock):**
```c
int lock = 0; //Признак блокировки данных  
while (true) {  
        while (test_and_set(&lock))  
                ;  // ждать

        // КРИТИЧЕСКАЯ СЕКЦИЯ 
        lock = 0;  

}
```

Для однопроцессорных систем хорошо. Для многопроцессорных – возможен одновременный захват.

**Свойства:**

- Атомарность гарантируется аппаратно.
    
- Простота реализации.
    
- **Недостаток:** Активное ожидание (busy waiting) — процессор занят циклом, что неэффективно при длительных ожиданиях.`
## 51. Использование аппаратной поддержки (swap)

**Идея:** Альтернативная атомарная инструкция для реализации взаимного исключения.

**Инструкция `swap` (или `exchange`):** Атомарно обменивает значение регистра и значение ячейки памяти.

**Реализация спин-блокировки:**

```c
int lock = 0; //Признак блокировки критических данных
int key;  
while(true){  
        key = 1;  
        do{  
                Swap(&lock, &key);  
        } while(key); //Делаем, пока ключ не будет 0        

        lock = 0;  

}
```

**Свойства:** Аналогичны `test_and_set`.

## 52. Активное ожидание (Busy Waiting)

**Определение:** Цикл, в котором поток постоянно проверяет условие входа в критическую секцию, не отдавая центральный процессор

**Определение:** Поток, ожидающий входа в КС, постоянно проверяет условие в цикле, не отдавая CPU.

**Плюсы:**

- Простота реализации.
    
- Нет накладных расходов на переключение контекста (если время ожидания очень мало).

**Минусы:**

- Трата процессорного времени впустую.
    
- Неэффективно при длительных ожиданиях.
    
- На однопроцессорной системе может привести к "зависанию" (если ожидающий поток не отдаёт CPU, а поток, удерживающий блокировку, не получает CPU).

**Когда допустимо:** Только когда время ожидания гарантированно **очень мало** (например, несколько тактов), иначе лучше использовать блокирующие примитивы.


**Когда эффективно (из лекции):**

- Критические секции очень короткие (несколько инструкций).
    
- Потоки выполняются на разных физических ядрах.


**Оптимизации (из лекции):**

- Добровольное освобождение CPU (`sleep(0)`, `sched_yield()`) после неудачной попытки.
    
- Двухфазный подход: сначала активное ожидание (заданное число итераций или время), затем засыпание.
## 53. Семафор, типы семафоров

**Определение:** Разделяемая целочисленная переменная с двумя атомарными операциями:

- **`P(S)` (wait, down, prolaag):** блокирующая. Уменьшает семафор. Если семафор == 0, поток блокируется до тех пор, пока семафор не станет > 0.
    
- **`V(S)` (signal, up, verhoog):** неблокирующая. `S++`. Если есть ждущие потоки, один пробуждается.
    

**Типы семафоров:**

|Тип|Значения|Назначение|
|---|---|---|
|**Двоичный (бинарный)**|0 или 1|Взаимное исключение (mutual exclusion). `1` — ресурс свободен, `0` — занят.|
|**Счётный (счётчик)**|от 0 до N|Контроль числа доступных единиц ресурса. Начальное значение = N.|

**Важное предостережение лектора (ОС13):** **Никогда не реализуйте очередь ожидания как FIFO!** Механизмы синхронизации должны использовать **случайный** выбор из числа ожидающих, иначе программы, которые рассчитывают на порядок, будут неправильно верифицироваться.
## 54. Мьютекс (Mutex)

**Определение:** Объект синхронизации, являющийся улучшенной версией двоичного семафора для организации эксклюзивного доступа. Он проще в использовании и позволяет избежать типовых ошибок.

**Базовые операции:**

- `acquire()` — блокирующий захват.
    
- `release()` — освобождение.
    
- `try_acquire()` — неблокирующая попытка захвата

**Дополнительные свойства Mutex (отличие от двоичного семафора):**

|Свойство|Описание|
|---|---|
|**Запоминание владельца**|`release()` может выполнить только поток-владелец. Чужой вызов `release()` ничего не делает.|
|**Рекурсивность**|Поток может многократно захватить mutex. Для полного освобождения нужно вызвать `release()` столько же раз. Удобно для библиотек (функции вызывают друг друга).|
|**Наследование приоритета**|Поток, захвативший mutex, временно наследует максимальный приоритет из ожидающих. Решает проблему **инверсии приоритетов**.|

## 55. Монитор, условные переменные. Мониторы Хоара и Меса

**Монитор:** Конструкция языка программирования (или пара "mutex + conditional variable"), которая инкапсулирует:

1. Разделяемые критические данные.
    
2. Функции (методы), использующие эти данные.
    
3. Автоматическую синхронизацию (вход в любой метод монитора блокирует его для других потоков).
    

**Правило:** В каждый момент времени только **один** поток может находиться внутри монитора (для данного объекта). Остальные блокируются на входе.

**Проблема без условных переменных:** Если первый вошедший поток не может выполнить операцию (буфер пуст), он "застревает" внутри монитора, и другие потоки не могут войти → **deadlock**.

**Условные переменные (Condition Variables):** Расширение монитора для ожидания событий.

|Операция|Действие|
|---|---|
|**`wait(cond)`**|Освобождает монитор и переводит поток в состояние ожидания на условной переменной `cond`.|
|**`signal(cond)`**|Пробуждает **максимум один** поток, ожидающий на `cond`. Если ожидающих нет — сигнал теряется.|
|**`broadcast(cond)`**|Пробуждает **все** потоки, ожидающие на `cond`.|

**Типы мониторов:**

|Тип|Поведение при `signal`|Следствие|
|---|---|---|
|**Мониторы Хоара (Hoare)**|Сигнализирующий поток немедленно блокируется, разбуженный получает монитор.|Разбуженный поток **гарантированно** обнаруживает условие выполненным. Не используются на практике.|
|**Мониторы Меса (Mesa)**|Разбуженный поток переводится в состояние "готов", но конкурирует за вход на общих основаниях.|Условие нужно проверять **в цикле `while`**, а не в `if` (возможны ложные пробуждения). **Современный стандарт**.|


## 56. Решение задачи "Производитель-потребитель" с использованием семафоров
#### А. Производитель-Потребитель (циклический буфер)

**Постановка задачи:** Есть ограниченный циклический буфер на `N` элементов. Один или несколько потоков-производителей добавляют данные в буфер, один или несколько потоков-потребителей забирают данные из буфера. Нельзя читать из пустого буфера и писать в полный.

**Используемые семафоры:**

- `empty` (счётный) = N — количество пустых ячеек.
    
- `full` (счётный) = 0 — количество заполненных ячеек.
    
- `access` (двоичный) = 1 — защита буфера и указателей.
    

**Код:**

```c
// Производитель
P(empty);        // ждём свободную ячейку
P(access);       // эксклюзивный доступ к буферу
// запись в буфер
V(access);       // освобождаем буфер
V(full);         // сигнализируем о новых данных
// Потребитель
P(full);         // ждём данные
P(access);       // эксклюзивный доступ к буферу
// чтение из буфера
V(access);       // освобождаем буфер
V(empty);        // сигнализируем о свободном месте
```

## 57. Решение задачи "Производитель-потребитель" с использованием мониторов

**Решение задачи "Производитель-Потребитель" на мониторах:**

```c
// В мониторе: буфер, указатели, условные переменные not_full, not_empty
void produce(item) {
    while (buffer_full) wait(not_full);
    add_to_buffer(item);
    signal(not_empty);
}
item consume() {
    while (buffer_empty) wait(not_empty);
    item = remove_from_buffer();
    signal(not_full);
    return item;
}
```
**Важные замечания (из лекции):**

- Вход в методы `produce` и `consume` автоматически блокирует монитор.
    
- Используется **цикл `while`**, а не `if`, потому что мы рассматриваем **мониторы Меса**. После пробуждения условие (`count == N` или `count == 0`) может снова стать истинным (другой поток мог изменить состояние), поэтому проверка повторяется.
    
- В мониторах Хоара можно было бы использовать `if`, но они сложнее в реализации и практически не используются.
## 58. Решение задачи "Читатели-писатели" с использованием семафоров

#### Б. Читатели-Писатели (с преимуществом читателей)
**Постановка задачи:** Есть разделяемые данные (критический ресурс). Потоки-читатели только читают данные (параллельно, не мешая друг другу), потоки-писатели изменяют данные (эксклюзивно, не допуская никого другого).
- Читатели могут работать параллельно (сколько угодно читателей одновременно).
- Писатель должен иметь эксклюзивный доступ (ни одного читателя или писателя)

**Инструменты:**

- `XS` (двоичный) — эксклюзивный доступ к данным. =1
-  RC` (двоичный) — защита `readcount`. =1
    
- `readcount` (целая) — счётчик активных читателей.  

**Код:**

c
```c

// Писатель
P(XS);
// запись
V(XS);
// Читатель
P(RC);
readcount++;
if (readcount == 1) P(XS);  // первый читатель захватывает данные
V(RC);
// чтение
P(RC);
readcount--;
if (readcount == 0) V(XS);  // последний читатель освобождает данные
V(RC);
```
**Проблема решения (из лекции):**  
Это решение дает **преимущество читателям**. Если читатели приходят постоянно, писатель может никогда не получить доступ (голодание писателя).

**Решение проблемы (улучшенный вариант):**  
Добавить еще один семафор `service` S  (изначально `1`).

- Писатель: `P(service)` перед `P(access)` и `V(service)` после `V(access)`.
    
- Читатель: `P(service)` перед входом и `V(service)` после выхода.  
    Теперь, если писатель ждет, он блокирует семафор `service`, и новые читатели не могут войти. Когда текущие читатели закончат, писатель получит доступ.

```c
// Писатель
P(S); P(XS); // запись V(XS); V(S);
// Читатель
P(S); // ... стандартный код читателя ... V(S);
```
## 59. Взаимоблокировка (определение)
**Определение:** Ситуация, когда два или более потоков (процессов) находятся в состоянии ожидания, и каждый ждёт ресурс, захваченный другим потоком из этой группы. Ни один из них не может продолжить выполнение.

**Пример из лекции (два флага, сначала установка, потом проверка):**

- Поток A: `flag[0] = true`
    
- Поток B: `flag[1] = true`
    
- Поток A: ждёт, пока `flag[1] == false`
    
- Поток B: ждёт, пока `flag[0] == false`
    
- **Результат:** Взаимоблокировка — оба потока никогда не выйдут из цикла.

**Пример:** Процесс A захватил ресурс 1, ждёт ресурс 2. Процесс B захватил ресурс 2, ждёт ресурс 1. Никто не может продолжить выполнение.

## 60. Необходимые условия возникновения тупика (4 условия Коффмана)


| Условие                                      | Смысл                                                                 | Как устранить                                                                                 |
| -------------------------------------------- | --------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| **1. Взаимное исключение**                   | Ресурс нельзя использовать совместно.                                 | Создать абстракцию для совместного использования (например, очередь печати).                  |
| **2. Удержание и ожидание (Hold and Wait)**  | Процесс удерживает ресурсы и ждёт другие.                             | Запрашивать все ресурсы сразу; освобождать все при новом запросе; использовать `try_acquire`. |
| **3. Отсутствие вытеснения (No Preemption)** | Ресурс нельзя отобрать принудительно.                                 | Разрешить отбирать ресурсы (требует сохранения контекста).                                    |
| **4. Циклическое ожидание (Circular Wait)**  | Существует цикл процессов, где каждый ждёт ресурс, занятый следующим. | Ввести порядок запроса ресурсов (например, по возрастанию номеров).                           |
## 61. Предотвращение тупиков

Нарушить **одно** из четырёх условий.

1. **Устранение условия "Взаимное исключение":**
    
    - Сделать ресурс разделяемым (например, очередь печати вместо самого принтера). Не всегда возможно

2. **Устранение условия "Удержание и ожидание" (Hold and Wait):**
    
    - **Вариант А:** Процесс должен запрашивать **все** необходимые ресурсы сразу, перед началом работы. Если все ресурсы недоступны, он не получает ничего и ждет.
        
    - **Вариант Б:** Если процессу нужен новый ресурс, он должен **освободить все** свои текущие ресурсы, а затем запросить их все заново вместе с новым.
        
    - **Проблема:** Неэффективное использование ресурсов (ресурсы могут быть заняты долго, даже если не используются) и риск голодания.
        
3. **Устранение условия "Отсутствие принудительного перераспределения":**
    
    - Позволить системе отбирать ресурсы у процессов (если состояние можно сохранить).
        
    - **Проблема:** Для большинства ресурсов (например, принтер, запись на диск) невозможно сохранить и восстановить контекст.
        
4. **Устранение условия "Циклическое ожидание":**
    
    - Ввести **нумерацию ресурсов**.
        
    - Обязать процессы запрашивать ресурсы **строго в порядке возрастания номеров**.
        
    - **Доказательство:** Если ресурсы запрашиваются в порядке возрастания, цикл невозможен (нарушается условие `R1 > R4 > R3 > ... > R1`).
        
    - **Проблема:** Не всегда возможно назначить порядок для всех типов ресурсов, и это также может привести к неэффективному использованию.


## 62. Избегание тупиков. Алгоритм банкира

**Идея:** Аккуратно распределять ресурсы, проверяя, останется ли система в **безопасном состоянии**.

**Безопасное состояние:**  существует некоторая последовательность выполнения процессов, при которой все они смогут завершиться, получив все необходимые ресурсы.

**Алгоритм банкира для одного типа ресурсов (пример из лекции):**

Когда процесс запрашивает ресурсы, система временно "предполагает", что запрос удовлетворен, и проверяет:

- "Будет ли новое состояние безопасным?"
- Если **ДА** — ресурсы выделяются.
- Если **НЕТ** — процесс блокируется до тех пор, пока выделение ресурсов не приведет к безопасному состоянию.

- Всего 12 единиц. Процессы: P0(макс=10, выделено=5), P1(4,2), P2(9,2). Свободно = 3.
    
- **Безопасная последовательность:** P1 (нужно 2, есть 3) → P0 (нужно 5, после P1 будет 5) → P2 (нужно 7, после P0 будет 10).
    
- **Запрос P2 на 1 единицу:** станет выделено 3, свободно 2. После P1 свободно 4, но ни P0 (нужно 5), ни P2 (нужно 6) не могут завершиться → **небезопасно**, запрос **отклоняется**.
## 63. Граф процесс-ресурс. Редукция графа

**Граф "Процесс-Ресурс" :**

- Вершины: процессы и ресурсы
    
- Дуги запроса: процесс → ресурс (процесс ждёт ресурс).
    
- Дуги распределения: ресурс → процесс (ресурс выделен процессу).
![[Pasted image 20260617224638.png]]

**Редукция графа:** Находим процесс, **все запросы которого могут быть удовлетворены** (ресурсы доступны). Удаляем процесс, все его дуги, его ресурсы возвращаются в пул свободных. Повторяем.

**Утверждения (из лекции):**

1. Если граф полностью редуцируем → **тупика нет**.
2. Если граф имеет цикл → тупик **возможен**, но не гарантирован (зависит от количества единиц ресурсов)
3. Если в графе остались вершины, которые нельзя редуцировать → **есть тупик**. Оставшиеся процессы находятся в тупике.

**Пример цикла без тупика (из лекции):** Процесс 1 запрашивает ресурс 3, есть цикл, но Процесс 3 можно редуцировать первым (он ничего не ждёт) → освобождается ресурс → тупика нет.
## 64. Устранение тупиков

Стратегия: допустить возникновение тупика, но затем его обнаружить и исправить.

**Обнаружение тупиков:**

- **С помощью графа процесс-ресурс:** Периодически строить граф и пытаться его редуцировать
    
- **Когда?**
    
    - При каждом запросе на ресурс (слишком дорого).
        
    - При запросе, который не может быть немедленно удовлетворен.
        
    - **Периодически** (например, раз в несколько секунд) — наиболее практичный подход.
        
    - По косвенным признакам (например, загрузка CPU упала ниже 40% — значит, процессы, возможно, заблокированы в тупике, а не выполняют полезную работу).


**Восстановление после тупика:**

- **Вариант 1: Уничтожение процессов.**
    
    - Убить один или несколько процессов, попавших в тупик.
        
    - Это самый распространенный подход.
        
    - Можно выбрать "жертву" (наименее затратный процесс). Пример из лекции: в СУБД при обнаружении тупика выбирается жертва (та, у которой самый короткий план выполнения), ее транзакция откатывается и перезапускается.
        
- **Вариант 2: Откат к контрольной точке.**
    
    - Периодически сохранять состояние процессов (контрольные точки).
        
    - При тупике откатить один или несколько процессов к последней контрольной точке и перезапустить их.
        
    - **Проблема:** При повторном запуске может снова возникнуть тот же тупик (если он детерминирован), но часто это помогает, так как меняется диаграмма выполнения потоков.
        
- **Вариант 3: Перераспределение ресурсов.**
    
    - Принудительно отобрать ресурсы у одного процесса и передать другому.
        
    - **Проблема:** Для большинства ресурсов это невозможно (невозможно сохранить и восстановить состояние принтера или записи на диск).ятью"

## 65. Схема с разделами фиксированного размера

При старте ОС распределяет ОЗУ на несколько разделов, один под ОС, остальная делится на разделы либо в соответствии с конфигурацией, которая хранится в файле, либо с конфигурацией, которая скомпилирована в ОС. Затем работает правило, одна программа - один раздел (один процесс - один раздел). При создании процесса, он может быть загружен в один из свободных разделов, если свободных разделов нет, то процесс не запускается. Есть ограничение на кол-во одновременно работающих процессов. Ограничение на размер памяти для программы (процесса). 

Два вида очередей. Слева к каждому разделу создается отдельная очередь, чтобы не запускать маленькие процессы в больших разделах. В правом общая очередь, при этом, если есть свободные разделы, но невозможно запустить первую программу в очереди, то пробую вторую, третью и т.д
Внутренняя фрагментация - память выделяется процессу блоками и процесс может не использовать выделенный блок целиком. Потери памяти в таком случае называются потерями из-за внутренней фрагментации, а сама эта проблема называется внутренняя фрагментация. 
Внешняя фрагментация - свободная память есть, но она представлена не в виде одного блока, а их набора и нет возможности использовать их, чтобы выполнить запрос какого-то процесса.

СВОПЛИНГ

Схема с фиксированными разделами допускает организацию выгрузки разделов на диск в случае необходимости.  
Выгруженный процесс может быть возвращен в то же самое адресное пространство или в другое — это ограничение диктуется методом связывания.

## 67. ВАП на основе страничного преобразования. Обработка страничного сбоя (блок-схема)
*(Лекция 7, раздел 3)*
**Основная идея:**  
Виртуальное адресное пространство (ВАП) процесса разбивается на блоки одинакового размера, называемые **страницами** (обычно 4 КБ). Оперативная память разбивается на блоки такого же размера — **физические страницы** (Page Frames). Таблица страниц отображает виртуальные страницы на физические.

**Ключевые элементы (ОС8):**

- **Таблица страниц:** Структура данных, которая хранит дескрипторы страниц. Каждый дескриптор содержит физический адрес страницы (или информацию о том, что страница на диске), а также биты защиты.
    
- **Страничный сбой (Page Fault):** Ситуация, когда страница есть в виртуальном адресном пространстве, но в данный момент отсутствует в оперативной памяти (например, выгружена на диск).
    
- **Многоуровневые таблицы:** Для 64-битных систем используются многоуровневые таблицы (2, 3, 4 уровня), чтобы не хранить огромную плоскую таблицу. Например, в 32-битной системе с размером страницы 4 КБ размер таблицы страниц составлял бы 4 МБ, а в 64-битной — неприемлемо много (512 ГБ), поэтому нужны уровни.
    
- **TLB (Translation Lookaside Buffer):** Кэш для дескрипторов страниц. Ускоряет преобразование адреса, так как без него каждое обращение к памяти требовало бы нескольких чтений из самой таблицы страниц.
    

**Преимущества страничной модели (ОС9):**

- Решает проблему внешней фрагментации (физическая память может быть разрывной).
    
- Позволяет организовать **разделяемую память** между процессами (разные ВАП отображаются на одни и те же физические страницы).
    
- Поддерживает механизм **Copy-on-Write** (копирование при записи), когда страницы кода и данных совместно используются процессами до момента первой записи.


**Блок-схема обработки page fault:**
1.  Аппаратное прерывание (page fault).
2.  Сохранение контекста.
3.  Определение виртуального адреса, вызвавшего сбой.
4.  Проверка: адрес принадлежит объявленному региону процесса?
    - **Нет** → Segmentation fault → убить процесс (SIGSEGV).
    - **Да** → Продолжить.
5.  Страница уже в памяти? (проверка PTE)
    - **Да** (другая ошибка, например, нарушение прав) → обработать.
    - **Нет** → Подкачка с диска.
6.  Найти свободный кадр в RAM. Если нет – выбрать жертву (алгоритм замещения), вытеснить на диск (если Dirty=1).
7.  Прочитать страницу с диска в кадр.
8.  Обновить PTE: Present=1, адрес кадра, сбросить Dirty.
9.  Перезапустить инструкцию, вызвавшую page fault.

## 68. Алгоритмы замещения страниц

**Классические алгоритмы:**

- **FIFO (First In, First Out):** Выгружается страница, которая загружена раньше всех. Прост в реализации, но страница может быть очень нужной («эффект Белади»: увеличение числа кадров может увеличить число сбоев).
    
- **LRU (Least Recently Used):** Выгружается страница, к которой дольше всего не обращались. Теоретически оптимален, но сложен в реализации (нужно отслеживать время каждого обращения или поддерживать список).
    
- **NFU (Not Frequently Used) / Алгоритм «Счетчика»:** Каждой странице присваивается счетчик, который увеличивается при каждом обращении. Выгружается страница с наименьшим счетчиком. Проблема: счетчики растут бесконечно, и информация устаревает.
    

**Усовершенствованные и практические алгоритмы:**

- **Second Chance (Второй шанс) / Алгоритм «Часов» (Clock):** Модификация FIFO, учитывающая бит обращения (Referenced). Если у страницы в начале очереди бит обращения = 1, она обнуляется и переносится в конец очереди (получает второй шанс). Если бит = 0, страница выгружается.
    
    - _«Алгоритм часов» (Clock):_ Кольцевой буфер. Указатель движется по кругу. Если у страницы бит = 1, обнуляем его и идем дальше; если 0 — выгружаем.
        
- **Biledy, Random:** Упоминаются, но не детализируются в тексте.

**4. Second Chance (Второй шанс)**

- Модификация FIFO, использующая бит обращения (Reference bit).
- Реализация: просматривается очередь FIFO. Если у головной страницы бит = 1, он сбрасывается в 0, страница перемещается в конец очереди (получает «второй шанс»). Если бит = 0 — страница выгружается.
    

**5. Clock (Алгоритм «Часов»)**

- Эффективная реализация Second Chance.
- Страницы организованы в кольцевой буфер. Указатель («стрелка часов») движется по кругу.
- При поиске жертвы проверяется бит обращения текущей страницы:
    
    - Бит = 1: сбросить бит, подвинуть указатель.
        
    - Бит = 0: выбрать эту страницу для выгрузки.
        
- Требует только одного бита на страницу и является стандартом в большинстве ОС.


- **Алгоритмы в Linux (из лекций):** В Linux используется вариант **LRU** (через список _active_ и _inactive_) и механизм **NFU** с периодическим делением счетчиков пополам, чтобы «старые» данные не забивали статистику. Ядро Linux (описывая алгоритм _goodness_ для планирования) также использует _aging_ для учета загрузки ЦП, что похоже на метод борьбы с устареванием счетчиков в NFU.

**Использование битов:**

- **Бит обращения (Reference bit / Использования):** Устанавливается аппаратно при любом обращении к странице. Используется для реализации LRU-подобных алгоритмов (Clock, Second Chance).
    
- **Бит изменения (Dirty bit / Модификации):** Устанавливается при записи в страницу. При выгрузке, если бит = 1, страницу нужно записать на диск; если 0 (не менялась) — можно просто удалить из памяти.

## 70. Структура жесткого диска, адресация
_(Лекция ОС14, раздел "Жёсткие диски")_

**Физическая структура:**

- **Пластины:** Жесткий диск состоит из нескольких пластин (дисков).
    
- **Поверхности:** Каждая пластина имеет 1 или 2 рабочих поверхности.
    
- **Дорожки (Tracks):** Концентрические окружности на поверхности.
    
- **Цилиндр (Cylinder):** Совокупность всех дорожек с одинаковым номером на разных поверхностях. Все головки, расположенные на одном цилиндре, могут читать данные без перемещения штанги.
    
- **Секторы (Sectors):** Минимальные адресуемые блоки данных на дорожке (обычно 512 байт или 4 КБ).
    

**Адресация:**

- **Физический адрес (CHS):** Тройка «Цилиндр — Головка — Сектор».
    
- **Линейная адресация (LBA):** Современный режим, где драйвер устройства предоставляет ОС просто массив блоков (блок 0, блок 1, ...). Преобразование CHS в LBA выполняется самим диском или его контроллером.

**Линеаризация (LBA — Logical Block Addressing):** Контроллер диска пересчитывает CHS в линейный номер блока. Вышележащие компоненты работают только с LBA.
## 71. Типы объектов ФС

В Unix определены следующие типы объектов (отображаются командой `ls -l` по первому символу):

1. **Файл (`-`):** Именованный набор данных.
    
2. **Каталог (`d`):** Контейнер для других объектов, обеспечивает иерархию.
    
3. **Файлы устройств:**
    
    - **Символьные устройства (`c`):** Потоковые (клавиатура, мышь). Не поддерживают произвольный доступ, нет размера. Содержат _major_ и _minor_ номера для идентификации драйвера и самого устройства.
        
    - **Блочные устройства (`b`):** Обеспечивают доступ к массиву блоков (жесткие диски).
        
4. **Именованный канал (Pipe, `p`):** Объект для межпроцессного взаимодействия.
    
5. **Сокет (`s`):** Интерфейс для сетевого или локального взаимодействия.
    
6. **Мягкая ссылка (Symbolic Link, `l`):** Файл, содержащий путь к другому объекту ФС. Может указывать на несуществующий объект (висячая ссылка). При открытии ОС перенаправляет запрос по указанному пути.

## 72. Концепция i-node

**Inode (index node):** Структура данных, содержащая **все характеристики объекта файловой системы, кроме его имени**.
   
- **Имя** хранится в каталоге, который связывает имя с номером `inode`.
    
- **Ссылки:**
    
    - **Жесткая ссылка (Hard Link):** Дополнительная запись в каталоге, ссылающаяся на тот же `inode`. Счетчик ссылок (`nlink`) в `inode` увеличивается.
        
    - **Удаление файла:** Функция называется `unlink()`. Удаляется запись в каталоге. Если счетчик ссылок стал `0`, данные и `inode` физически удаляются с диска.

**Атрибуты inode:**

- Тип объекта (файл, каталог, устройство, pipe, сокет, символьная ссылка).
    
- Пользователь-владелец (UID) и группа-владелец (GID).
    
- Права доступа (12 бит: `rwx rwx rwx` + доп. биты).
    
- Три времени: последнего доступа (atime), изменения данных (mtime), изменения метаданных (ctime).
    
- Счётчик жёстких ссылок.
    
- Размер (для файла — в байтах; для каталога — объём, выделенный под таблицу).
    
- Указатели на блоки данных (13 адресов в классической Unix).
    
- Для файлов устройств: старший (major) и младший (minor) номера.
    
- Для символьных ссылок: путь (может храниться в самом inode для коротких путей).
    

**Номер inode не хранится в самом inode** — это индекс в таблице inode.

## 73. Структура UNIX FS. Атрибуты суперблока. Атрибуты i-node
*(Лекция 4, раздел "Структура UNIX FS")*
_(Лекция ОС14, разделы "Внутренняя структура ФС" и "Суперблок")_

**Структура UNIX FS (классическая):**
- **Суперблок (Superblock):** атрибуты всей ФС. размер ФС, количество inode, количество блоков, список свободных inode, список свободных блоков, флаг чистоты.
- **Массив inode (inode table):** все i-node ФС.
- **Блоки данных (data blocks):** содержимое файлов и каталогов.

**Атрибуты i-node:** тип, права, число ссылок, UID, GID, размер, временные метки, указатели на блоки.

**Атрибуты суперблока:**

|Атрибут|Смысл|
|---|---|
|Идентификатор ФС|Уникальный ID (раньше — имя блочного устройства, теперь — внутренний).|
|Тип ФС|ext4, XFS и т.д.|
|Размер блока данных (кластера)|Задаётся при создании ФС.|
|Размер ФС|В блоках.|
|Размер таблицы inode|Сколько блоков отведено под inode.|
|Битовый массив свободных inode|Для быстрого поиска.|
|Битовый массив свободных блоков|Для быстрого поиска.|
|Номер inode корневого каталога|С него начинается обход дерева (обычно 2).|
|Флаг `mounted`|ФС примонтирована; при отмонтировании — сброс кэша на диск.|

**Важное соотношение:** Классика — **1 inode на 2 блока данных**. Можно изменить при форматировании. Если создавать много объектов без данных (файлы устройств, пустые каталоги), можно **исчерпать inode**, но свободное место ещё останется.

## 74. Хранение информации о размещении данных файла в UNIX FS
*(Лекция 4, раздел "Хранение информации о размещении данных файла в UNIX FS")*

**Механизм:** В i-node хранится массив указателей на блоки данных:
- **12 прямых указателей** (direct) – на первые 12 блоков файла.
- **1 косвенный указатель** (indirect) – на блок, содержащий указатели на блоки данных.
- **1 двойной косвенный** (double indirect) – на блок, содержащий указатели на блоки косвенных указателей.
- **1 тройной косвенный** (triple indirect) – для очень больших файлов.

Позволяет хранить большие файлы при небольшом размере i-node.

_(Лекция ОС14, раздел "Inode: 13 адресов")_

**Классическая Unix (размер блока = 512 байт, адрес = 4 байта):**

|Тип адресации|Индекс в массиве|Максимальный размер|
|---|---|---|
|Прямые адреса|0..9|10 × 512 = **5 КБ**|
|Одиночная косвенная|10|128 × 512 = **64 КБ** (128 адресов в блоке)|
|Двойная косвенная|11|128² × 512 = **8 МБ**|
|Тройная косвенная|12|128³ × 512 = **1 ГБ**|

**Итого:** максимальный размер файла ≈ 1 ГБ + 8 МБ + 64 КБ + 5 КБ.

**Если увеличить размер блока в 2 раза (до 1 КБ):**

- В блоке помещается 256 адресов (в 2 раза больше).
    
- Максимальный размер вырастает в **16 раз** (до ~16 ГБ), потому что:
    
    - Прямые адреса: ×2
        
    - Одиночная косвенная: ×4
        
    - Двойная косвенная: ×8
        
    - Тройная косвенная: ×16
## 75. Структура каталога в UNIX FS
*(Лекция 4, раздел "Структура каталога в UNIX FS")*

Каталог – это файл, содержащий массив записей (directory entries). Каждая запись содержит: **имя файла** (или ссылку) и **номер i-node** (inode number). Имя и i-node разделены – это позволяет создавать жёсткие ссылки (несколько имён на один i-node).

_(Лекция ОС14, раздел "Структура каталога")_

**Каталог — это таблица записей.** В оригинальной Unix (32-битной) запись состоит из 4 полей:

|Поле|Размер (байт)|Смысл|
|---|---|---|
|`size`|4|размер всей записи (включая все поля)|
|`name_len`|4|длина имени (без завершающего нуля)|
|`inode_num`|4|номер inode|
|`name`|переменная|имя + завершающий нуль (C-строка), выровнено до 4 байт|

**Специальные записи в любом каталоге:**

- `"."` (точка) — ссылается на сам каталог.
    
- `".."` (две точки) — ссылается на родительский каталог.
    

**Операции над каталогом:**

- **Добавление:** ищется свободное пространство в конце последней записи (запись имеет поле `size` больше своего заголовка). Если не влезает — выделяется новый блок (кластер).
    
- **Удаление:** предыдущая запись расширяется, "поглощая" удаляемую область как свободное пространство. Точка и две точки никогда не удаляются, поэтому перед удаляемой записью всегда есть предыдущая запись.

## 76. Структуры, используемые ядром для обеспечения доступа процессов к файлам. Атрибуты записей. Операции open() и close()
*(Лекция 4, раздел "Структуры, используемые ядром...")*

**Три уровня структур:**
1.  **Таблица дескрипторов процесса (Process Descriptor Table):** у каждого процесса своя. Каждая запись – указатель на запись в таблице открытых файлов.
2.  **Таблица открытых файлов (Open File Table):** общесистемная (или на процесс). Содержит: текущую позицию в файле, флаги открытия (чтение/запись), указатель на объект v-node/i-node, счётчик ссылок.
3.  **v-node / i-node:** описывает сам файл (права, размер, указатели на блоки).

**open():** по имени файла находит i-node, создаёт запись в таблице открытых файлов, создаёт дескриптор в таблице процесса, возвращает дескриптор.

**close():** уменьшает счётчик ссылок в таблице открытых файлов; если стал 0, закрывает запись.

_(Лекция ОС14, раздел "Структуры для доступа к файлам")_

**Три уровня структур (общесистемные + локальные для процесса):**

1. **Пользовательская таблица дескрипторов (FDT)** — у каждого процесса своя. Индекс в этой таблице — дескриптор (то, что возвращает `open`). Запись содержит указатель на запись в таблице файлов.
    
2. **Общесистемная таблица файлов (file table)** — содержит **контекст файлового ввода-вывода**:
    
    - Режим открытия (чтение/запись/добавление).
        
    - Текущая позиция в файле.
        
    - Код завершения последней операции.
        
    - Признак конца файла (EOF).
        
    - Ссылка на запись в таблице inode.
        
    - Счётчик ссылок из дескрипторов.
        
3. **Общесистемная таблица inode (кэш inode)** — закешированные inode с диска:
    
    - Идентификатор ФС и номер inode.
        
    - Атрибуты inode.
        
    - Счётчик ссылок из таблицы файлов.
        

**`open(path, flags)` — пошагово:**

1. Поиск inode по имени (проход по каталогам).
    
2. Найти или загрузить inode в кэш (увеличить счётчик в таблице inode).
    
3. Создать запись в таблице файлов (инициализировать контекст, установить счётчик = 1).
    
4. Создать запись в FDT процесса, указать на запись таблицы файлов.
    
5. Вернуть индекс записи в FDT (дескриптор).
    

**`close(fd)` — пошагово:**

1. Удалить запись из FDT.
    
2. Уменьшить счётчик в записи таблицы файлов. Если стал 0 → удалить запись таблицы файлов.
    
3. Уменьшить счётчик в записи inode. Если стал 0 → запись inode **остаётся в кэше** (она имеет собственную ценность — закешированные данные с диска).
    

**Связь `fork` и `dup`:**

- `fork()` — дочерний процесс получает **копию** FDT, но записи указывают на **те же** записи таблицы файлов (разделяют контекст, позицию в файле).
    
- `dup()` / `dup2()` — создаётся ещё один дескриптор в том же процессе, указывающий на ту же запись таблицы файлов.

## 77. Fast File System (FFS)

_(Лекция ОС14, конец)_

1. **Группы цилиндров (Cylinder Groups):**
    
    - Диск делится на группы блоков. Суперблок, битмапы, `inode` и данные одной группы хранятся рядом.
        
    - **Правила:**
        
        - Блоки данных одного файла лежат в одной группе.
            
        - Файлы одного каталога — в одной группе.
            
        - `inode` файла лежит в той же группе, что и его данные.
            
    - _Результат:_ Сокращается перемещение головок (в пределах одного "кольца" диска).
        
2. **Увеличение размера блока (до 4 КБ):**
    
    - Резко увеличивает максимальный размер файла (с 1 ГБ до ~4 ТБ).
        
    - _Проблема:_ Внутренняя фрагментация (средний размер файла в то время был ~1 КБ).
        
3. **Фрагменты (Fragments):**
    
    - Решение проблемы фрагментации. Блок (4 КБ) делится на фрагменты (обычно 1 КБ).
        
    - Хвост файла (менее 4 КБ) может занимать только необходимое количество фрагментов. Остальные фрагменты блока могут использоваться другими файлами.
        
4. **Учет характеристик железа:** Размещение блоков с пропусками ("через сектор"), чтобы ЦП успевал обрабатывать данные, не отставая от скорости вращения диска.

|Улучшение|Суть|Цель|
|---|---|---|
|**1. Группы цилиндров (cylinder groups)**|Диск разбит на группы (кольца). Правила: блоки одного файла — в одной группе; файлы одного каталога — в одной группе; inode файла — в той же группе, что и его данные.|Минимизировать перемещения головки при работе с одним каталогом.|
|**2. Увеличение размера блока до 4 КБ**|Блок = 4 КБ вместо 512 байт.|Увеличить максимальный размер файла (до ~4 ТБ) и уменьшить количество управляющих структур.|
|**3. Фрагменты (fragments)**|Кластер (4 КБ) делится на фрагменты (по 1 КБ). Если хвост файла < 3 КБ, он хранится не в целом кластере, а в нескольких фрагментах. Остальные фрагменты того же кластера могут быть отданы под хвосты других файлов.|Борьба с внутренней фрагментацией (средний размер файла в раннем Unix был ~1 КБ).|
|**4. Учёт характеристик аппаратуры**|Размещение данных на дорожке через сектор (пропуская один), если CPU медленный.|Дать CPU время на обработку данных, пока диск подводит следующий сектор.|

**Современное состояние (из лекции):** Фрагменты по умолчанию не используются (`fragment size = block size`), так как диски большие и потеря 4 КБ на файл не критична.
