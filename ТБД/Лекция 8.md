## 1. Повторение: ER-модель (сущность-связь)

**ER-модель** (Entity‑Relationship) — описание предметной области, независимое от конкретной СУБД.  
Состоит из двух типов объектов:

- **Сущность** (entity) — объект, про который нужно хранить данные (факультет, студент, преподаватель).
- **Связь** (relationship) — ассоциация между сущностями.

Типы связей:
- один ко многим (1:N) — самый частый
- один к одному (1:1)
- многие ко многим (M:N) — **нереализуема напрямую** в реляционной модели.

### Как реализовать M:N
Создаётся **сегмент связи** (промежуточная сущность), например `AB`. Каждый её экземпляр — факт наличия связи.  
На практике в таком сегменте часто появляются собственные атрибуты (вид занятий, опыт, семестр и т.п.).

---

## 2. Пример ER-модели университета (основные сущности)

- **Факультет/институт** — глобальный код.
- **Кафедра** — глобальный код, принадлежит одному факультету.
- **Преподаватель** — работает только на одной кафедре.
- **Специальность** — может повторяться в университете (например, ФИИТ на двух факультетах). Код — общероссийский.
- **Программа** (учебный план) — внутри специальности, свой идентификатор.
- **Предмет (курс)** — идентифицируется внутри программы. Одинаковые названия — разные дисциплины (разные часы, семестры, содержание).
- **Студенческая группа** — привязана к программе.
- **Студент** — экземпляр.
- **Вид работы** (лекции, практики, лабораторные, консультации).
- **Занятие (lesson)** — клетка расписания: предмет, группа, вид работы, преподаватель (аудитория опущена для простоты).

> Важно: если теоретически возможна повторяемость (специальность, код предмета), проектировщик обязан её предусмотреть. Иначе переделка всей схемы.

## 3. Алгоритм преобразования ER-диаграммы в реляционные таблицы

Алгоритм (из западной книги, теоретически обоснован):

1. **Каждый прямоугольник (сущность) → реляционная таблица**.
2. На каждом шаге берётся сегмент, который:
   - ещё не пройден;
   - **не содержит множественной связи (M:N) из непройденных сегментов**.
   - (Если такого нет — в графе есть цикл; тогда алгоритм не применим напрямую.)
3. Формируем таблицу из:
   - **потенциальных ключей** (подчёркнуты на ER-диаграмме);
   - **информационных атрибутов** (название, адрес и т.п.).
4. **Добавляем** потенциальные ключи от всех сегментов, из которых в текущий идёт **множественная связь** (даже если получится избыточность — потом уберём).

## 4. Пошаговый пример (университет)

Порядок обхода (от корневых к зависимым):

### 4.1. Факультет
- Первичный ключ: `id_faculty`
- Информационные атрибуты: название, адрес и пр.

### 4.2. Кафедра (зависит от факультета)
- Добавляем `id_department` (глобальный) → первичный ключ.
- Добавляем `id_faculty` (из факультета).
- Информационные атрибуты кафедры.

### 4.3. Преподаватель
- `id_teacher` (глобальный) → первичный ключ.
- Добавляем `id_department` (из кафедры).
- Данные преподавателя.

### 4.4. Специальность
- Потенциальный ключ: `id_specialty` (общероссийский) **+** `id_faculty` (из-за возможного повтора).
- Добавляем `id_faculty` (множественная связь от факультета).

### 4.5. Программа
- Потенциальный ключ: `id_program` + `id_specialty` + `id_faculty` (из-за повторяемости).
- Добавляем `id_specialty`, `id_faculty`.

### 4.6. Предмет (курс)
- Потенциальный ключ: `id_subject` + `id_program` + ... (идентификация внутри программы).
- Добавляем `id_program`, а также `id_department` (от кафедры — множественная связь).

### 4.7. Студенческая группа
- `id_group` (глобальный) → первичный ключ.
- Добавляем `id_program`, `id_specialty`, `id_faculty`.

### 4.8. Студент
- `id_student` (глобальный, номер студенческого) → первичный ключ.
- Добавляем `id_group` (и все вышестоящие ключи — временно).

### 4.9. Вид работы
- `id_work_type` → первичный ключ.

### 4.10. Занятие (lesson)
- Потенциальный ключ — составной (всё, что определяет клетку расписания).
- Добавляем `id_teacher`, `id_subject`, `id_group`, `id_work_type`.

## 5. Анализ и удаление избыточности (формирование первичных ключей)

**Принцип**: если атрибут может быть получен через `JOIN` из другой таблицы, его удаляем, чтобы избежать аномалий.

- **Факультет**: `id_faculty` — PK.
- **Кафедра**: `id_department` — PK; `id_faculty` — обычный атрибут (не PK).
- **Преподаватель**: `id_teacher` — PK; `id_department` — обычный атрибут; `id_faculty` — **удаляем** (восстанавливается через кафедру).
- **Специальность**: составной PK = (`id_faculty`, `id_specialty`) — из-за возможного повтора.
- **Программа**: составной PK = (`id_faculty`, `id_specialty`, `id_program`).
- **Предмет**: составной PK = (все вышестоящие + `id_subject`).
- **Группа**: `id_group` — PK; остальные (`id_faculty`, `id_specialty`, `id_program`) — обычные атрибуты (восстанавливаются через программу).
- **Студент**: `id_student` — PK; остаётся только `id_group` (остальное — через JOIN).
- **Вид работы**: `id_work_type` — PK.
- **Занятие**: оставляем `id_teacher`, `id_subject`, `id_group`, `id_work_type`. Остальное — через связи.

**Результат**: структура таблиц находится в **третьей нормальной форме (3НФ)**.  
> Важно: в БД нет «правильно/неправильно», есть работоспособность и корректность данных. Нормализация — рекомендация, а не догма.

## 6. Реляционная алгебра (теория)

Основа — теория множеств (Эдгар Кодд).

- **Домен** (`D₁, D₂, …`) — множество допустимых значений атрибута.
- **Отношение (таблица)** — подмножество декартова произведения доменов.
- **Кортеж (tuple)** — одна строка таблицы.
- **n-арное отношение** — таблица с n атрибутами.

### 8 фундаментальных операций реляционной алгебры

| Операция | Смысл |
|----------|-------|
| **Селекция (σ)** | выбор строк по условию (фильтр) |
| **Проекция (π)** | выбор столбцов + удаление дубликатов кортежей |
| **Объединение (∪)** | строки из двух таблиц (без дублей) |
| **Разность (−)** | строки из первой, которых нет во второй |
| **Пересечение (∩)** | строки, общие для двух таблиц |
| **Декартово произведение (×)** | все комбинации строк |
| **Соединение (⋈, JOIN)** | соединение по равенству (equijoin), естественное (natural join), левое, правое, полное (outer) |
| **Деление (÷)** | теоретическая операция для некоторых запросов |

> Пример обозначения:  
> `σ_{salary > 50000}(Employees)` — выбрать сотрудников с з/п > 50000.  
> `π_{department}(Employees)` — получить список отделов (уникальные).

## 7. Оптимизация запросов (кратко)

- **Стоимость запроса** — оценка времени/ресурсов.
- **Оптимизатор** СУБД строит **план выполнения** (evaluation plan) из операций реляционной алгебры.
- **Эквивалентные преобразования** (правила):
  - Селекция коммутативна: `σ₁(σ₂(R)) = σ₂(σ₁(R))`.
  - Селекцию можно проталкивать внутрь JOIN.
  - Проекции можно объединять: `πₐ(πᵦ(R)) = πₐ(R)` (если a ⊆ β).
  - и другие (всего 9 основных правил).

> Современные СУБД (Postgres и др.) оптимизируют запросы автоматически. Ручная оптимизация нужна в редких случаях (огромные таблицы, слабое оборудование).

## 8. Итог лекции

- ER-диаграмма → набор таблиц по алгоритму → 3НФ.
- Ключи бывают составными, если возможна повторяемость.
- Реляционная алгебра — математическая основа SQL.
- Оптимизатор использует эквивалентные преобразования для ускорения запросов.
- Главный критерий хорошей БД — **правильность данных**, а не «правильность» по учебнику.
