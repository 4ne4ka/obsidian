## Конспект лекции: Аналитические функции (оконные функции) в SQL

### 1. Организационные моменты
- В следующую пятницу лекции не будет (преподаватель в командировке).
- Лекция будет восстановлена позже (возможно, две лекции подряд через пятницу).
- Пропущенные видео с прошлых лекций уже восстановлены.

---

### 2. Новый раздел: «Профессиональный уровень»

**О чём раздел:**
- Техническое использование SQL на профессиональном уровне.
- Переход к более сложной теме: аналитика и базы данных в аналитике (частично в этом семестре, продолжение в следующем году на программной инженерии).

---

### 3. Что такое аналитические функции (Window Functions)

**История:**
- Относительно недавнее приобретение языка SQL (примерно 10–15 лет назад).
- Появились из-за роста задач в аналитике.
- Официальное название: **window functions** (в документации Oracle, Microsoft, Google Server).
- «Аналитические функции» — неофициальное, но прижившееся название.

**Важно:**
- Всё, что делают аналитические функции, можно сделать стандартными средствами SQL (подзапросы, сложные JOIN).
- Но аналитические функции позволяют это сделать **короче, проще и значительно быстрее** (пример: запрос на 16 минут против 0.4 секунды).

**Принципиальное отличие от обычного SELECT:**
- Обычный SELECT анализирует **только текущую строку**.
- Чтобы получить данные из другой строки, нужен подзапрос.
- Оконные функции позволяют формировать вычисления через **множество строк, связанных с текущей**.

**Пример задачи:**
Найти студентов, которые получили двойку на экзамене, а затем на пересдаче — пятёрку.

---

### 4. Синтаксис оконных функций

```sql
имя_функции(аргументы) OVER (параметры)
```

- Скобки после функции **обязательны**.
- Ключевое слово `OVER` (поверх чего-то) — обязательно.
- Скобки после `OVER` **обязательны**, даже если пустые.

**Порядок вычисления:**
- Оконные функции вычисляются **в самом конце**, после того как все колонки уже получены.

---

### 5. Учебная база данных «Телеком» (биллинг)

**Назначение:** подсчёт денег за связь (упрощённая, но похожа на реальную).

**Три основные таблицы:**

#### 5.1. Таблица тарифов
- Код тарифа
- Наименование тарифа
- Ежемесячная плата
- Количество минут в пакете
- Стоимость минуты сверх пакета
- Примечание: старые тарифы помечены (нельзя перейти, но они работают)

#### 5.2. Таблица абонентов (subscribers)
- Поля: имя, пол, рост, зарплата (в условных единицах), код тарифа (используется как код подразделения)

#### 5.3. Таблица звонков (фактов)
- ID звонка (автоматическая генерация)
- Кто звонил
- Когда звонил (timestamp)
- Продолжительность (в минутах, для простоты)

#### Представление (view) `billing`
- Содержит: количество минут, среднюю продолжительность, количество звонков.
- Можно использовать как обычную таблицу.

### 6. Первый пример: средняя зарплата

**Задача:** сравнить зарплату человека со средней по всем.

```sql
SELECT 
    *,
    AVG(salary) OVER () AS avg_salary,
    salary - AVG(salary) OVER () AS diff_from_avg
FROM subscribers;
```

**Пояснение:**
- `AVG(salary) OVER ()` — средняя по **всем** строкам.
- Без `OVER` — это обычная агрегатная функция (вернёт одну строку).
- С `OVER` — окно = все строки.

**Результат:** у каждой строки появляется колонка со средней зарплатой и разницей.

---

### 7. Разбиение на группы (PARTITION BY)

**Задача:** средняя зарплата внутри каждого подразделения (кода тарифа).

```sql
SELECT 
    *,
    AVG(salary) OVER (PARTITION BY tariff_code) AS avg_by_dept
FROM subscribers
ORDER BY tariff_code, last_name;
```

**Пояснение:**
- `PARTITION BY` делит строки на группы.
- Средняя считается **отдельно для каждой группы**.

---

### 8. Ранжирование: RANK()

**Задача:** определить место сотрудника по зарплате (как в спортивном соревновании).

```sql
SELECT 
    *,
    RANK() OVER (ORDER BY salary DESC) AS place
FROM subscribers;
```

**Особенности:**
- Если зарплаты одинаковые — они делят место (1, 2, 2, 4...).
- `ORDER BY` внутри `OVER` задаёт порядок ранжирования.

**Совместно с PARTITION BY:**
```sql
RANK() OVER (PARTITION BY tariff_code ORDER BY salary DESC)
```
Ранг считается внутри каждого подразделения.

---

### 9. Накопительный итог (сортировка внутри окна)

```sql
SELECT 
    *,
    SUM(salary) OVER (ORDER BY salary) AS running_sum
FROM subscribers;
```

**Эффект:** сумма нарастает с каждой строкой в порядке возрастания зарплаты.

---

### 10. Функции доступа к соседним строкам: LAG и LEAD

#### LAG — предыдущее значение

```sql
LAG(column, offset, default) OVER (ORDER BY ...)
```

**Пример:** показать предыдущую зарплату и разницу.

```sql
SELECT 
    last_name,
    salary,
    LAG(salary, 1, 0) OVER (ORDER BY salary DESC) AS prev_salary,
    salary - LAG(salary, 1, 0) OVER (ORDER BY salary DESC) AS diff
FROM subscribers;
```

- `1` — шаг назад.
- `0` — значение по умолчанию, если нет предыдущей строки.

#### LEAD — следующее значение (аналогично)

```sql
LEAD(salary, 1, 0) OVER (ORDER BY salary DESC)
```

**Применение:** выявление подозрений на мошенничество (фрод) — например, резкий рост активности.

### 11. Другие важные оконные функции

| Функция | Назначение |
|---------|-------------|
| `ROW_NUMBER()` | Номер строки в окне (без пропусков, уникальный) |
| `DENSE_RANK()` | Как RANK, но без пропусков (1,2,2,3) |
| `NTILE(n)` | Делит строки на n групп (квантили) |
| `FIRST_VALUE()` | Первое значение в окне |
| `LAST_VALUE()` | Последнее значение в окне |
| `NTH_VALUE(column, n)` | n-е значение в окне |

**Пример с NTILE (квартили):**
```sql
NTILE(4) OVER (ORDER BY salary DESC)
```
Делит на 4 группы по 25% строк.

**Пример с ROW_NUMBER:**
```sql
ROW_NUMBER() OVER (ORDER BY last_name)
```
Просто нумерует строки в заданном порядке.

### 12. Пример: доля зарплаты сотрудника в отделе

```sql
SELECT 
    last_name,
    tariff_code,
    salary,
    SUM(salary) OVER (PARTITION BY tariff_code) AS sum_in_dept,
    (salary * 100.0 / SUM(salary) OVER (PARTITION BY tariff_code)) AS percent_in_dept
FROM subscribers
ORDER BY tariff_code, last_name;
```

**Пояснение:**
- Считаем сумму по отделу.
- Делим зарплату сотрудника на эту сумму, умножаем на 100.
- Получаем процент зарплаты сотрудника внутри своего отдела.

---

### 13. Почему стоит использовать оконные функции (вывод)

1. **Короче и понятнее** — один `OVER` вместо сложных подзапросов.
2. **Быстрее** — оптимизатор хорошо умеет их ускорять.
3. **Пример из реальной жизни:**
   - Подзапрос на 250 тыс. звонков работал **16 минут**.
   - Тот же результат через оконные функции и JOIN — **0.4 секунды**.
4. Оконные функции вычисляются в конце, когда все данные уже готовы, и могут работать с уже вычисленными колонками.

---

### 14. Общее замечание

- В `OVER` можно использовать `PARTITION BY` и `ORDER BY`.
- Можно задать окно явно: `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` (от начала до текущей строки) и т.д.
- Конструкцию `OVER (...)` можно вынести в `WINDOW` для повторного использования (в одном запросе).
