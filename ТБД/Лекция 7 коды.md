Ниже приведены **практические примеры SQL-кода** на основе описанной в лекции базы данных деканата. Все примеры реалистичны, с объяснениями.

---

## 1. Создание таблиц (DDL)

```sql
-- 1. Таблица учебных дисциплин
CREATE TABLE Subjects (
    subject_id   INT PRIMARY KEY,
    name         VARCHAR(100) NOT NULL,
    credits      INT CHECK (credits BETWEEN 1 AND 12)
);

-- 2. Таблица преподавателей
CREATE TABLE Lecturers (
    lecturer_id   INT PRIMARY KEY,
    first_name    VARCHAR(50),
    last_name     VARCHAR(50),
    department_id INT,
    salary        DECIMAL(10,2)
);

-- 3. Таблица студентов (с внешним ключом на преподавателя-руководителя)
CREATE TABLE Students (
    student_id      INT PRIMARY KEY,
    first_name      VARCHAR(50),
    last_name       VARCHAR(50),
    birth_date      DATE,
    mentor_id       INT,
    FOREIGN KEY (mentor_id) REFERENCES Lecturers(lecturer_id)
);

-- 4. Таблица попыток сдачи экзаменов (составной первичный ключ)
CREATE TABLE ExamAttempts (
    student_id      INT,
    subject_id      INT,
    exam_date       DATE,
    examiner_id     INT,
    grade           INT CHECK (grade IN (0,2,3,4,5)),  -- 0 = неявка
    PRIMARY KEY (student_id, subject_id, exam_date),
    FOREIGN KEY (student_id) REFERENCES Students(student_id),
    FOREIGN KEY (subject_id) REFERENCES Subjects(subject_id),
    FOREIGN KEY (examiner_id) REFERENCES Lecturers(lecturer_id)
);
```

> **Пояснение:**  
> - `PRIMARY KEY` с несколькими полями обеспечивает уникальность попытки.  
> - `CHECK` ограничивает оценки.  
> - Внешние ключи поддерживают ссылочную целостность.

## 2. Вставка тестовых данных

```sql
-- Добавляем дисциплины
INSERT INTO Subjects VALUES
(101, 'Математический анализ', 6),
(102, 'Базы данных', 5);

-- Добавляем преподавателей
INSERT INTO Lecturers VALUES
(1001, 'Альберт', 'Шиттл', 10, 120000),
(1002, 'Николай', 'Юрьевич', 20, 95000);

-- Добавляем студентов (mentor_id = 1001)
INSERT INTO Students VALUES
(20001, 'Иван', 'Петров', '2000-01-15', 1001),
(20002, 'Мария', 'Сидорова', '2001-03-22', 1001);

-- Добавляем попытки экзаменов
INSERT INTO ExamAttempts VALUES
(20001, 101, '2024-01-20', 1002, 5),   -- сдал на 5
(20001, 102, '2024-01-22', 1001, 4),   -- сдал на 4
(20002, 101, '2024-01-20', 1002, 2),   -- получил 2
(20002, 101, '2024-02-10', 1002, 3);   -- пересдача
```

## 3. Простые SELECT-запросы

```sql
SELECT student_id, first_name, last_name, birth_date
FROM Students;
```

### 3.2. Преподаватели с зарплатой > 100 000

```sql
SELECT last_name, first_name, salary
```

## 4. DISTINCT — удаление дубликатов (упоминание в лекции)

> **Задача:** вывести список преподавателей, которые поставили хотя бы одну двойку.

```sql
SELECT DISTINCT examiner_id
FROM ExamAttempts
WHERE grade = 2;
```

Результат:  
| examiner_id |
|-------------|
| 1002        |

Если добавить фамилию (через JOIN — см. ниже), то повторяющиеся фамилии исчезнут.

---

## 5. JOIN — денормализация для отчётов

### 5.1. Кто и какую двойку поставил (с фамилиями)

```sql
SELECT 
    l.last_name AS преподаватель,
    s.last_name AS студент,
    sub.name AS предмет,
    e.grade
FROM ExamAttempts e
JOIN Lecturers l ON e.examiner_id = l.lecturer_id
JOIN Students s ON e.student_id = s.student_id
JOIN Subjects sub ON e.subject_id = sub.subject_id
WHERE e.grade = 2;
```

**Результат (денормализованный, удобный для человека):**

| преподаватель | студент   | предмет                | grade |
|---------------|-----------|------------------------|-------|
| Юрьевич       | Сидорова  | Математический анализ  | 2     |

> Это именно то, о чём говорилось в лекции: внутри хранятся ID, а пользователю показываем фамилии через JOIN.

### 5.2. Список студентов с их научными руководителями

```sql
SELECT 
    s.first_name AS student_first,
    s.last_name AS student_last,
    l.first_name AS mentor_first,
    l.last_name AS mentor_last
FROM Students s
LEFT JOIN Lecturers l ON s.mentor_id = l.lecturer_id;
```

`LEFT JOIN` нужен на случай, если у студента ещё нет руководителя (поле mentor_id = NULL).

## 6. Агрегация и GROUP BY

### 6.1. Количество двоек у каждого преподавателя

```sql
SELECT 
    examiner_id,
    COUNT(*) AS double_count
FROM ExamAttempts
WHERE grade = 2
GROUP BY examiner_id;
```

### 6.2. Средний балл студента

```sql
SELECT 
    student_id,
    AVG(grade) AS avg_grade
FROM ExamAttempts
WHERE grade > 0   -- исключаем неявки
GROUP BY student_id;
```

## 7. Представления (VIEW) — «постоянная денормализация»

```sql
CREATE VIEW StudentGradesReport AS
SELECT 
    s.last_name AS student,
    sub.name AS subject,
    e.exam_date,
    l.last_name AS examiner,
    e.grade
FROM ExamAttempts e
JOIN Students s ON e.student_id = s.student_id
JOIN Subjects sub ON e.subject_id = sub.subject_id
JOIN Lecturers l ON e.examiner_id = l.lecturer_id;
```

После создания можно использовать как таблицу:

```sql
SELECT * FROM StudentGradesReport WHERE grade = 5;
```

> VIEW не нарушает нормализацию внутри — это просто сохранённый запрос.

## 8. Пример транзакции (исправление ошибки)

> Из лекции: проблема рассинхрона названий дисциплин.

```sql
BEGIN TRANSACTION;

-- Меняем название предмета
UPDATE Subjects 
SET name = 'Математический анализ (углублённый)'
WHERE subject_id = 101;

-- Если всё правильно
COMMIT;

-- В случае ошибки
ROLLBACK;
```

## 9. Удаление неполной зависимости — пример декомпозиции (теория на практике)

**Было (плохо):** одна таблица `BadExamAttempts` с названием предмета.

```sql
-- Создаём правильные таблицы (уже есть Subjects)
CREATE TABLE GoodExamAttempts (
    student_id INT,
    subject_id INT,
    exam_date DATE,
    grade INT,
    PRIMARY KEY (student_id, subject_id, exam_date),
    FOREIGN KEY (subject_id) REFERENCES Subjects(subject_id)
);
```

Теперь название предмета хранится **один раз** в Subjects.

## 10. Функциональная зависимость в запросе (иллюстрация)

Запрос, где неключевое поле зависит от части ключа (в оригинальной плохой схеме):

```sql
-- Если бы название было в ExamAttempts, то:
SELECT student_id, exam_date, name
FROM BadExamAttempts
WHERE subject_id = 101;
-- название повторялось бы много раз
```

После нормализации такого не происходит.

## Итог по коду

| Концепция                 | SQL-механизм                              |
|---------------------------|-------------------------------------------|
| Первичный ключ            | `PRIMARY KEY`                             |
| Внешний ключ              | `FOREIGN KEY`                             |
| Атомарность (1NF)         | нет списков/массивов в колонках           |
| Удаление дубликатов       | `SELECT DISTINCT`                         |
| Денормализация для отчёта | `JOIN`                                    |
| Постоянный отчёт          | `CREATE VIEW`                             |
| Транзакционная целостность| `BEGIN TRANSACTION` / `COMMIT` / `ROLLBACK` |
