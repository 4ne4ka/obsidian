Ниже приведён **пример SQL-кода** для создания нескольких таблиц, полученных в результате алгоритма из лекции (университетская БД).  
Код написан для **PostgreSQL** (стандартный SQL), с пояснениями по каждому элементу.

---

## 1. Таблица «Факультет»

```sql
CREATE TABLE faculty (
    id_faculty   INTEGER PRIMARY KEY,      -- глобальный код факультета
    name         VARCHAR(200) NOT NULL,    -- название факультета
    address      VARCHAR(300),             -- адрес (может быть NULL)
    phone        VARCHAR(20)               -- контактный телефон
);
```

**Пояснения**:
- `PRIMARY KEY` — уникальный идентификатор, не может быть NULL.
- `VARCHAR(n)` — строка переменной длины.
- `NOT NULL` — обязательное поле.

---

## 2. Таблица «Кафедра»

```sql
CREATE TABLE department (
    id_department   INTEGER PRIMARY KEY,
    name            VARCHAR(200) NOT NULL,
    id_faculty      INTEGER NOT NULL REFERENCES faculty(id_faculty)
);
```

**Пояснения**:
- `REFERENCES faculty(id_faculty)` — **внешний ключ**: гарантирует, что кафедра принадлежит существующему факультету.
- `NOT NULL` — кафедра не может быть без факультета.

---

## 3. Таблица «Преподаватель» (после удаления избыточного `id_faculty`)

```sql
CREATE TABLE teacher (
    id_teacher      INTEGER PRIMARY KEY,
    full_name       VARCHAR(150) NOT NULL,
    birth_date      DATE,
    academic_degree VARCHAR(100),        -- учёная степень
    id_department   INTEGER NOT NULL REFERENCES department(id_department)
);
```

**Почему нет `id_faculty`?**  
Потому что факультет восстанавливается через `department`:
```sql
SELECT t.*, d.name AS department_name, f.name AS faculty_name
FROM teacher t
JOIN department d ON t.id_department = d.id_department
JOIN faculty f ON d.id_faculty = f.id_faculty;
```

---

## 4. Таблица «Специальность» (составной первичный ключ)

```sql
CREATE TABLE specialty (
    id_faculty      INTEGER NOT NULL REFERENCES faculty(id_faculty),
    id_specialty    INTEGER NOT NULL,   -- общероссийский код
    specialty_name  VARCHAR(200) NOT NULL,
    -- другие атрибуты (срок обучения, квалификация и т.п.)
    PRIMARY KEY (id_faculty, id_specialty)
);
```

**Почему составной ключ?**  
Одна и та же специальность (например, ФИИТ) может существовать на разных факультетах.  
`(id_faculty, id_specialty)` гарантирует уникальность.

## 5. Таблица «Студент» (минимально необходимая)

```sql
CREATE TABLE student (
    id_student      INTEGER PRIMARY KEY,   -- номер студенческого билета
    last_name       VARCHAR(80) NOT NULL,
    first_name      VARCHAR(80) NOT NULL,
    middle_name     VARCHAR(80),
    birth_date      DATE,
    id_group        INTEGER NOT NULL       -- внешний ключ на группу (пока без REFERENCES, т.к. группа создаётся позже)
);
```

Позже, когда таблица `student_group` будет создана:
```sql
ALTER TABLE student
    ADD CONSTRAINT fk_student_group
    FOREIGN KEY (id_group) REFERENCES student_group(id_group);
```

## 6. Таблица «Занятие» (ключевая для расписания)

```sql
CREATE TABLE lesson (
    id_lesson       SERIAL PRIMARY KEY,    -- искусственный суррогатный ключ
    id_teacher      INTEGER NOT NULL REFERENCES teacher(id_teacher),
    id_subject      INTEGER NOT NULL,      -- внешний ключ на предмет (см. ниже)
    id_group        INTEGER NOT NULL,
    id_work_type    INTEGER NOT NULL,      -- вид работы (лекция, практика и т.п.)
    lesson_datetime TIMESTAMP NOT NULL,    -- дата и время занятия
    -- составной уникальный ключ, чтобы не было дублей расписания
    UNIQUE (id_teacher, id_subject, id_group, id_work_type, lesson_datetime)
);
```

**Пояснения**:
- `SERIAL` — автоинкрементное целое (аналог `AUTO_INCREMENT`).
- `UNIQUE` — предотвращает одинаковые занятия в одно время у одного преподавателя/группы.
- Все внешние ключи связывают таблицы без избыточности (факультет/кафедра восстанавливаются через `teacher` или `subject`).

## 7. Таблица «Предмет» (пример с составным ключом, если строго следовать лекции)

```sql
CREATE TABLE subject (
    id_faculty      INTEGER NOT NULL,
    id_specialty    INTEGER NOT NULL,
    id_program      INTEGER NOT NULL,
    id_subject      INTEGER NOT NULL,
    subject_name    VARCHAR(200) NOT NULL,
    hours_total     INTEGER,
    PRIMARY KEY (id_faculty, id_specialty, id_program, id_subject),
    FOREIGN KEY (id_faculty, id_specialty) REFERENCES specialty(id_faculty, id_specialty)
    -- внешний ключ на программу (если она есть)
);
```

Но на практике часто используют **суррогатный ключ** `id_subject_global`, чтобы не таскать везде 4 поля.  
Однако по **теории лекции** — именно составной.

## 8. Простой SELECT с JOIN (восстановление факультета для преподавателя)

```sql
SELECT 
    t.full_name,
    d.name AS department,
    f.name AS faculty
FROM teacher t
JOIN department d ON t.id_department = d.id_department
JOIN faculty f ON d.id_faculty = f.id_faculty
WHERE t.id_teacher = 101;
```

**Результат** — все данные о преподавателе, включая факультет, без хранения `id_faculty` в таблице `teacher`.
