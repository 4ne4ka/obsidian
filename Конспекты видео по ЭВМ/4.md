### Введение в Ассемблер
### Ассемблер против языка Ассемблера

- **Язык ассемблера** — это язык программирования низкого уровня, на котором пишет программист. Он состоит из мнемонических представлений машинных инструкций (например, `MOV`, `ADD`).
- **Ассемблер** — это конкретная программа (инструмент), которая преобразует (компилирует) исходный код, написанный на языке ассемблера, в исполняемый **машинный код** (бинарный файл).
### Компилятор и перевод
Вы пишете программу на Java, затем компилятор Java переводит её в байт-код для Виртуальной машины Java (JVM). Для языка C другой компилятор C переводит её напрямую в машинный код.
- **Процесс для Ассемблера:** Это частный случай компиляции, но его называют **ассемблированием**. Вы пишете код на языке ассемблера, а специальная программа-**ассемблер** переводит его в машинный код.
### Машинный код и Архитектура
Машинный код — это конечный продукт компиляции, набор бинарных инструкций, которые процессор выполняет напрямую. Архитектура процессора определяет, какой именно машинный код он понимает.
- **Архитектура (ISA - Instruction Set Architecture)** — это набор команд, которые понимает конкретный процессор (например, x86-64, ARM, RISC-V). Каждая архитектура имеет свой уникальный машинный код. Программа, скомпилированная для ARM, не будет работать на процессоре x86.
- **Машинный код** — это низкоуровневое представление этих команд в виде двоичных чисел (единиц и нулей), которые загружаются в память и исполняются процессором.
### Особенности Ассемблера
1. **Мнемоники:** Человекочитаемые сокращения машинных команд. Вместо `10110000 01100101` (условно) пишется `mov al, 0x65`.
2. **Директивы:** Это команды не для процессора, а **для самого ассемблера**. Они нужны, чтобы структурировать программу:
    - Разметить, где данные, а где код.
    - Определить метки и символы (например, имена функций).
    - Объявить константы.
    - Подключить внешний код.     
3. **Макросы:** Возможность определить шаблон кода для повторного использования, чтобы избежать рутинного написания одних и тех же инструкций.
4. **Абстракция от синтаксиса процессора:** один и тот же ассемблер (например, GAS) может поддерживать разные синтаксисы для одной архитектуры (синтаксис AT&T vs. синтаксис Intel), скрывая от программиста неочевидные особенности описания команд самого процессора.
- **Синтаксис в Linux (GAS):** по умолчанию в ассемблере GNU (`as`, он же `gas`) используется **синтаксис AT&T**, а не тот, который можно найти в официальных мануалах от Intel. Это частая причина путаницы для новичков.
    - _Синтаксис AT&T:_ `movl $42, %eax`
    - _Синтаксис Intel:_ `mov eax, 42`

### Основные характеристики регистров
**Скорость доступа**
- Регистры находятся внутри процессора, что позволяет получать доступ к данным гораздо быстрее, чем к данным в основной памяти
**Именованный доступ**
- В отличие от ячеек памяти, которые адресуются по числовым адресам, регистры имеют специальные имена (например, AX, BX, CX, DX)
**Архитектурная зависимость**
- Набор и структура регистров определяются архитектурой конкретного процессора
### Типы регистров и их назначение
#### Регистры общего назначения
- **Аккумулятор (EAX/AX)**
    - Часто используется для хранения результатов арифметических и логических операций
- **Базовый регистр (EBX/BX)**
	- Применяется для хранения адреса некоторого объекта в памяти, действуя как указатель
- **Счетчик (ECX/CX)**
	- Обычно используется как счетчик в циклах или операциях, требующих повторений
- **Регистр данных (EDX/DX)**
	- Регистр общего назначения, который может использоваться для различных целей, иногда в паре с EAX для операций с большими числами или для ввода/вывода через порты
#### Регистры указателей и индексов
- **Указатель стека (ESP/SP)**
    - Содержит адрес вершины стека, который используется для управления стеком программы (добавление и извлечение данных) 
    - Стек обычно растет в сторону уменьшения адресов 
- **Базовый указатель (EBP/BP)**
    - Используется для указания на базу стекового кадра текущей функции, что позволяет легко получать доступ к аргументам функции и локальным переменным 
- **Исходный индекс (ESI/SI)**
    - В операциях с двумя объектами в памяти (например, копирование строк) указывает на адрес источника данных 

- **Целевой индекс (EDI/DI)**
    - В операциях с двумя объектами в памяти указывает на адрес назначения данных 
#### Сегментные регистры

Эти регистры используются для адресации памяти в сегментированной архитектуре, храня базовые адреса сегментов:
- **Сегмент кода (CS)**
    - Указывает на сегмент памяти, где находится исполняемый код программы 
- **Сегмент данных (DS)**
    - Указывает на сегмент памяти, где хранятся данные программы 
- **Сегмент стека (SS)**
    - Указывает на сегмент памяти, используемый для стека программы 
- **Дополнительные сегменты (ES, FS, GS)**
    - Могут использоваться для различных целей, предоставляя дополнительные возможности для адресации памяти

#### Регистр указателя инструкции (EIP/IP)
- Содержит адрес текущей команды, которая будет выполнена следующей 
- В реальном режиме адрес команды формируется путем сложения содержимого регистра сегмента кода (CS) и смещения из EIP

#### Регистр флагов (EFLAGS/FLAGS)
- Содержит различные флаги, которые отражают состояние процессора после выполнения операций 
    - **Флаг переноса (CF)**: Указывает на перенос или заем при арифметических операциях 
    - **Флаг нуля (ZF)**: Устанавливается, если результат операции равен нулю 
    - **Флаг переполнения (OF)**: Указывает на переполнение при арифметических операциях со знаковыми числами 
    - **Флаг направления (DF)**: Используется в строковых операциях для определения направления обработки (увеличение или уменьшение индексов)


### Стек и вызовы функций
`F(int x, int y)`
##### Шаг 1: Вызов функции (Caller's Job — задача того, кто вызывает)
Перед тем как передать управление функции, вызывающий код должен подготовить для нее "рабочее место". Это делается с помощью стека.
1. **Передача аргументов:** Аргументы функции (`x` и `y`) **заталкиваются (push)** в стек в обратном порядке (сначала `y`, потом `x`, если следовать соглашению cdecl).
2. **Вызов команды `CALL`:** Команда `CALL` делает две важные вещи:
    - **Автоматически кладет на стек адрес возврата** — адрес следующей команды в вызывающей функции, куда нужно будет вернуться после завершения работы вызванной функции.
    - **Переходит (jump) по адресу метки,** которая является началом вызванной функции.
**Итог шага 1:** В стеке теперь лежат аргументы функции и адрес возврата. Управление передано на первую инструкцию функции `F`.
##### Шаг 2: Пролог функции (Callee's Job — задача вызываемой функции)
Теперь функция `F` начинает работу. Ей нужно выделить себе место для локальных переменных и сохранить состояние, чтобы не испортить данные вызывающей функции.
1. **Создание стекового фрейма (кадра):**
    - Функция сохраняет значение регистра **EBP (Extended Base Pointer)** на стек. Этот регистр часто называют "указателем базы кадра".
    - Затем она копирует текущее значение **ESP (Stack Pointer)** в **EBP**. Теперь `EBP` указывает на начало _ее_ стекового фрейма.
    - **Результат:** `EBP` становится "якорем". Через него функция может обращаться к своим аргументам и локальным переменным, даже если стек продолжает двигаться (при `push`/`pop` внутри функции).
2. **Выделение места для локальных переменных:** "она ещё в стеке выделяет но уже под локальные переменные". Функция сдвигает указатель стека `ESP` на необходимое количество байт вниз, чтобы зарезервировать место для своих локальных переменных (например, для переменной `p`).
**Содержимое стекового фрейма после пролога:**  
(адреса условные, стек растет вниз)

|Адрес (пример)|Содержимое|Доступ через|
|---|---|---|
|`EBP + 12`|Аргумент `y`|`[ebp + 12]`|
|`EBP + 8`|Аргумент `x`|`[ebp + 8]`|
|`EBP + 4`|**Адрес возврата** (сохранен `CALL`)||
|`EBP`|**Сохраненное значение EBP** (старый EBP)||
|`EBP - 4`|Локальная переменная `p`|`[ebp - 4]`|
|`...`|... другие локальные переменные ...||
|`ESP`|**Вершина стека**||

##### Шаг 3: Работа функции и возврат значения
- Функция выполняет свой код, обращаясь к аргументам через положительные смещения от `EBP`, а к локальным переменным — через отрицательные.
- **Возврат значения:** По соглашению, часто возвращаемое значение помещается в регистр `EAX`. Для простых типов (`int`, `char`) этого достаточно. Для больших структур может использоваться стек.
##### Шаг 4: Эпилог функции и возврат
1. **Восстановление стека:** Функция сдвигает `ESP` обратно к значению `EBP`, "освобождая" память, выделенную под локальные переменные.
2. **Восстановление старого EBP:** Команда `POP EBP` восстанавливает значение `EBP` вызывающей функции. Теперь `EBP` снова указывает на ее стековый фрейм.
3. **Возврат:** Команда `RET` делает следующее:
    - **Извлекает со стека адрес возврата** (тот, что был сохранен командой `CALL`).
    - **Передает управление по этому адресу.** Выполнение продолжается в вызывающей функции.
4. **Очистка аргументов (Caller's Job):** После возврата вызывающая функция должна "очистить" аргументы, которые она передавала. Это часто делается командой `ADD ESP, 8` (если было передано 2 аргумента по 4 байта каждый).

### Переполнение стека (Stack Overflow)
- **Причина:** Если функция слишком глубоко вызывает сама себя (рекурсия) или выделяет огромные массивы как локальные переменные (например, `int huge_array[1000000];`), указатель стека `ESP` может выйти за пределы отведенной для стека области памяти.
- **Последствие:** Процесс начинает записывать данные в область памяти, предназначенную для других целей (например, в кучу или код программы). Это приводит к **неопределенному поведению** и почти гарантированному краху программы.



### Системные вызовы и прерывания
**Настоящая низкоуровневая работа с системой** (например, вывод строки на экран, чтение файла, работа с сетью) происходит через **системные вызовы (syscalls)**.
Причина в **безопасности и изоляции**:
- **Ядро (Kernel)** выполняется в привилегированном режиме (**kernel space**). У него есть прямой доступ ко всему hardware: всей памяти, жесткому диску, сетевой карте и т.д.
- **Программа пользователя** выполняется в изолированном пространстве (**user space**). Ей запрещено напрямую обращаться к устройствам.
Чтобы программа пользователя могла выполнить действие, требующее привилегий, она должна **попросить** ядро сделать это за нее. Механизмом этой "просьбы" и является системный вызов.
### Прерывания (Interrupts) — "Дверь" в ядро
**Как программа стучится в ядро?** Через **прерывание**.
- *Прерывание — это специальный сигнал процессору, который **немедленно приостанавливает** выполнение текущего кода и передает управление заранее заданной процедуре — **обработчику прерывания**.
- **Для системных вызовов в Linux:** Используется прерывание с номером **`0x80`**
**Что происходит при вызове `int 0x80`?**
1. Процессор завершает текущую инструкцию.
2. Сохраняет контекст (адрес следующей инструкции, регистры) чтобы потом можно было вернуться.
3. Переключается в привилегированный режим (kernel mode).
4. Переходит по заранее известному адресу, где находится **обработчик системных вызовов** ядра Linux.
5. **Ядро начинает исполняться**.
### Системные вызовы (Syscalls) — "Просьба" к ядру
Системный вызов — это конкретная услуга, которую предоставляет ядро. Например: `write` (вывод данных), `read` (ввод данных), `exit` (завершение программы).
**Как отличить один системный вызов от другого?**
- Каждый системный вызов имеет свой **уникальный номер**.
- Этот номер нужно поместить в регистр **`EAX`** перед вызовом `int 0x80`. "номер вызова здесь написано положить в `eax`".
Номера жестко зафиксированы и не меняются от версии к версии ядра, чтобы обеспечить обратную совместимость
**Как передать аргументы системному вызову?**  
Например, для вызова `write` нужно передать: куда выводить (1 — стандартный вывод), указатель на строку и ее длину.
- Аргументы передаются через регистры в определенном порядке: **`EBX`**, **`ECX`**, **`EDX`**, **`ESI`**, **`EDI`**.
- "параметры вот если их пять то они вот так в регистр записываются первый второй третий четвёртый пятый".
### Полный пример (Как вывести строку на ассемблере)
Вместо вызова библиотечной функции `puts` из `libc`, можно сделать прямой системный вызов `write`:
1. **Положить в `EAX` номер системного вызова.** Для `write` это номер 4.
2. **Подготовить аргументы:**
    - `EBX` = 1 (код файлового дескриптора для стандартного вывода)
    - `ECX` = адрес памяти, где лежит наша строка (метка в секции данных)
    - `EDX` = длина строки
3. **Вызвать прерывание:** `int 0x80`
Ядро получит управление, увидит номер 4 в `EAX`, поймет, что это запрос на `write`, возьмет аргументы из регистров, выполнит вывод и вернет управление вашей программе.


## Основные функции GDB
GDB — это **командный отладчик** (debugger).
**Отладка с помощью GDB (GNU Debugger) — это процесс, который позволяет программистам анализировать и исправлять ошибки в своих программах на языке C, C++ и других языках, поддерживаемых GDB.**
1. **Запуск программы**
    - GDB позволяет запускать программы в контролируемой среде, что дает возможность отслеживать выполнение кода и выявлять ошибки 
2. **Установка точек останова**
3. **Пошаговое выполнение**
    - GDB поддерживает **пошаговое выполнение** (step-by-step execution), что позволяет программистам исполнять код построчно и наблюдать за изменениями в переменных и состоянии программы 
4. **Просмотр значений переменных**
    - GDB позволяет просматривать значения переменных и регистров в любой момент времени, что помогает понять, как данные изменяются в процессе выполнения программы 
5. **Анализ стека вызовов**
    - Программа может быть остановлена в момент ошибки, и GDB предоставляет возможность просмотреть **стек вызовов** (call stack), что помогает определить, какие функции были вызваны до возникновения ошибки
### Ключевые концепции и команды GDB
##### a) Подготовка программы к отладке
программу нужно скомпилировать с **отладочной информацией**.
- **Ключ компилятора:** `-g` или `-g3`.
- **Что делает:** Компилятор добавляет в исполняемый файл специальные метаданные, которые связывают машинный код с вашим исходным кодом.
- **Что будет без `-g`:** Без отладочной информации GDB будет показывать только голый машинный код, что крайне неудобно.
##### b) Запуск и базовое управление
- **Запуск программы в GDB:** `gdb ./my_program`
- **Запуск программы:** `run` (или `r`)
- **Пошаговое выполнение:**
    - `step` (или `s`) — **шаг с заходом** (into). Выполняет следующую инструкцию и, если это вызов функции, заходит внутрь нее.
    - `next` (или `n`) — **шаг с обходом** (over). Выполняет следующую инструкцию, но _не заходит_ внутрь функций, treating them as single steps.
    - `continue` (или `c`) — **продолжить** выполнение до следующей точки останова или конца программы
#### c) Точки останова (Breakpoints)
Точки останова — это метки, которые вы устанавливаете в коде. Когда программа доходит до точки останова, ее выполнение приостанавливается, и управление возвращается в GDB.
- **Как поставить:** `break *адрес` или `break function_name` или `break file.c:line_number`
##### d) Анализ состояния программы
1. **Просмотр регистров:** `info registers` (или `i r`)
    - Показывает содержимое всех регистров процессора (EAX, EBX, ESP, EIP и т.д.).
2. **Просмотр памяти:** `x/NFU адрес`
    - `N` — количество единиц для отображения
    - `F` — формат вывода (например, `s` для строки, `x` для hex)
    - `U` — размер единицы (например, `b` — byte, `w` — word)
    - **Пример:** `x/s &hello_string` — посмотреть, что лежит по адресу метки `hello_string`, интерпретируя это как строку.

4. **Просмотр исходного кода/дизассемблирования:** `list` (или `l`) / `disassemble` (или `disas`)
    - `list` показывает исходный код (если есть отладочная информация).
    - `disassemble` показывает машинные инструкции по указанному адресу.
### Анализ аварийного завершения (Core Dumps)
- **Что такое core dump?** "`corum`... это образ памяти в момент краха". Когда программа падает, ядро ОС может сохранить весь ее образ памяти в файл (обычно называется `core` или `core.PID`). Это "моментальный снимок" состояния программы в момент смерти.
- **Как включить:** По умолчанию во многих системах создание core-файлов отключено. Чтобы включить, нужно выполнить команду: `ulimit -c unlimited`
- **Как анализировать:** Запустите GDB на исполняемом файле и core-файле: `gdb ./my_program core`. GDB автоматически остановится на той инструкции, которая вызвала крах, и вы сможете с помощью `bt`, `info registers` и `x` проанализировать причину.
