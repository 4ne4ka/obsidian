## Конспект лекции по синхронизации (часть 3) – для новичков

### 1. Что такое семафор (повторение)

**Семафор** – это счётчик (неотрицательное целое число) с двумя атомарными (неделимыми) операциями:

- **P(S)** – попытка уменьшить счётчик на 1. Если он был больше 0 – уменьшаем и идём дальше. Если был 0 – поток засыпает и ждёт.
- **V(S)** – увеличиваем счётчик на 1. Если кто-то спал на этом семафоре – один просыпается.

---

### 2. Два вида семафоров

#### А. Двоичный семафор (значения 0 или 1)
- **Для чего:** Обеспечить эксклюзивный доступ к ресурсу (например, к принтеру или к файлу).
- **Как работает:**
  - 1 = ресурс свободен
  - 0 = ресурс занят
- **Код:**
  ```
  P(S)   // захватить – если 1, станет 0 и входим; если 0 – спим
  // ... работа с ресурсом ...
  V(S)   // освободить – снова становится 1
  ```
- **Комментарий лектора:** "Это практически счастье" – самый простой вариант.

#### Б. Счётный семафор (значения от 0 до N)
- **Для чего:** Контролировать, сколько единиц ресурса свободно. Например, 5 принтеров, 10 слотов в буфере.
- **Начальное значение:** N (все ресурсы свободны).
- **Как работает:**
  - `P(S)` – зарезервировать одну единицу (уменьшить счётчик)
  - `V(S)` – вернуть единицу обратно (увеличить счётчик)

---

### 3. Типовая схема: пул ресурсов (например, 10 принтеров)

Нужны **два семафора**:

1. **Счётный семафор `free`** – сколько принтеров сейчас свободно (начальное значение 10).
2. **Двоичный семафор `access`** – защищает таблицу с описанием принтеров (кто каким занят).

**Алгоритм для потока, которому нужен принтер:**
```
P(free)           // 1) зарезервировать свободный принтер (если нет – спать)
P(access)         // 2) войти в таблицу принтеров
// найти свободный принтер, пометить его своим именем
V(access)         // 3) выйти из таблицы
// ... печатать ...
P(access)         // 4) снова войти в таблицу
// пометить принтер как свободный
V(access)         // 5) выйти из таблицы
V(free)           // 6) вернуть единицу в счётчик свободных
```

**Важное предупреждение:** Не пытайтесь "упростить" и вынести освобождение принтера из-под защиты. Если для освобождения нужно изменить несколько полей, компилятор или процессор могут переставить операции местами, и признак "свободен" запишется раньше, чем очистятся остальные поля. Это сломает программу.

### 4. Классическая задача: Производитель и Потребитель

#### Что такое циклический буфер
- Это массив фиксированного размера N.
- Есть два указателя: куда писать (свободный блок) и откуда читать (занятый блок).
- Когда доходим до конца, перескакиваем в начало.

#### Ограничения
- Если буфер пуст – читать нельзя.
- Если буфер полон – писать нельзя.

#### Где используется в реальности
- **Каналы (pipe)** в ОС – передача данных между программами.
- **Сетевой протокол TCP** – буферизация данных.
- Конвейерная обработка.

#### Важный вопрос: нужна ли синхронизация, если **один** производитель и **один** потребитель?

Лектор рассуждает:
- Производитель меняет только указатель на свободный блок. Потребитель – только указатель на занятый блок.
- Каждый из них другой указатель только читает.
- Кажется, что синхронизация не нужна: худшее, что случится – один из них "не заметит" изменений и подождёт лишний цикл. Данные не испортятся.

**НО возникает проблема:** когда буфер полностью заполнен, оба указателя совпадают. Когда буфер полностью пуст – они тоже совпадают. Состояния "полон" и "пуст" неразличимы!

**Решение:** добавить флаг `empty`. Но тогда этот флаг меняют оба потока – синхронизация становится строго обязательной.

**Хитрый способ обойтись без синхронизации:** Пожертвовать одним элементом буфера. Считать буфер полным, когда в нём N-1 элемент. Тогда для одного производителя и одного потребителя синхронизация действительно не нужна.

### 5. Решение на семафорах (общий случай – много потоков)

Используем **три семафора**:

| Семафор | Тип | Начальное значение | Что значит |
|---------|-----|-------------------|-------------|
| `empty` | счётный | N | количество свободных ячеек |
| `full`  | счётный | 0 | количество занятых ячеек |
| `access`| двоичный | 1 | защита самого буфера и указателей |

**Производитель:**
```
P(empty)      // ждём, пока появится свободная ячейка
P(access)     // захватываем доступ к буферу
// записываем данные
V(access)     // освобождаем буфер
V(full)       // сообщаем, что появилась новая запись
```

**Потребитель:**
```
P(full)       // ждём, пока появятся данные
P(access)     // захватываем доступ к буферу
// читаем данные
V(access)     // освобождаем буфер
V(empty)      // сообщаем, что освободилось место
```

**Комментарий лектора:** Это один из самых простых вопросов на экзамене. Нужно уметь его написать.

### 6. Задача "Читатели и Писатели"

#### Условие
- Есть общие данные.
- **Читатели** только читают, не меняют данные. Много читателей могут работать одновременно.
- **Писатели** меняют данные. Пока писатель работает, никто другой (ни читатели, ни писатели) не может получить доступ.

#### Пример из жизни
База данных. Сотни клиентов читают данные (отображают на экране). Иногда кто-то нажимает "Сохранить" – это писатель. Чтений гораздо больше, чем записей.

---

### 7. Решение "с преимуществом читателей" (простое, но нечестное)

**Инструменты:**
- `XS` – двоичный семафор для защиты самих данных
- `readcount` – счётчик читателей внутри данных
- `RC` – двоичный семафор для защиты счётчика `readcount`

**Код писателя (простой):**
```
P(XS)         // захватить данные
// ... запись ...
V(XS)         // освободить
```

**Код читателя:**
```
// ВХОД
P(RC)                     // зайти в счётчик
readcount++               // я – новый читатель
if (readcount == 1)       // если я первый
    P(XS)                 // захватываю данные для всех читателей
V(RC)                     // выйти из счётчика

// ... ЧТЕНИЕ ...

// ВЫХОД
P(RC)                     // зайти в счётчик
readcount--               // я ухожу
if (readcount == 0)       // если я был последним
    V(XS)                 // освобождаю данные для писателей
V(RC)                     // выйти из счётчика
```

**Проблема – "голодание писателей":**
Если читатели приходят непрерывно, `readcount` никогда не становится 0, и писатель вечно ждёт. Это называется **преимущество читателей**.

---

### 8. Честное решение (без голодания)

#### Способ 1 – добавить семафор `S`
И писатели, и читатели сначала конкурируют за семафор `S`. Когда писатель его захватывает, новые читатели не могут войти, пока он не закончит.

#### Способ 2 – очередь
Все желающие (и читатели, и писатели) встают в очередь в порядке поступления:
- Если в начале очереди писатель – он входит один.
- Если читатель – все читатели из очереди входят вместе.

Это гарантирует, что никто не будет голодать вечно.

### 9. Мьютекс (Mutex)

Mutex (MUTual EXclusion) – это специализированный объект для эксклюзивного доступа. Он похож на двоичный семафор, но имеет **три важных дополнительных свойства**.

#### Свойство 1: Запоминание владельца
- Мьютекс помнит, какой поток его захватил.
- **Освободить его может только поток-владелец.** Если чужой поток вызовет `release` – ничего не произойдёт.
- Это защищает от ошибки, когда семафор освобождает не тот поток.

#### Свойство 2: Рекурсивность
- Поток может захватить мьютекс несколько раз подряд.
- Чтобы полностью освободить, нужно вызвать `release` столько же раз.
- **Зачем:** Если функция A вызывает функцию B, и обе должны защищать одни и те же данные, без рекурсивности произойдёт deadlock (зависание).

#### Свойство 3: Наследование приоритета (кратко)
- Решает проблему "инверсии приоритетов" – когда высокоприоритетный поток ждёт ресурс, захваченный низкоприоритетным.
- Суть: низкоприоритетный поток временно получает высокий приоритет, чтобы быстрее освободить ресурс.

### 10. Мониторы

#### Что это такое
Монитор – это конструкция языка программирования, которая **объединяет в одном месте**:
1. Защищаемые данные
2. Функции для работы с ними
3. Механизмы синхронизации

**Главное правило:** Доступ к данным монитора возможен **только через его функции**. Компилятор сам добавляет код синхронизации.

**Важно:** Монитор – это объект, а не класс. У разных объектов – разные мониторы. Проблемы возникают, только когда два потока пытаются работать с **одним и тем же объектом**.

#### Как работает монитор (базово)
В каждый момент времени только один поток может выполнять любую функцию монитора для данного объекта. Остальные ждут снаружи.

---

### 11. Проблема монитора без условных переменных

**Пример:** Буфер, функции `produce` и `consume`. Буфер пуст.

- Первым входит **Потребитель**, вызывает `consume`.
- Он видит, что данных нет, но не может выйти из функции (по логике должен ждать).
- Потребитель застрял внутри монитора.
- **Производитель** пытается войти, но монитор занят.

**Результат:** Взаимоблокировка (deadlock). Оба потока ждут друг друга.

---

### 12. Условные переменные (решение проблемы)

**Условная переменная** – это объект, символизирующий событие (например, "появились данные" или "появилось место").

**Операции:**
- **`wait`** – поток внутри монитора говорит: "я жду этого события". При этом он **освобождает монитор** (пускает других) и засыпает.
- **`signal`** – пробуждает **один** спящий поток. **Важно:** если никто не ждёт, сигнал теряется.
- **`broadcast`** – пробуждает **все** спящие потоки.

---

### 13. Решение задачи на мониторах

**Структура:**
- Буфер и указатели
- Условная переменная `not_full` (буфер не полон)
- Условная переменная `not_empty` (буфер не пуст)

**Производитель:**
```
while (буфер полон) {
    wait(not_full)    // заснуть, освободив монитор
}
// записать данные
signal(not_empty)     // разбудить потребителей
```

**Потребитель:**
```
while (буфер пуст) {
    wait(not_empty)   // заснуть, освободив монитор
}
// прочитать данные
signal(not_full)      // разбудить производителей
```

**Почему `while`, а не `if`?** Это важно!

---

### 14. Два типа мониторов

#### Мониторы Хоара (теоретические, не используются)
- При `signal` управление немедленно передаётся проснувшемуся потоку.
- Если проснулся – условие **гарантированно** выполнено. Можно писать `if`.
- Но непонятно, как сделать `broadcast`.

#### Мониторы Mesa (реальные, используются везде)
- При `signal` проснувшийся поток переводится в состояние "готов", но **не входит в монитор** сразу.
- Поток, пославший сигнал, продолжает работать.
- Когда монитор освобождается, все готовые потоки (и старые, и новые) конкурируют за вход.
- **Следствие:** Проснувшись, нельзя быть уверенным, что условие всё ещё выполнено (другой поток мог уже забрать данные).
- **Правило:** Проверять условие **в цикле `while`** – после каждого пробуждения проверять заново.

**Именно поэтому в реальном коде пишут `while`, а не `if`.**

---

### 15. Как реализовать монитор вручную

В языках, где нет встроенных мониторов, используют **пару: мьютекс + условная переменная**.

- **Windows:** Critical Section + Conditional Variable
- **Linux (POSIX):** `pthread_mutex_t` + `pthread_cond_t`

Функция `wait` атомарно делает две вещи: освобождает мьютекс и засыпает на условной переменной.

## Главные выводы всей лекции

1. **Семафоры** – простой и мощный инструмент. Двоичный – для взаимоисключения, счётный – для пулов ресурсов.
2. **Задача "Производитель-Потребитель"** – классика, решается тремя семафорами или монитором с двумя условными переменными.
3. **Задача "Читатели-Писатели"** – простое решение даёт голодание писателей. Честное решение требует очереди или дополнительного семафора.
4. **Мьютекс** – как двоичный семафор, но с защитой от ошибок: запоминает владельца, может быть рекурсивным, умеет наследовать приоритет.
5. **Мониторы** – объединяют данные и синхронизацию в одной конструкции. На практике везде используются мониторы типа Mesa, где проверку условия нужно делать в цикле `while`.
6. **Условные переменные** нужны, чтобы поток внутри монитора мог заснуть, освободив монитор, и проснуться при наступлении события.

**Совет лектора:** Для большинства программистов лучше использовать готовые высокоуровневые механизмы (мьютексы, мониторы в языках), чем писать свои алгоритмы с нуля.

## Конспект: Файловые системы (для новичков)

### 1. Архитектура файловой подсистемы (как всё устроено снизу вверх)

Представьте пирамиду из 5 уровней. Каждый уровень "прячет" сложность нижнего и даёт удобные кирпичики для верхнего.

**Уровень 1: Железо** (HDD, SSD, флешка, CD-ROM)
- Это физическое устройство, которое хранит данные.
- Умеет адресовать данные по-своему (например, у жёсткого диска: номер цилиндра, номер головки, номер сектора).
- *Сложность:* Разные устройства ведут себя по-разному.

**Уровень 2: Драйвер устройства**
- Маленькая программа, которая знает, как говорить с конкретным железом.
- Его задача: **скрыть физическую сложность** и представить любое устройство вышестоящему уровню как **простой линейный массив блоков** (например, миллион блоков по 512 байт).
- *Абстракция:* Верхние уровни больше не думают про цилиндры и головки, они просто говорят "дай мне 5-й блок".

**Уровень 3: Драйвер файловой системы**
- Решает главную задачу: как на **линейном массиве блоков** (с уровня 2) организовать **иерархическую структуру** — папки, файлы, ссылки.
- Создаёт логическую структуру: корень, подпапки, файлы внутри.

**Уровень 4: Виртуальная файловая система (VFS)**
- Позволяет работать вместе нескольким разным файловым системам (например, диск C: и флешка).
- Создаёт **единое дерево объектов**, которое видит пользователь.
  - **Windows:** "Этот компьютер" → C:\, D:\ → папки и файлы.
  - **Linux:** Единый корневой каталог `/`, в который "монтируются" (подключаются) все остальные диски и разделы.

**Уровень 5: Система управления файлами**
- Это интерфейс для программ (системные вызовы): `open`, `read`, `write`, `close`.
- Управляет открытыми файлами и тем, где в файле мы сейчас находимся (курсор чтения/записи).

---

### 2. Как жёсткий диск (HDD) адресует данные (физический уровень)

**Что внутри диска:**
- Несколько пластин (блинов).
- У каждой пластины есть две стороны (поверхности).
- На каждой поверхности — дорожки (концентрические круги от края к центру).
- Каждая дорожка разбита на **сектора** (маленькие блоки данных).

**Трёхкомпонентный адрес (CHS):**
- **C (Cylinder — цилиндр):** Это все дорожки, которые находятся на одинаковом расстоянии от центра, на всех поверхностях сразу (стопка дорожек друг над другом).
- **H (Head — головка):** Номер поверхности (или головки, которая читает эту поверхность).
- **S (Sector — сектор):** Номер сектора на дорожке.

**Как с этим работают:**
- Физический адрес — это тройка чисел (C, H, S).
- Контроллер диска (или драйвер) превращает этот сложный адрес в **простой линейный номер** (LBA — Logical Block Addressing). Например, превращает `(C=2, H=1, S=5)` в блок номер `12345`.
- Все уровни выше драйвера работают только с линейными номерами, не зная про цилиндры и головки.

---

### 3. Объекты файловой системы (что мы видим на диске)

#### 3.1. Файл (File)
- **Что это:** Именованный набор данных с набором характеристик (атрибутами: размер, дата создания, владелец, права доступа).
- **Важное уточнение:** Раньше считалось, что файл — это просто массив байтов. Но современные файловые системы (например, NTFS в Windows) поддерживают **альтернативные потоки** — внутри одного файла может быть несколько независимых массивов данных.
- **Операции:** создание, открытие, чтение, запись, закрытие, удаление.

#### 3.2. Каталог (Directory, он же папка)
- **Что это:** Единственный тип объектов, который может **содержать других** (файлы и другие папки). Служит для создания дерева.
- **Зачем нужен:** Чтобы давать объектам имена через путь. Например: `C:\Мои документы\Письмо.txt`.

**Как называть файлы:**
- **Абсолютное имя:** Полный путь от корня. Например, `/home/vasya/photo.jpg` (в Linux) или `D:\Фото\отпуск.png` (в Windows).
- **Текущий каталог:** Для каждого процесса система помнит, в какой папке он "находится".
- **Относительное имя:** Путь от текущей папки. Если мы в `/home/vasya/`, то файл `photo.jpg` — это относительное имя.
- **Специальные обозначения:** `.` — текущая папка, `..` — родительская папка.

**Как внутри устроен каталог:**
- Он хранит список имён дочерних объектов и их атрибуты (или ссылки на них). Как именно — зависит от типа файловой системы (FAT32, NTFS, ext4).

#### 3.3. Как открыть файл `/one/two/three` (процесс поиска пути)
1. Найти и открыть **корневой каталог**. (Способ зависит от ФС, но он всегда известен.)
2. В корне найти объект с именем `one`. Проверить, что это каталог, и открыть его.
3. В каталоге `one` найти `two`. Открыть его как каталог.
4. В каталоге `two` найти `three`. Если `three` — файл, открыть его. Иначе — ошибка.
- **Проверки:** На каждом шаге система проверяет, есть ли у вас право заходить в этот каталог.
- **Проблема:** Это **дорогая операция**. Поэтому ОС кэширует содержимое каталогов (в Linux есть специальный кэш `dentry`), чтобы не искать каждый раз заново.

#### 3.4. Файлы устройств (Device Files)
- **Что это:** Это **не настоящие файлы** с данными. Это "дверь" к драйверу устройства.
- **Два типа:**
  - **Символьные:** Доступ к устройству как к потоку байтов (клавиатура, мышь, принтер).
  - **Блочные:** Доступ к устройству как к массиву блоков (жёсткий диск, SSD).
- **Как они устроены:**
  - **Старший номер (major):** Какое это устройство? (Код драйвера). Например, все диски одного типа имеют один major-номер.
  - **Младший номер (minor):** Какой именно экземпляр? (Первый диск, второй диск).
- **Как это работает:**
  1. При загрузке драйвер регистрируется в системе и говорит: "Я отвечаю за устройства со старшим номером 42, и вот мои функции: открыть, прочитать, записать, закрыть".
  2. Когда программа открывает файл устройства (например, `/dev/sda`), ОС смотрит на его старший номер, находит нужный драйвер и вызывает его функцию `open`.
  3. Драйверу передаётся младший номер, чтобы он понял, с каким конкретно диском работать.

- **Знаменитые примеры в Linux:**
  - `/dev/null` — свалка данных. Всё, что туда пишете, исчезает. При чтении даёт пустоту.
  - `/dev/zero` — при чтении выдаёт бесконечные нули.
  - `/dev/random` и `/dev/urandom` — при чтении дают случайные числа.
  - У всех этих файлов **один и тот же старший номер** (1) — значит, один драйвер. А разные младшие номера задают разное поведение.

#### 3.5. Именованные объекты для связи между программами (IPC)
- **Идея:** Использовать файловую систему, чтобы две программы нашли друг друга.
- **Как работает:** При создании объекта (например, сокета или именованного канала) в файловой системе появляется специальный файл-заглушка. Он не хранит данные, он просто **имя**. Другая программа открывает этот файл и получает доступ к тому же объекту в ядре.
- **Пример:** Локальные Unix-сокеты (вы их видите в папке как файлы с типом `s` в выводе `ls -la`).

#### 3.6. Мягкая ссылка (Symbolic Link)
- **Что это:** Файл-ярлык на уровне файловой системы.
- **Как устроен:** Это маленький файл, внутри которого хранится **текстовый путь** к другому объекту (абсолютный или относительный). Например: "Я ссылаюсь на `/home/vasya/photo.jpg`".
- **Как работает:**
  - Программа обращается к символьной ссылке.
  - Файловая система сама (прозрачно для программы) читает путь из ссылки и начинает поиск заново.
- **Отличие от ярлыка Windows:** Ярлык — это обычный файл, который обрабатывает программа (проводник). Символьная ссылка обрабатывается **самой файловой системой** на низком уровне.
- **Свойства:**
  - Может указывать на объект на **другом диске**.
  - **Не гарантирует, что цель существует.** Если цель удалить или переименовать, ссылка "повисает" (указывает в никуда).
  - Может указывать на объект другого типа. Удалили файл, создали на его месте папку — ссылка теперь указывает на папку.
- **Защита от бесконечного цикла:** Ограничивается количество переходов по ссылкам (например, не более 40). Если ссылка ведёт на другую ссылку, и так по кругу — система остановится.

#### 3.7. Жёсткая ссылка (Hard Link)
- Упомянута, но не разобрана в этом фрагменте. Будет позже. Пока просто знайте, что это другой механизм, при котором несколько имён указывают на одни и те же данные на диске.

## Главное, что нужно запомнить

| Уровень | Что делает |
|---------|-------------|
| Железо | Физически хранит данные, умеет адресовать блоки (C, H, S) |
| Драйвер устройства | Превращает (C,H,S) в линейный массив блоков |
| Драйвер ФС | На массиве блоков строит папки и файлы |
| VFS | Объединяет разные ФС в одно дерево |
| Управление файлами | Даёт программам вызовы open, read, write |

**Объекты ФС:**
- **Файл** — набор данных.
- **Каталог** — контейнер для других объектов.
- **Файл устройства** — дверь к драйверу (major = тип, minor = экземпляр).
- **Символьная ссылка** — путь к другому объекту (обрабатывается ФС).
- **IPC-объекты** — имена для межпрограммного общения.
