### 1. Многоуровневые таблицы страниц: зачем они нужны?

Мы выяснили, что у каждого процесса есть своё **виртуальное адресное пространство** (ВАП), поделенное на блоки по 4 КБ (страницы). Таблица, переводящая виртуальные адреса в физические, называется **таблицей страниц**.

**Проблема:** В 64-битных системах одна общая таблица страниц была бы невероятно огромной (сотни гигабайт), что невозможно.

**Решение:** Таблицы делают многоуровневыми.
*   Всё ВАП делится на крупные **блоки** (например, по 4 МБ).
*   Создается **каталог** (верхний уровень таблиц), где записано, где лежат таблицы страниц для каждого блока.
*   **Главный принцип:** таблицы страниц создаются **только для тех блоков, где реально есть данные** (код, стек, куча). Для пустых блоков (а их большинство) ничего не создается.

Это уменьшает накладные расходы памяти до ничтожных **~0.1%** от полезных данных.

**Как работает адрес?**
Виртуальный адрес делится на три части:
1.  **Индекс в каталоге** (указывает на нужный блок).
2.  **Индекс в таблице страниц блока** (указывает на физическую страницу).
3.  **Смещение** (конкретная ячейка внутри этой страницы).

**Важно:** Главная плата за это — скорость. Чтобы прочитать одну ячейку данных, процессору нужно совершить **несколько обращений к памяти** (пройти каталог, таблицу страниц и только потом сами данные). Для ускорения используют кэш **TLB**, который запоминает недавние переводы адресов.

### 2. Два главных трюка страничной памяти

**Трюк №1: Разделяемая память (Shared Memory)**
Это способ сделать так, чтобы **разные процессы видели один и тот же кусок физической памяти** (как общую доску для записей). Самый быстрый способ обменяться данными.
*   ОС просто настраивает таблицы страниц двух процессов так, чтобы их виртуальные адреса указывали на одни и те же физические страницы.
*   **Copy-on-Write (Копирование при записи):** Пока все только читают общую память (например, код программы) — копия одна. Как только кто-то пытается что-то записать, только для него создается личная копия страницы, которую он меняет. Это экономит память.

**Трюк №2: Большие страницы (Huge Pages)**
Обычная страница — 4 КБ. Если программа работает с гигабайтами данных, ей нужно миллионы записей, которые не помещаются в кэш TLB.
*   **Идея:** Использовать "большие страницы" размером 2 МБ или даже 1 ГБ.
*   **Плюс:** Одна запись в таблице покрывает огромный участок памяти, резко снижая нагрузку на TLB.
*   **Минусы:** Требует незанятого **непрерывного** куска физической памяти (проблема из-за фрагментации). Подходит не для всех задач, иногда даже вредит.

### 3. Как совместить сегменты и страницы

Современные системы используют **сегментно-страничную модель**.
*   **Главный принцип:** Сначала работает страничная модель (создается общее линейное адресное пространство), а уже **поверх него** накладываются сегменты.
*   **Линейный адрес** — это полный адрес (например, от 0 до 4 ГБ), с которым работает страничная модель.
*   **Виртуальный адрес** (с точки зрения программы) — это пара: **(селектор сегмента + смещение)**.

**Классический пример (32-битные системы):**
*   Всё пространство в 4 ГБ делят на две части:
    *   **Верхний 1 ГБ** — только для ОС (код ядра, доступ запрещен программам).
    *   **Нижние 3 ГБ** — для пользовательских программ.
*   Это разделение настраивается в **Таблице дескрипторов сегментов (GDT)**, где каждому сегменту присваивается уровень привилегий.

**Почему программисты этого не видят?**
Всё работает аппаратно. В процессоре есть специальные сегментные регистры (`CS` для кода, `DS` для данных). Когда программа обращается по адресу, процессор сам подставляет нужный селектор из регистра, проверяет права доступа и смещение. Это прозрачно и очень быстро.

> **Современность (64 бита):** Механизм сегментов сильно урезали за ненадобностью. Всю защиту памяти теперь делают сами страницы через биты прав доступа и привилегий. Сегменты остались только для служебных задач.
