### 3. Inode – главная структура

**Inode** – это описание объекта файловой системы.

**Самое важное правило:** inode хранит **все характеристики объекта, кроме его имени**!

- Почему имя не в inode? Потому что имя хранится в **каталоге** (об этом дальше).

---

### 4. Как устроен каталог (на примере)

#### Корневой каталог и его inode

- Есть корневой каталог `/`, внутри него каталог `root`.
- У каждого объекта есть свой inode с **уникальным номером** (в пределах одной файловой системы).
- Если файловых систем несколько, то уникальный идентификатор – это пара `(ID_ФС, номер_inode)`.

**Пример:** inode каталога `/root` имеет номер 5.

#### Что хранится в каталоге

Данные каталога – это таблица из двух полей:
1. **Имя** дочернего объекта.
2. **Номер inode** этого объекта.

**Именно поэтому имя не хранится в inode!** Оно лежит в родительском каталоге.

#### Служебные записи в любом каталоге

- **`.`** (точка) – ссылается на сам каталог.
- **`..`** (две точки) – ссылается на родительский каталог.

**Пример содержимого каталога `/root` (inode 5):**

| Имя | Номер inode |
|-----|-------------|
| `.` | 5 |
| `..` | 2 (корневой каталог) |

#### Добавляем подкаталог `dir`

- Создаём каталог `dir` в `/root`. Ему выделяется новый inode (номер 10).
- В родительском каталоге `/root` появляется запись: `dir` → inode 10.
- В самом каталоге `dir` (inode 10) появляются записи:
  - `.` → inode 10 (ссылка на себя)
  - `..` → inode 5 (ссылка на родителя – `/root`)

#### Добавляем файл `file` в каталог `dir`

- Файл получает новый inode (номер 20), тип – файл.
- В каталоге `dir` появляется запись: `file` → inode 20.

---

### 5. Другие типы объектов в каталоге (примеры из лекции)

В каталоге `dir` могут быть записи и на другие типы объектов:

| Имя | Тип | Номер inode | Особенности |
|-----|-----|-------------|--------------|
| `mydev` | символьное устройство (c) | 30 | нет данных на диске, только major/minor номера |
| pipe | именованный канал (p) | 40 | нет данных, имя для объекта ядра |
| socket | локальный сокет (s) | 50 | нет данных, создаётся в ядре при открытии |
| link | мягкая ссылка (l) | 60 | хранит путь к другому объекту (в данных) |

> **Важно:** мягкая ссылка – это **отдельный объект** (в отличие от жёсткой ссылки, которая просто запись в каталоге).

---

### 6. Жёсткая ссылка (hardlink) – подробно

#### Что это такое

**Жёсткая ссылка** – это просто **запись в каталоге**, которая связывает имя с номером inode.

- У одного inode может быть **несколько жёстких ссылок** (несколько имён из разных каталогов).
- Каждая такая запись – это одна жёсткая ссылка.

#### Важные свойства

- Жёсткие ссылки можно создавать **только внутри одной файловой системы** (одного раздела). Нельзя ссылаться на другой диск или раздел.
- У каждого inode есть **счётчик жёстких ссылок** – сколько имён на него ссылается.

#### Как удаляется файл в Unix

Функция называется **`unlink()`**, а не `delete`. Почему?

- `unlink()` удаляет **одну жёсткую ссылку** (одно имя из каталога).
- Сам inode и его данные удаляются **только когда счётчик жёстких ссылок станет равен 0**.

**Пример:** если на файл есть две жёсткие ссылки (два имени), то после первого `unlink()` файл всё ещё существует (у него осталось одно имя). Только после второго `unlink()` данные действительно исчезают.

#### Пример со счётчиком ссылок в нашей иерархии

У inode каталога `/root` (номер 5) есть жёсткие ссылки:
1. Из корневого каталога (запись `root`)
2. Из самого себя (запись `.`)
3. Из подкаталога `dir` (запись `..`)

**Итого: 3 жёсткие ссылки.**

---

### 7. Разделы диска и файловые системы

#### Структура диска

На каждом жёстком диске есть:
1. **Блок начальной загрузки** (boot block) – для запуска компьютера.
2. **Таблица разделов** (partition table) – описывает, на какие части (разделы) разбит диск.
3. **Сами разделы**.

#### Что может быть в разделе

Каждый раздел может содержать **либо**:
- **Файловую систему** (данные, папки, файлы).
- **Пространство подкачки** (swap) – временное хранилище для выгруженной из памяти информации.

> **Важное отличие:** в Windows подкачка – это **файл** (pagefile.sys), который можно увеличивать. В Unix подкачка – это **отдельный раздел** фиксированного размера.

---

### 8. Внутреннее устройство файловой системы (Unix)

Файловая система внутри раздела – это **массив блоков** (кластеров) одинакового размера. Внутри этого массива три логические области:

| Область | Что хранит |
|---------|------------|
| **Суперблок (superblock)** | Атрибуты всей файловой системы (размеры, битовые карты, номер корневого inode) |
| **Таблица inode** | Массив структур inode (описания всех объектов) |
| **Блоки данных** | Содержимое файлов и данные каталогов |

> **Внимание:** это логические блоки (кластеры) файловой системы, не путать с физическими секторами диска.

---

### 9. Физические блоки (секторы) vs логические блоки (кластеры)

**Физический блок (сектор)**
- Минимальная единица передачи данных между памятью и диском.
- Размер: 512 байт или 4 КБ.

**Логический блок (кластер)**
- Минимальная управляемая единица файловой системы.
- Размер = целое число секторов (1, 2, 4, 8...).
- В оригинальной Unix: сектор = 512 б, кластер = 512 б (совпадали).
- Сегодня типичный размер кластера: **4 КБ**.

---

### 10. Выбор размера кластера – компромисс (важный пример лектора)

#### Если увеличить размер кластера (4 КБ вместо 512 б)

**Становится хуже:** внутренняя фрагментация (потери внутри последнего кластера файла).

**Пример:** файл 100 байт занимает целый кластер 4 КБ.  
Потеря: 4096 − 100 = **3996 байт** (около 97,5% места впустую).

**Реальный эксперимент лектора с каталогом System32 в Windows:**
- 2000 файлов меньше 50 байт.
- Потери: `2000 × 4046 ≈ 8 МБ`.
- При сотне тысяч мелких файлов потери достигнут **4 ГБ**.

#### Если уменьшить размер кластера (512 б)

**Становится хуже:** увеличивается количество управляющих структур.

**Пример:** файл 100 КБ
- При кластере 4 КБ → нужно ≈25 указателей.
- При кластере 512 б → нужно 200 указателей (почти в 10 раз больше).

#### Итог

| Критерий | Крупный кластер (4 КБ) | Мелкий кластер (512 б) |
|----------|------------------------|------------------------|
| Потери на мелких файлах | **Большие** | **Маленькие** |
| Размер управляющих структур | **Маленький** | **Большой** |
| Когда лучше | Файлы в основном крупные | Файлы в основном мелкие |

---

### 11. Суперблок – главные атрибуты ФС

Что хранит суперблок (пример из Unix):

| Параметр | Что значит |
|----------|------------|
| Идентификатор ФС | Уникальный ID |
| Тип ФС | Например, ext4 |
| Размер блока (кластера) | Задаётся при создании |
| Размер ФС | В блоках |
| Число блоков данных | Общее минус служебные |
| Битовый массив свободных inode | Для быстрого поиска |
| Битовый массив свободных блоков | Для быстрого поиска |
| Номер inode корневого каталога | Обычно 2 |
| Флаг "примонтирована" | При отмонтировании – сброс кэша |

> **Важно:** битовые массивы при монтировании загружаются в память и работают там (для скорости).

#### Соотношение inode и блоков

В классическом Unix: **на 2 блока данных → 1 inode** (среднее по статистике).

**Следствие (важное!):** если создавать много объектов без данных (файлы устройств, пустые каталоги, сокеты), можно **исчерпать inode**, а свободное место на диске ещё будет. То есть создать новый файл нельзя, хотя место есть.

#### Что происходит при отмонтировании

- Флаг `mounted` сбрасывается.
- Все данные из кэша (оперативной памяти) сбрасываются на диск.
- Если внезапно отключили питание – кэш может не сброситься.
- Современные ОС сбрасывают кэш каждые ~5 секунд.

### 12. Что хранит inode (атрибуты)

Лектор: «Почти всё, что выводит `ls -la` (кроме имени), хранится в inode».

| Атрибут | Пояснение |
|---------|-----------|
| Тип объекта | файл, каталог, устройство и т.д. |
| Пользователь-владелец (UID) | Кто создал |
| Группа-владелец (GID) | |
| Права доступа (12 бит) | rwx rwx rwx + дополнительные биты |
| Время последнего доступа (atime) | Когда читали |
| Время последнего изменения данных (mtime) | Когда меняли содержимое |
| Время последнего изменения метаданных (ctime) | Когда меняли права, владельца |
| Счётчик жёстких ссылок | Сколько имён ссылается |
| Размер файла | В байтах |
| Указатели на блоки данных | Где на диске лежат данные |

> **Важно:** самого номера inode внутри inode **нет** – это индекс в таблице.

### 13. Как устроен каталог на диске (в деталях)

В оригинальной Unix каждая запись каталога состоит из 4 полей:

| Поле | Размер | Смысл |
|------|--------|-------|
| `size` | 4 байта | размер всей записи |
| `name_len` | 4 байта | длина имени |
| `inode_num` | 4 байта | номер inode |
| `name` | переменная | имя + завершающий нуль |

#### Пример каталога с записями

| Запись | name_len | inode | Размер записи (байт) |
|--------|----------|-------|---------------------|
| `.` | 1 | 5 | 16 |
| `..` | 2 | 2 | 16 |
| `file` | 4 | 10 | 20 |
| `dir` | 3 | 20 | 16 |

**Общий выделенный размер под каталог – 512 байт** (один кластер). После последней записи (`dir`, 16 байт) остаётся **444 свободных байта**, которые считаются приписанными к ней.

#### Добавление новой записи

- Ищется запись, после которой есть свободное пространство.
- В начало свободного пространства пишется новая запись.
- Остаток свободного места приписывается к новой записи.

#### Удаление записи (важный механизм)

При удалении записи `file`:
- Запись `..` (16 байт) **расширяется** до 36 байт (16 + 20), "поглощая" удалённое место.
- После `..` появляется свободное пространство (20 байт), а затем запись `dir`.

> **Ключевое:** точка и две точки никогда не удаляются, поэтому перед любой удаляемой записью всегда есть предыдущая, к которой можно присоединить свободное место.

#### Вставка после удаления

При создании нового файла система сначала пытается поместить запись в свободное пространство. Если запись не влезает – переходит к следующему свободному участку.

### 14. Хранение данных в inode: 13 адресов (классическая Unix)

Inode содержит **массив из 13 элементов** – адресов блоков на диске.

#### Прямые адреса (10 штук)

- Элементы 0–9 → адреса первых 10 блоков данных.
- Максимальный размер: `10 × 512 = 5 КБ`

#### Одиночная косвенная адресация (элемент 10)

- Элемент 10 → блок, который содержит список адресов блоков данных.
- В одном блоке 512 байт, адрес = 4 байта → помещается 128 адресов.
- Максимальный размер: `128 × 512 = 64 КБ`

#### Двойная косвенная (элемент 11)

- Элемент 11 → блок первого уровня (128 адресов)
- Каждый адрес ведёт на блок второго уровня (ещё 128 адресов)
- Максимальный размер: `128 × 128 × 512 = 8 МБ`

#### Тройная косвенная (элемент 12)

- Три уровня косвенности.
- Максимальный размер: `128 × 128 × 128 × 512 = 1 ГБ`

**Итог:** максимальный размер файла в оригинальной Unix (блок 512 б) – чуть больше **1 ГБ**.

> **Исторический контекст:** ФС разработана около 1969 года. В 90-х диск на 1 ГБ уже был реален, а в 60-х – нет (лектор показывает картинку: 5-мегабайтный диск грузят в самолёт).

### 15. Что если увеличить блок (например, до 1 КБ)

**Вопрос:** во сколько раз вырастет максимальный размер файла?

- Размер блока увеличился в 2 раза.
- Количество адресов в блоке: было 128, стало 256 (тоже в 2 раза).

**Результат:** максимальный размер вырастает не в 2, а в **16 раз** (до ~16 ГБ).

| Тип адресации | Множитель |
|---------------|-----------|
| Прямые | ×2 |
| Одиночная косвенная | ×4 |
| Двойная косвенная | ×8 |
| Тройная косвенная | ×16 |

### 16. Три таблицы для доступа к файлам

Операционная система использует три уровня структур.

#### Уровень 1: таблица дескрипторов (у каждого процесса)

- `open()` возвращает дескриптор – это **индекс** в этой таблице.
- Каждая запись содержит указатель на запись в **общесистемной таблице файлов**.

#### Уровень 2: общесистемная таблица файлов (file table)

Хранит **контекст открытия**:

| Поле | Смысл |
|------|-------|
| Режим открытия | чтение / запись / добавление |
| Текущая позиция в файле | откуда читать/писать |
| Ссылка на inode | указатель на кэшированный inode |
| Счётчик ссылок из дескрипторов | сколько процессов/дескрипторов ссылаются |

#### Уровень 3: кэш inode (общесистемный)

- Inode, загруженные с диска и хранящиеся в памяти.
- Даже если счётчик ссылок из таблицы файлов стал 0, inode **остаётся в кэше** – он может скоро понадобиться снова.

#### Связь между таблицами

```
[Процесс]
  └─ дескриптор 3 → [запись в file table] → [запись в кэше inode]
```

### 17. Операции `open()` и `close()` – полный механизм

#### `open(path)`

1. **Поиск inode по имени** – проходим по пути от корня, находим inode файла.
2. **Работа с кэшем inode** – если inode не в кэше, загружаем с диска. Увеличиваем счётчик ссылок из таблицы файлов.
3. **Создаём запись в таблице файлов** – заполняем режим, позицию (обычно 0), указатель на inode. Счётчик ссылок из дескрипторов = 1.
4. **Создаём запись в таблице дескрипторов процесса** – находим свободный индекс, ставим указатель на запись из таблицы файлов. Возвращаем этот индекс.

#### `close(fd)`

1. **Удаляем запись из таблицы дескрипторов** процесса.
2. **Уменьшаем счётчик** в записи таблицы файлов.
   - Если стал 0 → удаляем запись таблицы файлов → уменьшаем счётчик у inode.
   - Если не 0 → останавливаемся (другие дескрипторы всё ещё используют).
3. **Inode остаётся в кэше** (даже если счётчик стал 0) – может пригодиться позже.

---

### 18. Важные случаи (примеры лектора)

#### Процесс открыл один файл дважды
- Один inode, **два разных контекста** (две позиции).
- **Предупреждение:** запись через `fd1`, чтение через `fd2` может не работать – данные могут не синхронизироваться.

#### `dup()` – копия дескриптора в том же процессе
- **Один контекст** (одна позиция) разделяется между двумя дескрипторами.
- Используется для перенаправления ввода/вывода (например, в конвейерах).

#### `fork()` – дочерний процесс
- Наследует таблицу дескрипторов, но записи указывают на **те же** записи таблицы файлов.
- Родитель и ребёнок **разделяют позицию** в файле.

### 19. Улучшения файловых систем: FFS (Fast File System, BSD Unix)

Оригинальная Unix FS была медленной – inode и данные далеко друг от друга. FFS ввела 4 улучшения.

#### Улучшение 1: Группы цилиндров

**Проблема:** диск разделён на дорожки. Inode – в одной области, данные – в другой. Головка диска постоянно прыгает.

**Решение:** разбить диск на **группы цилиндров** (близкие дорожки). В каждой группе – свой суперблок, своя таблица inode, свои данные.

**Три правила FFS:**
1. Все блоки одного файла – в одной группе.
2. Все файлы одного каталога – в одной группе.
3. Inode файла – в той же группе, что и его данные.

**Результат:** головка диска почти не прыгает, работа ускоряется.

**Цена:** нужно резервировать ~10% свободного места, чтобы соблюдать правила.

#### Улучшение 2: Увеличение блока до 4 КБ

- Блок стал в 8 раз больше.
- Количество адресов в блоке: было 128, стало 256.
- Максимальный размер файла вырос с ~1 ГБ до **~4 ТБ**.

**Цена:** увеличилась внутренняя фрагментация (потери на мелких файлах).

#### Улучшение 3: Фрагменты (fragments)

**Проблема:** средний файл в Unix тех лет был ~1 КБ, а блок стал 4 КБ. Потеря 3 КБ на файл – много.

**Решение:** блок 4 КБ делится на фрагменты по 1 КБ.
- Если файл маленький, он занимает только 1–3 фрагмента.
- Остальные фрагменты блока могут быть отданы другим файлам.

**Современное состояние:** диски большие, потерями 4 КБ пренебрегают. Фрагменты сейчас обычно не используются.

#### Улучшение 4: Учёт скорости CPU

**Исторический факт:** раньше CPU не всегда успевал обрабатывать данные, идущие с диска.

**Решение:** размещать данные на дорожке **через сектор** (пропуская один). У CPU есть время на обработку одного сектора, пока диск подводит следующий (пропущенный).

---

### 20. Реальный пример потерь (Windows)

Лектор посчитал на своём компьютере:

- Каталог `C:\Windows`: 235 000 файлов, 49 ГБ данных.
- Реальный занятый объём на диске: **на 110 ГБ больше** (всего ~159 ГБ).
- Потери = 110 ГБ.
- В среднем: `110 / 235 000 ≈ 468 байт потерь на файл`.

**Вывод:** для современных дисков это терпимо.

## Итог по всей теме (самое важное)

| Понятие | Суть |
|---------|------|
| **Inode** | Описание объекта (всё, кроме имени) |
| **Каталог** | Таблица `имя → inode` |
| **Жёсткая ссылка** | Запись в каталоге; у inode – счётчик ссылок |
| `unlink()` | Удаляет одну жёсткую ссылку |
| **Мягкая ссылка** | Отдельный объект, хранит путь |
| **Суперблок** | Атрибуты всей ФС (размеры, битовые карты, корневой inode) |
| **13 адресов inode** | 10 прямых + косвенные → максимум ~1 ГБ (512 б) или ~16 ГБ (1 КБ) |
| **Размер кластера** | Компромисс: крупный – меньше управления, но больше потерь |
| **Три таблицы** | дескрипторы процесса → таблица файлов (контекст) → кэш inode |
| **FFS** | Группы цилиндров, блок 4 КБ, фрагменты, учёт CPU |
