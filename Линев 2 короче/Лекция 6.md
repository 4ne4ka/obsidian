### Процессы и потоки: внутреннее устройство и управление

#### 1. Контекст и переключение (как ОС жонглирует задачами)

**Контекст** — это полный «слепок» состояния процесса или потока в данный момент. Он нужен, чтобы потом можно было возобновить работу ровно с того же места.

**Что входит в контекст процесса:**
- **Дескриптор** (паспорт процесса: ID, права, приоритеты).
- **Виртуальное адресное пространство** (карта памяти: где лежат код, данные, стеки).
- **Ресурсы** (список открытых файлов, сетевых соединений, окон).

**Что входит в контекст потока:**
- **Дескриптор потока** (ID, приоритет).
- **Аппаратный контекст** — самая важная часть: значения всех регистров процессора (программный счетчик IP, указатель стека SP, регистры общего назначения). Именно эти данные физически сохраняются в память и загружаются обратно при каждом переключении.

**Переключение контекста** происходит так:
1. ОС сохраняет регистры текущего потока в его дескриптор.
2. Если новый поток из другого процесса — переключает «вид на память» (таблицы виртуальной памяти) на новый процесс.
3. Загружает регистры нового потока в процессор.
4. Передает управление. Процессор «думает», что ничего не прерывалось.

**Важно:** Переключение контекста — дорогая операция (надо сохранить и загрузить килобайты данных), но она неизбежна для многозадачности.

#### 2. Дескриптор процесса (паспорт процесса в ядре)

В Linux это структура `task_struct`. В ней около 140–150 полей. Вот ключевые:

- **PID** — уникальный номер процесса.
- **PPID** — PID родителя (кто создал этот процесс).
- **Групповые параметры** — объединение процессов в группы (GID), сессии (SID) и задания (Job ID в Windows) для управления скопом (например, ограничить время CPU всем процессам задания).
- **Приоритет** — влияет на то, кому достанется процессор (конкурируют потоки) и память (конкурируют процессы).
- **Состояние процесса** — создание, выполнение, остановлен, завершается, завершён.
- **Статистика** — сколько времени CPU съел процесс и его пиковое использование памяти.
- **Описание адресного пространства** — карта виртуальной памяти (где лежат код, данные, куча, стеки, аргументы командной строки).
- **Контекст ввода-вывода** — какие файлы открыты, с какими правами, на какой позиции сейчас курсор.
- **Контекст безопасности** — состоит из двух частей:
    1.  Что **сам процесс** может делать (привилегии: менять время, завершать чужие процессы).
    2.  Что **можно делать с этим процессом** другим (права: кто может его приостановить или убить).
- **Код завершения** — 0 (успех) или не ноль (ошибка).
- **Текущий каталог** — нужен для работы с относительными путями к файлам.

#### 3. Дескриптор потока

В Linux отдельного дескриптора нет — для каждого потока создается своя `task_struct`, где поля, общие для процесса, просто дублируются.

Ключевые поля:
- **TID** — уникальный идентификатор потока.
- **PID процесса-владельца** — чтобы быстро найти его ресурсы (файлы, память). Потому что ресурсы принадлежат процессу, а не потоку.
- **Приоритет** — для конкуренции за время CPU.
- **Аппаратный контекст** — сохраненные регистры (хранятся в спецструктуре TSS, а в дескрипторе лежит указатель на нее).
- **Код завершения** — пока работает, там спецзначение `STILL_ACTIVE`.

#### 4. Пользовательские потоки (быстрые, но опасные)

Многопоточность можно сделать **без участия ядра** — силами библиотеки внутри самого процесса. Ядро видит только один процесс (и один его поток), а библиотека внутри крутит несколько «пользовательских» потоков.

**Плюсы:**
- **Очень быстро.** Создание, переключение и удаление потока — без системных вызовов, в 10–100 раз быстрее, чем у потоков ядра.
- **Свой планировщик.** Можно придумать свои правила, кому когда работать.

**Минусы:**
- **Нечестное распределение CPU.** Ядро не знает про внутренние потоки и дает процессу фиксированный квант времени. Если у одного процесса 10 потоков ядра, а у другого 1 поток ядра + 100 пользовательских, второй получит гораздо меньше времени.
- **Катастрофа с блокировкой.** Если хоть один пользовательский поток делает блокирующий системный вызов (например, `read` с диска), ядро «усыпляет» **весь процесс** (ведь для ядра внутри процесса только один поток). В результате замирают **все** пользовательские потоки.

**Как чинят проблему блокировок:** Библиотека перехватывает все системные вызовы и вместо блокировки использует неблокирующий режим + периодический опрос («данные уже пришли?»). Это работает, но сильно всё усложняет.

**Где используется:** Корутины (асинхронное программирование в Python, Go, Kotlin) — современная реализация похожей идеи.

#### 5. Создание и завершение (по шагам)

##### Создание процесса (Windows)
1. Создать дескриптор процесса и заполнить его поля (PID, PPID, приоритеты наследуются от родителя).
2. Создать пустое виртуальное адресное пространство (ВАП).
3. Сформировать структуру ВАП, разместив в нем код и данные. Адреса регионов берет **линковщик** на этапе сборки программы, записывая их в заголовок исполняемого файла. (DLL добавляют свои регионы кода и данных отдельно).
4. Выделить ресурсы по умолчанию (например, стандартные потоки ввода-вывода `stdin`, `stdout`).
5. Оповестить все подсистемы ядра о новом процессе.
6. Создать первичный поток.

##### Создание потока
1. Создать дескриптор потока и заполнить его поля (TID, приоритет).
2. Создать в ВАП процесса **стек** для этого потока.
3. Инициализировать **аппаратный контекст**: в его структуру записывают начальные значения регистров.
    - **Указатель стека (SP)** — на вершину только что созданного стека.
    - **Программный счетчик (IP)** — адрес первой инструкции. Для первичного потока берется из заголовка исполняемого файла (это не `main`, а стартовый код, который потом вызовет `main`). Для дополнительного потока — адрес функции, переданной в `CreateThread`.
4. Перевести поток в состояние «Готов».

##### Завершение потока
1. Сохранить в дескриптор статистику и код возврата.
2. Перевести ресурсы потока (например, захваченные блокировки) в стабильное состояние.
3. Освободить его личные ресурсы.
4. Установить состояние «Завершён».
5. **Важно:** Дескриптор потока не удаляется сразу. Он висит в системе, пока все процессы не закроют свои хэндлы на этот поток (т.к. кому-то может понадобиться узнать код возврата).
6. Если этот поток был последним в процессе — запускается завершение процесса.

##### Завершение процесса
1. Завершить (возможно, принудительно) все его потоки.
2. Сохранить в дескриптор статистику и код возврата.
3. Освободить **все** ресурсы процесса (память, файлы, сокеты, мьютексы).
4. Уничтожить виртуальное адресное пространство.
5. Установить состояние «Завершён».
6. **Важно:** Дескриптор процесса тоже остается висеть (аналог «зомби» в Linux), пока родительский процесс не заберет код возврата (через `wait`). В Windows — пока все процессы не закроют свои хэндлы на этот процесс.
