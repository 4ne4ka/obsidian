#### Напоминание о командном конвейере (Command Pipeline)

*   **Цель конвейера:** Ускорение выполнения инструкций процессором за счет их параллельного выполнения.
*   **Стадии конвейера (5 этапов):**
    1.  **IF (Instruction Fetch):** Выборка инструкции.
    2.  **ID (Instruction Decode):** Декодирование инструкции.
    3.  **EX (Execute):** Выполнение (например, арифметической операции).
    4.  **MEM (Memory Access):** Доступ к памяти (чтение или запись).
    5.  **WB (Write Back):** Запись результата в регистр.
*   **Идеальная работа:** В идеале, завершение выполнения одной инструкции происходит на каждом тактовом цикле. Хотя латентность (время выполнения одной инструкции) не меняется (все равно 5 шагов), пропускная способность (throughput) увеличивается до 1 инструкции/цикл.
*   **Определения:**
    *   **Латентность (Latency):** Количество шагов (тактов), необходимое для выполнения одной инструкции.
    * **Пропускная способность (Throughput):** Среднее количество инструкций, выполняемых за один тактовый цикл.
#### Конфликты в конвейере
Идеальная работа нарушается из-за конфликтов.

*   **Структурные конфликты (Structural Hazards):**
    *   **Причина:** Нехватка аппаратных ресурсов.
    *   **Типичные ситуации:**
        1.  Две стадии пытаются использовать один и тот же ресурс (например, алу) в одном цикле.
        2.  Некоторые стадии (например, доступ к памяти) могут занимать более одного такта. Если следующие инструкции "догонят" предыдущую на этой стадии, возникнет конфликт.
    *   **Результат:** Ожидание (stall).

*   **Конфликты по данным (Data Hazards):**
    *   **Причина:** Инструкции используют одни и те же данные (речь идет в первую очередь о регистрах).
    *   **Классификация:**
        *   **RAW (Read After Write):** Самая частая ситуация. Последующая инструкция читает данные, которые должна произвести предыдущая. Это истинная зависимость, а не конфликт, но она требует механизмов обхода.
        *   **WAR (Write After Read):** Последующая инструкция пытается записать данные в регистр, который еще не прочитала предыдущая. Может возникать при изменении порядка исполнения.
        *   **WAW (Write After Write):** Две инструкции пытаются записать в один регистр. Важен порядок записи.
    *   **В текущем конвейере:** WAR и WAW конфликты невозможны, так как стадии выполняются строго последовательно.

#### Метод обхода данных (Data Forwarding / Bypassing)

Передача результата с выхода стадии, где он был вычислен, напрямую на вход стадии, где он нужен, минуя стадию записи в регистр (WB) и последующего чтения (ID).
*   **Как это работает:**
    *   Результат арифметической операции готов в конце стадии **EX**. Он может быть передан сразу на вход стадии **EX** следующей инструкции.
    *   Результат чтения из памяти готов в конце стадии **MEM**. Он может быть передан на вход стадий **EX** или **MEM** следующих инструкций.
*   **Реализация:** В конвейер добавляются дополнительные "обходные" пути (forwarding paths):
    1.  С выхода **EX** на вход **EX**.
    2.  С выхода **MEM** на вход **EX**.
    3.  *(Отмечено лектором)* Теоретически, может потребоваться путь с выхода **MEM** на вход **MEM** (например, для цепочки `load` -> `store`), но на практике такие ситуации часто обрабатываются на уровне контроллера памяти и очередей чтения/записи.
*   **Примеры с Forwarding:**
    *   **Цепочка арифметических операций:** Результат первой операции, полученный в **EX**, передается сразу для второй операции. Задержек нет.
    *   **Операция `load` с последующей операцией:** Данные из памяти готовы в конце **MEM**. Если следующая инструкция использует их в **EX**, ей придется подождать один такт (появится "пузырь" - stall), даже с forwarding.

#### Планирование исполнения команд (Instruction Scheduling)

*   **Цель:** Избежать или минимизировать простои (stalls) в конвейере.
*   **Метод:** Изменение порядка исполнения инструкций без изменения семантики программы.
*   **Типы планирования:**
    *   **Статическое планирование (Static Scheduling):** Выполняется компилятором.
        *   Компилятор должен хорошо знать архитектуру целевого процессора.
        *   Он анализирует зависимости между инструкциями и переупорядочивает их так, чтобы инструкции, не зависящие друг от друга, выполнялись в "простои", вызванные зависимостями.
    *   **Динамическое планирование (Dynamic Scheduling):** Выполняется аппаратно самим процессором.
        *   Процессор сам меняет порядок исполнения декодированных инструкций "на лету".
        *   Стадии IF и ID обычно выполняются в порядке программы, а стадии EX, MEM, WB — в порядке готовности операндов.

#### Практический пример статического планирования

*   **Исходный код (неоптимальный):**
    ```
    // Блок 1
    load R0, B    // Загрузить B в R0
    load R1, C    // Загрузить C в R1
    add R0, R0, R1 // R0 = R0 + R1
    store R0, A    // Сохранить R0 в A

    // Блок 2
    load R0, D    // Загрузить D в R0
    load R1, E    // Загрузить E в R1
    sub R0, R0, R1 // R0 = R0 - R1
    store R0, F    // Сохранить R0 в F
    ```
*   **Проблема:** После инструкции `load` данные будут готовы только в конце стадии **MEM**. Инструкция `add` (или `sub`), которая использует эти данные в стадии **EX**, будет вынуждена ждать, создавая "пузырь" в конвейере, даже при использовании forwarding.

*   **Решение (перепланирование компилятором):**
    1.  **Разделение регистров:** Чтобы смешать инструкции из двух независимых блоков, нужно использовать разные регистры. Например, для Блока 1 использовать R0, R1, а для Блока 2 — R2, R3.
    2.  **Перестановка инструкций:** Компилятор помещает независимые инструкции между `load` и инструкцией, которая от него зависит.

*   **Оптимизированный код:**
    ```
    load R0, B    // Блок 1
    load R1, C    // Блок 1
    load R2, D    // Блок 2 (не зависит от Блока 1, выполняется в "простой")
    load R3, E    // Блок 2 (не зависит от Блока 1, выполняется в "простой")
    add R0, R0, R1 // Блок 1 (данные из R0 и R1 уже готовы)
    store R0, A    // Блок 1
    sub R2, R2, R3 // Блок 2 (данные из R2 и R3 уже готовы)
    store R2, F    // Блок 2
    ```
*   **Результат:** "Пузыри" в конвейере заполнены полезной работой, общее время выполнения сокращается.

#### Конфликты управления (Control Hazards)

*   **Причина:** Инструкции, изменяющие поток выполнения программы (значение счетчика команд `PC`).
*   **Типы инструкций:**
    *   Безусловные переходы (`jump`).
    *   Условные переходы (`branch`).
    *   Вызовы функций (`call`).
    *   Возвраты из функций (`return`).
*   **Фокус на условных переходах:** Именно они создают наибольшие проблемы, так как их результат (будет переход или нет) предсказать сложнее. Остальные типы переходов, как правило, хорошо предсказуемы.

*   **Суть проблемы:** Для определения следующей инструкции необходимо дождаться, пока инструкция условного перехода вычислит свой результат (условие и адрес перехода). Пока это вычисление не завершено, конвейер не может корректно выбрать и начать обработку следующей инструкции.
*   **Пример в базовом конвейере:** Предполагается, что условный переход завершает обработку на стадии **MEM**.
    *   **Результат:** Три такта простоя (три "пузыря" - `stall`), пока не станет известен адрес следующей инструкции.
*   **Зависимость потерь от стадии завершения перехода:**
    *   Результат известен в **MEM** -> 3 такта простоя.
    *   Результат известен в **EX** -> 2 такта простоя.
    *   Результат известен в **ID** -> 1 такт простоя.
    *   Результат известен в **IF** -> 0 тактов простоя.

#### Возможно ли определить переход на стадии IF?

Можно ли реализовать конвейер так, чтобы результат условного перехода был известен уже в конце стадии **IF**?
Теоретически да, но для этого потребуется:
    1.  **Декодирование в IF:** Добавить в стадию **IF** логику, которая сразу распознает, что текущая инструкция - это условный переход.
    2.  **Проверка условия:** На стадии **IF** необходимо иметь логику для проверки условия перехода (например, проверку регистра на равенство нулю).
    3.  **Вычисление адреса:** Добавить в **IF** дополнительный сумматор для вычисления относительного адреса перехода.
Реализовать это на практике в стадии **IF** очень сложно из-за ограниченности ресурсов и времени на этом этапе. Вместо этого используется другой подход — **предсказание переходов**.

#### Стратегии уменьшения потерь от конфликтов управления

Узнать, произошел ли переход, и вычислить адрес следующей инструкции как можно раньше в конвейере.

*   **A. Базовый метод (Наивный):**
    *   Остановка конвейера (`stall`) до тех пор, пока инструкция перехода не завершит свою работу (в примере - 3 такта). Не требует дополнительной аппаратной логики, но неэффективен.

*   **B. Перенос логики на стадию ID (Рассматриваемый пример):**
    *   **Что нужно сделать:** Переместить вычисление условия перехода и адреса на стадию **декодирования (ID)**.
    *   **Как это реализуется:**
        1.  В стадию **ID** добавляется логика проверки условия (в примере - проверка регистра на ноль).
        2.  Добавляется дополнительный сумматор для вычисления адреса перехода (для относительной адресации).
        3.  Сигнал о результате перехода и вычисленный адрес подаются на мультиплексер, который выбирает адрес следующей инструкции для стадии **IF**.
    *   **Результат:** Количество тактов простоя сокращается с 3 до **1**.

*   **C. Отложенное ветвление (Delayed Branch):**
    *   **Идея:** Инструкция, следующая *непосредственно* за инструкцией условного перехода (называемая "слотом отложенного перехода" - `branch delay slot`), **всегда выполняется**, независимо от того, произошел переход или нет.
    *   **Как используется:** Компилятор помещает в этот слот инструкцию, которую необходимо выполнить в любом случае, или полезную инструкцию, не зависящую от результата перехода. Таким образом, такт, который был бы потерян как "пузырь", заполняется полезной работой.
    *   **Пример выполнения:**
        *   **Переход не произошел:** Выполняются: переход -> инструкция в слот -> следующая инструкция.
        *   **Переход произошел:** Выполняются: переход -> инструкция в слот -> инструкция по целевому адресу перехода.
    *   **Недостаток для современных процессоров:** В глубоких конвейерах (15+ стадий) обработка перехода может завершаться только на 4-м такте. Чтобы эффективно использовать эту технику, потребовалось бы несколько слотов отложенного перехода (3 и более), что сильно усложняет работу компилятора.
    *   **Примечание:** Эта техника была характерна для архитектуры MIPS. Современные архитектуры (например, ARM, RISC-V) от нее отошли.

#### Заключительные замечания

*   В современных процессорах для борьбы с конфликтами управления повсеместно используется **предсказание переходов** на стадии **IF**, что позволяет сразу начать выборку инструкций по предсказанному адресу.
*   Стадии конвейера в современных процессорах часто разделены (например, `Fetch 1`, `Fetch 2`, `Decode 1`, `Decode 2`), и логика обработки переходов может быть распределена между ними.
*   Если предсказание оказывается неверным, конвейер очищается ("сбрасывается" - `flush`), что приводит к потере тактов, но в среднем правильное предсказание значительно ускоряет работу.

#### Предсказание переходов (Branch Prediction) и спекулятивное выполнение

Процессор пытается предугадать, будет ли выполнен условный переход, и начинает выполнение инструкций по предсказанному адресу *до того*, как станет известен реальный результат условия перехода.
*   **Механизм работы:**
    1.  **Предсказание:** При встрече с инструкцией условного перехода процессор на стадии **IF** предсказывает направление перехода (будет переход или нет).
    2.  **Спекулятивное выполнение:** Процессор начинает выбирать и выполнять инструкции по предсказанному адресу. Результаты этого выполнения являются спекулятивными и пока никуда не записываются (не фиксируются).
    3.  **Проверка:** Когда инструкция условного перехода реально вычисляет условие и адрес (например, на стадии **ID** или **EX**), происходит проверка.
    4.  **Два сценария:**
        *   **Предсказание верное:** Спекулятивно выполненные инструкции фиксируются, и выполнение продолжается без задержек.
        *   **Предсказание неверное:** Все результаты спекулятивного выполнения отбрасываются, конвейер частично очищается, и начинается выполнение инструкций с правильного адреса.

*   **Стратегии статического предсказания (простейшие):**
    *   **«Переход никогда не происходит» (Predict Not Taken):** Процессор всегда предполагает, что условие перехода ложно, и продолжает последовательное выполнение инструкций, следующих за переходом.
        *   **Простота:** Легко реализуется на аппаратном уровне.
    *   **«Переход всегда происходит» (Predict Taken):** Процессор всегда предполагает, что условие перехода истинно, и сразу начинает выборку с целевого адреса.
        *   **Сложность:** Требует быстрого вычисления адреса перехода на ранней стадии, что сложнее в реализации.

*   **Взаимодействие с компилятором:** Эффективность статического предсказания зависит от компилятора. Если процессор использует стратегию "переход никогда не происходит", компилятор должен располагать код так, чтобы наиболее вероятный путь выполнения был путем последовательных инструкций (т.е. путь, где условие ложно).

*   **Пример для стратегии "Predict Not Taken":**
    *   **Переход не произошел (предсказание верное):** Инструкции выполняются последовательно, без простоев.
    *   **Переход произошел (предсказание неверное):**
        1.  Инструкция, следующая за переходом, выполняется спекулятивно.
        2.  Как только на стадии **ID** вычисляется, что переход должен произойти, результат спекулятивного выполнения отбрасывается.
        3.  Начинается выборка инструкции по правильному (целевому) адресу.
        4.  **Результат:** Возникает **один такт простоя** (в сравнении с тремя тактами в наивной реализации).

#### Оценка производительности конвейера (Расчет CPI)

*   **Цель:** Оценить среднее количество тактов, затрачиваемых на одну инструкцию (Cycles Per Instruction - CPI), при различных комбинациях оптимизаций.
*   **Исходные данные (статистика):**
    *   Арифметические операции: 40%
    *   Операции загрузки (`load`): 30%
    *   Операции сохранения (`store`): 10%
    *   Операции условного перехода: 20%
    *   Дополнительно: 25% инструкций `load` сразу же используют загруженные данные.
    *   Дополнительно: 65% условных переходов выполняются (переход происходит).

*   **Оптимизации конвейера (8 комбинаций):**
    1.  Пересылка данных (Data Forwarding).
    2.  Раннее определение перехода (вычисление на стадии **ID**).
    3.  Предсказание "переход не происходит".

*   **Формула:** `CPI = Идеальный CPI + Потери от конфликтов данных + Потери от конфликтов управления + Потери от структурных конфликтов`
    *   **Идеальный CPI** в конвейере равен 1.

*   **Пример расчета (с определенными допущениями):**
    *   **Потери от конфликтов данных (загрузка + использование):**
        *   Доля `load` инструкций: 30%
        *   Доля `load`, после которых данные используются сразу: 25%
        *   Даже с пересылкой возникает 1 такт простоя.
        *   **Потери:** `0.30 * 0.25 * 1 = 0.075`
    *   **Потери от конфликтов управления (неверное предсказание):**
        *   Доля инструкций перехода: 20%
        *   Доля переходов, которые происходят: 65%
        *   При раннем определении и предсказании "not taken" потери = 1 такт за неверное предсказание.
        *   **Потери:** `0.20 * 0.65 * 1 = 0.13`
    *   **Суммарный CPI (предварительный):** `1 + 0.075 + 0.13 = 1.205`

*   **Важные замечания лектора:**
    1.  **Структурные конфликты:** В расчете не учтены структурные конфликты. Например, если инструкция обращения к памяти (`load`/`store`, 40%) блокирует доступ к памяти для стадии **IF**, это добавляет еще один такт простоя на каждую такую инструкцию. Это может значительно увеличить CPI (например, до `1 + 0.4 = 1.6`).
    2.  **Нереалистичное суммирование:** Механическое сложение потерь от разных конфликтов некорректно. На практике конфликты могут "перекрываться" (например, простой из-за перехода может "поглотить" простой из-за конфликта данных). Реальные потери будут меньше арифметической суммы.
    3.  **Заниженные значения:** Полученное значение CPI ~1.2 является очень оптимистичным и на практике для простых процессоров будет выше.

#### Резюме по структурным конфликтам

*   **Пример:** Инструкции `load` и `store` (40% от всех) используют блок работы с памятью. Если этот же блок нужен для стадии **IF** для выборки следующей инструкции, возникает структурный конфликт. Это приводит к приостановке стадии **IF** и добавлению одного такта простоя.


#### Углубление в статическое предсказание переходов

Направление предсказания (будет переход или нет) указывается непосредственно в самой инструкции условного перехода с помощью специального бита. Это указание устанавливается компилятором.
*   Компилятор анализирует программу и для каждого перехода решает, какое направление более вероятно.
*   **Два метода анализа для компилятора:**
    1.  **Статический анализ исходного кода:** Компилятор анализирует логику программы. Классический пример — циклы. Условие выхода из цикла (переход) обычно ложно на всех итерациях, кроме последней. Следовательно, компилятор может предсказать "переход не происходит".
    2.  **Профилирование (Profiling):**
        *   **Процесс:** Программа компилируется без предсказаний, затем запускается на репрезентативных тестовых данных.
        *   **Сбор статистики:** Во время прогона собирается статистика — сколько раз каждый условный переход был выполнен. Это можно сделать с помощью инструментации кода или аппаратных счетчиков.
        *   **Внесение предсказаний:** Специальная программа обрабатывает собранную статистику и модифицирует исполняемый файл, проставляя в инструкциях переходов биты предсказания.
*   **Эффективность:** Профилирование дает более точные предсказания, чем статический анализ, так как основано на реальном поведении программы.
*   **Историческая справка:** Такая техника активно использовалась около 20 лет назад и поддерживалась в некоторых архитектурах (не в x86). Эффективность статического предсказания на основе профиля составляла примерно 5-20% неправильных предсказаний.

#### Динамическое предсказание переходов (Краткий обзор)

Процессор самостоятельно, "на лету", предсказывает направление переходов на основе истории их выполнения в ходе работы программы. Основная логика работает на стадии **IF**.
*   **Механизм:** Процессор хранит историю выполнения предыдущих переходов (например, в специальном буфере — Branch History Buffer, BHB) и использует эту информацию для предсказания следующего шага.
*   **Эффективность:** Современные алгоритмы динамического предсказания позволяют достичь точности около **97%** (т.е., всего 2-3% неправильных предсказаний).
*   **Пример для циклов:** Даже простейшие алгоритмы динамического предсказания хорошо работают с циклами. После первой итерации они "запоминают", что переход происходит, и правильно предсказывают все последующие итерации, кроме последней (выхода из цикла). Более сложные алгоритмы минимизируют количество ошибок.
