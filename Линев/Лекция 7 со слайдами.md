# Конспект лекции по параллелизму на уровне инструкций (ILP) и разворачиванию циклов

## Введение в ILP и базовые блоки

**ILP (Instruction Level Parallelism)** — это свойство программы, позволяющее процессору выполнять несколько инструкций одновременно. Задача компилятора и программиста — максимизировать ILP для ускорения выполнения.

**Базовый блок инструкций** — это последовательность инструкций:
- Без переходов внутрь (кроме точки входа)
- Без переходов наружу (кроме точки выхода)
- Пример: тело цикла или ветка условия

Компиляторы обычно оптимизируют код в пределах **базового блока**, так как общая оптимизация всей программы (со множеством возможных путей выполнения) слишком сложна.

## Конфликты и зависимости между инструкциями

**Зависимые инструкции нельзя переупорядочивать** — ни компилятором, ни аппаратурой. Зависимости делятся на три типа:

### Зависимости по данным (Data Dependencies)
Возникают, когда инструкция `j` использует результат инструкции `i`.
- **Пример**: `DADD F0, F1, F2` → `DADD F8, F2, F9` (вторая инструкция использует F2 из первой)
- **Конфликт**: RAW (Read After Write) — если порядок нарушен, результат будет неверным
- **Обозначение на графе**: обычная стрелка `→`

**Важно**: Зависимость по данным — это *истинная зависимость*, её нельзя устранить, можно только учитывать при планировании.

### Зависимости по именам (Name Dependencies)
Возникают, когда инструкции используют один и тот же регистр или ячейку памяти, но **без потока данных** между ними.

#### a) Антизависимость (Anti-dependency)
- Инструкция `j` записывает в регистр, который читает инструкция `i`
- **Конфликт**: WAR (Write After Read) — если `j` запишет раньше, чем `i` прочитает
- **Обозначение**: стрелка с крестом `- - - →`

#### b) Выходная зависимость (Output Dependency)
- Обе инструкции `i` и `j` записывают в один и тот же регистр
- **Конфликт**: WAW (Write After Write) — если порядок записи нарушен
- **Обозначение**: стрелка с кружком `○→`

**Устранение**: зависимости по именам можно устранить **переименованием регистров** (использованием разных регистров).

### Зависимости по управлению (Control Dependencies)
Связаны с условными переходами — рассматриваются отдельно.

## Разворачивание циклов как метод увеличения ILP

### Пример исходного цикла
```assembly
for (i=1000; i>0; i=i-1)
    x[i] = x[i] + s;

Loop:
    LD   [R1], F0      ; загрузка элемента массива
    FADD F2, F0, F4    ; добавление константы из F2
    ST   F4, [R1]      ; сохранение результата
    ADD  R1, R3, R1    ; уменьшение указателя (R3 = -8)
    BNE  R1, R2, Loop  ; переход, если R1 != R2
```
- Базовый блок: 5 инструкций
- Без оптимизации: 10 тактов на итерацию (из-за простоев от конфликтов данных)

### Оптимизация внутри базового блока
Переупорядочивание инструкций для заполнения простоев:
```assembly
Loop:
    LD   [R1], F0
    ADD  R1, R3, R1    ; переместили ADD выше
    FADD F2, F0, F4
    BNE  R1, R2, Loop  ; использовали слот задержки перехода
    ST   F4, [R1+8]    ; учли, что R1 уже уменьшен
```
Результат: **6 тактов на итерацию** (вместо 10).

### Механическое разворачивание цикла (без оптимизации)
Берём 4 итерации исходного цикла и объединяем их в одну:
- Используем разные регистры для каждой "под-итерации" (F0,F4; F6,F8; F10,F12; F14,F16)
- Убираем 3 перехода и 3 изменения R1 (остаётся одно уменьшение на 32 вместо 4×8)
- Размер блока: с 5 до 14 инструкций
- Результат: **7 тактов на исходную итерацию** (28 тактов на 4 итерации)

**Выгоды**:
- Увеличивается базовый блок → больше инструкций для перепланирования
- Меньше инструкций переходов и служебных инструкций цикла

### Разворачивание + переупорядочивание
```assembly
Loop:
    LD   [R1], F0      ; 4 загрузки подряд
    LD   [R1-8], F6
    LD   [R1-16], F10
    LD   [R1-24], F14
    FADD F2, F0, F4    ; 4 сложения подряд
    FADD F2, F6, F8
    FADD F2, F10, F12
    FADD F2, F14, F16
    ST   F4, [R1]      ; сохранения (частично переупорядочены)
    ST   F8, [R1-8]
    ADD  R1, R3, R1    ; R3 = -32
    ST   F12, [R1+16]
    BNE  R1, R2, Loop
    ST   F16, [R1+8]
```
Результат: **3.5 такта на исходную итерацию** (14 тактов на 4 итерации).

**Сравнение**:
- Исходный: 10 тактов/итерация
- С оптимизацией: 6 тактов/итерация
- С разворачиванием: 7 тактов/итерация
- С разворачиванием + оптимизацией: 3.5 такта/итерация
- **Ускорение**: 6 / 3.5 ≈ 1.7 раза

## Ключевые требования для разворачивания циклов

1. **Независимость итераций** — программист должен гарантировать, что итерации не зависят друг от друга (компиляторы пока плохо анализируют это автоматически).

2. **Большое количество регистров** — чтобы избежать конфликтов по именам (WAR, WAW) при использовании разных регистров для каждой "под-итерации".

   *Примечание*: современные компиляторы и процессоры могут переиспользовать регистры при разворачивании, но это сложнее и требует учёта зависимостей.

## Графическое представление зависимостей

**Правило переупорядочивания**: инструкцию можно переместить вверх или вниз, если **все стрелки зависимостей продолжают указывать вниз** (от более ранних к более поздним инструкциям).

**Пример графа для двух итераций цикла (с общими регистрами)**:
```
    LD [R1], F0     → (RAW) → FADD F2, F0, F4 → (RAW) → ST F4, [R1]
      ↓ (WAW)           ↑ (WAR)                  ↓ (WAW)    ↑ (WAR)
    LD [R1-8], F0  → (RAW) → FADD F2, F0, F4 → (RAW) → ST F4, [R1-8]
```
- Сплошные стрелки → зависимости по данным (RAW)
- Стрелки с кружком ○→ выходные зависимости (WAW)
- Стрелки с крестом - - - → антизависимости (WAR)

**После переименования регистров** зависимости по именам исчезают, остаются только истинные зависимости по данным.

## Практические рекомендации

1. **Для программиста**: пишите циклы с максимально независимыми итерациями.
2. **Для компилятора**: используйте разворачивание в сочетании с переупорядочиванием.
3. **Для архитектуры**: больше регистров → больше возможностей для разворачивания.
4. **Альтернатива**: векторные инструкции (если поддерживаются архитектурой) — работают с массивами данных за одну инструкцию.



## Зависимости по управлению (Control Dependencies)

### Определение
Зависимости по управлению определяют порядок выполнения инструкций с учётом **условных переходов** (if, циклы).

**Правила:**
1. **Инструкция, зависимая от перехода**, не может быть перемещена **перед** переходом так, чтобы она выполнялась всегда (вне управления этим переходом).
   
   *Пример:*
   ```c
   if (p1) {
       S1;  // Зависит от p1 — нельзя вынести перед if
   }
   ```

2. **Инструкция, независимая от перехода**, не может быть перемещена **после** перехода так, чтобы она стала зависимой от него.

   *Пример:* Если есть код после if, который выполняется всегда, его нельзя переместить внутрь условия.

### Пример с вложенными условиями
```c
if (p1) {
    S1;
}
if (p2) {
    S2;
}
```
- `S1` зависит по управлению **только от `p1`**
- `S2` зависит **только от `p2`**, но не от `p1` (если нет else)

**Важный момент:** если в ветках `if` и `else` есть **одинаковый код**, его можно **вынести за пределы условия** — это один из способов оптимизации.

### Практическое значение
Зависимости по управлению **ограничивают переупорядочивание инструкций**. Современные процессоры и компиляторы стараются:
- **Предсказывать переходы** (branch prediction)
- **Выполнять спекулятивно** (speculative execution) — выполнять инструкции до того, как известно условие

Но программист должен писать код так, чтобы минимизировать количество условных переходов, если это возможно.

## Пример анализа зависимостей в развёрнутом цикле

### Исходный код (2 итерации)
```assembly
1: LD   [R1], F0
2: DADD F2, F0, F4
3: ST   F4, [R1]
4: LD   [R1-8], F0      // та же пара регистров F0, F4
5: DADD F2, F0, F4
6: ST   F4, [R1-8]
```

###  Зависимости:
**По данным (RAW):**
- 2 → 1 (использует F0)
- 3 → 2 (использует F4)
- 5 → 4 (использует F0)
- 6 → 5 (использует F4)

**По именам:**
- **WAW** (выходные): 
  - 1 и 4 (пишут в F0)
  - 2 и 5 (пишут в F4)
- **WAR** (антизависимости):
  - 2 и 4 (2 читает F0, 4 пишет в F0)
  - 3 и 5 (3 читает F4, 5 пишет в F4)

### Граф зависимостей:
```
  1(LD F0) →(RAW)→ 2(DADD F4) →(RAW)→ 3(ST F4)
    ↓(WAW)           ↓(WAR)           ↓(WAR)
  4(LD F0) →(RAW)→ 5(DADD F4) →(RAW)→ 6(ST F4)
    ↑(WAR)           ↑(WAW)           ↑(WAW)
```

### Ответы на вопросы лектора:

#### Вопрос 1: Можно ли переместить инструкцию 3 (первый ST) после инструкции 4 (второй LD)?
**Ответ: Да, можно.**  
Проверяем стрелки:
- Зависимость 3 → 2 (RAW) сохранится (2 выше 3)
- Антизависимость 3 → 5 (WAR) не нарушится (3 будет выше 5)
- **Итог:** все стрелки идут вниз → перестановка допустима.

#### Вопрос 2: Можно ли переместить инструкцию 4 (второй LD) после инструкции 1 (первый LD)?
**Ответ: Нет, нельзя.**  
Нарушится **антизависимость (WAR)** между 2 и 4:
- Инструкция 2 читает F0
- Инструкция 4 пишет в F0
Если 4 поставить сразу после 1, то 4 запишет в F0 **раньше**, чем 2 прочитает из него → конфликт WAR.

## Разворачивание цикла при нехватке регистров

### Проблема
Если регистров мало, приходится использовать **одни и те же регистры в разных итерациях** развёрнутого цикла.

**Пример (4 итерации, регистры F0, F4 переиспользуются):**
```assembly
1:  LD   [R1], F0
2:  DADD F2, F0, F4
3:  ST   F4, [R1]
4:  LD   [R1-8], F0      // снова F0!
5:  DADD F2, F0, F4      // снова F4!
...
```

**Появляются дополнительные зависимости по именам (WAW, WAR) между итерациями:**
- WAW: 1 ↔ 4 ↔ 7 ↔ 10 (все пишут в F0)
- WAR: 2 ↔ 4 (2 читает F0, 4 пишет в F0), 5 ↔ 7, 8 ↔ 10 и т.д.

### Помогает ли такое разворачивание?
**Ответ: Почти нет.**  
- Основная выгода от разворачивания — **устранение переходов** и увеличение базового блока.
- Но если регистры переиспользуются, возникают **зависимости по именам**, которые ограничивают параллелизм.
- Единственный небольшой выигрыш — **лучшее использование слота задержки перехода** (после BNE сразу идёт полезная инструкция).

**Вывод:** без переименования регистров такое разворачивание почти бесполезно.

## Устранение зависимостей по именам через переименование регистров

### Решение
Использовать **разные регистры** для каждой "под-итерации" в развёрнутом цикле.

**Пример после переименования:**
```assembly
1:  LD   [R1], F0      // итерация 1
2:  DADD F2, F0, F4
3:  ST   F4, [R1]
4:  LD   [R1-8], F6    // итерация 2 — новые регистры!
5:  DADD F2, F6, F8
6:  ST   F8, [R1-8]
7:  LD   [R1-16], F10  // итерация 3
...
```

### Результат
- **Исчезают все зависимости по именам** (WAW, WAR)
- **Остаются только истинные зависимости по данным** (RAW) внутри каждой тройки инструкций
- **Появляется возможность переупорядочивания** между итерациями

**Граф становится простым:**
```
[Итерация 1: 1→2→3]   [Итерация 2: 4→5→6]   [Итерация 3: 7→8→9] ...
```
Все эти блоки **независимы** и могут перемешиваться.

## Параллелизм уровня цикла (Loop-Level Parallelism)

### Определение
Это возможность **параллельного выполнения разных итераций цикла**.

**Требование:** итерации должны быть **независимыми** (никаких данных, произведённых на одной итерации, не используются на другой).

**Пример идеального независимого цикла:**
```c
for (i = 0; i < N; i++) {
    A[i] = B[i] + C[i];  // каждая итерация работает со своим i
}
```

### Роль программиста
- Писать циклы с **максимально независимыми итерациями**
- Избегать зависимостей между итерациями (например, `A[i] = A[i-1] + ...`)

### Роль компилятора
- **Анализировать** зависимости между итерациями
- **Разворачивать** циклы
- **Переименовывать** регистры
- **Переупорядочивать** инструкции

### Важное замечание про указатели
Если функция принимает **три указателя** `A, B, C`, компилятор **не знает**, перекрываются ли эти массивы в памяти.

**Пример опасной ситуации:**
```c
void func(int *A, int *B, int *C) {
    for (i = 0; i < N; i++) {
        A[i] = B[i] + C[i];
    }
}
```
Если `B = A + 1` и `C = A + 2`, то итерации **зависимы** — изменение `A[i]` влияет на `B[i-1]` и `C[i-2]`.

**Решение:** использовать ключевые слова типа `restrict` (в C) или указывать компилятору, что массивы не перекрываются.

## Выводы

1. **Зависимости по управлению** ограничивают перестановку инструкций вокруг условных переходов.
2. **Разворачивание циклов** эффективно **только с переименованием регистров**.
3. **Параллелизм уровня цикла** требует **независимости итераций**.
4. **Программист** должен:
   - Минимизировать условные переходы
   - Писать независимые итерации циклов
   - Указывать компилятору на неперекрывающиеся данные
5. **Компилятор** делает:
   - Разворачивание
   - Переименование регистров
   - Переупорядочивание
   - Спекулятивное выполнение


## Анализ циклов с межитерационными зависимостями

### Пример проблемного цикла

**Исходный код:**
```c
for (i = 1; i <= 100; i = i + 1) {
    A[i] = A[i] + B[i];     // S1
    B[i+1] = C[i] + D[i];   // S2
}
```

### Анализ зависимостей

**Межитерационные зависимости (между разными итерациями):**
1. **S1 → S1 по A[i]**: НЕТ — каждая итерация работает со своим A[i]
2. **S2 → S2 по B[i]**: НЕТ — каждая итерация записывает B[i+1], читает B[i]
3. **Ключевая проблема: S2 → S1 по B[i+1]**:
   - На итерации `i`: S2 вычисляет `B[i+1]`
   - На итерации `i+1`: S1 читает `B[i+1]`
   - **Зависимость через B[i+1] между итерациями**

**Визуализация:**
```
Итерация i:   S1: A[i] = A[i] + B[i]
              S2: B[i+1] = C[i] + D[i]
                    ↓
Итерация i+1: S1: A[i+1] = A[i+1] + B[i+1]  ← использует B[i+1] из предыдущей итерации
```

### Граф зависимостей
- **Внутри одной итерации**: S1 → S2? НЕТ, независимы
- **Между итерациями**: S2(i) → S1(i+1) через B[i+1]

Это создаёт **цепочку зависимостей** от первой до последней итерации → цикл нельзя распараллелить.

### Решение: изменение границ итераций

**Идея:** перестроить цикл так, чтобы все зависимости были **внутри одной итерации**.

**Модифицированный код:**
```c
// Инициализация
A[1] = A[1] + B[1];

// Основной цикл (99 итераций вместо 100)
for (i = 1; i <= 99; i = i + 1) {
    B[i+1] = C[i] + D[i];       // S2 текущей итерации
    A[i+1] = A[i+1] + B[i+1];   // S1 следующего элемента
}

// Завершение
B[101] = C[100] + D[100];
```

**Что изменилось:**
1. **Первая операция S1** вынесена перед циклом
2. **Внутри цикла**: S2(i) → S1(i+1) (но теперь это внутри одной итерации!)
3. **Последняя S2** вынесена после цикла

**Новый граф выполнения:**
```
Инициализация: A[1] = A[1] + B[1]

Итерация 1:  B[2] = C[1] + D[1]  →  A[2] = A[2] + B[2]
Итерация 2:  B[3] = C[2] + D[2]  →  A[3] = A[3] + B[3]
...
Итерация 99: B[100] = C[99] + D[99] → A[100] = A[100] + B[100]

Завершение: B[101] = C[100] + D[100]
```

**Результат:** Все зависимости теперь **внутриитерационные** → цикл можно распараллеливать!

## Тест GCD для обнаружения межитерационных зависимостей

### Формальная постановка задачи

Компилятор анализирует, есть ли зависимость между:
- **Сохраняемым** элементом с индексом: `a × j + b`
- **Загружаемым** элементом с индексом: `c × k + d`

Где:
- `j, k` — номера итераций
- `j < k` (более ранняя итерация пишет, более поздняя читает)

### Условие зависимости

Зависимость существует, если уравнение имеет решение:
```
a × j + b = c × k + d
```
при условиях:
1. `m ≤ j < k ≤ n` (в пределах границ цикла)
2. `j` и `k` — целые числа

### Пример применения

Для нашего цикла:
- S2 записывает: `B[i+1]` → индекс: `1 × i + 1` (a=1, b=1)
- S1 читает: `B[i]` → но в следующей итерации: `B[(i+1)]` → индекс: `1 × (i+1) + 0` (c=1, d=0)

Уравнение:
```
1 × j + 1 = 1 × k + 0
j + 1 = k
```
Решение: `k = j + 1` → зависимость существует между соседними итерациями.

## Динамическое планирование выполнения инструкций

### Почему нужно динамическое планирование?

**Ограничения статического (компиляторного) планирования:**
1. Компилятор не знает **конкретный путь выполнения** при условных переходах
2. Не знает **задержки доступа к памяти** (кэш-промахи)
3. Не может учесть **конфликты ресурсов** во время выполнения

**Пример:** Вложенные условные операторы создают дерево возможных путей. Компилятор оптимизирует каждый базовый блок отдельно, но не оптимизирует "стыки" между блоками при случайных переходах.

### Что такое динамическое планирование?

**Динамическое планирование** — это **аппаратный механизм**, который:
- **Переупорядочивает выполнение инструкций** во время выполнения программы
- Начинает выполнение инструкции, как только **стали доступны её операнды**
- Позволяет **скрывать задержки** (доступ к памяти, длинные операции)

**Ключевая идея:** "Out-of-order execution" (неупорядоченное выполнение)
Инструкция начинает выполняться, как только **готовы все её операнды**, а не в порядке программы.

### Пример
```assembly
DIV F2, F4, F0   ; Долгая операция
ADD F0, F8, F10  ; Зависит от F0 → ждёт DIV
SUB F8, F14, F12 ; Независима → может выполняться параллельно
```
**Динамическое планирование:** SUB может начать выполнение раньше, пока ADD ждёт результат DIV.

### Как это работает?

**Поток обработки инструкций:**
1. **Выборка** (Fetch) → **Декодирование** (Decode) → **Определение зависимости**
2. **Ожидание операндов** (Operands wait)
3. **Как только операнды готовы** → инструкция поступает на исполнение
4. **Исполнение** может происходить **не в порядке программы**

**Важное уточнение:** "Неупорядоченное" выполнение — это относительно **исполнения** (EX), а не всей цепочки:
- Выборка и декодирование — всегда по порядку
- После декодирования процессор анализирует зависимости
- Инструкции с готовыми операндами идут на исполнение раньше

### Пример преимущества

```c
// Компилятор видит:
LD  [R1], F0    // Загрузка из памяти (долгая)
DADD F2, F0, F4 // Ждёт F0
DADD F6, F8, F10 // Независимая инструкция
```

**Статически:** компилятор может переставить, если поймёт, что третья инструкция независима.

**Динамически:** процессор сам обнаружит, что третья инструкция не зависит от первых двух, и выполнит её раньше, пока ждёт загрузку из памяти.

### Особенности динамического планирования

1. **Сохраняет логическую корректность** — конечный результат как в последовательном выполнении
2. **Требует сложной аппаратной поддержки**:
   - Таблицы зависимостей
   - Буферы переупорядочивания
   - Механизмы переименования регистров
3. **Эффективно скрывает задержки** доступа к памяти

## Изменение структуры конвейера

### Старая структура (MIPS 5-стадийный)
```
IF → ID → EX → MEM → WB
```

### Новая структура с динамическим планированием
```
IF → ID1 → ID2 → EX → WB
        ↓
    Динамическое
   планирование
```

### Подробнее о новых стадиях:

#### а) ID1 — Выдача (Issue)
- **Декодирование инструкции**
- **Проверка структурных конфликтов** (есть ли свободное функциональное устройство)
- **Проверка WAW-конфликтов** (регистр назначения не должен быть помечен для записи другой инструкцией)
- **Выполняется строго по порядку программы**
- **В табло создаётся запись** о зависимости

#### б) ID2 — Чтение операндов (Read Operands)
- **Ожидание готовности операндов** (разрешение RAW-конфликтов)
- **Чтение операндов**, как только они доступны
- **Может происходить не по порядку программы**
- **Нет поддержки пересылки (forwarding)** — операнды читаются только из регистрового файла

#### в) EX — Исполнение
- **Функциональное устройство выполняет операцию**
- **Сообщает табло о завершении**

#### г) WB — Запись результата
- **Проверка WAR-конфликтов** (если другой инструкции нужно прочитать этот регистр)
- **Запись в регистровый файл**
- **Освобождение функционального устройства**
## Сравнение подходов

| Аспект | Статическое (компилятор) | Динамическое (процессор) |
|--------|--------------------------|--------------------------|
| **Когда** | Во время компиляции | Во время выполнения |
| **Информация** | Только статическая | Реальная во время выполнения |
| **Гибкость** | Ограничена предсказаниями | Адаптируется к ситуации |
| **Сложность** | Сложный компилятор | Сложная аппаратура |
| **Пример** | Разворачивание циклов | Out-of-order execution |

## Выводы

1. **Межитерационные зависимости** — главное препятствие для распараллеливания циклов.
2. **Перестройка циклов** (изменение границ итераций) может превратить межитерационные зависимости во внутриитерационные.
3. **Тест GCD** помогает компилятору анализировать зависимости в циклах.
4. **Динамическое планирование** дополняет статическое:
   - Компилятор делает грубую оптимизацию
   - Процессор выполняет тонкую настройку во время работы
5. **Современные процессоры** используют оба подхода для максимизации ILP.
## Алгоритм Scoreboard (Табло)

### Основная идея
**Scoreboard** — централизованная аппаратная структура, которая:
- Отслеживает **зависимости между инструкциями**
- Управляет **доступностью функциональных устройств**
- Разрешает **неупорядоченное выполнение**, но сохраняет **логическую корректность**

### Архитектура процессора с Scoreboard
```
[Регистровый файл] ←→ [Scoreboard] ←→ [Функциональные устройства]
       ↓                       ↓                 ↓
   Чтение/запись        Контроль зависимостей   Integer Unit
                                             FP Add Unit
                                             FP Mult Unit1
                                             FP Mult Unit2
                                             FP Divide Unit
```

**Характеристики:**
- **Один порт чтения/записи** на регистровый файл
- **Несколько функциональных устройств** (могут быть одинаковые)
- **Scoreboard** управляет всем

### Что хранится в Scoreboard? / Структура таблицы

#### 1. Для каждой инструкции (Раздел инструкций):
1. **Регистр назначения** (Destination)
2. **Источники операндов** (Source1, Source2)
3. **Функциональное устройство**, на котором выполняется
4. **Статус готовности операндов**

**Детализация полей таблицы инструкций:**

|Поле|Описание|
|---|---|
|**Dest**|Регистр назначения|
|**Src1, Src2**|Исходные регистры|
|**FU for j, k**|Какое функциональное устройство будет производить операнды|
|**Fj?, Ek?**|Готовность операндов|

#### 2. Для каждого функционального устройства (Раздел ФУ):
1. **Занято/свободно** (Busy)
2. **Тип операции** (Op)
3. **Регистры-операнды** (Dest, Src1, Src2)
4. **Статус выполнения**

**Детализация таблицы функциональных устройств:**

|Устройство|Busy|Op|Dest|Src1|Src2|
|---|---|---|---|---|---|
|Integer|false|||||
|Multiply1|false|||||
|Multiply2|false|||||
|Add|false|||||
|Divide|false|||||

**Поля:**
- **Busy**: Занято ли устройство
- **Op**: Какая операция выполняется
- **Dest, Src1, Src2**: Регистры операции

**Важное предположение для упрощения анализа:**
- **Функциональные устройства НЕ конвейеризованы** — каждая операция занимает устройство на всё время своего выполнения
  - *Причина*: упрощение таблиц и анализа выполнения
  - *В реальности*: современные процессоры имеют конвейеризованные функциональные устройства

#### 3. Для каждого регистра (Раздел состояния регистров):
1. **Кто пишет в этот регистр** (какая инструкция/какое ФУ)
2. **Статус** (забронирован для записи или свободен)

**Таблица состояния регистров:**

|Регистр|FU|
|---|---|
|F0||
|F2||
|F4||
|F6||
|F8||
|F10||
|F12||

**Поле FU:** Какое функциональное устройство будет писать в этот регистр (если есть)

**Порядок заполнения таблиц:**
1. **В конце каждого такта** обновляем все состояния
2. **По вертикали** — такты выполнения
3. **По горизонтали** — ресурсы процессора
4. **Важно:** Функциональное устройство помечается как свободное **только после завершения WB**

## Этапы выполнения инструкции в Scoreboard

### Стадия 1: Выдача (Issue) — ID1
**Условия для выдачи:**
1. **Свободное функциональное устройство** нужного типа
2. **Регистр назначения не забронирован** другой инструкцией (предотвращение WAW)

**Важность второго условия:**
```assembly
ADD F2, F0, F1    ; Инструкция A — пишет в F2
MUL F2, F3, F4    ; Инструкция B — тоже пишет в F2
```
Если разрешить выдачу обеих инструкций одновременно, возникнет **WAW-конфликт** (Write After Write). Scoreboard **предотвращает это**, не выдавая инструкцию B, пока инструкция A не освободит регистр F2.

**Действия при выдаче:**
- Забронировать функциональное устройство
- Забронировать регистр назначения
- Записать информацию в Scoreboard
- Определить зависимости от предыдущих инструкций

### Стадия 2: Чтение операндов (Read Operands) — ID2
**Условия:**
- **Все операнды готовы** (RAW разрешены)
- **Нет WAR-конфликтов** на данном этапе

**Важное ограничение:** **НЕТ ПЕРЕСЫЛКИ (forwarding)**
- Операнды читаются **только из регистрового файла**
- Инструкция может перейти к чтению операндов **только когда**:
  - Все инструкции, которые должны были записать значения в её операнды, **уже записали их в регистры**

**Последствие:** Большие задержки из-за RAW-конфликтов, так как инструкция ждёт полной записи в регистр, а не может получить результат напрямую с выхода функционального устройства.

**Действия при чтении операндов:**
- Прочитать операнды из регистрового файла
- Начать выполнение на функциональном устройстве

### Стадия 3: Исполнение (EX)
**Условия:** (автоматически, после ID2)

**Действия:**
- Выполнение операции
- По завершении — уведомить Scoreboard

### Стадия 4: Запись результата (Write Result) — WB
**Условия:**
1. **Операция завершена**
2. **Нет WAR-конфликтов** (другие инструкции не должны читать этот регистр в данный момент)

**Сложная проверка для предотвращения WAR-конфликтов:**

Когда инструкция хочет записать результат:
1. Scoreboard смотрит **все инструкции, которые ещё не прочитали свои операнды** (находятся на стадии Issue или ждут операндов)
2. Для каждой такой инструкции проверяется:
   - Использует ли она **наш выходной регистр как входной**?
   - **Ожидает ли она результат именно от нас**?
3. Если **хотя бы одна инструкция** использует наш выходной регистр, но ожидает результат **не от нас** → запись результата **откладывается**
4. Запись возможна **только когда** все инструкции, использующие этот регистр, ожидают результат именно от текущей инструкции

**Пример:**
```assembly
ADD F2, F0, F1    ; Инструкция A — пишет в F2
MUL F3, F2, F4    ; Инструкция B — читает F2 (зависит от A)
SUB F2, F5, F6    ; Инструкция C — пишет в F2 (позже в программе)
```
При попытке инструкции C записать в F2:
- Инструкция B уже прочитала старый F2 (до выполнения A) → можно записывать
- Если бы B ещё не читала операнды и ожидала результат от A → C должна ждать

**Действия при записи результата:**
- Записать результат в регистровый файл
- Освободить функциональное устройство
- Обновить Scoreboard (разблокировать зависимые инструкции)

## Пример работы Scoreboard

### Исходный код
```assembly
1: LD   [34+R2], F6     ; Загрузка из памяти в F6
2: LD   [45+R3], F2     ; Загрузка из памяти в F2  
3: DMUL F2, F4, F0      ; Умножение: F0 = F2 * F4
4: DSUB F6, F2, F8      ; Вычитание: F8 = F6 - F2
5: DDIV F0, F6, F10     ; Деление: F10 = F0 / F6
6: DADD F8, F2, F6      ; Сложение: F6 = F8 + F2
```

**Характеристики функциональных устройств:**
- **Integer/Load Unit**: 1 такт на операцию
- **Multiply Unit**: 10 тактов на операцию
- **Add Unit**: 2 такта на операцию
- **Divide Unit**: 40 тактов на операцию

### Таблица Scoreboard будет содержать:

**Информация об инструкциях:**

|Инстр.|Dest|Src1|Src2|FU|Status|
|---|---|---|---|---|---|
|1 (LD)|F6|R2|-|Mem|Executing|
|2 (LD)|F2|R3|-|Mem|Issued|
|3 (DMUL)|F0|F2|F4|Mult|Waiting(F2)|
|...|...|...|...|...|...|

**Информация о функциональных устройствах:**

|FU Type|Busy|Op|Dest|Src1|Src2|
|---|---|---|---|---|---|
|Integer|Yes|LD|F6|R2|-|
|Mem Unit|Yes|LD|F2|R3|-|
|Mult1|No|-|-|-|-|
|...|...|...|...|...|...|

**Информация о регистрах:**

|Регистр|FU|
|---|---|
|F0|Mult1|
|F2|Mem|
|F6|Integer|
|...|...|

## Важные особенности и ограничения Scoreboard

### Ограничения
1. **Нет пересылки (forwarding)** — операнды читаются только из регистрового файла
2. **Последовательная выдача** — не более одной инструкции за такт
3. **Централизованное управление** — может стать узким местом
4. **Сложные проверки WAR** — дополнительные логические цепи и задержки

### Почему Scoreboard не используется в чистом виде сегодня?
1. **Нет пересылки** → большие задержки
2. **Централизованное управление** → сложно масштабировать
3. **Последовательная выдача** → ограничение пропускной способности
4. **Сложные проверки WAR** → дополнительные задержки

### Преимущества перед статическим планированием
1. **Адаптация к реальным задержкам** (кэш-промахи)
2. **Обработка неизвестных во время компиляции зависимостей**
3. **Лучшее использование нескольких функциональных устройств**

## Сравнение с современными процессорами

### Архитектура vs Микроархитектура
**Архитектура (неизменна):**
- Система команд (инструкции)
- Набор регистров
- Режимы адресации

**Микроархитектура (изменилась кардинально):**
- Внутренняя организация процессора
- Конвейерная структура
- Механизмы планирования (Scoreboard → Tomasulo → современные)

### Современные процессоры имеют:
- **Множество функциональных устройств** (как на слайде с микроархитектурой)
- **Более сложные механизмы** (переименование регистров, спекулятивное выполнение)
- **Распределённое управление** (не централизованное как Scoreboard)

### Современные аналоги
Современные процессоры используют **усовершенствованные механизмы**, включающие:
- **Переименование регистров** (устранение WAR/WAW)
- **Распределённое планирование**
- **Спекулятивное выполнение**
- **Глубокие буферы переупорядочивания**

**Пример современной микроархитектуры:**
```
[Frontend] → [Rename/Allocate] → [Scheduler] → [Execution Units] → [Retire]
                    ↓
              Множество специализированных блоков:
              - ALU (целочисленные)
              - FPU (вещественные)
              - Vector Units (векторные)
              - Load/Store Units (память)
```

### Реальные реализации
- **Первый Scoreboard**: CDC 6600 (1963) — первый суперкомпьютер, цена $7 млн
- **Альтернатива**: Алгоритм Томасуло (IBM 360/91, 1966) — более сложный, но эффективный

**Эволюция:** от Scoreboard → к алгоритму Томасуло → к современным механизмам с переименованием регистров

## Выводы / Ключевые выводы по Scoreboard

1. **Scoreboard — первый практический механизм** динамического планирования
2. **Основная идея:** выдача по порядку, выполнение по готовности операндов
3. **Динамическое планирование** — аппаратный механизм для увеличения ILP
4. **Ключевые принципы**:
   - Выдача по порядку, выполнение по готовности операндов
   - Отслеживание всех типов зависимостей (RAW, WAR, WAW)
   - Централизованное управление всеми ресурсами

5. **Решает все три типа конфликтов:**
   - RAW: через ожидание готовности операндов
   - WAR: через сложную проверку при записи результата
   - WAW: через запрет выдачи инструкции с тем же регистром назначения

6. **Ограничения Scoreboard**:
   - Нет пересылки → дополнительные задержки
   - Централизованное управление → сложность масштабирования
   - Последовательная выдача → ограничение пропускной способности
