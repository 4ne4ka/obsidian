### Краткое повторение предыдущей темы: Метрики производительности и бенчмарки

*   **Рассмотренные метрики:**
    *   **Среднее число тактов на инструкцию (CPI):** Основная метрика, характеризующая, сколько тактов в среднем требуется для выполнения одной инструкции. Лектор подчеркивает, что эта характеристика будет часто использоваться в дальнейшем.
    *   **Среднее число миллионов инструкций в секунду (MIPS):** Сколько миллионов инструкций процессор выполняет за секунду.
    *   **Число операций с плавающей точкой в секунду (FLOPS):** Производительность процессора в вычислениях с плавающей запятой.

*   **Типы бенчмарков:**
    1.  **Целевые приложения:** Измерение производительности на конкретном приложении.
    2.  **Подобранные наборы приложений (напр., SPEC):** Набор тестов, репрезентирующих типовую нагрузку.
    3.  **Типовые алгоритмы:** Например, умножение матриц или быстрое преобразование Фурье (FFT).
    4.  **Микробенчмарки:** Тестирование производительности отдельных узлов системы.

*   **Важные замечания:**
    *   Производители также заявляют **теоретические характеристики** (например, частота).
    *   **CPI не является абсолютной характеристикой.** Низкое CPI не всегда означает быстрый процессор. Эта метрика полезна для сравнения процессоров с одинаковой архитектурой (например, RISC vs. RISC), но не для сравнения разных архитектур (например, RISC vs. CISC).
    *   **Ключевой вывод для оптимизации:** Операции работы с памятью в среднем занимают наибольшее время. Поэтому **первичная оптимизация должна быть направлена на работу с памятью**.

### Конвейеризация (Pipelining)

#### Основная идея конвейера

*   **Центральный процессор** выполняет несколько шагов (стадий) для обработки каждой инструкции.
*   **Идея конвейера** — организовать обработку так, чтобы выполнение шагов для разных инструкций **перекрывалось во времени**.
*  Конвейер исполнения инструкций состоит из нескольких **стадий (ступеней)**, на каждой из которых завершается определенный этап работы над инструкцией.

#### Устройство конвейера

* **Ограничение:** Стадии соединены **линейно**. Инструкции вводятся с одного конца, последовательно проходят все стадии и выходят с другого.
*   **Архитектурное влияние:** Конвейерная организация накладывает ограничения и влияет как на **микроархитектуру**, так и на **архитектуру набора команд (ISA)**. Необходимо, чтобы все инструкции можно было выполнить на одном и том же наборе стадий.

#### Пример: 5-стадийный конвейер MIPS

**Стадии конвейера:**
1.  **IF (Instruction Fetch):** Выборка инструкции из памяти.
2. **ID (Instruction Decode):** Декодирование инструкции и чтение значений из регистров.
3.  **EX (Execution):** Исполнение операции (например, арифметической).
4.  **MEM (Memory Access):** Обращение к памяти (чтение или запись).
5.  **WB (Write Back):** Запись результата обратно в регистровый файл.

**Важные уточнения по работе стадий:**
*   **Не все инструкции используют все стадии.**
    *   **Пример:** Инструкции записи в память (`store`) **не используют стадию WB** (данные записываются в память на стадии MEM).
    *   **Пример:** Логические инструкции **не используют стадию MEM** (они не работают с памятью).
    *   **Пример:** Операции с памятью с простой адресацией могут **не использовать стадию EX** (если не нужно вычислять адрес).
*   **Стадия не "пропускается".** Даже если содержательной работы на стадии не производится, инструкция все равно занимает эту стадию на один такт, и результат (например, неизмененные данные) передается дальше.

#### Принцип работы конвейера на примере

Рассматривается идеальный случай заполнения конвейера на примере выполнения инструкций `I`, `I+1`, `I+2` и т.д.

*   **Такт 1:** Инструкция `I` проходит стадию **IF**.
*   **Такт 2:**
    *   Инструкция `I` переходит на стадию **ID**.
    *   Инструкция `I+1` начинает стадию **IF**.
*   **Такт 3:**
    *   Инструкция `I` переходит на стадию **EX**.
    *   Инструкция `I+1` переходит на стадию **ID**.
    *   Инструкция `I+2` начинает стадию **IF**.
*   **Такт 4:**
    *   Инструкция `I` переходит на стадию **MEM**.
    *   Инструкция `I+1` переходит на стадию **EX**.
    *   Инструкция `I+2` переходит на стадию **ID**.
    *   Инструкция `I+3` начинает стадию **IF**.
*   **Такт 5 (полное заполнение):**
    *   Инструкция `I` переходит на стадию **WB** и завершается.
    *   Инструкция `I+1` переходит на стадию **MEM**.
    *   Инструкция `I+2` переходит на стадию **EX**.
    *   Инструкция `I+3` переходит на стадию **ID**.
    *   Инструкция `I+4` начинает стадию **IF**.

| Инструкция \ Такт | 1   | 2   | 3   | 4   | 5   | 6   | 7   | 8   |
|:-----------------:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| **I**             | IF  | ID  | EX  | MEM | WB  |     |     |     |
| **I+1**           |     | IF  | ID  | EX  | MEM | WB  |     |     |
| **I+2**           |     |     | IF  | ID  | EX  | MEM | WB  |     |
| **I+3**           |     |     |     | IF  | ID  | EX  | MEM | WB  |
| **I+4**           |     |     |     |     | IF  | ID  | EX  | MEM |
| **I+5**           |     |     |     |     |     | IF  | ID  | EX  |
**Главный вывод:** При полном заполнении конвейера на каждом такте завершается выполнение одной инструкции, что значительно повышает производительность по сравнению с последовательным выполнением.


#### Фазы работы конвейера и расчет производительности

*   **Три фазы работы конвейера:**
    1.  **Заполнение (Fill):** Конвейер постепенно заполняется инструкциями.
    2.  **Установившийся режим (Steady State):** Все стадии конвейера заняты работой.
    3.  **Опустошение (Drain):** После окончания программы конвейер завершает выполнение всех начатых инструкций.

*   **Расчет времени выполнения:**
    *   Для конвейера с **N** стадий:
        *   Фазы заполнения и опустошения длятся **N-1** тактов каждая.
    *   **Пример для 5-стадийного конвейера:**
        *   Выполнение 5 инструкций займет **9 тактов** (4 на заполнение + 5 инструкций * 1 такт/инструкцию в установившемся режиме? *Лектор уточняет расчет: 5 инструкций за 9 тактов = CPI = 9/5 = 1.8*).
        *   При **последовательном** выполнении (без конвейера) CPI был бы равен **5**.
    *   **Идеальный случай для длинной программы:** Чем длиннее программа, тем ближе CPI к **1**, так как накладные расходы на заполнение и опустошение становятся незначительными. Для 1000 инструкций: (1000 + 4) / 1000 = 1.004.

#### Определение тактовой частоты и проблема дисбаланса стадий

*   **Время такта** определяется **максимальным временем обработки** среди всех стадий конвейера.
*   **Процесс работы стадии:** На каждом такте стадия получает входные данные (сигналы), которые проходят через комбинационную логику. Результат должен установиться и быть стабильным до конца такта. Время такта должно быть достаточно для завершения этого процесса на **самой медленной** стадии.
*   **Проблема дисбаланса:** Если стадии имеют разное время выполнения (например, IF=1нс, ID=2нс, EX=5нс, MEM=2нс, WB=1нс), то общая тактовая частота будет ограничена самой долгой стадией (EX, 5нс).
*   **Решение — балансировка конвейера:** Самые длинные стадии (как EX) можно **разбить на несколько более мелких стадий**.
    *   **Пример:** Стадию EX (5нс) разбивают на 4 стадии по ~2нс.
    *   **Результат:** Хотя общее время прохождения инструкции увеличивается (латентность растет), **время такта можно значительно уменьшить** (с 5нс до ~2нс), что резко повышает **пропускную способность** (инструкций за секунду).

#### Ключевые характеристики конвейера

*   **Латентность (Latency):** Время (в тактах), необходимое для выполнения одной инструкции от начала до конца. **Конвейер НЕ уменьшает латентность.** (Для 5-стадийного конвейера латентность = 5 тактов).
*   **Пропускная способность (Throughput):** Количество инструкций, завершаемых за такт. В идеальном случае конвейера стремится к **1 инструкции/такту** (CPI = 1).

#### Упорядоченный конвейер и микроархитектурные детали

*   **Упорядоченный конвейер (In-order pipeline):** Инструкции **начинают и заканчивают выполнение строго в том порядке**, в котором они указаны в программе.
    *  **Важно:** Все стадии для всех инструкций выполняются в программном порядке.
    *  Лектор отмечает, что бывают и неуправляемые конвейеры, где порядок выполнения стадий может меняться, но это тема для будущего.

*   **Микроархитектурная хитрость для обхода зависимостей:**
    *   **Стадия ID (Декодирование):** Чтение регистров происходит во **второй половине** такта декодирования.
    *   **Стадия WB (Запись результата):** Запись в регистровый файл происходит в **первой половине** такта.
    *   **Цель:** Это позволяет "обойти" проблему, когда одна инструкция должна записать результат, а следующая — сразу его прочитать. Так как запись происходит раньше, чем чтение в следующем такте, данные успевают обновиться. Это создает иллюзию "передачи" результата напрямую.

#### Реализация конвейера

*   Конвейер состоит из **стадий** (комбинационная логика) и **промежуточных регистров** (серые прямоугольники на схеме).
*   **Назначение промежуточных регистров (pipeline registers):** Они не видны программисту и служат для **сохранения и передачи результатов** одной стадии на вход следующей стадии на каждом такте.
*   **Принцип работы:**
    1.  **В течение такта:** Комбинационная логика стадии обрабатывает входные данные из предыдущего регистра и формирует выходной сигнал.
    2.  **По тактовому импульсу:** Все промежуточные регистры одновременно "защелкивают" данные со своих входов и передают их на выход, который становится входом для следующей стадии на новый такт.
    3.  **Цикл повторяется:** Этот процесс синхронного переключения позволяет каждой стадии работать над своей инструкцией одновременно с другими.

#### Детализация стадий конвейера (на примере MIPS)

**Стадия IF (Instruction Fetch - Выборка инструкции):**
* **Ключевой компонент:** Регистр **PC (Program Counter)**, хранящий адрес следующей инструкции.
*   **Процесс:**
    1.  Адрес из PC отправляется в память команд.
    2.  Из памяти загружаются 4 байта (фиксированный размер команды в RISC) и помещаются в промежуточный регистр между IF и ID.
    3. Значение PC одновременно подается на **сумматор**, где к нему прибавляется 4 (расчет адреса следующей последовательной инструкции).
*   **Обработка переходов:** Результат сумматора (PC+4) и потенциальный новый адрес от инструкции перехода подаются на **мультиплексор**. Управляющий сигнал (пришел ли переход) решает, какое значение (PC+4 или адрес перехода) будет записано обратно в регистр PC на следующем такте.
*   **Важное замечание:** Значение PC (или PC+4) также передается на следующую стадию (ID), так как оно может потребоваться для вычисления **относительных адресов** при выполнении переходов.

**Стадия ID (Instruction Decode - Декодирование инструкции):**
*   **Процесс:** Бинарный код инструкции декодируется. На основе полей инструкции (например, биты 6-10 и 11-15 в MIPS) определяются:
    *   Номера регистров-источников операндов.
    *   Тип операции (управляющие сигналы для ALU).
*   **Чтение регистров:** Значения из указанных регистров считываются и, вместе с управляющими сигналами и другими данными (например, непосредственной константой из команды), помещаются в промежуточный регистр для стадии EX.

**Стадия EX (Execution - Исполнение):**
*   **Ключевой компонент:** **АЛУ (Арифметико-логическое устройство)**.
* **Процесс:** На АЛУ подаются входные данные (из регистров или через мультиплексоры) и управляющий сигнал о типе операции. АЛУ производи вычисление.
*   **Обработка переходов:** В классическом MIPS условные переходы проверяли регистр на равенство нулю. Для этого требовалось отдельное устройство сравнения. Для более сложных сравнений (> , < и т.д.) потребовалось бы второе АЛУ для операции вычитания/сравнения.
*   **Результат:** Результат операции, а также сигнал о том, произошел ли переход, передаются на следующие стадии (MEM и обратно в IF).

**Стадии MEM (Memory Access) и WB (Write Back):**
*   **MEM:** На этой стадии происходит обращение к памяти данных (запись или чтение). Если обращение не требуется, данные просто передаются дальше.
*   **WB:** Результат (из АЛУ или из памяти) записывается обратно в регистровый файл. Устройство на этой стадии "понимает", в какой именно регистр нужно произвести запись.

##### Общее замечание о сложности проектирования

Лектор отмечает, что собрать учебный процессор, следуя этой схеме, **возможно**. Существуют учебные курсы (например, "nand2tetris"), где students поэтапно создают процессор, начиная с базовых логических элементов.
**Ключевое отличие** такого проектирования от программирования на языках высокого уровня — необходимость мыслить в терминах **параллельно работающих аппаратных блоков**, а не последовательности команд.

#### Пример расчета времени выполнения программы на конвейере

*   Лектор разбирает пример программы, состоящей из:
    *   5 инструкций до цикла.
    *   Цикла, выполняющегося 16 раз, с 8 инструкциями внутри.
    *   2 инструкций после последней итерации цикла.
    *   Инструкций после завершения цикла.
*   **Общее количество инструкций (N):** Рассчитывается исходя из структуры программы.
*   **Расчет времени для конвейера (идеальный случай):**
    *   Время = (Число стадий - 1) + N
    *   Для 5-стадийного конвейера: `Время = 4 + N` тактов.
    *   В примере при N=137, время составило **141 такт**.
*   **Сравнение с неконвейерным процессором:**
    *   Время = N * (Число стадий)
    *   Для 5-стадийного процессора: `Время = N * 5` тактов.
    *   Для того же примера (N=137) время составило бы **685 тактов**, что **значительно больше**, чем при конвейерной организации. Это наглядно демонстрирует выигрыш в пропускной способности.

## Конвейерные конфликты (Hazards)

#### Практические аспекты глубины конвейера

*   **Теоретическое преимущество длинного конвейера:** Увеличение числа стадий позволяет уменьшить время такта (так как каждая стадия становится проще) и, следовательно, повысить тактовую частоту. В идеале процессор с 31 стадией мог бы работать в ~1.5-2 раза чаще, чем процессор с 14 стадиями, завершая больше инструкций в секунду.
*   **Практическая проблема — конфликты (Hazards):** В реальности выполнение на конвейере **не идеально**. Возникают ситуации (конфликты), которые приводят к **простоям (stalls)** конвейера.
*   **Штраф за сброс конвейера:** При возникновении конфликта (например, из-за неверного предсказания перехода) конвейер может быть вынужден отбросить часть начатых инструкций ("всё сломалось"). На его повторное заполнение (**разгон**) требуется время.
    *   **Пример:** Конвейеру с 14 стадиями для выхода на полную мощность нужно **13 тактов**, а конвейеру с 31 стадией — **30 тактов**.
*   **Исторический итог:** Из-за высоких **штрафов** за конфликты гонка за увеличение количества стадий прекратилась. В современных процессорах используется порядка **14-15 стадий**, что является компромиссом между высокой частотой и приемлемыми потерями от конфликтов.

#### Введение в типы конфликтов (Hazards)

*   **Определение:** Конфликт — это ситуация в конвейере, которая препятствует продвижению следующей инструкции в предназначенный для нее такт.
*   **Следствие:** Конфликты уменьшают идеальное ускорение от конвейера и **увеличивают CPI (количество тактов на инструкцию)**.
*   **Три класса конфликтов:**
    1.  **Структурные конфликты (Structural Hazards)**
    2.  **Конфликты по данным (Data Hazards)**
    3.  **Конфликты по управлению (Control Hazards)**

#### Структурные конфликты (Structural Hazards)

*   **Причина:** Возникают из-за **недостатка аппаратных ресурсов**, когда две или более инструкции пытаются использовать один и тот же функциональный блок одновременно.
*   **Классический пример — конфликт за доступ к памяти:**
    *   В простейшем процессоре есть **одна общая память** (или один порт доступа к ней) для команд и данных.
    *   **Стадия IF** инструкции `i+1` пытается считать следующую команду из памяти.
    *   **Одновременно стадия MEM** инструкции `i` (например, `load` или `store`) пытается обратиться к памяти для чтения или записи данных.
    *   Возникает **конфликт**, так как обе стадии требуют единственный ресурс — канал доступа к памяти.
*   **Решение (в простейшем случае):** Процессор вынужден делать **простой (stall)**.
    *   Инструкция на стадии IF задерживается на один такт.
    * Все последующие инструкции также сдвигаются, создавая "пузырь" (`Bubble`) в конвейере.
    * На диаграмме это выглядит как вставка одного такта простоя между выборкой инструкций, что нарушает идеальный поток выполнения.

#### Структурные конфликты из-за многотактовых операций

*   **Причина конфликта:** Некоторые операции (например, **целочисленное умножение или деление**) на стадии исполнения (EX) могут занимать **не один, а несколько тактов**.
*   **Проблема:** Если инструкция `i` (например, умножение) занимает стадию EX на два такта, то инструкция `i+1`, которая на следующем такте должна перейти на стадию EX, **будет заблокирована**. Это структурный конфликт, так как аппаратный ресурс (АЛУ) занят.
*   **Эффект "пробки":** Блокировка одной стадии вызывает остановку **всех предыдущих стадий**.
    *   Инструкция `i+1` не может покинуть стадию ID, так как не может перейти на занятую стадию EX.
    *   Поскольку стадия ID остается занятой, инструкция `i+2` не может перейти на стадию ID из IF.
    *   Таким образом, **весь конвейер останавливается**, создавая "пузырь" (bubble).
*   **Корректное изображение на временной диаграмме:** Лектор подчеркивает важность построения диаграммы **по тактам**, а не по инструкциям. На правильной диаграмме видно, как все стадии после блокированной "наползают" друг на друга, создавая вертикальный столбец простоя, а не просто горизонтальную линию.

#### Пример структурного конфликта при работе с памятью

*   **Рассматривается программа:** Загрузка трех чисел в регистры, а затем их сложение.
*   **Проблема:** Инструкции загрузки (`load`) на стадии MEM и инструкция сложения на стадии IF пытаются одновременно обратиться к памяти.
*   **Разрешение конфликта:** Приоритет отдается инструкции, которая **уже находится в конвейере** (т.е., `load`).
*   **Результат:** Выборка инструкции сложения откладывается на три такта (по одному на каждую инструкцию `load`).
*   **Итоговый CPI:** Вместо идеального CPI=1, реальный CPI становится больше (в примере 1.58), так как к 4 тактам заполнения конвейера добавляются 3 такта простоя из-за структурных конфликтов.

#### Методы устранения структурных конфликтов доступа к памяти

*   **Проблема:** Один канал доступа к унифицированной памяти для команд и данных.
*   **Решение:** Разделение кэшей верхнего уровня.
    *   Создаются **раздельные кэши: кэш инструкций (L1I)** и **кэш данных (L1D)**.
    *   **Стадия IF** обращается к **кэшу инструкций**.
    *   **Стадия MEM** обращается к **кэшу данных**.
    *   Это позволяет обеим стадиям работать **одновременно и без конфликтов**.
*   **Важное замечание:** Оперативная память по-прежнему остается унифицированной, но конфликт разрешается на уровне быстрых буферов (кэшей), близких к ядру процессора. Конфликты при обращении к общей оперативной памяти могут возникать, но они менее критичны и скрыты от конвейера.

#### Введение в конфликты по данным (Data Hazards)

*   **Причина:** Инструкции используют одни и те же регистры, и при конвейерном выполнении **порядок чтения и записи** этих регистров меняется, что может привести к использованию некорректных (еще не обновленных) данных.
*   **Типичный сценарий:** Последовательность инструкций, где каждая следующая зависит от результата предыдущей (например, `A = B + C; D = A * E; F = D / G`). Это создает **зависимость по данным**.
* **Термин:** Инструкции, которые **не зависят** друг от друга, называются **независимыми** и обладают **параллелизмом на уровне инструкций (Instruction-Level Parallelism, ILP)**. Чем больше ILP в коде, тем эффективнее он выполняется на конвейере.
*   **Основной способ борьбы (на данном этапе):** Процессор может обнаружить конфликт и принудительно **остановить конвейер (вставить "пузыри")**, чтобы дождаться готовности данных.

#### Конкретный пример конфликта по данным

* **Рассматривается последовательность инструкций,** где каждая следующая зависит от результата предыдущей (например, `instr1: R1 = R2 + R3`; `instr2: R4 = R1 + R5`; `instr3: R6 = R1 + R7`).
*   **Проблема при идеальном конвейере:**
    *   Инструкция `instr1` запишет результат в регистр `R1` только на стадии **WB**.
    *   Однако инструкции `instr2` и `instr3` должны прочитать значение `R1` уже на стадии **ID**.
    *   При идеальном выполнении они прочтут **устаревшее (старое) значение** `R1`, что приведет к ошибке.
*   **Частичное решение за счет временного финта:** Ранее упомянутая хитрость (запись в регистр в **первой половине** такта WB и чтение во **второй половине** такта ID) позволяет **только последней** инструкции в цепочке (`instr3`) получить корректное значение без задержек. Для первых зависимых инструкций (`instr2`) этот метод не работает.

#### Классификация конфликтов по данным

Рассматриваются четыре типа ситуаций при совместном использовании регистра двумя инструкциями (A - более ранняя, B - более поздняя):

1.  **WAW (Write After Write):** B пишет в регистр после A.
    *   *Статус:* В **упорядоченном конвейере** это **невозможно**, так как стадия WB инструкции B всегда выполняется после стадии WB инструкции A. Станет актуально в неуправляемых конвейерах.

2.  **WAR (Write After Read):** B пишет в регистр после того, как A его прочитала.
    *   *Статус:* В **упорядоченном конвейере** это **невозможно**, так как стадия чтения (ID) инструкции A всегда выполняется раньше стадии записи (WB) инструкции B. Также станет актуально в неуправляемых конвейерах.

3.  **RAR (Read After Read):** B читает регистр после того, как A его прочитала.
    *   *Статус:* **Конфликта нет**. Это безопасная операция.

4.  **RAW (Read After Write):** B читает регистр, в который пишет A.
    *   *Статус:* Это **истинная зависимость по данным (true data dependency)**. Это **основной тип конфликта** в упорядоченном конвейере, который **нельзя устранить**, можно только минимизировать его последствия. Он отражает реальную последовательную логику программы.

#### Методы борьбы с конфликтом RAW

1.  **Вставка тактов простоя (Stalling):**
    *   Процессор приостанавливает выполнение зависимых инструкций, вставляя "пузыри" в конвейер, пока данные не будут готовы.
    *   Это простое, но **неэффективное** решение, так как снижает общую производительность.

2.  **Пересылка данных (Forwarding / Bypassing):**
    *   **Суть метода:** Результат выполнения инструкции передается на вход стадии исполнения **напрямую**, как только он был вычислен, **не дожидаясь** его формальной записи в регистровый файл на стадии WB.
    *   **Реализация:** На схеме конвейера это отображается **дополнительными стрелками (зеленые)**:
        *   С выхода стадии **EX** одной инструкции на вход стадии **EX** следующей инструкции.
        *   С выхода стадии **MEM** одной инструкции на вход стадии **EX** следующей инструкции.
    *   **Эффект:** Позволяет значительно сократить или полностью избежать простоев для цепочек зависимых инструкций. Данные используются "в полете".

**Итог:** Комбинация **временного финта** (половинки тактов) и механизма **пересылки данных** позволяет эффективно бороться с конфликтами типа RAW в конвейере, минимизируя необходимость в остановках.
