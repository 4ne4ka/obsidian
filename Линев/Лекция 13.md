### Когда кэш снижает производительность?
- **Cache Miss Time = Penalty + Hit Time**
- В некоторых задачах (например, "Решето Эратосфена", тест Донгарра для решения систем линейных уравнений) при больших размерах данных каждое обращение к памяти может приводить к **кэш-промаху**.
- Если промахи происходят постоянно, то **Hit Time** (время проверки кэша) становится лишней нагрузкой, замедляющей работу.
- Однако в большинстве практических задач **относительные накладные расходы на кэш составляют не более 1%**, поэтому кэш в целом эффективен.
## Многоуровневый кэш

### Зачем нужны несколько уровней?
- **Первый уровень (L1)** должен обеспечивать **минимальную задержку** для конвейера ЦП.
- **Второй (L2) и третий (L3) уровни** появляются из-за невозможности совместить **большую ёмкость** и **низкое время доступа** в одном уровне.
- **Задача кэшей L2 и выше** — сократить штраф от промахов предыдущего уровня, перехватывая дополнительные запросы к основной памяти, тем самым **снижая среднее время доступа к памяти (AMAT)**.
## Пример: эффективность раздельного кэша L1

### Условия примера
- У ЦП есть **отдельные кэши L1 для инструкций и данных**.
- Промах в кэше приводит к простою в **200 тактов**.
- **CPI (идеальной памяти) = 1.1**
- Состав инструкций:
  - 50% — арифметика/логика
  - 30% — загрузка/сохранение (load/store)
  - 20% — управление (control)
- Частота промахов:
  - **0.5%** для инструкций
  - **6%** для данных

## Стратегии записи в кэш

### Write Through (сквозная запись)
- **Данные записываются одновременно в кэш и в основную память.**
- **Преимущества:**
  - Нижние уровни памяти всегда содержат актуальные данные.
  - Важно для **ввода-вывода (I/O)** и многопроцессорных систем.
  - Проще в реализации.
- **Недостатки:**
  - Медленнее, так как каждая запись идёт в память.
  - Для ускорения используется **буфер записи (Write Buffer)**.

### Write Back (отложенная запись)
- **Данные обновляются только в кэше.**
- Запись в основную память происходит только при **вытеснении** модифицированного блока из кэша.
- **Преимущества:**
  - Запись происходит на скорости кэша.
  - Меньше обращений к памяти, выше пропускная способность.
- **Механизм:**
  - Каждая строка кэша имеет **dirty bit (бит модификации)**, который показывает, был ли блок изменён.
  - Если бит установлен, при вытеснении блок записывается в память.

## Стратегии обработки промахов при записи

### Write Allocate (размещение при записи)
- При промахе **блок загружается в кэш**, затем происходит запись.
- **Используется в связке с Write Back**, чтобы последующие записи в ту же строку кэша происходили быстро.

### No-Write Allocate (без размещения при записи)
- При промахе **данные записываются напрямую на нижний уровень** (память или следующий кэш), **без загрузки в кэш**.
- **Используется чаще с Write Through**, так как последующие записи всё равно идут в память.

### Практический пример: инициализация больших массивов
- Если нужно заполнить нулями массив из 100 млн элементов, **No-Write Allocate** эффективнее:
  - Не засоряет кэш.
  - Не вызывает ненужных промахов.
- В современных процессорах есть **векторные инструкции**, позволяющие записывать в память, минуя кэш.

## Оптимизация доступа к памяти на уровне программного кода

### Методы оптимизации (компилятор и программист)
1. **Слияние массивов** (Array Merging)
   - Объединение нескольких массивов в один массив структур для улучшения **пространственной локальности**.
   - Выбор между "массив структур" и "структура массивов" зависит от шаблонов доступа.

2. **Согласование вложенности циклов** (Loop Interchange)
   - Изменение порядка циклов для соответствия **порядку хранения данных в памяти** (например, обход матрицы по строкам, а не по столбцам).
   - Пример:
     ```c
     // До (плохо: обход по столбцам)
     for (j = 0; j < 100; j++)
       for (i = 0; i < 5000; i++)
         x[i][j] = 2 * x[i][j];
     
     // После (хорошо: обход по строкам)
     for (i = 0; i < 5000; i++)
       for (j = 0; j < 100; j++)
         x[i][j] = 2 * x[i][j];
     ```

3. **Слияние циклов** (Loop Fusion)
   - Объединение независимых циклов, работающих с одними данными, чтобы **увеличить временную локальность**.
   - Пример:
     ```c
     // До (два прохода по данным)
     for (i) for (j) a[i][j] = 1/b[i][j] * c[i][j];
     for (i) for (j) d[i][j] = a[i][j] + c[i][j];
     
     // После (один проход)
     for (i) for (j) {
       a[i][j] = 1/b[i][j] * c[i][j];
       d[i][j] = a[i][j] + c[i][j];
     }
     ```

4. **Поблочная обработка** (Blocking / Tiling)
   - Разбиение данных на **блоки**, которые помещаются в кэш, для увеличения **временной локальности**.
   - Особенно эффективно для операций с матрицами (например, умножение матриц).
   - Пример умножения матриц:
     ```c
     // После разбиения на блоки размером BxB
     for (jj = 0; jj < N; jj += B)
       for (kk = 0; kk < N; kk += B)
         for (i = 0; i < N; i++)
           for (j = jj; j < min(jj+B, N); j++) {
             r = 0;
             for (k = kk; k < min(kk+B, N); k++)
               r += y[i][k] * z[k][j];
             x[i][j] += r;
           }
     ```
   - Оценка промахов:
     - Без блокировки: ~\(2N^3 + N^2\)
     - С блокировкой: ~\(2N^3/B + N^2\)

## Буфер записи (Write Buffer)

### Назначение
- **Уменьшает простой ЦП при записи в память** в стратегии Write Through.
- ЦП записывает данные в буфер и продолжает работу, а контроллер памяти асинхронно сбрасывает буфер в память.

### Устройство
- Обычно это **FIFO-буфер** ёмкостью около 4 записей.
- Эффективен, если **частота записей значительно меньше, чем скорость работы DRAM**.


## Предсказание ветвлений (Branch Prediction)

### Цель
Уменьшение простоев конвейера из-за **задержек при выполнении условных переходов**.

### Методы статического предсказания (на этапе компиляции)
1. **На основе направления перехода**:
   - **Ветвления назад** (уменьшающие PC) предсказываются как **выполняющиеся** (типично для циклов).
   - **Ветвления вперёд** (увеличивающие PC) предсказываются как **не выполняющиеся**.
2. **На основе профилирования**:
   - Программа запускается несколько раз, собирается статистика по ветвлениям.
   - Результаты профилирования **встраиваются в код** как подсказки для предсказателя.

## Выводы

- **Кэш-память** критически важна для производительности, но требует правильного управления.
- **Многоуровневая организация** позволяет балансировать между скоростью и объёмом.
- **Стратегии записи** (Write Through / Write Back) и обработки промахов влияют на пропускную способность и согласованность данных.
- **Программные оптимизации** (блокировка, слияние циклов и др.) могут **увеличить производительность в разы**, особенно при работе с большими массивами данных.
- **Предсказание ветвлений** и **буферы записи** — важные аппаратные механизмы для скрытия задержек памяти и управления потоком команд.

## Почему предсказание ветвлений критично

### Проблема конфликтов управления в конвейере
- В конвейере ЦП **каждая 7-я инструкция — это условный переход**.
- **Потери из-за условных переходов могут быть огромными**: простой конвейера на несколько тактов.
- До сих пор использовали:
  1. **Статическое предсказание "переход не происходит"** — спекулятивное продолжение выполнения.
  2. **Ранняя обработка переходов** — перенос логики условного перехода со стадии MEM на стадию ID.
  3. **Слот задержки (delay slot)** — инструкция после перехода выполняется всегда.
  4. **Статическое предсказание, закодированное в инструкции** (бит в коде операции).

### Ограничения статического предсказания
- **Компилятор** может использовать:
  - **Направление перехода**: переходы "назад" (в циклах) — предсказывать "выполнится", переходы "вперёд" — "не выполнится".
  - **Профилирование**: программа запускается на тестовых данных, собирается статистика по каждому переходу.
- **Но!** Траектория выполнения программы зависит от **конкретных входных данных**. Статическое предсказание может быть неточным.

### Необходимость динамического предсказания
- **Динамическое предсказание** — аппаратный механизм, который **анализирует поведение программы во время её выполнения** и адаптируется под текущие данные.
- **Цель**: снизить простой конвейера из-за переходов до **нуля** (в идеале).

## Динамическое предсказание: базовые понятия

### Алгоритм vs Механизм
- **Алгоритм** — математическая/логическая схема предсказания (например, "двухбитный счётчик").
- **Механизм** — аппаратная реализация алгоритма **в конвейере ЦП** (например, Branch Target Buffer, Branch History Table).
- **Задача**: реализовать механизм на стадии **IF (Instruction Fetch)**, чтобы результат предсказания был готов **к началу следующего такта**.

### Branch Target Buffer (BTB) — буфер целей переходов
- **Проблема**: чтобы вычислить адрес перехода (цель), нужно дождаться стадии **ID (Instruction Decode)**, где происходит декодирование и вычисление смещения.
- **Решение**: **BTB** — это **ассоциативная память**, которая хранит:
  - **Адреса инструкций переходов**, которые уже выполнялись.
  - **Адреса их целей** (куда они переходили).
- **Работает на стадии IF**:
  1. При выборке инструкции по адресу PC проверяется, есть ли этот адрес в BTB.
  2. Если есть — **сразу выдаётся предсказанный адрес следующей инструкции**.
  3. Если нет — работа продолжается как обычно.

#### Алгоритм работы BTB
```
IF стадия:
  1. Отправляем PC в память (кэш инструкций) и в BTB.
  2. Если запись в BTB найдена → используем предсказанный PC.
  3. Если не найдена → продолжаем обычное выполнение.

ID стадия:
  1. Если инструкция — условный переход, и он "выполнился" (taken), но его не было в BTB → добавляем запись в BTB.
  2. Если переход был предсказан BTB, и предсказание верное → продолжаем без простоев.
  3. Если предсказание неверное → очищаем конвейер, обновляем BTB, продолжаем с правильного адреса.
```
- **Задержка при обновлении BTB**: 1–2 такта.
- **Идеальный случай**: переход предсказан верно → **простой конвейера = 0**.

## Одноуровневое (бимодальное) предсказание

### Основная идея
- Использует **таблицу истории переходов** (Branch History Table, BHT), также называемую **Pattern History Table (PHT)**.
- **Индексируется N младшими битами адреса инструкции перехода**.
- Каждая запись в таблице — это **счётчик с насыщением**, обычно **2-битный**.

### Алгоритм Смита (Smith Algorithm, 1985)
- **Таблица PHT имеет 2^N записей** (по индексу из N бит).
- **Каждая запись — 2-битный счётчик**:
  ```
  00: уверенно "не выполнится" (Strongly Not Taken)
  01: слабо "не выполнится" (Weakly Not Taken)
  10: слабо "выполнится" (Weakly Taken)
  11: уверенно "выполнится" (Strongly Taken)
  ```
- **Старший бит определяет предсказание**:
  - 0 → предсказываем "не выполнится".
  - 1 → предсказываем "выполнится".

#### Обновление счётчика
- **Если переход выполнился (taken)** → увеличиваем счётчик (но не выше 11).
- **Если переход не выполнился (not taken)** → уменьшаем счётчик (но не ниже 00).
- **Преимущество перед 1-битной схемой**: нужно **два неверных предсказания подряд**, чтобы изменить решение. Это снижает ошибки в циклах.

### Branch History Table (BHT) vs Pattern History Table (PHT)
- **PHT (прямое отображение)**: индекс = N бит адреса, **нет тегов**, возможны коллизии (разные переходы с одинаковыми младшими битами).
- **BHT (ассоциативная)**: хранит **полный адрес перехода как тег**, позволяет точнее идентифицировать инструкцию. **Работает как кэш**.

## Почему именно 2-битные счётчики?
- Наблюдения показывают: **эффективность 2-битной BHT сравнима с n-битной** (n > 2).
- **3-битные и более счётчики** редко используются, так как дают небольшой прирост точности при значительном усложнении схемы.

## Практические выводы по эффективности кэша и предсказания

### Когда кэш снижает производительность?
- **Cache Miss Time = Penalty + Hit Time**
- Если программа **постоянно промахивается** (например, "Решето Эратосфена", умножение больших матриц), то **Hit Time становится лишней нагрузкой**.
- **Но** в реальных задачах накладные расходы на кэш обычно **не превышают 1%**.

### Ценность кэширования: пример с цифрами
- **Идеальный кэш** (все попадания) быстрее **неидеального кэша** примерно в **5 раз**.
- **Работа без кэша** медленнее идеального кэша примерно в **250 раз**.
- **Вывод**: даже неидеальный кэш даёт **ускорение в 50 раз** по сравнению с отсутствием кэша. Пренебрегать оптимизацией работы с памятью нельзя.

## Краткий итог по кэшам (повторение)
1. **Задача L1 кэша** — обеспечить **минимальную задержку** для конвейера ЦП.
2. **Причина появления L2, L3** — невозможность совместить **большую ёмкость** и **низкое время доступа**.
3. **Задача нижележащих кэшей** — **сократить штраф промаха** предыдущего уровня, перехватывая часть запросов к памяти.

## Что дальше?
- **Двухуровневое адаптивное предсказание** (Two-Level Adaptive Prediction) — использует **глобальную историю переходов**.
- **Алгоритм gshare** — комбинирует индекс из адреса перехода и глобальной истории.
- **Гибридные (турнирные) предсказатели** — используют **несколько алгоритмов одновременно**, выбирая лучший для каждого перехода динамически.

Эти методы позволяют достичь **точности предсказания выше 95%** в реальных приложениях, что критично для современных глубоких конвейеров.


## Как работает Branch Target Buffer (BTB)

### Структура BTB
BTB — это **специальный ассоциативный кэш** (часто полностью ассоциативный), расположенный на **стадии IF (Instruction Fetch)**. Каждая запись содержит три поля:

| Поле | Описание |
|------|----------|
| **Адрес инструкции перехода** | Адрес в памяти, по которому находится инструкция условного перехода |
| **Адрес цели перехода** | Адрес, куда должен перейти процессор, если переход выполняется |
| **Признак предсказания** | Флаг, указывающий, предсказываем ли мы, что этот переход **выполнится** (taken) |

---

## Подробный алгоритм работы BTB в конвейере

### Стадия IF (выборка инструкции)
1. **Обычный поток**:
   - PC (Program Counter) отправляется в **кэш инструкций L1** и одновременно в **BTB**.
   - Если адрес **не найден в BTB** → считаем, что это не переход (или переход нам неизвестен).
   - Вычисляем **PC + 4** (следующая инструкция по порядку).

2. **Если адрес найден в BTB**:
   - **И признак предсказания = "выполнится"**:
     - **Сразу заменяем PC+4 на адрес цели из BTB**.
     - Начинаем **спекулятивное выполнение** по этому адресу.
   - **Декодирование инструкции не производится** — мы ещё не знаем, что это за инструкция!
### Последующие стадии
1. **Инструкция продолжает движение по конвейеру**.
2. На стадии **ID (декодирование)** становится известно:
   - Является ли инструкция **условным переходом**.
   - Вычислился ли реальный **адрес перехода**.
   - Должен ли переход **выполниться** на основе условий.

### Проверка предсказания и обработка ошибок
#### Случай 1: Предсказание было верным
- **Переход предсказан как "выполнится" и действительно выполнился**:
  - Всё спекулятивно выполненное **остаётся**.
  - **Простой конвейера = 0 тактов**.
- **Переход предсказан как "не выполнится" и действительно не выполнился**:
  - Тоже **0 простоев**.

#### Случай 2: Предсказание было неверным
1. **Мы предсказали "выполнится", а переход не выполнился**:
   - **Отменяем всё спекулятивно выполненное** (очистка конвейера).
   - **Начинаем выполнение с адреса PC+4**.
   - **Штраф: 2 такта**:
     - 1 такт на определение ошибки на стадии ID/EX
     - 1 такт на переключение на правильный адрес

2. **Мы предсказали "не выполнится", а переход выполнился**:
   - Аналогично: **очистка конвейера**, переход на правильный адрес.
   - **Штраф: 2 такта**.

#### Случай 3: Перехода не было в BTB
- Если инструкция оказалась переходом **впервые**:
  - **Останавливаем конвейер** на стадии ID.
  - Вычисляем, выполнился ли переход.
  - **Добавляем запись в BTB** с адресом перехода, целью и признаком (например, "выполнится", если он выполнился).
  - В следующий раз для этого адреса предсказание **уже будет работать**.

## Поведение BTB на циклах

### Типичный цикл:
```c
for (int i = 0; i < 10; i++) {
    // тело цикла
}
```
1. **Первая итерация (вход в цикл)**:
   - В прошлый раз (если цикл не выполнялся) мы **вышли из цикла** → переход не выполнился.
   - BTB **помнит "не выполнится"** → предсказываем выход.
   - Но сейчас мы **должны войти в цикл** → **ошибка предсказания** (штраф 2 такта).
   - Обновляем BTB: теперь "выполнится".

2. **Средние итерации (2–9)**:
   - BTB помнит "выполнится" → всегда правильно предсказывает.
   - **0 простоев**.

3. **Последняя итерация (выход из цикла)**:
   - BTB всё ещё думает "выполнится".
   - Но мы **должны выйти из цикла** → **ошибка предсказания** (штраф 2 такта).
   - Обновляем BTB: теперь "не выполнится".

**Итого для цикла из 10 итераций**:
- 2 ошибки предсказания × 2 такта = 4 такта штрафа.
- Без BTB: каждая итерация могла бы стоить 2 такта простоя → 20 тактов.
- **Выигрыш значительный**.

## Усовершенствования BTB

### Branch Target Instruction Cache (BTIC)
- **Идея**: кэшировать не только **адрес цели**, но и **саму инструкцию по этому адресу**.
- **Преимущество**:
  - Избегаем обращения к **L1-кэшу инструкций** (даже если он быстрый, это всё равно задержка).
  - Получаем следующую инструкцию **мгновенно**.

### Современный подход: кэширование микроопераций
- В современных процессорах инструкции **декодируются в микрооперации** (μops).
- **BTB может кэшировать не инструкции, а готовые микрооперации** по адресу перехода.
- **Двойная выгода**:
  1. **Пропускаем стадию выборки** из L1.
  2. **Пропускаем стадию декодирования**.
- **Результат**: мгновенная готовность потока μops для исполнения.

## Эффективность BTB: когда она оправдана

### Простое правило
- Если BTB **угадывает правильно в >50% случаев** → средние потери **меньше**, чем без предсказания.
- В реальных программах с циклами **точность предсказания достигает 80–90%**.
- **Особенно эффективно для циклов** — большинство итераций предсказываются идеально.

### Сравнение с отсутствием BTB
| Метод | Штраф за переход | Для цикла из 10 итераций |
|-------|------------------|---------------------------|
| Без предсказания (всегда ждём) | 2 такта × каждый переход | 20 тактов |
| С BTB (2 ошибки на цикл) | 2 такта × 2 ошибки | 4 такта |
| **Выигрыш** | **5 раз** | |

## Связь с другими алгоритмами предсказания

BTB — это **механизм доставки адреса перехода**. Сам **алгоритм предсказания** (будет ли переход) может быть разным:

1. **Простой 1-битный флаг** (как в примере выше).
2. **2-битные счётчики с насыщением** (алгоритм Смита).
3. **Двухуровневые адаптивные предсказатели**.
4. **Турнирные (гибридные) предсказатели**.

**BTB может работать с любым из этих алгоритмов** — просто вместо "флага" в записи BTB будет храниться **состояние счётчика или более сложная структура**.


## Практические выводы

1. **BTB — аппаратный механизм**, расположенный на **стадии IF**.
2. **Цель**: дать адрес следующей инструкции **до её декодирования**.
3. **Ключевая идея**: запоминать **куда переходили в прошлый раз** и повторять это поведение.
4. **Ошибки предсказания**:
   - Стоят **2 такта** (очистка конвейера + переключение).
   - Но в циклах случаются **только на входе и выходе**.
5. **Современные процессоры** кэшируют не инструкции, а **микрооперации**, что ускоряет работу ещё сильнее.


## Проблемы исходного Branch Target Buffer (BTB)

### Большой объём хранимых данных
Каждая запись BTB содержит:

| Поле | Размер (бит) | Комментарий |
|------|--------------|-------------|
| **Адрес инструкции перехода** | 32 или 64 | Полный адрес в памяти |
| **Адрес цели перехода** | 32 или 64 | Полный адрес, куда переходим |
| **Признак предсказания** | 1–2 бита | Флаг или счётчик |
| **Тег для ассоциативности** | Часть адреса | Для поиска в ассоциативном кэше |

**Итого**: каждая запись занимает **65–130+ бит**.

### Полностью ассоциативный поиск — дорого
- **Сложность логики**: поиск адреса во всём буфере.
- **Задержка**: сравнение со всеми записями требует времени.
- **Площадь на кристалле**: много компараторов и памяти.


## Упрощение: Branch History Table (BHT) / Pattern History Table (PHT)

### Основная идея
**Отказываемся от хранения адресов** и используем **прямое отображение**:

- Берем **N младших бит адреса инструкции**.
- Используем их как **индекс в массиве бит**.
- В массиве храним только **один бит**: 
  - `0` = последний раз переход **не выполнился**.
  - `1` = последний раз переход **выполнился**.

### Структура PHT
```
Адрес инструкции: [ старшие биты | N младших бит (индекс) ]
                        ↓
                  +---------------+
                  | PHT (массив)  |
                  | размер = 2^N  |
                  | каждый эл-т = 1 бит |
                  +---------------+
                        ↓
                  0 или 1 → предсказание
```

**Пример для N=10**:
- PHT имеет **1024 записи** (2^10).
- Размер памяти: **1024 бита = 128 байт** — очень мало!
### Алгоритм работы
1. **На стадии IF**:
   - Берем **N младших бит PC** → получаем индекс.
   - Читаем бит из PHT по индексу.
   - Если бит = 1 → **предсказываем "переход выполнится"**.
   - Если бит = 0 → **предсказываем "переход не выполнится"**.

2. **На стадии ID/EX** (после декодирования):
   - Определяем, был ли это действительно переход.
   - Определяем, выполнился ли он на самом деле.
   - **Обновляем PHT**:
     - Если переход выполнился → записываем `1` по индексу.
     - Если не выполнился → записываем `0` по индексу.
### Где взять адрес перехода?
**Проблема**: PHT хранит только **будет/не будет переход**, но не **куда переходить**.

**Решение**:
- **Декодируем инструкцию** (всё равно нужно).
- **Вычисляем адрес цели** из полей инструкции (смещение).
- **Задержка**: адрес цели становится известен только на стадии ID → **штраф в 1 такт** (не 2, как в BTB при ошибке).


## Проблема PHT: интерференция адресов

### Суть проблемы
Если два разных перехода имеют **одинаковые N младших бит адреса**, они будут **использовать одну и ту же ячейку PHT**.

**Пример**:
- Переход A по адресу `0x1004` (младшие 10 бит: `0000000100`)
- Переход B по адресу `0x2004` (младшие 10 бит: `0000000100` — те же!)
- Оба будут читать/писать **один и тот же бит в PHT** → **конфликт**.

### Последствия
- **История одного перехода портит предсказание другого**.
- Особенно плохо в больших программах с множеством переходов.
- **Увеличение N** уменьшает вероятность конфликта, но увеличивает размер таблицы.

## Улучшение: двухбитные счётчики с насыщением (Алгоритм Смита)

### 4.1. Почему 1 бит недостаточно?
**Пример с циклом**:
```
for (int i = 0; i < 10; i++) {
    // тело
}
```
1. **Первая итерация**:
   - В прошлый раз вышли из цикла → бит = 0.
   - Сейчас нужно войти → **ошибка**, обновляем в 1.

2. **Вторая–девятая итерации**:
   - Бит = 1 → всегда угадываем.

3. **Последняя итерация**:
   - Бит = 1 → предсказываем "войти".
   - Но нужно выйти → **ошибка**, обновляем в 0.

**Итого**: **2 ошибки за цикл**.

### Двухбитный счётчик
Каждая запись PHT теперь — **2-битный счётчик**:

```
Состояния:
  00: Strongly Not Taken (уверенно "не выполнится")
  01: Weakly Not Taken   (слабо "не выполнится")
  10: Weakly Taken       (слабо "выполнится")
  11: Strongly Taken     (уверенно "выполнится")
```

**Старший бит = предсказание**:
- `0` → предсказываем "не выполнится"
- `1` → предсказываем "выполнится"

### Правила обновления (с насыщением)
- **Если переход выполнился (Taken)**:
  - Увеличиваем счётчик (но не выше 11).
  - `00 → 01`, `01 → 10`, `10 → 11`, `11 → 11`

- **Если переход не выполнился (Not Taken)**:
  - Уменьшаем счётчик (но не ниже 00).
  - `11 → 10`, `10 → 01`, `01 → 00`, `00 → 00`

### Почему это лучше для циклов?
**Тот же цикл из 10 итераций**:

1. **Перед циклом**: счётчик = `00` (после выхода в прошлый раз).

2. **Первая итерация**:
   - Предсказание: `00` → "не выполнится" (ошибка).
   - Обновление: `00 → 01` (уже не Strongly Not Taken).

3. **Вторая итерация**:
   - Предсказание: `01` → "не выполнится" (опять ошибка!).
   - Обновление: `01 → 10` (теперь Weakly Taken).

4. **Третья–девятая итерации**:
   - Предсказание: `10` → "выполнится" (правильно).
   - Обновление: каждый раз `10 → 11` (на первой правильной итерации), потом `11 → 11`.

5. **Последняя итерация**:
   - Предсказание: `11` → "выполнится" (ошибка).
   - Обновление: `11 → 10` (Weakly Taken).

**Итого**: 
- **Ошибки только на 1-й и 2-й итерациях** (вход в цикл).
- **На выходе из цикла ошибки нет** — счётчик стал `10`, что всё ещё предсказывает "выполнится", но это некритично, т.к. цикл закончился.
- **На следующем входе в цикл**: счётчик `10` → сразу правильно предсказывает "выполнится"!

**Фактически, двухбитный счётчик "запоминает"**, что цикл обычно выполняется много раз подряд, и после первого входа быстро переходит в режим "выполнится".

## Сравнение методов

| Метод | Размер записи | Интерференция | Ошибок в цикле | Простота |
|-------|---------------|---------------|----------------|----------|
| **BTB (ассоциативный)** | 65–130+ бит | Нет (теги) | 2 | Сложно |
| **PHT (1 бит)** | 1 бит | Есть | 2 | Очень просто |
| **PHT (2 бита)** | 2 бита | Есть | 1–2 | Просто |

## Выводы по одноуровневому предсказанию

1. **PHT с прямым отображением** — очень компактный, быстрый, но страдает от **интерференции адресов**.
2. **Двухбитные счётчики** значительно улучшают точность за счёт **гистерезиса** (нужны две ошибки подряд, чтобы изменить предсказание).
3. **Основная проблема остаётся**: предсказание основано **только на последнем поведении этого конкретного перехода**, без учёта контекста (какие переходы выполнялись до этого).
4. **Это приводит к двухуровневым и адаптивным предсказателям**, которые учитывают **глобальную историю переходов**.


## Двухбитные счётчики с насыщением: диаграммы состояний

### Две схемы обновления счётчиков

#### Схема 1 (Простая, 1 бит меняется)
```
Состояния: 00 → 01 → 10 → 11
Правила:
- При выполнении перехода (T): двигаемся ВПРАВО (00→01, 01→10, 10→11, 11→11)
- При невыполнении (NT): двигаемся ВЛЕВО (11→10, 10→01, 01→00, 00→00)

Предсказание:
- 00, 01 → предсказываем "не выполнится" (старший бит = 0)
- 10, 11 → предсказываем "выполнится" (старший бит = 1)
```

#### Схема 2 (С пропуском)
```
Состояния: 00 → 01 → 10 → 11
Но переходы:
- 00 → 10 (при T)
- 10 → 11 (при T)
- 11 → 01 (при NT)
- 01 → 00 (при NT)
```

### Содержательное отличие
**Схема 2** имеет важную особенность:
- **Двойная ошибка → переход в уверенное предсказание противоположного направления**
  - Из 00 (Strongly NT) после двух T подряд сразу в 10 (Weakly T)
  - Из 11 (Strongly T) после двух NT подряд сразу в 01 (Weakly NT)

### Техническое отличие
- **Схема 1**: проще реализовать — изменение одного бита (инкремент/декремент)
- **Схема 2**: требует сложнее логики, но может быстрее адаптироваться

## N-битные счётчики: почему не используют больше 2 бит?

### Наблюдения
- **Эффективность 2-битной схемы сравнима с n-битной** (n > 2)
- На практике редко используют более 3 бит

### Почему больше бит может быть хуже
1. **Медленное забывание**:
   - Если программа долго ходила одним путём (счётчик на максимуме)
   - Потом данные поменялись, нужно ходить другим путём
   - **N-битному счётчику нужно n ошибок подряд** чтобы изменить предсказание
   - **2-битному — всего 2 ошибки**

2. **Усложнение логики** без значительного выигрыша


## Одноуровневые предсказатели: Pattern History Table (PHT) vs Branch History Table (BHT)

### PHT (Pattern History Table) — "бестеговая" версия
- **Индекс = N младших бит адреса инструкции**
- Каждая запись = 2-битный счётчик
- **Проблема**: интерференция адресов

### BHT (Branch History Table) — с тегами
- **Добавляется поле "тег адреса"**
- **Как работает**:
  1. Адрес инструкции разбивается:
     - Младшие 2 бита = 00 (выравнивание)
     - Следующие M бит = индекс в таблице
     - Остальные биты = тег
  2. При обращении:
     - Сравниваем тег в таблице с тегом адреса
     - Если совпадает → используем счётчик
     - Если нет → это другая инструкция

### Пример разбиения адреса (32 бита)
```
Адрес: [ 20 бит - тег ][ 10 бит - индекс ][ 2 бита - 00 ]
       ↑                ↑                 ↑
     Для сравнения    Номер записи     Всегда 0
```

**Размер записи BHT**:
- 2 бита (счётчик) + 20 бит (тег) = 22 бита
- PHT: только 2 бита

**Размер таблицы 1024 записи**:
- PHT: 1024 × 2 бита = 256 байт
- BHT: 1024 × 22 бита ≈ 2.8 КБ

## Интерференция адресов: что это и как бороться

### 4.1. Суть проблемы (для PHT)
Если две разные инструкции перехода имеют **одинаковые N младших бит**, они:
- Используют **одну и ту же ячейку PHT**
- **Портят предсказания друг другу**

### 4.2. Пример интерференции
```
Переход A: адрес 0x1234 (младшие 10 бит: 0x034)
Переход B: адрес 0x2234 (младшие 10 бит: 0x034) ← совпали!
```
Обе инструкции будут читать/писать **одну ячейку PHT** → непредсказуемое поведение.

### Решения
1. **Увеличить N** → больше ячеек, меньше вероятность коллизий
2. **Использовать BHT с тегами** → полное устранение интерференции (но дороже)
3. **Использовать двухуровневые предсказатели** (рассмотрим далее)

## Практическая эффективность

### Цифры точности предсказания
| Метод | Ошибки для integer | Ошибки для float |
|-------|-------------------|------------------|
| **Статическое предсказание** | ~30% | ~30% |
| **PHT (4000 записей)** | 5–20% (среднее 11%) | 1–9% (среднее 4%) |
| **BHT (с тегами)** | Лучше на ~5% | Лучше на ~5% |

### Что даёт увеличение размера таблицы?
- **Для PHT размером 4000+ записей**: дальнейшее увеличение даёт **0–1% улучшения**
- **Вывод**: 4000 записей достаточно для большинства приложений

### Современные процессоры (конец 90-х)
- Intel заявляла **>97% точности предсказания**
- Достигалось **двухуровневыми и гибридными алгоритмами** (не только простой PHT/BHT)

## Специальные схемы для циклов

### Проблема с циклами
Даже с двухбитными счётчиками:
- **Ошибка при входе в цикл** (1–2 раза)
- **Ошибка при выходе из цикла** (всегда!)

### Современные решения
1. **Счётчики итераций**:
   - Запоминают, сколько итераций обычно выполняется
   - Предсказывают выход на последней итерации

2. **Проблема**:
   - Работает для **статических циклов** (for i=0; i<N; i++)
   - Не работает для **динамических обходов** (списки, деревья, while(p != NULL))

## Итоги по одноуровневым предсказателям

### Что мы имеем
1. **PHT**:
   - Компактный (2 бита/запись)
   - Быстрый (прямое отображение)
   - Страдает от интерференции

2. **BHT**:
   - Решает проблему интерференции (теги)
   - Больше памяти (20+ бит/запись)
   - Точнее на ~5%

3. **Двухбитные счётчики**:
   - Лучше однобитных (гистерезис)
   - Достаточно для большинства случаев
   - N-битные не дают значительного преимущества

### Ограничения
- **Не учитывают контекст**: предсказание основано только на истории данного перехода
- **Не знают о других переходах**, которые выполнялись до этого
- **Не адаптируются к паттернам** (например, T, T, NT, T, T, NT...)

## Что дальше? Двухуровневые предсказатели

### Идея
Использовать не только **адрес перехода**, но и **глобальную историю переходов**:
- Какие переходы выполнялись до этого?
- Какой "путь" по программе мы прошли?

### Пример алгоритмов
1. **Gshare** (1993): комбинирует адрес перехода и глобальную историю через XOR
2. **Алгоритм McFarling**: несколько предсказателей + мета-предсказатель
3. **Турнирные (гибридные) предсказатели**: несколько алгоритмов конкурируют, выбирается лучший

### Результат
- **Точность >97%** в реальных приложениях
- **Минимизация штрафов** за неправильные предсказания
- **Адаптация** к разным типам программ (integer vs float, серверные vs десктопные)
### Ключевые выводы сегодняшней лекции
1. **BTB** — механизм доставки адреса перехода (ассоциативный, дорогой).
2. **PHT/BHT** — алгоритмы предсказания (прямое отображение, дешёвые).
3. **Двухбитные счётчики** — оптимальный баланс сложности/эффективности.
4. **Интерференция адресов** — серьёзная проблема для PHT, решается тегами (BHT).
5. **Современные процессоры** используют сложные многоуровневые алгоритмы для точности >97%.
