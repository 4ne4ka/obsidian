# Динамическое предсказание условных переходов

## Введение и контекст
*   **Цель:** Борьба с **конфликтами управления** в конвейере, вызванными условными переходами (ветвлениями). Предсказание позволяет минимизировать простои (stalls).
*   **Два ключевых компонента:**
    1.  **Алгоритм предсказания** (логика принятия решения).
    2.  **Механизм реализации** в конвейере (аппаратная часть).
*   **Идеальное место для предсказания:** Стадия **IF (Instruction Fetch — выборка инструкции)**. Это позволяет максимально рано определить адрес следующей инструкции и избежать остановки конвейера.

## Базовый механизм: Branch Target Buffer (BTB)
BTB — это аппаратная таблица (буфер), которая позволяет предсказывать переходы на стадии IF.

### **Как работает BTB:**
1.  **На стадии IF:** Подается адрес текущей инструкции (PC) в BTB.
2.  **Проверка:** Если адрес найден в BTB, значит, эта инструкция — **условный переход**, и в BTB хранится:
    *   **Предсказанное направление** (будет переход/не будет).
    * **Целевой адрес** (адрес, куда перейти, если переход предсказан как "произошедший").
3.  **Действие:** Если переход предсказан как "произошедший" (taken), следующий PC загружается из BTB. Если не найден или предсказан как "непроизошедший", PC просто инкрементируется.

### **Алгоритм работы BTB в конвейере:**
*   **IF:** Отправка PC в память (кэш инструкций) и в BTB.
* **ID (Instruction Decode — декодирование):** Определяется, является ли инструкция условным переходом.
    *   **Если это переход И в BTB была запись:** Используется предсказанный PC.
    *   **Иначе:** Работа продолжается как с обычной инструкцией.
*  **EX (Execution — выполнение):** Фактически вычисляется условие перехода и его целевой адрес.
    *   **Если предсказание было верным:** Конвейер работает без задержек.
    *   **Если предсказание было неверным:** Возникает **штраф (penalty)**.
        *   Нужно "убить" (kill) ошибочно выбранные инструкции.
        *   Начать выборку с правильного адреса.
        *   Обновить или удалить запись в BTB.
*   **Штраф за неверное предсказание в BTB:** **2 такта** (задержка обнаружения ошибки на стадии EX + перезапуск выборки).

##### **Цель BTB:** Добиться ситуации **«переходов без простоев»** (zero-cycle branches).

## Алгоритмы предсказания направления (для хранения в BTB)

### Одноуровневый (Бимодальный) предсказатель
*   **Суть:** Использует одну **Таблицу Истории Переходов (Branch History Table, BHT)**.
*   **Структура:** Таблица двухбитных **счетчиков с насыщением**, индексируемая младшими битами адреса перехода.
*   **Принцип работы счетчика (2 бита):**
    *   Состояния `11` и `10`: **Предсказание "переход произойдет" (Taken)**.
    *   Состояния `01` и `00`: **Предсказание "перехода не будет" (Not Taken)**.
    *   **При фактическом выполнении перехода:** Если переход *произошел* — значение счетчика увеличивается (до максимума 3). Если *не произошел* — уменьшается (до минимума 0). Это обеспечивает "гистерезис": нужно два неожиданных события подряд, чтобы изменить предсказание.
*   **Эффективность (на тестах SPEC89):**
    *   С таблицей на **4096 записей** (2^12) достигается точность, близкая к таблице "бесконечного" размера.
    *   **Средняя вероятность ошибки:** ~11% для целочисленных программ (Integer), ~4% для программ с плавающей точкой (FP).
    *   **Вывод:** Для этих тестов **увеличение размера таблицы (снижение интерференции адресов) не дает существенного выигрыша.** FP-программы ошибаются реже из-за большего количества предсказуемых циклов.

### Проблема корреляции переходов и двухуровневые предсказатели
*   **Проблема:** Поведение одного перехода может зависеть от результатов **предыдущих переходов** (например, в цепочках `if-then-else`).
*   **Пример из лекции:**
    ```c
    if (aa == 2) aa = 0;  // Переход B1
    if (bb == 2) bb = 0;  // Переход B2
    if (aa != bb) { ... } // Переход B3
    ```
    *   Если оба перехода **B1** и **B2** **НЕ произошли** (ветви `else` не выполнялись), то мы точно знаем, что `aa=0` и `bb=0`. Следовательно, условие `aa != bb` ложно, и переход **B3 НЕ произойдет**.
    *   Одноуровневый предсказатель, видящий только адрес B3, эту закономерность **не улавливает**.

### Двухуровневое адаптивное предсказание с учетом корреляции (GAg, gshare)
Использует **два уровня** информации для принятия решения.

*   **Первый уровень (глобальный): Branch History Register (BHR)**
    *   **Что это:** `m-битный регистр сдвига`.
    *   **Что хранит:** Историю (`1`/`0`) `m` последних **глобально** выполненных условных переходов (независимо от их адреса).
    *   **Как обновляется:** При выполнении каждого перехода его результат (`1` если был переход, `0` если не был) записывается в младший бит, остальные биты сдвигаются.

*   **Второй уровень (локальный для каждого "шаблона" истории): Pattern History Tables (PHTs)**
    *   **Что это:** Не одна, а **2^m** отдельных таблиц (по числу возможных значений `m-битного` BHR). Каждая таблица — это массив `n-битных` счетчиков с насыщением (обычно 2-битных).
    *   **Как работает:**
        1.  **Шаг 1:** Текущее значение **BHR** выбирает, **какую именно** из 2^m таблиц PHT использовать для предсказания. Разные истории ведут к разным таблицам.
        2.  **Шаг 2:** Внутри выбранной таблицы PHT младшие `N` бит **адреса текущего перехода** используются как индекс для выбора конкретного счетчика.
        3.  **Шаг 3:** Значение этого счетчика дает итоговое предсказание (Taken/Not Taken).

*   **Объем памяти:** `Общее число бит = (2^m) * (размер счетчика n) * (2^N)`

*   **Обозначение GAg(m,n):** `m` — размер BHR, `n` — битность счетчиков в PHT.
    *   **Пример:** Обычный одноуровневый предсказатель — это частный случай **GAg(0,2)** (истории нет, 2-битные счетчики).
    *   **Пример GAg(2,2):** BHR = 2 бита (помнит 2 последних перехода) → 4 таблицы PHT (для шаблонов 00, 01, 10, 11). В каждой таблице, например, 16 записей (если N=4).

*   **Разновидность: gshare (McFarling, 1993)**
    *   Усовершенствование: Индекс для PHT формируется как **XOR** между глобальной историей (BHR) и адресом перехода, что позволяет эффективнее использовать одну общую таблицу PHT и уменьшить конфликты.
## Гибридные (Турнирные) предсказатели
*   **Суть:** Комбинация двух (или более) разных механизмов предсказания (например, GAg и gshare).
*   **Как работает:** Добавляется **мета-предсказатель** — дополнительная таблица, которая отслеживает, какой из основных предсказателей был более точен для **конкретного перехода** в недавнем прошлом.
*   **Итоговое предсказание** выдает тот алгоритм, который мета-предсказатель считает более точным для данной ситуации.
*   **Цель:** Адаптация к разным типам переходов в программе (одни лучше предсказывает один алгоритм, другие — другой).

## Итоги эволюции методов
1.  **Одноуровневый (бимодальный):** Таблица 2-битных счетчиков по адресу перехода. Просто, эффективно, основа.
2.  **Двухуровневой GAg/gshare:** Учитывает глобальную историю переходов для улавливания корреляций. Точнее, но сложнее.
3.  **Гибридный (турнирный):** Объединяет сильные стороны разных алгоритмов для максимальной точности. Используется в современных высокопроизводительных процессорах.

**Ключевой вывод:** Динамическое предсказание радикально (в разы) снижает количество ошибок по сравнению со статическим, что критически важно для поддержания высокой скорости работы глубокого конвейера. Плата за это — увеличение аппаратной сложности (BTB, BHT, логика предсказания) и штраф в 2+ такта при ошибке.

## Подробный разбор примера с корреляцией переходов (GAp(1,1))

### Анализ исходного кода
Рассматривается фрагмент кода:
```c
if (d == 0)
    d = 1;
if (d == 1)
    // Какое-то действие
```

**Реализация на псевдоассемблере:**
```
BNEZ  R1, L1     ; branch b1: if (d != 0) переход на L1 (пропускаем then)
DADDIU R1, R0, #1 ; then-часть: d = 1 (R1 = 0 + 1)
L1:
DADDIU R3, R1, #-1 ; R3 = d - 1 (эквивалент проверки d == 1)
BNEZ  R3, L2     ; branch b2: if (d != 1) переход на L2 (пропускаем then)
L2:
```
*   **Логика:**
    *   **B1:** Переход происходит (`BNEZ` = taken), если `d != 0` (мы пропускаем `then` и идём в `else`).
    *   **B2:** Переход происходит (`BNEZ` = taken), если `d != 1` (мы пропускаем `then` и идём в `else`).
*   **Ключевая корреляция:** Поведение перехода **B2** зависит от результата перехода **B1**. Если B1 *не произошёл* (т.е. `d` был равен 0 и мы зашли в `then`), то после выполнения `d = 1` условие для B2 (`d != 1`) становится ложным, и B2 *также не произойдёт*.

### Принцип работы предсказателя GAp(1,1)
*   **m = 1, n = 1:**
    *   **Первый уровень (BHR):** 1-битный регистр глобальной истории. Запоминает результат (`1`/`0`) **последнего** выполненного в программе условного перехода.
    *   **Второй уровень (PHTs):** `2^m = 2^1 = 2` таблицы предсказаний (для BHR=0 и BHR=1). В каждой таблице счётчики 1-битные (`n=1`). Это значит:
        *   `0` = предсказание "перехода не будет" (Not Taken).
        *   `1` = предсказание "переход будет" (Taken).
        *   **Обновление:** Счётчик сразу меняется на противоположное значение в случае ошибки предсказания (нет "гистерезиса", как у 2-битного).

*   **Схема работы для двух переходов (B1 и B2):**
    *   Для каждого перехода в таблице PHT есть своя запись (индексируется младшими битами адреса инструкции).
    *   Для **B1** актуальны две ячейки (по одной в каждой из двух таблиц PHT).
    *   Для **B2** актуальны две другие ячейки (также по одной в каждой таблице PHT).
    *   **Алгоритм предсказания:**
        1.  Смотрим текущее значение BHR (0 или 1).
        2.  Выбираем соответствующую таблицу PHT (0 или 1).
        3.  По адресу перехода (B1 или B2) находим нужную ячейку в выбранной таблице.
        4.  Значение в ячейке (0 или 1) — это и есть предсказание.

### Пошаговый разбор таблицы выполнения

**Исходное состояние:** `d = 2`. BHR = 0 (например, предположим, предыдущий переход не происходил). Все счётчики в PHT инициализированы в 0 ("предсказываем Not Taken").

**Итерация 1 (`d=2`):**
*   **B1:** BHR=0 → берём предсказание из PHT0 для B1 = **"Нет"** (Not Taken). Фактически `d=2` (не равно 0), поэтому переход **ДОЛЖЕН произойти (Taken)** → **ОШИБКА ПРЕДСКАЗАНИЯ**.
    *   Обновляем счётчик для B1 в PHT0 на противоположное: теперь там **"Да"** (Taken).
    *   Обновляем BHR: записываем результат B1 = 1 (произошёл). Теперь BHR = 1.
*   **B2:** Теперь BHR=1 → берём предсказание из PHT1 для B2 = **"Нет"**. Фактически `d` всё ещё =2 (не равно 1), переход **ДОЛЖЕН произойти (Taken)** → **ОШИБКА**.
    *   Обновляем счётчик для B2 в PHT1 на **"Да"**.
    *   Обновляем BHR: записываем результат B2 = 1. BHR остаётся = 1.

**Итерация 2 (`d=0`):**
*   **B1:** BHR=1 → предсказание из PHT1 для B1 = **"Да"**. Фактически `d=0` (равно 0), переход **НЕ должен происходить (Not Taken)** → **ОШИБКА**.
    *   Обновляем счётчик для B1 в PHT1 на **"Нет"**.
    *   Обновляем BHR: записываем 0 (не произошёл). Теперь BHR = 0.
*   **B2:** BHR=0 → предсказание из PHT0 для B2 = **"Да"**. Фактически после выполнения then-части B1 теперь `d=1`, условие `d != 1` ложно, переход **НЕ должен происходить (Not Taken)** → **ОШИБКА**.
    *   Обновляем счётчик для B2 в PHT0 на **"Нет"**.
    *   Обновляем BHR: записываем 0. BHR = 0.

**Итерация 3 (`d=2`):** Ситуация аналогична итерации 1, и предсказатель, "наученный" на предыдущих шагах, начнёт угадывать правильно.
**Итерация 4 (`d=0`):** Ситуация аналогична итерации 2, и предсказатель также будет предсказывать верно.

**Вывод:** После начальной "обучающей" фазы (2 итерации) двухуровневый предсказатель GAp(1,1) **начинает правильно учитывать корреляцию** между B1 и B2. Он запоминает, что *если предыдущий переход (B1) не произошёл, то и следующий (B2) не произойдёт*. Одноуровневый предсказатель (Gap(0,1)) так делать не умел и постоянно ошибался на чередующихся значениях `d`.

## Усовершенствование: Предсказатель gshare

### Проблемы схемы GAp:
1.  **Неэффективное использование памяти:** Для `m` бит истории нужно `2^m` отдельных таблиц PHT. Большинство ячеек в этих таблицах остаются пустыми или редко используются (принцип локальности кода).
2.  **Интерференция адресов:** Чтобы её избежать, нужны большие таблицы.

### Решение — gshare (McFarling, 1993):
*   **Идея:** Использовать **одну общую таблицу PHT**, но формировать индекс в неё с учётом и истории, и адреса.
*   **Механизм:** Индекс = **XOR (исключающее ИЛИ)** между:
    *   `m-битным` Branch History Register (BHR)
    *   `m` младшими битами адреса текущей инструкции перехода (обычно отбрасываются выравнивающие нули, если длина инструкции фиксирована, например, 2 младших бита для 4-байтных инструкций).
*   **Структура:** Одна таблица из `2^m` записей (каждая запись — `n-битный` счётчик, обычно 2-битный).

*   **Преимущества перед GAp:**
    *   **Экономия памяти:** Вместо `2^m` таблиц — одна.
    *   **Меньше логики:** Не нужно выбирать из множества таблиц.
    *   **Можно использовать более длинную историю:** Увеличить `m` стало дешевле.
    *   **Эффективное хеширование:** Операция XOR помогает лучше распределять записи в одной таблице, снижая конфликты.

*   **Результат:** Схема **gshare** показала сравнимую, а часто и лучшую точность, чем **GAp**, особенно при небольших размерах таблицы. Она стала и остаётся очень популярной в реальных процессорах из-за своего соотношения эффективности и простоты.

## Турнирные (Гибридные) предсказатели

*   **Идея (также предложена McFarling):** Ни один алгоритм не является идеальным для всех типов переходов в программе.
*   **Реализация:** Используется **два разных предсказателя** (например, `gshare` и простой биморальный) параллельно.
*   **Мета-предсказатель (selector):** Дополнительная таблица (часто тоже 2-битные счётчики), которая отслеживает, **какой из двух основных предсказателей был более точен для данного конкретного перехода** в недавнем прошлом.
*   **Принцип работы:**
    1.  Оба предсказателя вычисляют свой вердикт для текущего перехода.
    2.  Мета-предсказатель по адресу перехода смотрит, какому из двух алгоритмов стоит доверять **в данном конкретном случае**.
    3.  Итоговое предсказание берётся от того алгоритма, на который указывает мета-предсказатель.

*   **Цель:** Адаптироваться к различным шаблонам поведения переходов в программе. Например, одни переходы (в циклах) лучше предсказывает простой алгоритм, а другие (в сложных цепочках `if-else`) — `gshare`.

## Общие итоги по точности

*   Одноуровневый (биморальный) предсказатель на таблице в **4096 записи** даёт точность:
    *   ~89% (11% ошибок) для целочисленных программ (INT).
    *   ~96% (4% ошибок) для программ с плавающей точкой (FP).
*   Двухуровневые предсказатели (**GAp**, **gshare**) позволяют **значительно улучшить точность для INT-программ** (где много коррелирующих переходов типа `if-then-else`), иногда сокращая количество ошибок вдвое.
*   Гибридные предсказатели стремятся взять лучшее от обоих миров.
*   **Важный вывод лекции:** К концу 90-х годов (на материале SPEC89) с использованием таких техник, как `gshare`, удавалось достичь точности предсказания **выше 96%**, что для того времени было выдающимся результатом и критически важным для развития глубокого конвейерирования.

## Гибридные (Турнирные) предсказатели

### Идея и мотивация
*   **Проблема:** Ни один алгоритм предсказания (биморальный, `gshare`, `GAp`) не является универсально лучшим для всех типов переходов в программе.
*   **Наблюдение (McFarling, 1993):** Для некоторых переходов (например, в простых циклах) достаточно простого адресного предсказателя. Для других (сложные цепочки `if-else`) критически важна глобальная история. Причём на маленьких таблицах простой предсказатель иногда может быть даже эффективнее сложного.
*   **Аналогия:** Как в библиотеках линейной алгебры: для маленьких матриц используется один алгоритм умножения, для больших — другой, более оптимизированный под кэш. Система сама выбирает лучший в данной ситуации.

### Принцип работы
*   **Структура:**
    *   Используются **два (или более) разных предсказателя** параллельно (например, простой биморальный и `gshare`).
    *   Добавляется **мета-предсказатель (селектор)** — отдельная таблица, обычно из двухбитных счётчиков с насыщением. Каждая запись в ней соответствует конкретному переходу (индексируется адресом).
*   **Алгоритм принятия решения:**
    1.  Оба основных предсказателя вычисляют свой вердикт для текущего перехода.
    2.  Мета-предсказатель по адресу перехода смотрит, какой из двух алгоритмов был более точен для **этого конкретного перехода** в прошлом.
        *   **Старший бит счётчика селектора решает:** например, `10` или `11` → выбираем предсказатель P1, `00` или `01` → выбираем P2.
    3.  Итоговое предсказание берётся от "победившего" алгоритма.
*   **Обновление мета-предсказателя (турнирная логика):**
    *   **Если P1 ошибся, а P2 угадал:** Счётчик селектора **уменьшается** (сдвигаемся в сторону выбора P2).
    *   **Если P1 угадал, а P2 ошибся:** Счётчик селектора **увеличивается** (сдвигаемся в сторону выбора P1).
    *   **Если оба угадали или оба ошиблись:** Состояние счётчика **не меняется**.
*   **Результат:** Гибридный предсказатель автоматически адаптируется, выбирая для каждого перехода наиболее эффективный в данный момент алгоритм. Он **всегда работает не хуже, а часто лучше** каждого из своих компонентов в отдельности.

## Реальные примеры из процессоров 90-х

### Intel Pentium (1993)
*   **Точность:** Около 78% (довольно низкая для своего времени).
*   **Механизм:**
    *   **Одноуровневый (биморальный) предсказатель:** BHT (Branch History Table) с двухбитными счётчиками Смита.
    *   **BTB (Branch Target Buffer):** 4-канальный наборно-ассоциативный кэш на 256 записей.
*   **Ограничения:**
    *   Нет предсказания для непроизошедших переходов.
    *   Нет **RAS (Return Address Stack)** для предсказания адресов возврата из процедур — критически важная оптимизация для `call`/`ret`.
    *   Не может обрабатывать несколько переходов одновременно.
*   **Штраф за ошибку:** Всего **3-4 такта** (благодаря очень короткому конвейеру).

### HP PA-8000
*   **Гибридная стратегия:** Комбинирует **статическое** и **динамическое** предсказание.
*   **Статическая часть:**
    *   Компилятор может явно указывать направление перехода (на основе профилирования или эвристик).
    *   По умолчанию: **прямые** переходы предсказываются как "непроизошедшие" (Not Taken), **обратные** (типичные для циклов) — как "произошедшие" (Taken).
    *   Может динамически включать/выключать динамический предсказатель для отдельных страниц памяти.
*   **Динамическая часть:**
    *   Одноуровневая BHT на 256 записей.
    *   **Не счётчик, а 3-битный регистр сдвига:** Хранит результаты **трёх последних** выполнений данного перехода. Предсказание, вероятно, основано на анализе этого паттерна.
*   **BTAC (Branch Target Address Cache):** Отдельный кэш целей переходов на 32 записи, чтобы избежать 1-тактовой задержки при расчёте адреса.

**Вывод:** К середине 90-х индустрия экспериментировала с разными подходами (`gshare`, гибридные), но даже флагманские процессоры (Pentium) часто использовали относительно простые схемы из-за соображений стоимости и сложности.

## Введение в суперскалярное исполнение

### Мотивация и основная идея
*   **Лимит обычного конвейера:** CPI (Clocks Per Instruction) ≥ 1. Максимум **одна инструкция завершается за такт**.
*   **Цель суперскалярности:** Выполнять **более одной инструкции за такт** (IPC > 1). Это путь к повышению производительности без бесконечного наращивания тактовой частоты.
*   **Суть:** Оборудование (аппаратура) процессора анализирует поток инструкций и **выдает на выполнение несколько независимых инструкций параллельно** в одном такте.

### Ключевые требования и проблемы
Для работы суперскалярного конвейера **все его стадии должны поддерживать параллельную обработку** нескольких инструкций:
1.  **IF (Выборка):** Должна уметь выбирать несколько инструкций из кэша за такт.
2.  **ID (Декодирование):** Должен уметь декодировать несколько инструкций параллельно.
3.  **EX (Исполнение):** Нужно иметь **несколько функциональных устройств** (АЛУ, FPU, блоки загрузки/сохранения), чтобы выполнять инструкции одновременно.
4.  **MEM (Доступ к памяти):** Шина памяти и кэш должны поддерживать несколько параллельных обращений (сложно!).
5.  **WB (Запись результата):** Порт(ы) в регистровый файл должны позволять записывать несколько результатов за такт.

**Главное ограничение:** Конвейер будет работать со скоростью **самой медленной ("узкой") стадии**. Если стадия ID может обработать 4 инструкции/такт, а стадия WB — только 2 инструкции/такт, то реальный IPC не превысит 2.

### Роль компилятора и аппаратуры
*   **Компилятор (статическая оптимизация):** Может переупорядочивать инструкции (`instruction scheduling`) ещё на этапе компиляции, чтобы увеличить параллелизм на уровне независимых инструкций (ILP). **Но:** он не знает динамических данных (результатов условных переходов, адресов памяти).
*   **Аппаратура (динамическая оптимизация):** Процессор должен на лету:
    *   **Анализировать зависимости:** Находить независимые инструкции, которые можно выполнить параллельно.
    *   **Разрешать конфликты:** Обходить зависимости по данным (RAW, WAR, WAW) с помощью переименования регистров и внеочередного исполнения.
    *   **Обеспечивать корректность:** Гарантировать, что результат программы будет таким же, как при последовательном выполнении.
*   **Фактическая производительность:** Динамически меняется от **0 до теоретического максимума** в зависимости от:
    *   Наличия независимых инструкций в коде.
    *   Количества конфликтов по данным (истинных зависимостей RAW).
    *   Задержек доступа к памяти ("промахов" в кэше).
    *   Ошибок предсказания переходов.

### Визуализация суперскалярного конвейера
Представьте расширенную временную диаграмму конвейера:
*   **По вертикали — стадии конвейера.**
*   **По горизонтали — такты.**
*   **В каждой ячейке (стадия/такт) теперь может находиться не одна, а несколько инструкций.**
    *   Например, на такте 3: стадия EX обрабатывает две инструкции сложения (1 такт), а стадия MEM обрабатывает одну инструкцию загрузки (много тактов).
    *   Инструкции разных типов расходятся по времени выполнения.
    *   Несколько инструкций могут одновременно завершаться (стадия WB) в одном такте, если позволяет аппаратура.

**Итог по суперскалярности:** Это мощный, но сложный механизм. Он требует значительных аппаратных ресурсов (дублирование функциональных устройств, сложные схемы анализа зависимостей и планирования) и умного кода от компилятора, чтобы раскрыть свой потенциал и реально добиться IPC > 1.

# Суперскалярность и векторизация

## Суперскалярность на практике

### Пример: Intel Pentium (первый суперскалярный x86)
*   **Архитектура:** Два конвейера (**U-конвейер** и **V-конвейер**).
*   **Стратегия выдачи инструкций:**
    1.  Первая инструкция в паре всегда отправляется на **главный (U) конвейер** (может выполнять любые инструкции).
    2.  Вторая инструкция анализируется: если она **простая** (определённый набор) и **независима** от первой, она отправляется на **вспомогательный (V) конвейер**.
    3.  Если условия не выполняются, V-конвейер простаивает, и выполняется только одна инструкция за такт.
*   **Ключевой момент:** Эта суперскалярность **не требует внеочередного исполнения (out-of-order execution)**. Инструкции начинают и завершают выполнение в строгом порядке.
*   **Роль компилятора:** Критически важна. Для эффективности компилятор должен **переупорядочивать код**, стараясь "припарковать" простую независимую инструкцию **после** сложной, чтобы загрузить оба конвейера. Пример: размещать целочисленные операции загрузки/сохранения данных рядом с вещественными вычислениями.

### Пример оптимизации под суперскалярный конвейер
Рассматривается знакомый цикл DAXPY. Исходная версия (без оптимизаций) выполняла итерацию за **3.5 такта**.

**После развёртывания цикла и переупорядочивания инструкций под суперскалярный конвейер (где целочисленные и вещественные инструкции могут выполняться параллельно):**
*   **Целочисленный конвейер (Int):** Занимается загрузкой данных (`LD`), сохранением результатов (`SD`), управлением циклом (`SUBI`, `BNEZ`).
*   **Вещественный конвейер (FP):** Занимается исключительно арифметикой (`ADDD`).
*   **Результат:** Итерация выполняется за **~2.4 такта**. Это ускорение примерно в **1.5 раза**, достигнутое за счёт параллельной работы двух конвейеров.

**Вывод:** Суперскалярность даёт ощутимый выигрыш, но для его получения **код должен содержать достаточное количество независимых инструкций разных типов**, которые аппаратура или компилятор могут распределить по параллельным конвейерам.

### Проблемы и сложности реализации суперскалярности
*   **Сложная аппаратная логика:** Требуются мощные блоки для предсказания переходов, анализа зависимостей (для поиска независимых инструкций) и спекулятивного исполнения.
*   **"Узкие места" в конвейере:** Необходимо дублировать ресурсы на всех стадиях (дешифраторы, порты регистрового файла, функциональные устройства, шины записи).
*   **Ограничение по памяти:** Суперскалярность увеличивает потребность в данных, обостряя проблему **ограниченной пропускной способности оперативной памяти (RAM Bound)**. Процессор начинает "голодать" из-за ожидания данных.

## Векторизация (SIMD)

### Основная идея
*   **Вектор:** Набор **однотипных** данных в памяти (массив `int`, `float`, `double`).
*   **SIMD (Single Instruction, Multiple Data):** Одна машинная инструкция выполняет **одну и ту же операцию** над **несколькими парами операндов одновременно**.
    *   Пример: Инструкция `VADDPD ymm1, ymm2, ymm3` складывает **4 пары** 64-битных вещественных чисел (всего 8 чисел) за один такт.
*   **Длина вектора:** Определяется архитектурой (например, 128 бит — SSE, 256 бит — AVX2, 512 бит — AVX-512). Фиксирована для конкретного набора инструкций.

### Как добиться векторизации в коде
1.  **Автоматическая векторизация компилятором:** Самый простой способ. Современные компиляторы (GCC, Clang, ICC) анализируют простые циклы и автоматически генерируют векторные инструкции, если это безопасно.
2.  **Директивы компилятора (pragma):** Дают компилятору подсказку.
    *   `#pragma ivdep` (Intel): Указывает, что в следующем цикле **нет зависимостей по данным** между итерациями, что разрешает векторизацию.
    *   `#pragma omp simd` (OpenMP 4.0+): Директива стандарта OpenMP для явного запроса векторизации цикла.
3.  **Векторные встроенные функции (intrinsics):** Низкоуровневый способ, дающий полный контроль.
    *   Это специальные функции на языке C (например, `_mm256_add_pd`), которые компилятор напрямую заменяет на конкретную векторную инструкцию.
    *   **Зачем это нужно?** Когда компилятор не справляется с автоматической векторизацией или генерирует неоптимальный код, программист может вручную написать векторную версию алгоритма с помощью интринсиков.
    *   **Ресурс:** [Intel Intrinsics Guide](https://software.intel.com/sites/landingpage/IntrinsicsGuide/) — полный справочник по векторным инструкциям и соответствующим им интринсикам с описанием, латентностью и пропускной способностью (CPI).

### Анализ результата векторизации
*   **Проверка:** После компиляции с высоким уровнем оптимизации (`-O3 -march=native`) можно сгенерировать ассемблерный листинг (ключ `-S` у GCC/Clang) и посмотреть, появились ли в коде инструкции с префиксами `V` (AVX) или похожие.
*   **Пример из слайда:** В ассемблерном коде видны инструкции типа `vmovupd` (векторная загрузка), `vfmadd132pd` (векторное умножение-сложение fused multiply-add), `vaddpd` (векторное сложение). Это явные признаки успешной векторизации компилятором.

### Ограничения и потенциал векторизации
*   **Теоретическое ускорение:** Равно **коэффициенту векторизации** (например, 4 для `float` при AVX, 8 для `float` при AVX-512). Однако на практике упирается в:
    *   **Пропускную способность памяти:** Если данные не помещаются в кэш, ускорение будет меньше.
    *   **Зависимости в алгоритме:** Не все циклы можно векторизовать.
    *   **Смешанные типы данных.**
*   **Практический результат:** В вычислительно-насыщенных задачах (обработка изображений, научные расчёты, линейная алгебра) векторизация даёт **кратное ускорение** и является обязательным приёмом высокопроизводительного программирования.

## Итог лекции: Иерархия методов повышения производительности

1.  **Конвейеризация:** Параллелизм на уровне стадий обработки **одной** инструкции. CPI стремится к 1.
2.  **Суперскалярность:** Параллелизм на уровне **нескольких независимых инструкций** внутри одного ядра. Позволяет добиться **IPC > 1**.
3.  **Векторизация (SIMD):** Параллелизм на уровне **данных**. Одна инструкция обрабатывает несколько элементов массива. Даёт ускорение, пропорциональное ширине вектора.
4.  **Многоядерность (Многопоточность):** Параллелизм на уровне **независимых потоков исполнения** (threads). Это тема следующих лекций.

**Заключение:** Современный высокопроизводительный код должен быть написан с учётом **всех этих уровней параллелизма**. Компиляторы помогают, но понимание принципов (конфликты данных, независимость итераций, выравнивание памяти) необходимо для того, чтобы заставить процессор работать на полную мощность.

## Углубление в векторные встроенные функции (Intrinsics)

### Особенности использования Intrinsics
*   **Соответствие между Intrinsics и инструкциями:**
    *   **Не один к одному:** Один `intrinsic` может компилироваться в **несколько** машинных инструкций (например, если требуется подготовка данных или обработка граничных случаев).
    *   **Неоднозначность:** **Несколько разных `intrinsics`** могут компилироваться в **одну и ту же** машинную инструкцию. Это связано с требованием строгой типизации в C/C++. Например, инструкция `VADDPD` (сложение `double`) может быть представлена разными `intrinsics` для работы с `__m256d`, `__m128d` и т.д., в зависимости от контекста использования типов.
*   **Цель Intrinsics:** Предоставить программисту на языке высокого уровня (C/C++) **низкоуровневый, но типобезопасный доступ** к конкретным векторным инструкциям процессора, скрывая некоторые детали ассемблера.

### Важность контроля генерируемого кода
*   **Обязательная проверка:** При использовании `intrinsics` **необходимо всегда проверять итоговый ассемблерный код** (ключ `-S`).
*   **Потенциальные проблемы, которые нужно искать:**
    1.  **Избыточные пересылки между памятью и регистрами:** Компилятор может неудачно распределить регистры, вызывая лишние операции `load`/`store`.
    2.  **Неконсистентность наборов инструкций:** Если основной код использует, например, AVX2 (`ymm` регистры), а компилятор вставляет служебный код, использующий SSE (`xmm` регистры), это вызывает **штрафные такты** на переключении между разными векторными режимами (State Switch Penalty).
*   **Вывод:** Программирование на `intrinsics` — это **низкоуровневая оптимизация**. Она даёт полный контроль, но требует глубокого понимания и тщательной проверки результата.

### Пример анализа векторизованного кода (скалярное произведение)
Рассмотрим эффективный векторизованный код из слайда:
*   **Суть алгоритма:** Развёрнутый и векторизованный цикл для вычисления скалярного произведения.
*   **Детали реализации:**
    *   **Шаг цикла `rax += 32`:** Обрабатывается **4 элемента типа `double` (8 байт * 4 = 32 байта)** за одну итерацию. Используются 256-битные регистры (`ymm`).
    *   **Загрузка данных:** Инструкции `vmovupd` загружают по **4 значения `double`** за раз.
    *   **Вычисления:** Инструкции `vfmadd132pd` выполняют **4 операции "умножение-сложение" (FMA)** одновременно.
    *   **Развёртывание (unrolling):** Внутри одной итерации цикла выполнена обработка **32 пар чисел** (8 векторных операций FMA по 4 элемента в каждой). Это соответствует **32 операциям умножения и 32 операциям сложения**.
    *   **Редукция (сведение):** После цикла 32 частичные суммы, хранящиеся в векторных регистрах, необходимо свести к одному скалярному значению. Это делается последовательностью операций горизонтального сложения (`vaddpd`, `vshufpd`, `vextractf128`, `vaddsd`).
*   **Эффективность:** Такой код **в десятки раз быстрее** наивной скалярной реализации, так как:
    1.  Использует **векторные инструкции** (коэффициент ускорения ~4 для `double` при AVX/AVX2).
    2.  Использует **развёртывание цикла**, что снижает накладные расходы на проверку условия цикла.
    3.  Использует **мощную инструкцию FMA** (одна инструкция делает две операции).

## Переносимость и архитектурная зависимость Intrinsics

*   **Условная переносимость в пределах одного вендора/семейства:**
    *   `Intrinsics` из **Intel Intrinsics Guide** гарантируют переносимость **между разными поколениями процессоров Intel**, поддерживающими один и тот же набор инструкций (например, AVX2).
    *   Компилятор может сгенерировать **разный машинный код** для одного `intrinsic` на разных микроархитектурах (например, Haswell vs. Skylake), оптимизируя под их особенности, но семантика операции остаётся той же.
*   **Нет переносимости между разными архитектурами:**
    *   Наборы `intrinsics` для **x86-64 (Intel/AMD)**, **ARM (NEON, SVE)** и **PowerPC (AltiVec/VSX)** — **совершенно разные**.
    *   **Имена функций, типы данных и доступные операции** различаются.
*   **Как с этим работать?**
    *   Использовать **макросы условной компиляции** (`#ifdef __AVX2__`, `#ifdef __ARM_NEON`).
    *   Писать **несколько версий кода** для разных архитектурных targets.
    *   Рассматривать использование **кросс-платформенных библиотек** (например, Eigen, xsimd), которые абстрагируют векторные операции.

## Итоги лекции: Путь к высокопроизводительному коду

**Цепочка оптимизаций для одного вычислительного ядра:**

1.  **Выбор алгоритма:** Самый большой выигрыш даёт асимптотически оптимальный алгоритм.
2.  **Базовая оптимизация:** Устранение лишних операций, кэширование промежуточных результатов.
3.  **Учёт иерархии памяти:** Локализация данных, работа с кэшами (блочные алгоритмы).
4.  **Векторизация (SIMD):** Самый эффективный способ ускорить регулярные вычисления над массивами. Способы:
    *   Надеяться на **автоматическую векторизацию** компилятора.
    *   Помогать компилятору **директивами** (`#pragma omp simd`, `#pragma ivdep`).
    *   В крайнем случае — писать **вручную на intrinsics**.
5.  **Суперскалярность:** Обеспечивается аппаратно и компилятором. Задача программиста — предоставить **поток инструкций с высокой степенью параллелизма (ILP)**: независимые операции, развёртывание циклов.
6.  **Многопоточность (многоядерность):** Распараллеливание на несколько ядер/потоков. Следующий уровень параллелизма, выходящий за рамки одного ядра.

**Ключевой вывод:** Современный процессор — сложнейший инструмент. Чтобы извлечь из него максимальную производительность, код должен быть **"дружественным"** ко всем уровням параллелизма: **конвейеру, суперскалярному исполнительным устройствам, векторным блокам и, наконец, нескольким ядрам.** Понимание принципов, изложенных в лекции, — основа для написания такого кода.
