# Детальный конспект лекции по выполнению программы на конвейере

## 1. Введение в тему
Тема практического занятия — выполнение программы на конвейере. Рассматривается **пятистадийный конвейер команд**:
1. **IF** (Instruction Fetch) — выборка инструкции.
2. **ID** (Instruction Decode) — декодирование.
3. **EX** (Execution) — исполнение.
4. **M** (Memory Access) — обращение к памяти.
5. **WB** (Write Back) — запись результата.

**Идеальное выполнение**: на каждом такте на каждую стадию поступает новая инструкция. Одновременно в конвейере находятся 5 инструкций на разных стадиях.

## 2. Типы конфликтов в конвейере
### 2.1. Структурные конфликты
- Возникают, когда две инструкции пытаются использовать **один и тот же аппаратный ресурс** одновременно.
- Решение: добавление дополнительных аппаратных компонентов (например, дополнительных портов памяти, дополнительных АЛУ).

### 2.2. Конфликты данных
- Возникают, когда инструкции **используют одни и те же регистры или ячейки памяти**.
- Пример: инструкция B использует результат инструкции A, который ещё не записан в регистр.
- **Основной способ решения — пересылка (forwarding)**.

### 2.3. Конфликты управления
- Возникают при выполнении **инструкций переходов** (безусловных и условных).
- Проблема: следующая инструкция неизвестна до завершения инструкции перехода.
- **Способы уменьшения потерь**:
  1. **Ранняя обработка условных переходов** (early branch resolution).
  2. **Спекулятивное выполнение** (speculative execution) с предсказанием переходов.

## 3. Пример программы и её выполнение
Рассматривается пример программы на псевдоассемблере:

| Адрес | Инструкция         |
|-------|---------------------|
| 0     | LD [1000], R1      |
| 4     | LD [1004], R2      |
| 8     | CMP R1, R2         |
| 12    | JGE [+32]          |
| 16    | LD [1000], R1      |
| 20    | LD [1008], R3      |
| 24    | ADD R1, R3, R1     |
| 28    | ST R1, [1008]      |
| 32    | LD [1000], R1      |
| 36    | INC R1             |
| 40    | ST R1, [1000]      |
| 44    | JMP [0]            |
| 48    | NOP                |

**Важное замечание**: 
- **Программа** — это последовательность инструкций в памяти.
- **Фактическая последовательность выполнения** — это порядок, в котором инструкции реально выполняются процессором (с учётом переходов).

В примере:
- Инструкция `JGE [+32]` (адрес 12) — условный переход.
- Первые 3 итерации цикла: переход **не происходит**, выполняется тело цикла (адреса 16–44).
- 4-я итерация: переход **происходит**, выполняется инструкция NOP по адресу 48.

## 4. Конфликты данных и пересылка (Forwarding)
### 4.1. Без пересылки
Пример: инструкция загрузки (`LD`) и следующая за ней арифметическая инструкция (`ADD`), использующая загруженное значение.

Стадии:
- `LD`: IF → ID → EX → M → WB.
- `ADD`: должна ждать, пока значение не будет записано в регистр на стадии WB.

**Оптимизация**: чтение регистров на стадии ID происходит во **второй половине такта**, а запись на стадии WB — в **первой половине**. Это позволяет выполнить ID для `ADD` в тот же такт, что и WB для `LD`.

### 4.2. С пересылкой
**Идея**: данные передаются на стадию, где они нужны, сразу после их вычисления, **не дожидаясь записи в регистр**.

Пример:
- `LD` загружает значение в регистр R1.
- `ADD` использует R1 на стадии EX.
- Значение готово после стадии M инструкции `LD`.
- `ADD` на стадии EX получает значение напрямую из выхода стадии M инструкции `LD`.

**Задержка**:
- Если `ADD` требует данные с выхода стадии M, то она должна **ожидать один такт** (вставить "пузырь" — stall).
- Если данные готовы раньше (например, результат арифметической операции), задержки может не быть.

**Важно**: при пересылке **ожидает та стадия, которой данные нужны** (обычно EX), а не стадия ID.

## 5. Конфликты управления и оптимизации переходов
Рассматриваются **4 варианта** в зависимости от поддержки:
1. **Ранняя обработка** (early branch resolution).
2. **Предсказание переходов** (branch prediction).

### 5.1. Без ранней обработки и без предсказания
- Адрес перехода определяется после стадии **M**.
- До этого следующая инструкция **не выбирается**.
- **Потери**: 3 такта простоя (после ID, EX, M).

### 5.2. С ранней обработкой, но без предсказания
- Адрес перехода определяется после стадии **ID**.
- Потери сокращаются до **1 такта** (после ID).

### 5.3. С предсказанием, но без ранней обработки
- Процессор **предсказывает**, произойдёт переход или нет, и начинает выбирать инструкции по предсказанному адресу.
- Если предсказание **верное**, потерь нет.
- Если **ошибочное**, то:
  - Все инструкции, начатые спекулятивно, **аннулируются** (их результаты отбрасываются).
  - Начинается выборка правильной инструкции.
  - Потери: **3 такта** (время, потраченное на спекулятивное выполнение).

### 5.4. С предсказанием и с ранней обработкой
- Предсказание выполняется, но проверка происходит уже на стадии **ID**.
- При ошибке предсказания аннулируется **только одна инструкция** (выбранная спекулятивно).
- Потери: **1 такт**.

## 6. Примеры с диаграмм (слайды)
На слайдах показаны **диаграммы выполнения** для четырёх случаев:

### 6.1. Предсказание: нет, ранняя обработка: есть
- Инструкция перехода (`Jxx`) проходит стадии: IF → ID → EX → M → WB.
- Следующая инструкция начинает выбираться только после стадии ID перехода.
- Задержка 1 такт.

### 6.2. Предсказание: нет, ранняя обработка: нет
- Следующая инструкция начинает выбираться только после стадии M перехода.
- Задержка 3 такта.

### 6.3. Предсказание: переход не происходит, ранняя обработка: нет
- Процессор предсказывает, что переход **не произойдёт**, и выбирает инструкции по следующему адресу.
- Если предсказание верное, выполнение идёт без задержек.
- Если предсказание ошибочное, спекулятивно выбранные инструкции аннулируются после стадии M, начинается выборка правильных инструкций.

### 6.4. Предсказание: переход не происходит, ранняя обработка: есть
- Предсказание проверяется на стадии ID.
- При ошибке аннулируется только одна спекулятивно выбранная инструкция, задержка 1 такт.

## 7. Итоги и рекомендации для самостоятельной работы
- **Важно различать** программу в памяти и фактические инструкции выполнения.
- При анализе конвейера **учитывать все зависимости** между инструкциями.
- Для условных переходов **определять**, происходил ли переход в данной итерации.
- В самостоятельной работе **не будет дана программа на псевдоассемблере** — нужно восстановить последовательность выполнения по списку фактически выполненных инструкций.
- При ошибке предсказания **аннулированные инструкции не включаются** в итоговую диаграмму выполнения (они не записывают результаты).

## 8. Ключевые термины
- **Конвейеризация** — параллельное выполнение нескольких инструкций на разных стадиях.
- **Forwarding (пересылка)** — передача данных между стадиями конвейера, минуя регистры.
- **Спекулятивное выполнение** — выполнение инструкций до того, как известно, нужны ли они.
- **Предсказание переходов** — предугадывание направления условного перехода.
- **Ранняя обработка переходов** — определение адреса перехода на более ранней стадии (ID вместо M).
- **Пузырь (stall)** — такт простоя в конвейере из-за конфликта.

# Детальный конспект лекции: Выполнение программы на конвейере — зависимость между инструкциями и пересылка (forwarding)

## 1. Введение в конвейер и конфликты
Конвейер из 5 стадий:
- **IF** (Instruction Fetch) — выборка инструкции из памяти
- **ID** (Instruction Decode) — декодирование, чтение регистров
- **EX** (Execution) — исполнение (арифметика/логика)
- **M** (Memory Access) — обращение к памяти
- **WB** (Write Back) — запись результата в регистр

**Идеальное выполнение**: на каждом такте каждая стадия обрабатывает новую инструкцию. В реальности возникают конфликты.

## 2. Конфликты данных и решение через пересылку

### 2.1. Пример зависимости: LD (загрузка) → ADD (сложение)
```
LD [address], REG1    # Загружаем значение в REG1
ADD REG1, REG2, REG3  # Используем REG1 сразу
```

### 2.2. Случай 1: БЕЗ пересылки (no forwarding)

#### Диаграмма конвейера:
```
Такты:       1   2   3   4   5   6   7   8
LD:         IF  ID  EX  M   WB
ADD:            IF  ID  -   -   EX  M   WB
```

**Пошаговое объяснение:**

1. **Такт 1-5**: `LD` выполняется нормально через все 5 стадий
2. **Ключевая проблема**: Результат загрузки будет записан в регистр REG1 только на стадии **WB** (такты 5, первая половина)
3. **ADD** нуждается в значении REG1 на стадии **ID** (чтение регистров) — такт 3
4. Но на такте 3 значение еще не готово! Оно появится только на такте 5
5. **Решение с половинными тактами**:
   - Запись в регистр (WB) происходит в **первой половине** такта 5
   - Чтение регистра (ID) происходит во **второй половине** такта 5
   - Поэтому `ADD` может выполнить ID на такте 5, а не на такте 3
6. **Итог**: между ID `ADD` и предыдущей стадией появляются **2 такта простоя** ("пузыри" на тактах 3-4)

**Почему 2 пузыря, а не 3?**
- Без оптимизации пришлось бы ждать 3 такта (такты 3, 4, 5)
- Благодаря половинным тактам экономим 1 такт
- Формула: задержка = (стадия_готовности - стадия_потребности) - 1

---

### 2.3. Случай 2: С пересылкой (with forwarding)

#### Диаграмма конвейера:
```
Такты:       1   2   3   4   5   6   7
LD:         IF  ID  EX  M   WB
ADD:            IF  ID  -   EX  M   WB
```

**Пошаговое объяснение:**

1. **Изменение философии**: 
   - Не ждем записи в регистр
   - Передаем данные **напрямую** между стадиями

2. **Когда готовы данные**: 
   - Для инструкции `LD` данные готовы после стадии **M** (так как читаем из памяти)
   - Данные доступны в **конце такта 4**

3. **Когда нужны данные**:
   - Инструкции `ADD` данные нужны на стадии **EX** (для выполнения сложения)
   - `ADD` достигает стадии EX в начале **такта 5**

4. **Механизм пересылки**:
   - Выход стадии M `LD` подключается ко входу стадии EX `ADD`
   - В такте 5: данные передаются напрямую, минуя регистровый файл

5. **Итог**: только **1 такт простоя** (пузырь на такте 4)

**Почему все еще есть 1 пузырь?**
- Данные готовы в конце такта 4 (после M `LD`)
- Данные нужны в начале такта 5 (для EX `ADD`)
- Нужен 1 такт для "выравнивания" времени

---

## 3. Другие типы зависимостей

### 3.1. Арифметическая → Арифметическая зависимость
```
ADD REG1, REG2, REG3  # Пишем в REG3
SUB REG3, REG4, REG5  # Читаем REG3
```

#### БЕЗ пересылки:
```
ADD:    IF  ID  EX  M   WB
SUB:        IF  ID  -   EX  M   WB
```
- Задержка: 1 такт (пузырь на такте 4)

#### С пересылкой:
```
ADD:    IF  ID  EX  M   WB
SUB:        IF  ID  EX  M   WB
```
- **Идеально, без задержек!**
- Почему: данные готовы после EX `ADD` (такт 3), нужны для EX `SUB` (такт 4)

### 3.2. Загрузка → Сохранение (Store)
```
LD [addr], REG1    # Загружаем
ST REG1, [addr2]   # Сохраняем то же значение
```

#### БЕЗ пересылки:
```
LD:    IF  ID  EX  M   WB
ST:        IF  ID  -   -   EX  M   WB
```
- Задержка: 2 такта

#### С пересылкой:
```
LD:    IF  ID  EX  M   WB
ST:        IF  ID  EX  M   WB
```
- **Идеально, без задержек!**
- Почему: данные нужны для ST на стадии M (так как пишем в память), а готовы после M `LD`

---

## 4. Общие правила для пересылки

### 4.1. Когда данные готовы:
- **Арифметические операции**: после стадии **EX**
- **Операции загрузки (LD)**: после стадии **M**
- **Все операции**: окончательно после стадии **WB** (для последующих инструкций)

### 4.2. Когда данные нужны:
- **Арифметические/логические операции**: на стадии **EX**
- **Операции сохранения (ST)**: на стадии **M**
- **Операции сравнения (CMP)**: на стадии **EX** (для вычисления флагов)
- **Условные переходы (Jxx)**: 
  - Без ранней обработки: на стадии **M**
  - С ранней обработкой: на стадии **ID**

### 4.3. Правило определения задержки:
```
Если (стадия_готовности < стадия_потребности + временное_смещение):
    добавляем пузыри
Иначе:
    пересылка возможна
```

---

## 5. Практические выводы для анализа программ

### 5.1. Как строить диаграммы:
1. **Начинаем с первой инструкции**, заполняем ее стадии
2. **Для каждой следующей инструкции**:
   - Проверяем зависимости с предыдущими
   - Определяем, нужна ли пересылка
   - Вычисляем необходимое количество пузырей
   - Сдвигаем инструкцию на нужное количество тактов

### 5.2. Пример полного анализа:
```assembly
0:  LD [1000], R1    ; Загрузка в R1
4:  ADD R1, R2, R3   ; Используем R1
8:  ST R3, [1004]    ; Используем R3
```

**С пересылкой:**
```
Такт: 1   2   3   4   5   6   7   8
LD:   IF  ID  EX  M   WB
ADD:      IF  ID  -   EX  M   WB
ST:           IF  ID  EX  M   WB
```
- LD→ADD: 1 пузырь (данные после M, нужны для EX)
- ADD→ST: идеально (данные после EX, нужны для M)

**БЕЗ пересылки:**
```
Такт: 1   2   3   4   5   6   7   8   9
LD:   IF  ID  EX  M   WB
ADD:      IF  ID  -   -   EX  M   WB
ST:           IF  ID  -   -   EX  M   WB
```
- LD→ADD: 2 пузыря
- ADD→ST: 2 пузыря (ждем записи R3 в WB)

---

## 6. Визуальное представление механизма пересылки

### 6.1. Без пересылки (традиционный путь):
```
     LD: IF → ID → EX → M → WB → Регистровый файл
                                        ↓
     ADD: IF → ID → EX → M → WB
```

### 6.2. С пересылкой (короткий путь):
```
     LD: IF → ID → EX → M → WB
                    ↓    ↓
     ADD: IF → ID → EX → M → WB
```

**Аппаратная реализация**:
- Дополнительные проводники между стадиями
- Мультиплексоры на входах функциональных устройств
- Логика определения, откуда брать данные (регистр или предыдущая стадия)

---

## 7. Ключевые моменты для запоминания

1. **Пересылка устраняет, но не полностью удаляет задержки**
2. **Половинные такты** (half-cycle timing) — важная оптимизация
3. **Тип операции определяет**:
   - Когда данные готовы
   - Когда данные нужны
   - Сколько пузырей потребуется
4. **Худший случай** — загрузка с последующим использованием: даже с пересылкой 1 пузырь
5. **Лучший случай** — арифметическая цепочка: с пересылкой может быть идеально

---

## 8. Практический совет для самостоятельных работ

При анализе конвейера:
1. Сначала отметьте ВСЕ зависимости между инструкциями
2. Для каждой зависимости определите:
   - Тип операций (load/store/alu)
   - Стадию готовности данных
   - Стадию потребности данных
3. В зависимости от поддержки пересылки в системе:
   - Рассчитайте количество пузырей
   - Постройте диаграмму
4. Проверьте: общее время выполнения должно быть:
   ```
   Общее время = (количество инструкций + сумма пузырей) * время такта
   ```

Это фундаментальное понимание необходимо для оптимизации программ под конкретные архитектуры процессоров.


# Детальный конспект лекции: Зависимость между CMP и Jxx. Построение диаграмм конвейера

## 7. Зависимость между инструкцией сравнения (CMP) и условным переходом (Jxx)

**Важная проблема**: между инструкцией сравнения (`CMP REG1, REG2`) и следующей за ней инструкцией условного перехода (`Jxx [address]`) существует **зависимость по данным**.

### 7.1. Принцип работы инструкции CMP
- `CMP` выполняет вычитание `REG1 - REG2`, **но результат не записывается в регистр**.
- Результат сравнения (знаки, флаги: нулевой флаг ZF, флаг знака SF, флаги переполнения) **обновляются в регистре флагов (FLAGS/CCR)**.
- Инструкция `Jxx` использует эти **флаги** для принятия решения о переходе.

### 7.2. Анализ 4 вариантов выполнения в зависимости от поддержки пересылки (forwarding) и ранней обработки (early branch resolution)

**Базовое предположение**: инструкция `CMP` выполняется идеально (без задержек до неё).

---

#### **Вариант 1: Пересылка НЕТ, Ранняя обработка НЕТ**
```
Такты:     1   2   3   4   5   6   7
CMP:      IF  ID  EX  M   WB
Jxx:          IF  -   -   ID  EX  M   WB
```
**Объяснение:**
- `CMP` завершает запись флагов в стадии **WB** (в первой половине такта 5).
- `Jxx` может прочитать обновлённые флаги на стадии **ID** только во второй половине такта 5 (благодаря половинной временной схеме: запись в первой половине, чтение во второй).
- Поэтому стадия **ID** для `Jxx` сдвигается на такт 5.
- **Итог**: между `CMP` и `Jxx` появляется **2 такта задержки** (пузыри на тактах 3 и 4 для `Jxx`).

---

#### **Вариант 2: Пересылка НЕТ, Ранняя обработка ДА**
```
Такты:     1   2   3   4   5   6
CMP:      IF  ID  EX  M   WB
Jxx:          IF  -   ID  EX  M   WB
```
**Объяснение:**
- Ранняя обработка означает, что `Jxx` пытается определить адрес перехода и проверить условие уже на стадии **ID**.
- Для этого ей нужны флаги, которые всё ещё не готовы до стадии **WB** инструкции `CMP`.
- Используется та же оптимизация с половинными тактами: запись флагов в первой половине такта 5, чтение во второй половине того же такта.
- Поэтому `Jxx` может выполнить **ID** на такте 4 (не дожидаясь полного завершения `CMP`).
- **Итог**: задержка уменьшается до **1 такта** (пузырь на такте 3 для `Jxx`).

---

#### **Вариант 3: Пересылка ДА, Ранняя обработка НЕТ**
```
Такты:     1   2   3   4   5   6
CMP:      IF  ID  EX  M   WB
Jxx:          IF  ID  EX  M   WB
```
**Объяснение:**
- С пересылкой данные (флаги) передаются на стадию, где они **фактически нужны**, сразу после их вычисления.
- Флаги формируются после стадии **EX** инструкции `CMP` (так как это арифметическая операция).
- Инструкции `Jxx` флаги нужны для принятия решения на стадии **M** (где определяется адрес перехода, если нет ранней обработки).
- **Прямая передача**: результат с выхода стадии **EX** `CMP` пересылается на вход стадии **M** `Jxx`.
- **Итог**: **Нет задержек**. Конвейер работает идеально.

---

#### **Вариант 4: Пересылка ДА, Ранняя обработка ДА**
```
Такты:     1   2   3   4   5   6
CMP:      IF  ID  EX  M   WB
Jxx:          IF  -   ID  EX  M   WB
```
**Объяснение:**
- Ранняя обработка требует, чтобы `Jxx` проверила условие и вычислила адрес уже на стадии **ID**.
- Для этого ей нужны флаги сразу после стадии **EX** инструкции `CMP`.
- Однако пересылка организует передачу данных между стадиями, но **стадия ID `Jxx` начинается на такте 3**, а результат `CMP` готов только **в конце такта 3** (после EX).
- Возникает конфликт: `Jxx` на стадии ID нужны данные, которые ещё только формируются в том же такте.
- **Решение**: добавить один такт ожидания (пузырь) для `Jxx`, чтобы данные успели переслаться.
- **Итог**: появляется **1 такт задержки** (пузырь на такте 3 для `Jxx`).

---

## 8. Сводная таблица задержек между CMP и Jxx

| Пересылка | Ранняя обработка | Задержка (такты) | Причина                                                                 |
|-----------|------------------|------------------|-------------------------------------------------------------------------|
| Нет       | Нет              | 2                | Ожидание записи флагов в WB и чтения в ID.                              |
| Нет       | Да               | 1                | Ранняя проверка на ID, но всё равно ожидание флагов до WB.              |
| **Да**    | **Нет**          | **0**            | Идеальная пересылка с EX(CMP) на M(Jxx).                                |
| Да        | Да               | 1                | Ранняя проверка требует данные на ID, но они готовы только после EX.    |

## 9. Общие выводы и замечания лектора

1. **Зависимость CMP-Jxx — критически важна** для понимания работы конвейера.
2. **Без пересылки** всегда возникают задержки, так как нужно ждать записи в регистр флагов.
3. **С пересылкой** задержки минимизируются, но при ранней обработке может оставаться 1 такт из-за временного несоответствия.
4. **Парадокс**: лучший результат (0 тактов задержки) достигается при **пересылке ДА, но ранней обработке НЕТ**. Это компенсируется тем, что **следующая после Jxx инструкция** будет начинаться позже (так как адрес перехода определяется только на стадии M).
5. **Ранняя обработка** ускоряет работу со следующей инструкцией, но может замедлить саму пару CMP-Jxx.
6. На практике часто **упрощают**, считая, что пара CMP-Jxx выполняется идеально (без задержек), чтобы не усложнять расчёты. Однако для точного анализа нужно учитывать все зависимости.

## 10. Практическое применение для самостоятельной работы

При построении диаграмм конвейера для программы с условными переходами:

1. **Определите все зависимости** между инструкциями (данные, управление).
2. **Учтите тип конвейера**: поддерживает ли он пересылку, раннюю обработку, предсказание.
3. **Для пары CMP-Jxx** используйте соответствующую схему из 4 рассмотренных вариантов.
4. **Помните**: инструкции, начатые спекулятивно и отменённые при ошибке предсказания, **не включаются** в итоговую диаграмму фактического выполнения.
5. **Отмечайте пузыри (stalls)** явно, указывая причину (например, "ожидание флагов", "ожидание результата загрузки").

---

**Ключевое сообщение**: Конвейер — это баланс между сложностью аппаратной реализации (пересылка, предсказание) и производительностью. Понимание этих механизмов позволяет предсказывать поведение реальных процессоров и оптимизировать программы.



# Детальный конспект лекции: Построение диаграмм конвейера на примере конкретной программы

## 9. Практическое построение диаграмм конвейера для программы

### 9.1. Вариант 1: Без оптимизаций
**Условия конвейера:**
- Пересылка (forwarding) **ОТСУТСТВУЕТ**
- Адрес перехода вычисляется на стадии **M** (после стадии исполнения)
- Предсказание переходов **НЕ ИСПОЛЬЗУЕТСЯ**

#### Пошаговое построение диаграммы:

**Такт 1-3: Начало выполнения**
```
Такт 1: I0 (LD) - IF
Такт 2: I0 - ID, I4 (LD) - IF
Такт 3: I0 - EX, I4 - ID, I8 (CMP) - IF
```

**Такт 4: Первый конфликт**
- I0 - M (обращение к памяти - **реальное**)
- I4 - EX
- I8 (CMP) - **ЖДЕТ** (зависимость от I0 и I4)
- I12 (JGE) - **НЕЛЬЗЯ считать** по двум причинам:
  1. **Структурный конфликт**: память занята (I0 в M)
  2. **Структурный конфликт**: стадия IF занята (I8 не перешла с IF)

**Такт 5: Продолжение конфликтов**
- I0 - WB
- I4 - M (обращение к памяти)
- I8 - **ВСЕ ЕЩЕ ЖДЕТ**
- I12 - **НЕЛЬЗЯ считать** (память занята I4 в M)

**Такт 6: Разрешение зависимости**
- I4 - WB
- I8 - **МОЖЕТ выполнить ID** (оба регистра R1 и R2 теперь доступны)
- I12 - IF

**Такты 7-11: Зависимость CMP → JGE**
```
Такт 7: I8 - EX, I12 - ID
Такт 8: I8 - M (ничего не делает), I12 - ЖДЕТ (нужен результат CMP)
Такт 9: I8 - WB, I12 - ID (читает флаги)
Такт 10: I12 - EX
Такт 11: I12 - M
Такт 12: I12 - WB, I16 (LD) - IF (адрес стал известен после M)
```

**Важное наблюдение**: К 12-му такту выполнено всего **4 инструкции**!

**Такты 13-45: Аналогичные конфликты продолжаются**

#### Типичные паттерны конфликтов:
1. **Двойной структурный конфлит при LD/ST**:
   - Память занята (реальное обращение)
   - Стадия IF занята (предыдущая инструкция не ушла с IF)

2. **Зависимости данных без пересылки**:
   - Каждая зависимость добавляет 1-2 такта простоя

3. **Условные переходы без оптимизаций**:
   - 3 такта простоя между Jxx и следующей инструкцией

#### Итог первого варианта:
- Всего тактов: 45
- Завершено инструкций: 18
- **CPI = 45/18 = 2.5**

### 9.2. Вариант 2: С оптимизациями
**Условия конвейера:**
- Пересылка (forwarding) **ЕСТЬ**
- Ранняя обработка переходов **ЕСТЬ** (адрес вычисляется на ID)
- Предсказание переходов **ЕСТЬ** (предсказываем "не происходит")

#### Пошаговое построение диаграммы:

**Такт 1-3: Аналогично первому варианту**

**Такт 4: Меньше простоев с пересылкой**
- I0 - M
- I4 - EX
- I8 (CMP) - **ID** (не ждет, т.к. данные понадобятся позже)
- I12 - **НЕЛЬЗЯ считать** (память занята)

**Такт 5:**
- I0 - WB
- I4 - M
- I8 - EX
- I12 - **ВСЕ ЕЩЕ НЕЛЬЗЯ** (память занята)

**Такт 6:**
- I4 - WB
- I8 - M
- I12 - **IF** (память освободилась)

**Такт 7: Ранняя обработка перехода**
- I8 - WB
- I12 - **ID** (ранняя обработка)
- I16 - **IF** (спекулятивно, предсказание "не происходит")

**Ключевое преимущество**: Уже на 7-м такте мы **спекулятивно выбрали следующую инструкцию**!

**Такты 8-10:**
```
Такт 8: I12 - EX, I16 - ID, I20 - IF
Такт 9: I12 - M, I16 - EX, I20 - ID, I24 - IF
Такт 10: I12 - WB, I16 - M, I20 - EX, I24 - ID, I28 - IF
```

**Обратите внимание**: На 10-м такте в конвейере **5 инструкций**!

#### Преимущества оптимизаций:

1. **Пересылка уменьшает задержки данных**:
   - Зависимость разрешается на стадии, где данные фактически нужны
   - Меньше "пузырей"

2. **Ранняя обработка переходов**:
   - Адрес перехода вычисляется на ID вместо M
   - Сокращает задержку с 3 до 1 такта

3. **Предсказание переходов**:
   - Позволяет спекулятивно выполнять следующие инструкции
   - Если предсказание верное - никаких потерь

**Итог второго варианта:**
- Всего тактов: 31
- Завершено инструкций: 18
- **CPI = 31/18 ≈ 1.72**

## 14. Ответы на вопросы студентов

### Вопрос: "Можно ли на 30-м такте выполнить EX для 19-й операции?"
**Ответ**: Нет. Для инструкции сложения (19-я) нужны результаты обеих загрузок (17-й и 18-й). 18-я загрузка находится в стадии M, результат будет доступен только после её завершения. С пересылкой данные доступны сразу после M, но не раньше.

### Вопрос: "Почему при загрузке данные доступны только после M, а не после EX?"
**Ответ**: Инструкция загрузки (LD) должна:
1. Вычислить адрес (EX)
2. Обратиться к памяти (M)
3. Получить данные

Только после M данные физически получены из памяти и могут быть использованы.
