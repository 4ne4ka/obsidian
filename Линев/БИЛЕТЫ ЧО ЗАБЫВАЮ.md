#### **Определение ISA (Architecture / Instruction Set Architecture)**
**ISA** — это формальное описание компьютера с точки зрения программиста или компилятора
*   Набор доступных **инструкций** (команд).
*   **Программируемые места хранения** данных (регистры, память).
*   **Типы данных**, поддерживаемые на аппаратном уровне.
*   **Форматы инструкций** (как команды кодируются в битах).
*   **Режимы адресации** (способы указания операндов).
*   Реакцию на **исключения и прерывания**.

| Режим адресации                                               | Пример (LOAD)              | Описание и применение                                                                                                                   |
| :------------------------------------------------------------ | :------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------- |
| **Регистровый**                                               | `ADD R1, R2, R3`           | Операнд находится в регистре. Самый быстрый режим.                                                                                      |
| **Непосредственный (Immediate)**                              | `ADD R1, R2, #5`           | Операнд — константа, закодированная в самой инструкции.                                                                                 |
| **Прямой (Абсолютный)**                                       | `LOAD R1, [0x1000]`        | Адрес памяти задан явной константой в команде.                                                                                          |
| **Косвенный регистровый**                                     | `LOAD R1, [R2]`            | В регистре содержится **адрес** операнда в памяти. Работа с указателями.                                                                |
| **Базовый со смещением (Base + Displacement)**                | `LOAD R1, [R2 + 8]`        | Адрес = Содержимое базового регистра + константа. **Крайне распространён** для доступа к полям структур и локальным переменным в стеке. |
| **Индексный (Base + Index)**                                  | `LOAD R1, [R2 + R3]`       | Адрес = Базовый регистр + Индексный регистр. Для массивов.                                                                              |
| **Индексный с масштабированием (Base + Index*Scale + Disp.)** | `LOAD R1, [R2 + R3*8 + 4]` | Адрес = Базовый регистр + (Индекс * Размер элемента) + Смещение. **Идеален для массивов структур** (например, `array[i].field`).        |


#### Кодирование инструкций
*   **Основные подходы:**
    1.  **Фиксированная длина (характерно для RISC):** Все инструкции имеют одинаковый размер (например, 32 бита в ARM, RISC-V). Поля внутри инструкции жестко закреплены за опкодом, регистрами и константами.
        *   **Преимущество:** Простота и скорость декодирования, упрощает конвейеризацию.
    2.  **Переменная длина (характерно для CISC, x86):** Разные инструкции кодируются разным количеством байт (от 1 до 15+). Часто используемые команды короче.
        *   **Преимущество:** Высокая плотность кода (программа занимает меньше памяти).
        *   **Недостаток:** Сложное и медленное декодирование.

*   ==**Типичные поля в формате инструкции:**==
    *   **Opcode (Код операции):** Определяет, какую операцию выполнять (ADD, LOAD и т.д.).
    *   **Register fields (Поля регистров):** Номера регистров-источников и регистра-приёмника.
    *   **Immediate field (Поле непосредственного значения):** Константа, встроенная в команду.
    *   **Addressing mode field (Поле режима адресации):** Указывает, как вычислять адрес операнда в памяти.


- **==Латентность== (Latency)** — время (в тактах), необходимое для полного выполнения одной инструкции от начала до конца. В 5-стадийном конвейере латентность равна 5 тактам.
- **Пропускная способность (Throughput)** — количество инструкций, завершаемых за такт. В идеальном конвейере стремится к 1 инструкции/такту

==**Методы устранения структурных конфликтов**==  
- **Разделение ресурсов** — использование отдельных кэшей инструкций (L1I) и данных (L1D).
- **Дублирование функциональных блоков** — несколько АЛУ, блоков умножения и т.д.
- **Конвейеризация внутри стадий** — разбиение долгих операций на более мелкие стадии (например, суперконвейеризация)


### ==Аппаратные== способы сокращения числа тактов простоя (+пример)

#### 1. **Раннее определение перехода (Early Branch Resolution)**
- **Идея:** Вычислять условие и адрес перехода на более ранней стадии конвейера.
- **Пример:**  
  В базовом 5-стадийном конвейере MIPS переход завершается на стадии **MEM** → 3 такта простоя.  
  Если перенести логику на стадию **ID**, то простой сокращается до **1 такта**.

#### 2. **Предсказание переходов (Branch Prediction)**
- **Идея:** Угадать направление перехода до того, как он будет вычислен.
- **Пример:**  
  - **Статическое предсказание:** «Переход никогда не происходит» (Predict Not Taken) → если предсказание верно, простой = 0; если ошибка → 1 такт простоя.  
  - **Динамическое предсказание:** Использует историю выполнения переходов (Branch History Buffer) → точность до 97%.

#### 3. **Спекулятивное выполнение (Speculative Execution)**
- **Идея:** Выполнять инструкции по предсказанному адресу до подтверждения правильности предсказания.  
- **Пример:**  
  Инструкции выполняются «вслепую», результаты временно сохраняются. Если предсказание верно — результаты фиксируются, если нет — отбрасываются.
### ==Статические== методы обработки условных переходов

#### 1. Отложенное ветвление (Delayed Branch)
**Идея:** Инструкция, следующая непосредственно за переходом (слот задержки), выполняется **всегда**, независимо от исхода перехода.

**Пример (MIPS):**
```assembly
BEQ R1, R2, Label  # Условный переход
ADD R3, R4, R5     # Выполняется ВСЕГДА (слот задержки)
```
**Недостатки:** Требует от компилятора поиска независимой инструкции для заполнения слота. В глубоких конвейерах может потребоваться несколько слотов.

#### 2. Вынос кода за пределы условного блока (If-conversion)
**Идея:** Преобразовать условные переходы в последовательность инструкций с условным выполнением (predicated execution) или вынести общий код за пределы ветвления.

**Пример:**
```c
// До оптимизации
if (x > 0) {
    a = b + c;
    d = e * f;
} else {
    a = b - c;
    g = h + i;
}
// После: вынос общей операции
temp1 = b + c;
temp2 = b - c;
if (x > 0) {
    a = temp1;
    d = e * f;
} else {
    a = temp2;
    g = h + i;
}
```



#### **==Требования== к разворачиванию циклов:**

1. **Независимость итераций:** Отсутствие зависимостей по данным между итерациями.
2. **Достаточное количество регистров:** Разные итерации используют разные регистры во избежание конфликтов (WAR, WAW).
3. **Известное или статическое число итераций:** Компилятор должен знать, на сколько итераций разворачивать цикл.
4. **Отсутствие сложных зависимостей по управлению:** Цикл не должен содержать вложенные переходы или сложные условия выхода.


#### **==Задержка== (Latency)**
- **Определение:** Количество тактов между началом выполнения инструкции, производящей результат, и моментом, когда этот результат становится доступным для следующей инструкции (при использовании forwarding).
- **Формула:** `Задержка = Время_выполнения - 1`

#### **Период запуска (Issue Latency / Throughput)**
- **Определение:** Минимальное количество тактов между началом выполнения двух последовательных операций одного типа.
- **Показывает, насколько конвейеризировано функциональное устройство.**

#### **1. Определение динамического планирования**
**Динамическое планирование** — это механизм **аппаратного переупорядочивания выполнения инструкций во время работы программы** с целью уменьшения простоев конвейера и улучшения CPI (Clock Cycles Per Instruction).  
Оно позволяет:
- Выполнять инструкции **не в порядке программы (out-of-order)**, но с соблюдением истинных зависимостей по данным (RAW).
- Устранять конфликты **WAR (Write-After-Read)** и **WAW (Write-After-Write)** за счёт отслеживания состояния регистров и функциональных устройств.
- Эффективнее использовать ресурсы процессора при неизвестных во время компиляции зависимостях.

### **Как работает BTB:**
1.  **На стадии IF:** Подается адрес текущей инструкции (PC) в BTB.
2.  **Проверка:** Если адрес найден в BTB, значит, эта инструкция — **условный переход**, и в BTB хранится:
    *   **Предсказанное направление** (будет переход/не будет).
    * **Целевой адрес** (адрес, куда перейти, если переход предсказан как "произошедший").
3.  **Действие:** Если переход предсказан как "произошедший" (taken), следующий PC загружается из BTB. Если не найден или предсказан как "непроизошедший", PC просто инкрементируется.

### **Алгоритм работы BTB в конвейере:**
*   **IF:** Отправка PC в память (кэш инструкций) и в BTB.
* **ID (Instruction Decode — декодирование):** Определяется, является ли инструкция условным переходом.
    *   **Если это переход И в BTB была запись:** Используется предсказанный PC.
    *   **Иначе:** Работа продолжается как с обычной инструкцией.
*  **EX (Execution — выполнение):** Фактически вычисляется условие перехода и его целевой адрес.
    *   **Если предсказание было верным:** Конвейер работает без задержек.
    *   **Если предсказание было неверным:** Возникает **штраф (penalty)**.
        *   Нужно "убить" (kill) ошибочно выбранные инструкции.
        *   Начать выборку с правильного адреса.
        *   Обновить или удалить запись в BTB.
*   **Штраф за неверное предсказание в BTB:** **2 такта** (задержка обнаружения ошибки на стадии EX + перезапуск выборки).


#### Структура PHT (Pattern History Table)
PHT — это **таблица прямого отображения**, где индекс формируется из **N младших бит адреса перехода**, а каждая запись — это **2-битный счётчик с насыщением**.
Используется в **двухуровневых предсказателях** (GAg, gshare).  
Хранит предсказания для **конкретных паттернов истории переходов**.

**Что хранит:**
- 2^m таблиц (где m — длина глобальной истории)
- В каждой таблице — массив 2-битных счётчиков (как в BHT)
**Алгоритм использования:**
- На стадии IF: берём N бит PC → индекс в PHT → читаем счётчик.
- Старший бит счётчика = предсказание (0 → Not Taken, 1 → Taken).

**Проблема: Интерференция адресов**
- Разные переходы с **одинаковыми N младшими битами** используют одну ячейку PHT.
- Приводит к конфликтам и снижению точности.
#### Структура BHT (Branch History Table)
BHT — это **ассоциативная таблица**, которая хранит **полный адрес перехода как тег** и связанный с ним счётчик.

**Структура записи:**
- **Тег адреса** (старшие биты адреса).
- **Счётчик** (2-битный).
- **Дополнительно:** может хранить историю переходов (для двухуровневых предсказателей).

**Преимущества перед PHT:**
- Решает проблему интерференции адресов (теги исключают коллизии).
- Точнее на ~5%.

**Недостатки:**
- Больше аппаратных затрат (память на теги, компараторы).


#### **Двухуровневый механизм с учётом корреляции (GAp)**
GAp (*Global History, Per-address Pattern Table*) — это двухуровневый предсказатель, который учитывает не только адрес текущего перехода, но и **глобальную историю переходов** программы.

**Структура GAp(m,n):**
1. **Первый уровень: Branch History Register (BHR)**
   - **m-битный регистр сдвига**, хранящий результаты (`1`/`0`) **m последних выполненных переходов** в программе.
   - Обновляется при каждом выполненном переходе.

2. **Второй уровень: Pattern History Tables (PHTs)**
   - **2^m отдельных таблиц**, каждая из которых соответствует определённому шаблону в BHR.
   - Каждая таблица содержит **2^N записей** (индексируются младшими битами адреса перехода).
   - Каждая запись — **n-битный счётчик с насыщением** (обычно 2-битный).

**Алгоритм работы GAp:**
1. По текущему значению BHR выбираем одну из 2^m таблиц PHT.
2. По адресу перехода (N младших бит) выбираем счётчик в выбранной таблице.
3. Предсказание = старший бит счётчика.

#### **Схема McFarling’а gshare**
улучшенная версия GAp

**Структура:**
- **Одна общая таблица PHT** (вместо 2^m таблиц в GAp).
- **Индекс в PHT** формируется как **XOR** между:
  - **m-битным Branch History Register (BHR)**
  - **m младшими битами адреса перехода**

**Преимущества gshare перед GAp:**
1. **Экономия памяти:** Вместо 2^m таблиц — одна.
2. **Улучшенное хеширование:** XOR уменьшает конфликты адресов.
3. **Возможность использовать более длинную историю** при тех же аппаратных затратах.
4. **Сравнимая или лучшая точность** при меньшей сложности.

**Алгоритм:**
1. BHR XOR (адрес перехода & маска) → индекс в PHT.
2. По индексу читаем 2-битный счётчик.
3. Старший бит = предсказание.


*   **Кэш-попадание (Cache Hit):** Ситуация, когда запрашиваемые данные найдены в текущем уровне кэша (например, в L1).
    *   **Частота попаданий (H₁):** Процент обращений, завершившихся попаданием. Измеряется **эмпирически** при профилировании программы.
*   **Кэш-промах (Cache Miss):** Ситуация, когда данные отсутствуют в текущем уровне кэша.
    *   **Частота промахов:** Рассчитывается как `1 - H₁`.
    *   Поиск продолжается на следующем, более низком уровне иерархии.
*   **Страничный сбой (Page Fault):** Аварийная ситуация, когда данные не найдены даже в оперативной памяти и должны быть подгружены с диска (например, код исполняемого файла).
    *   **Обработка:** Управляется **операционной системой**
    *   **Задержка:** Чрезвычайно велика (миллионы тактов).


#### Стратегии размещения блоков в кэше

**1. Кэш прямого отображения (Direct Mapped Cache)**
*   **Принцип:** Каждый блок основной памяти может быть размещен **строго в одной-единственной** строке кэша.
*   **Функция проецирования:** `Номер_строки_кэша = (Адрес_блока) MOD (Число_блоков_в_кэше)`
*   **Пример:** Кэш из 8 строк. Блоки с адресами 29, 37, 45 и т.д. будут проецироваться на строку `29 MOD 8 = 5`.
*   **Проблема:** **Конфликты.** Если программа попеременно обращается к нескольким блокам, которые проецируются на одну и ту же строку, они будут постоянно вытеснять друг друга, даже если в кэше есть свободные места.
*   **Структура строки кэша:** Каждая строка хранит:
    *   **Данные (Data)**
    * **Тег (Tag)** — Старшие биты адреса. Это уникальный идентификатор блока. Все блоки с одинаковым индексом, но из разных областей памяти, будут иметь разные теги.
    * **Валидный бит (Valid bit, V)** — показывает, содержит ли строка актуальные данные.

**2. Полностью ассоциативный кэш (Fully Associative Cache)**
*   **Принцип:** Блок основной памяти может быть размещен **в любой** свободной строке кэша.
* **Преимущество:** Полностью исключает конфликты из-за функции проецирования.
*   **Недостаток:** Высокая сложность и стоимость. Для поиска блока необходимо одновременно сравнить его тег с тегами **всех** строк кэша, что требует большого объема компараторной логики.

**3. Множественно-ассоциативный кэш (Set-Associative Cache)**
*   **Принцип:** Компромиссный вариант. Кэш разбивается на **наборы (sets)**, каждый из которых содержит `n` строк (каналов).
*   **Функция проецирования:** Блок памяти проецируется **строго на один набор**, но внутри этого набора может быть размещен **в любой** из `n` строк.
*   **Функция:** `Номер_набора = (Адрес_блока) MOD (Число_наборов_в_кэше)`
* **Терминология:** Такой кэш называется **`n`-канальным наборно-ассоциативным** (`n`-way set-associative).
*   **Пример:** 4-канальный ассоциативный кэш. Память проецируется на наборы, но внутри каждого набора есть 4 кандидата на замену, что резко снижает вероятность конфликтов по сравнению с кэшем прямого отображения.


#### **Методы уменьшения промахов**

1. **Victim Cache (Кэш «жертв»)**  
   — Небольшой полностью ассоциативный буфер для блоков, вытесненных из основного кэша.  
   — Позволяет спасти блоки, вытесненные из-за конфликтных промахов.

2. **Разделение кэша прямого отображения на 2 части**  
   — Используется в наборно-ассоцитивных кэшах (например, 2-way, 4-way).  
   — Уменьшает конфликтные промахи за счёт увеличения мест для блоков с одним индексом.

3. **Программная предвыборка (Software Prefetching)**  
   — Компилятор или программист вставляет инструкции предвыборки данных, которые понадобятся в будущем.  
   — Эффективно для регулярных шаблонов доступа (например, циклы по массивам).

4. **Аппаратная предвыборка (Hardware Prefetching)**  
   — Контроллер памяти автоматически определяет шаблоны доступа и заранее подгружает данные.  
   — **Для команд**: загрузка нескольких блоков сразу (например, Alpha 21064).  
   — **Для данных**: эффективна при последовательном доступе (например, научные расчёты).


#### **==Условия== успешной векторизации**

1. **Отсутствие зависимостей между итерациями** (loop-carried dependencies).
2. **Прямой последовательный доступ к памяти** (stride-1).
3. **Выровненные данные** (alignment) для максимальной скорости.
4. **Однотипные операции** над данными.
5. **Достаточное количество итераций** для окупания накладных расходов.
